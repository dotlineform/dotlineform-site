# Standalone Docs Viewer Concept

## Problem

Docs Viewer has been repeatedly refactored as its reader, management, public, import, Data Sharing, and Docs Review responsibilities grew. Those changes created useful seams, but they did not answer the fundamental question: **what is the core Docs Viewer before repository and adjacent-application integrations are attached?**

Data Sharing made this ambiguity particularly visible. Storage and publication adapters partially separated workflows from artifact location, but location is only one architectural boundary. An adapter can answer where bytes live and how they are read or written without answering which document, scope, build, reader, and management behavior belongs to Docs Viewer itself.

The result is a capable application whose browser composition is increasingly explicit while its Python startup, paths, builders, imports, exports, and optional services still inherit repository and cross-application context.

## Desired Outcome

One concrete Docs Viewer core supports two application compositions from the same public repository:

- the **integrated composition** combines the core with repository, public-site, R2, Data Sharing, Docs Review, and Studio integrations;
- the **standalone composition** combines the same core with a manage-only, external-local application environment and direct local import/export.

The standalone application is not a fork or reduced rewrite. It is the first independent consumer and executable proof that the core boundary is real. Extracting that boundary should make the integrated application easier to understand, test, and extend even if no user ever installs the standalone release.

## Working Core Hypothesis

No authoritative core definition exists yet. The [Core Definition Analysis Delivery](/docs/?scope=studio&doc=d-20260718-121833-9fa2d4) must test this working hypothesis against current code, composition consumers, environment assumptions, and open roadmap proposals before Transition Architecture begins.

The core is the concrete Docs Viewer product behavior shared by both compositions:

- document and scope identity, hierarchy, metadata, and source rules;
- Markdown, scope-owned media, build, published-data, and logical-reference contracts;
- browsing, search, recent activity, rendering, and local reader state;
- document creation, editing, movement, deletion, rebuild, and scope lifecycle;
- generated document and search payloads;
- external-local artifact access and confinement;
- ordinary direct import and export;
- capability, route, view, action, and extension contracts needed to compose the manage application.

This list may be revised by the analysis and later implementation evidence. Core is not a generic documentation framework and should not gain abstractions for hypothetical consumers. Interfaces belong only at demonstrated application, provider, path, and workflow seams.

## Target Shape

```text
Docs Viewer core
├── reader and manage UI
├── source model and builders
├── document and scope mutations
├── external-local artifact access
├── direct local import and export
└── loopback service primitives

Integrated composition
├── repository and R2 locations
├── public routes and projections
├── static-site publishing
├── Data Sharing and Docs Review
└── Studio and repository bootstrap

Standalone composition
├── manage route only
├── external-local scopes only
├── app-owned config, state, logs, staging, and exports
└── bundled runtime and dependencies
```

This is one codebase with two deliberately different dependency closures. The integrated and standalone compositions may depend on core; core must not import either composition.

## Boundary Layers

| boundary | question answered |
| --- | --- |
| artifact adapter | where are artifacts stored and how are they listed, read, written, verified, and referenced? |
| Docs Viewer core | which document, scope, build, reader, and management behaviors constitute Docs Viewer? |
| application composition | which optional providers, routes, workflows, adjacent applications, and delivery assumptions are attached here? |

The current provider-neutral artifact direction is valuable and should remain the I/O boundary. It is not by itself an application boundary: repository and R2 implementations, authentication, public exposure, Data Sharing formats, and route composition must still be attached from outside core.

## Current Assessment

### Browser

The browser is closest to the target:

- shared boot and application-composition modules accept route context, collection providers, source adapters, view definitions, control renderers, shell renderers, and other contributions;
- the manage entrypoint is already a thin assembly of management contributions;
- route features, access projection, live backend capabilities, and registered views/actions are distinct decisions.

Standalone remains `app_kind: manage`; it is a distribution and application composition, not a fourth route kind beside public, manage, and review.

The public-safe shared runtime currently lives under `site/docs-viewer/`. That is not a runtime blocker: the repository is the source and release platform, so a standalone build may bundle those canonical modules. The installed application must not resolve them from a repository checkout.

### Python And Services

The backend has useful focused modules but is not yet package-safe:

- `docs_viewer_service.py` discovers a repository root, maps runtime and CSS from repository paths, and imports Docs Review at startup;
- `docs_management_service.py` assembles core mutations alongside public publishing, Review sessions, and repository-oriented export;
- `docs_import_source_service.py` imports Data Sharing package recognition, collection apply, review handoff, and Data Sharing workspace paths directly;
- builder common code derives repository paths from `__file__` and knows browser config under `site/`;
- the current static HTML export accepts repository-backed local scopes and rejects external-local scopes;
- global scope configuration and repeated `repo_root` parameters carry application-environment assumptions through otherwise reusable workflows.

`docs_artifact_locations.py` is the strongest existing backend seam. Its common location contract is reusable, but standalone packaging should include only external-local support while repository and R2 providers remain integrated-composition contributions.

This assessment is a starting map, not a frozen module manifest. Exact dependency direction must be re-derived from current code before each extraction slice.

## Composition Rules

- Select behavior once at an application composition root; do not scatter `standalone` branches through shared workflows.
- Give core explicit application paths and services for assets, configuration, state, logs, staging, exports, temporary files, builders, and artifact access instead of letting it discover repository context.
- Register optional routes, import formats, export formats, providers, UI contributions, and capability records. Do not import an excluded integration and then merely hide it with a capability flag.
- Derive capability truth from the components actually registered and available at startup.
- Keep one source implementation for each shared behavior. The standalone application may have a new launcher, installer, configuration shell, and composition root, but not parallel document or management logic.
- Enforce the dependency direction mechanically so that standalone packaging cannot accidentally acquire repository, public, R2, Data Sharing, or Docs Review imports.

## Standalone Product Boundary

Every standalone scope is external-local by definition. Its source, media, generated data, and other scope-owned artifacts live in an explicitly configured local workspace outside the installed application.

Within that boundary the application retains full management:

- browse, search, create, edit, move, and delete documents;
- create, rename, configure, rebuild, and delete scopes;
- manage scope-owned local media and generated data;
- use ordinary direct local import and export.

Import and export do not embed Data Sharing modules, reviewed-package workflows, return-import contracts, or Data Sharing route and application context. The standalone application also has no public scopes, public routes, public projections, static-site publishing, repository-backed scopes, R2 locations, or Docs Review surface.

## Install, Data, And Upgrade Boundary

A download-and-overwrite upgrade is safe only when the installed application is disposable. Mutable configuration and user-owned data must not live beneath `<install-root>/docs-viewer/`.

| root | owns | upgrade behavior |
| --- | --- | --- |
| `<install-root>/docs-viewer/` | runtime, bundled dependencies, browser assets, schemas, immutable defaults, and migration code | replace completely with the downloaded release |
| `<app-data-root>/` | user configuration, scope registry, application-data schema version, logs, durable local state, import staging, and default export locations | preserve across application replacement and migrate explicitly |
| `<scope-root>/` | external-local documents, media, generated data, and scope-owned settings | remain user-owned and outside application upgrade lifecycle |
| `<cache-root>/` | reproducible caches and temporary material | safe to discard and recreate |

The launcher should resolve app-data and cache roots through platform conventions or an explicit absolute override. Neither root should be inferred as a child of the installed application. Scope-specific settings that must travel with a workspace should live with that scope; the central registry should retain only application-level settings and the information needed to locate scopes.

Configuration should have three layers:

1. immutable defaults and schemas bundled with the installed release;
2. a small mutable application overlay in the app-data root;
3. scope-owned configuration in each external-local workspace where the setting belongs to that scope.

Do not copy the complete bundled defaults into mutable configuration. New application defaults should normally arrive by replacing the install, while the user overlay stores only deliberate differences.

App-data configuration must carry an explicit schema version. A built-in migration runner, shipped with the new release, should apply ordered migrations from the recorded version to the installed version. It should reject state written by a newer application, preserve a recoverable pre-migration copy, write changes atomically, and record successful completion. Deterministic non-destructive migrations may run during first startup; a migration that discards or reinterprets user choices must stop for explicit confirmation.

Application-data migration must not silently become scope-content migration. Any future change to document, media, generated-data, or scope-owned configuration layouts needs its own previewed and confirmed scope migration boundary. Long-lived compatibility aliases are not a substitute for either migration path.

## Integrated Repository Mapping

The same structural ownership model must apply when the core runs inside the repository. Otherwise core workflows would still receive one undifferentiated `repo_root` and the standalone composition would need special behavior throughout the application.

The integrated composition has an additional repository-project role, but it should map every path explicitly:

| logical role | current integrated examples | rule |
| --- | --- | --- |
| application source and immutable defaults | `docs-viewer/runtime/`, `docs-viewer/services/`, `docs-viewer/build/`, templates, schemas, and default config | running Docs Viewer reads these; only development changes them |
| repository-owned project data | tracked scope and route config, repository-backed source, `docs-viewer/published/`, and public projection under `site/` | changed only through explicit source, lifecycle, build, or publish workflows and reviewed through Git |
| machine-local application data | `.env.local`, `var/docs/`, logs, suppressions, backups, caches, and temporary state | untracked, replaceable where derived, and never treated as project source |
| external scope workspaces | configured external-local source, media, and published data | independently owned and outside repository lifecycle |
| adjacent-application data | Data Sharing staging, review, and export workspaces | owned by that integration and supplied only when the integrated composition registers it |

The repository checkout is therefore not the integrated equivalent of one large app-data directory. Tracked project configuration is source-controlled product data; machine-local operational state is not. Repository-backed scopes are an integrated provider capability, while external-local scopes retain their own lifecycle.

Core should consume the same explicit application-environment and location contracts in both compositions. The integrated adapter may map some roles into the checkout, but core must not infer those mappings from its own file location or gain broader write authority merely because a repository root exists.

## Configuration Ownership

Core extraction should replace the current broad idea of “Docs Viewer config” with explicit configuration roles. Every setting needs one owner, one canonical location, one mutation path, and a clear rule for any generated projection.

| configuration role | owner and behavior |
| --- | --- |
| core defaults and schemas | code-owned, bundled with the application, immutable at runtime, and limited to behavior implemented by core |
| composition configuration | integrated- or standalone-application-owned declarations of registered routes, providers, integrations, and capabilities |
| repository project configuration | tracked declarations for repository scopes, routes, and publication behavior; changed only by explicit reviewed workflows |
| application overlay | user-owned settings and scope locations stored in the mutable app-data root rather than copied defaults |
| scope-owned configuration | settings whose meaning and lifecycle belong to one external-local workspace and should travel with it |
| lifecycle evidence | tool-owned records used to prove which resources a managed operation may rename or delete |
| runtime projection | generated browser-safe configuration derived from canonical declarations; rebuildable and never edited as authority |
| secrets and machine environment | machine- or platform-owned values outside tracked config and ordinary user-editable JSON |

The present repository tree mixes several of these roles beneath `docs-viewer/config/`:

- `config/defaults/docs-viewer-service.json` and schemas behave like immutable application defaults;
- `config/scopes/docs_scopes.json` and route config are canonical tracked project declarations and lifecycle mutation targets;
- `config/scopes/docs_scope_manifest.json` is tool-managed lifecycle evidence;
- report and semantic-reference registries are code/product extension definitions;
- `config/defaults/docs-viewer-config.json` and its public counterpart are written by the docs builder as browser-facing projections despite living under `defaults/`;
- `.env.local`, `var/docs/`, and external workspaces hold machine-local or adjacent-application state outside that tree.

Those current paths are evidence for the architecture, not a proposed rename plan. The first requirement is to make authority and mutation explicit in code and documentation. Physical reorganisation should follow only where it makes that ownership more truthful without introducing unnecessary copy or compatibility machinery.

Core should define the schema and defaults for its own behavior, not the complete configuration of every composition. The integrated and standalone compositions decide which optional capabilities exist; users choose their allowed settings and scope locations; scopes retain configuration that belongs to their own data lifecycle.

## User Configuration Surface

Configuration classified as user-editable needs a supported manage-mode UI. Requiring users to edit JSON would make the standalone app incomplete and would preserve the same ownership confusion in the integrated application.

The surface should remain narrower than the complete configuration model:

- **Application settings** edit the mutable application overlay and show the effective value after bundled defaults are applied.
- **Scope settings** edit allowlisted settings owned by the selected scope or its registered project configuration.
- **Scope lifecycle** remains the separate preview/apply workflow for creating, locating, renaming, and deleting scopes or changing structural storage/publication contracts.
- **Configuration diagnostics** may show effective values, owners, canonical sources, projections, and blocked fields without making them editable.

Do not provide a generic JSON editor. Each editable field should come from a code-owned settings contract that states its owner, level, value type, default and current value, canonical write target, validation, sensitivity, and whether the change requires rebuild, restart, or migration. The backend remains authoritative for allowlisting and writes.

The integrated and standalone compositions may expose different fields because they register different capabilities and write authorities, but they should consume the same settings contract and UI primitives. A repository-project setting must be clearly labelled as a tracked project change; an application-overlay setting writes outside the checkout; a scope-owned setting writes with that workspace. Immutable defaults, generated projections, secrets, and composition-owned policy are never ordinary editable fields.

The current implementation is a useful starting point:

- `docs_source_config_settings.py` already declares editable, blocked, and deferred fields with source/projection paths, validation, and rebuild effects;
- the settings endpoint performs allowlisted atomic writes and can coordinate rebuilds;
- the manage UI has a Settings action and modal;
- the current workflow selects only the first editable field from the first returned scope, so it is not yet a complete multi-scope, multi-field, or application-settings surface;
- `default_doc_id` is currently the only editable scope field and `recent_limit` is explicitly deferred until global settings are supported.

Completing this surface should extend that contract rather than replace it with form-specific write logic. The UI should group fields by ownership and selected scope, submit only changed values, report their effects before confirmation when material, and refresh effective configuration after a successful write.

## Proof Of The Boundary

The extraction is not complete merely because a package can be assembled. A standalone installation must be able to:

1. start without a repository checkout, repository-shaped directory, or separately installed Python;
2. survive complete replacement of the install while preserving and, when required, migrating application configuration;
3. create or open an external-local scope;
4. browse, search, edit, move, and rebuild its documents;
5. manage scope-owned local media;
6. complete one direct import and one direct export;
7. inspect and change its allowlisted application and scope settings without editing JSON;
8. report only capabilities that are actually present;
9. operate with repository, public-projection, R2, Data Sharing, and Docs Review modules absent from the distribution.

The integrated application must continue to compose the same core with its additional providers and workflows. That is the matching proof that extraction produced a shared product boundary rather than moving behavior into a standalone fork.

## Benefit To The Integrated Application

- Data Sharing and Docs Review become explicit integrations instead of ambient Docs Viewer context.
- Repository storage, R2, public projection, and site publishing become attached capabilities rather than assumptions inside ordinary management.
- Builders and mutations can be tested against an explicit application environment.
- Configuration becomes classifiable by owner, authority, mutation path, and projection instead of accumulating in one broad config tree.
- Users gain one supported settings surface that makes those ownership boundaries visible in both compositions.
- Startup and capability reporting become truthful about which components were assembled.
- Optional integrations can fail or be unavailable without making the core application unimportable.
- Future work gains one dependency rule: integrations depend on core; core does not depend on integrations.

## Delivery And Licence Model

The public dotlineform repository remains the canonical source, collaboration space, and release platform for both compositions. The standalone release may be built from files whose canonical source lives anywhere appropriate in that repository; self-contained means only that the installed artifact bundles its runtime and dependencies and resolves no files from a checkout.

The repository currently uses CC0 1.0 Universal. Confirm that this remains the intended software-distribution basis before the first packaged release. Licence governance is repository-wide and is not owned by this feature.

## Python Runtime And Release Package

End users should not need to install or select Python. A GitHub audience may commonly have Python, but the application cannot safely assume the required version, architecture, environment, compiled dependencies, or command-path behavior. The standalone release should bundle a pinned CPython runtime and its complete runtime dependency set.

The first packaging target should be a one-folder application bundle rather than a self-extracting one-file executable:

- the folder naturally contains the launcher, embedded runtime, Python extensions, browser assets, immutable defaults, schemas, and migration code;
- replacing that folder matches the intended download-and-overwrite upgrade;
- static assets and collected dependencies remain inspectable and easier to diagnose;
- startup does not need to unpack executable support files into a temporary directory;
- the package can carry third-party licence notices and a machine-readable dependency inventory beside the application.

This is a user-facing application distribution, not a portable Python development environment. Do not expose the embedded interpreter as a general `python` command, ask users to run `pip`, or share packages with a system environment. Repository development may continue to use the configured project Python and development requirements independently of the packaged runtime.

The standalone composition needs its own pinned runtime dependency manifest. It should include only packages reachable from core and standalone integrations. Repository-wide development, test, spreadsheet, Data Sharing, public-publishing, and other integrated-only dependencies must not enter the bundle merely because they appear in the repository `requirements.txt`.

Current process execution requires architectural follow-through. `docs_write_rebuild.py` and `docs_live_rebuild_watcher.py` use `sys.executable` to launch builder scripts. In a frozen application `sys.executable` identifies the application launcher rather than a separately callable Python interpreter. Builders should therefore become importable core services invoked in-process, or explicit internal subcommands dispatched by the same packaged executable when process isolation is genuinely required. The package must not depend on loose `.py` entrypoints being executable by an external Python.

Build one release artifact per supported operating-system and architecture combination using that target environment. GitHub Releases can carry the resulting archives, checksums, version metadata, and notices. Replacing the application folder upgrades the bundled Python and dependencies together; the separate app-data migration contract then advances mutable configuration.

PyInstaller one-folder mode is the leading first implementation candidate because it bundles its active interpreter and dependencies and fits this folder-replacement model. That tool choice should be proven with a packaging spike rather than made part of the core architecture. A native installer or application-bundle tool can be considered later if platform integration, signing, or distribution requirements justify the extra lifecycle.

## Refactor Execution

Run the extraction in an isolated feature worktree while the normal checkout remains the stable application and documentation control plane. The root-level [Docs Viewer Core Refactor](/docs/?scope=studio&doc=d-20260718-114645-7b9d24) parent owns main-based planning and handoffs; its working method owns branch flow, local runtimes, document and generated-output ownership, checkpoints, and controlled integration.

## Risks And Guardrails

- A broad file move can create the appearance of a core without changing dependency direction. Prove boundaries through imports, composition, and clean-environment behavior.
- A global standalone-mode flag would preserve coupling and create two behaviors inside every workflow. Branch only at composition roots and registered integrations.
- A generic plugin platform would make this larger without answering the known product boundary. Extract only seams required by the two real compositions.
- A generic configuration editor would expose implementation structure as product UI and make generated or policy-owned fields appear safely editable. Render only registered user settings.
- Requiring system Python would turn user environment repair into an application support contract and make releases non-reproducible. Bundle and patch one runtime with the application.
- Bundling the repository-wide requirements would obscure the standalone dependency boundary and enlarge its security/update surface. Produce a dedicated runtime dependency closure.
- Packaging can distract from core extraction. First start a standalone composition from the repository against one external-local scope; package it only after that dependency boundary is true.
- Mutable state under the install root would make overwrite upgrades destructive. Treat install replacement and app-data migration as separate operations from the first standalone slice.
- Treating the repository root as every logical root would leave the integrated application structurally different from standalone and keep repository assumptions inside core. Apply the same role contracts to both compositions.
- Two copied implementations would drift in security, validation, generated contracts, and management behavior. Shared capabilities must retain one authority.
- Existing repository behavior can regress during extraction. Recompose the integrated application through the new boundary before treating standalone success as sufficient.

## Direction Before A Delivery

The likely order is to freeze the capability/dependency map, introduce an explicit application environment and install/data roots, separate optional registrations, recompose the integrated application without product changes, start the external-local-only composition, and finally package and verify replacement plus migration outside a checkout.

That order is architectural direction, not a delivery checklist. A proposed architecture should name the exact ownership boundaries and a first delivery should end in one complete, independently verifiable outcome.
