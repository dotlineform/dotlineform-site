# Inline Mermaid Rendering Delivery

## Outcome

A document in a repo-backed `local` scope renders fenced Mermaid after its generated HTML is mounted. The canonical source remains inline Markdown. Public and `local_external` scopes continue to receive only the existing static SVG media workflow.

The concept and dependency decisions are settled. This is an implementation-ready tracker, but work remains bounded by its checkpoints and begins at IMD-0 only when implementation is authorized.

## Settled Dependency Contract

`docs-viewer/build/mermaid/package.json` owns two exact direct development dependencies:

```json
"@mermaid-js/mermaid-cli": "11.16.0",
"mermaid": "11.16.0"
```

The adjacent lockfile remains the resolved dependency authority. A feature-local `sync-browser-asset` npm script verifies the direct Mermaid package version, then copies:

```text
node_modules/mermaid/dist/mermaid.min.js
node_modules/mermaid/LICENSE
```

to checked output beneath:

```text
site/docs-viewer/runtime/vendor/mermaid/11.16.0/
```

The update command is:

```bash
npm --prefix docs-viewer/build/mermaid run sync-browser-asset
```

The tracked runtime copy is generated output and is not edited by hand. Reader code lazy-loads only that versioned asset and never imports from `node_modules`, a CDN, the CLI, Puppeteer, or Node. IMD-0 accepted the upstream distribution: its 3,565,102 decoded bytes loaded and executed from a cold local-preview request in 83.6 ms on the reference development machine. This is implementation evidence, not a general performance guarantee. If later measurement is unacceptable, stop and revise this delivery around one small Mermaid-only bundle; do not introduce a general frontend bundler solely for this feature.

## Settled Diagram Host Contract

A successful inline render replaces its code block with this adapter-owned semantic shape:

```html
<div class="docsViewer__diagram" data-docs-viewer-diagram-kind="inline-mermaid">
  <svg><!-- Mermaid output --></svg>
</div>
```

The host adds no image role because the SVG owns its accessible title and description. The inline adapter owns the host and direct SVG child. The sibling Diagram Detail View may later add an adjacent control without changing this marker or taking ownership of rendering.

## Explicitly Deferred

- pre-rendering inline fences during static HTML export;
- choosing rendered-versus-inline behavior per export profile;
- public publication conversion;
- automatic extraction to canonical `.mmd` and SVG;
- inline editing widgets or live source-editor preview;
- a general frontend bundler, application-wide asset pipeline, or independent browser Mermaid version;
- per-scope inline-rendering flags, a Studio-only runtime branch, or Processing proof-content changes;
- Retry controls, inline parser diagnostics, or source-edit actions in the failure state;
- the sibling Diagram Detail View control or open-in-new-tab behavior;
- rendering inside reports, review packages, or embedded detail documents without an explicit second delivery need.

The [concept](/docs/?scope=studio&doc=d-20260720-102658-b55348) owns the product and export decisions. The shipped [Mermaid Document Diagrams Delivery](/docs/?scope=studio&doc=d-20260718-214206-27df44) remains the implementation authority for separate `.mmd` source and published SVG.
