# Docs Viewer Modal Consistency Delivery

## Status And Outcome

**Complete. DMC-P0 through DMC-P5 are closed.**

Deliver one consistent modal system across the integrated Docs Viewer. Establish a permanent production-backed Docs Viewer pack in the federated [UI Workbench](/docs/?scope=studio&doc=d-20260723-093229-dd63ab), then use it to remove duplicated lifecycle rules and correct visible, tactile, and implementation drift between the shared ephemeral shell and persistent Metadata/Import/Settings modals.

Every current management modal now uses the shared lifecycle, DMC-P4 normalized the agreed presentation and feedback contracts, and DMC-P5 completed focused automated evidence plus user-approved tactile review. The retained Workbench remains the cheap inspection surface for future modal changes. This delivery does not block Docs Viewer Core Refactor.

Start only after [Docs Viewer Manage Theme Restoration](/docs/?scope=studio&doc=d-20260723-101224-e62173) closes. That delivery owns the initial manage-route and overlay dark-mode repair; this delivery owns modal lifecycle and component consistency against the restored theme baseline.

## Benefits And Risks

The main benefit is one lifecycle and styling contract plus a cheap permanent inspection surface for current and future modals. This reduces accessibility regressions, duplicated controls, route-local correction work, and reliance on complete workflow setup for visual review.

The main risks are creating a second demo implementation, letting fixtures drift from production, or changing focus return, nested Escape handling, page scroll, action muscle memory, Import busy/cancel transitions, and long-content layout while removing duplication. Specimens must use production renderers and styles, and migration must preserve explicit workflow targets while proceeding from the simplest modal family to Import.

## Accepted Contract

### Shared Lifecycle

- Every open modal has a labelled `role="dialog"` with `aria-modal="true"`.
- Each workflow supplies an explicit safe initial-focus target. Destructive confirmation does not gain primary focus merely for consistency.
- Tab and Shift-Tab remain inside the active modal. Keyboard and pointer input produce the established focus treatment without leaving stale modality state.
- Escape closes the active modal unless a workflow-owned nested control consumes it first. Metadata's parent picker continues to close before its enclosing modal.
- Backdrop behavior and visible Cancel/Close actions invoke the same workflow cancellation path.
- Opening locks background page scroll; closing restores the prior overflow state without moving the document.
- Closing restores focus to the invoking control or the workflow's explicit current fallback.
- Lifecycle listeners, scroll changes, status state, and temporary markup are cleaned up exactly once.

### Presentation And Feedback

- Compact, standard, document, import, and wide cards have deliberate widths, vertical placement, maximum height, and overflow ownership.
- Headers, titles, metadata, fields, notes, status, lists, and action rows use one modal styling owner unless a workflow genuinely requires an additional local component.
- Action placement is role-driven. Auxiliary and standalone destructive actions retain their declared order in a leading group; Cancel immediately precedes the single rightmost primary action, which may itself be destructive; terminal Close stands alone. Caller array order does not set layout.
- Enabled, hover, focus, disabled, busy, and destructive states remain distinguishable in light and dark presentation.
- Validation status uses the existing modal status region and clears when its owning field is edited or focus moves to another operation.
- Busy feedback prevents duplicate submission and preserves the workflow's valid cancellation policy.
- Terminal success and failure results remain visible until the user closes them.
- Narrow screens stack fields and wrap actions without horizontal overflow, concealed controls, or competing scroll containers.

### Ownership

- The existing management modal shell remains the Docs Viewer rendering and lifecycle starting point.
- A small lifecycle adapter may manage persistent pre-rendered markup; complex Import markup is not rewritten into generic HTML strings.
- Metadata, Import, Settings, scope lifecycle, document packages, source editing, media, and document actions retain their current domain owners.
- No current product service endpoint, request payload, package selection, index selection, scope lifecycle, import, or metadata behavior changes. The abandoned Review Sessions endpoint family is removed without aliases.

### Permanent UI Workbench

- Admin owns the only visible Workbench route, exact specimen and named review-recipe selection, any required internal specimen-frame surface, and temporary audit-output reading.
- App-neutral workbench support lives outside `shared/frontend/`; only behavior proven across real app packs belongs under `shared/ui-workbench/`.
- Docs Viewer owns its specimen registry, fixture data, injected clients, adapters, and pack entrypoint under its test/workbench support. Admin loads that app-owned code; Docs Viewer does not expose a Workbench application route.
- `/docs/ui-workbench/` is retired in DMC-P1A. The Docs Viewer service does not own Workbench navigation, a standalone pack page, or serving workbench-only assets merely to prove a test surface can route them.
- Serving the local inspection document through Admin does not transfer ownership of Docs Viewer production modules, styles, fixtures, or adapters to Admin.
- Each specimen imports the real production renderer, component, and styles. No production markup, modal body, list, field, status, or action row is copied into a demo implementation.
- Each specimen also names a current product composition or explicitly approved delivery path. A production source location alone does not make dormant UI a current specimen.
- The Docs Viewer pack exposes normal, empty, selected, validation, busy, success, failure, and long-content states where the production workflow supports them.
- Family, state, and app remain useful registry metadata, search labels, and validation inputs. They do not form a cascading navigation interface; app selection is visible only when more than one pack is available.
- The normal UI selects one specimen from a complete exact-name list with native keyboard type-ahead. Theme and viewport remain explicit review conditions.
- A new or materially changed modal records whether comparison is required. A comparison names its question, exact A and B specimens, review dimensions, and sequential or side-by-side mode; `no comparison required` is a valid decision.
- Sequential review owns lifecycle, keyboard, state-transition, and narrow-interaction questions. Side-by-side review is optional and limited to presentation questions where simultaneous viewing reduces reliance on visual memory.
- Named review recipes replace arbitrary pairing and broad related-control groups. The Workbench does not invite random comparison across every possible specimen combination.
- The pack supports desktop and constrained narrow-viewport review plus light and dark presentation without writing source, calling mutation services, or requiring an external workspace.
- The narrow condition is a layout stress test inside the desktop-hosted Workbench. It protects fit, overflow, zoom, small-window, and future-hosting resilience; it does not declare mobile deployment or mobile support for the Python-backed Admin and Docs Viewer management applications.
- A lightweight permanent check proves every registered specimen mounts without a browser error. It does not approve appearance or replace focused production behavior checks.
- Selected browser-assisted screenshots are tracked centrally under `admin-app/app/assets/ui-workbench/screenshots/<app>/`, not embedded in documents or treated as exhaustive visual baselines.
- `var/admin/ui-workbench/` is reserved for untracked scratch captures, computed-style comparisons, measurements, exports, and generated review reports.
- Studio specimens and Studio UI corrections remain a separate delivery.

The [UI Workbench Guide](/docs/?scope=studio&doc=d-20260723-155929-ef9c14) owns the operational workflow and comparison practice.

### Control Equivalence

- Comparison begins with an explicit review question and exact named specimens, not a registry category or visual resemblance.
- Selection cardinality, focus model, keyboard navigation, state semantics, loading/empty/error behavior, actions, and data ownership determine whether controls implement the same idea.
- Same-contract controls should converge on one production primitive where the result is simpler.
- Different-contract controls may share tokens or presentation while retaining separate behavior.
- The exact specimen and implementation inventory remains code-owned.

### Workbench Boundary

- Admin serves `/admin/ui-workbench/` and, only if isolation requires a separate document, an internal Admin-owned specimen-frame endpoint. No Workbench route is added to Docs Viewer.
- Docs Viewer registry, fixtures, injected clients, adapters, and pack entrypoint live under `docs-viewer/tests/workbench/`. They import production shell renderers, modules, and styles; fixtures create data and callbacks but do not copy modal bodies.
- Admin may load and narrowly serve the app-owned pack support needed for local inspection. This serving boundary does not move production or fixture ownership out of Docs Viewer and does not expose Docs mutation APIs.
- Exact specimen or named recipe, theme, and viewport state belongs to the Admin-hosted Workbench. Family and state remain app registry metadata. Service data and mutation requests may not enter the pack.
- Create `shared/ui-workbench/` only if more than one real app pack later proves a shared contract. DMC-P1A removed the provisional cross-service protocol without an alias because Admin-hosted pack loading does not need it.
- Reviewed captures remain under `admin-app/app/assets/ui-workbench/screenshots/docs-viewer/`; scratch captures and reports remain under `var/admin/ui-workbench/`.

### Lifecycle And Presentation Boundary

- One focused Docs Viewer lifecycle owner supplies labelled dialog setup, explicit initial focus, Tab trapping, keyboard/pointer modality, close requests, scroll locking, focus restoration, and exactly-once cleanup to dynamic and persistent renderers.
- Escape, backdrop, Cancel, and Close enter the same workflow close policy. A Metadata parent popup consumes the first Escape and closes only itself. An Import collision or confirmation consumes Escape as its internal Cancel first; busy Import cannot be closed through Escape or backdrop when its visible cancellation policy is disabled.
- Compact is for a simple confirmation, choice, or short field, including Settings; standard is for Metadata; document provides the long-body structure used by Import; import extends that structure with Import-owned internals; wide is for previews and reports. All variants must fit desktop and a constrained approximately `390 × 844` frame in light and dark presentation without implying a supported mobile runtime.
- Validation uses the modal's status region and clears on edit of its owning field or focus moving to another operation. Busy state prevents duplicate submission. Terminal results remain until Close.

### Required Differences And Current Drift

Required workflow differences are explicit initial-focus and focus-return targets, Metadata's nested parent picker, Import's internal renderer and cancellation states, and the content needs represented by the five card variants.

DMC-P4 removes the recorded presentation and feedback drift. DMC-P5 verifies the resulting tactile matrix rather than reopening the fixed contract.
