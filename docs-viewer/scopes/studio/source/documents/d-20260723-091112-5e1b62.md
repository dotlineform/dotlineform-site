# Docs Viewer Modal Consistency Delivery

## Status And Outcome

**Proposed. Documentation only; implementation has not started.**

Deliver one consistent modal system across the integrated Docs Viewer. Establish a permanent production-backed Docs Viewer pack in the federated [UI Workbench](/docs/?scope=studio&doc=d-20260723-093229-dd63ab), then use it to remove duplicated lifecycle rules and correct visible, tactile, and implementation drift between the shared ephemeral shell, persistent Metadata/Import/Settings modals, and Review Sessions.

This delivery is ready for implementation after explicit approval. It does not block Docs Viewer Core Refactor.

Start only after [Docs Viewer Manage Theme Restoration](/docs/?scope=studio&doc=d-20260723-101224-e62173) closes. That delivery owns the initial manage-route and overlay dark-mode repair; this delivery owns modal lifecycle and component consistency against the restored theme baseline.

## Benefits And Risks

The main benefit is one lifecycle and styling contract plus a cheap permanent inspection surface for current and future modals. This reduces accessibility regressions, duplicated controls, route-local correction work, and reliance on complete workflow setup for visual review.

The main risks are creating a second demo implementation, letting fixtures drift from production, or changing focus return, nested Escape handling, page scroll, action muscle memory, Import busy/cancel transitions, and long-content layout while removing duplication. Specimens must use production renderers and styles, and migration must preserve explicit workflow targets while proceeding from the simplest modal family to Import.

## Accepted Contract

### Shared Lifecycle

- Every open modal has a labelled `role="dialog"` with `aria-modal="true"`.
- Each workflow supplies an explicit safe initial-focus target. Destructive confirmation does not gain primary focus merely for consistency.
- Tab and Shift-Tab remain inside the active modal. Keyboard and pointer input produce the established focus treatment without leaving stale modality state.
- Escape closes the active modal unless a workflow-owned nested control consumes it first. Metadata's parent picker continues to close before its enclosing modal.
- Backdrop behavior and visible Cancel/Close actions invoke the same workflow cancellation path.
- Opening locks background page scroll; closing restores the prior overflow state without moving the document.
- Closing restores focus to the invoking control or the workflow's explicit current fallback.
- Lifecycle listeners, scroll changes, status state, and temporary markup are cleaned up exactly once.

### Presentation And Feedback

- Compact, standard, document, import, and wide cards have deliberate widths, vertical placement, maximum height, and overflow ownership.
- Headers, titles, metadata, fields, notes, status, lists, and action rows use one modal styling owner unless a workflow genuinely requires an additional local component.
- Action order is consistent. The renderer, not each caller's array order, owns placement of primary, secondary, destructive, Cancel, and Close actions.
- Enabled, hover, focus, disabled, busy, and destructive states remain distinguishable in light and dark presentation.
- Validation status uses the existing modal status region and clears when its owning field is edited or focus moves to another operation.
- Busy feedback prevents duplicate submission and preserves the workflow's valid cancellation policy.
- Terminal success and failure results remain visible until the user closes them.
- Narrow screens stack fields and wrap actions without horizontal overflow, concealed controls, or competing scroll containers.

### Ownership

- The existing management modal shell remains the Docs Viewer rendering and lifecycle starting point.
- A small lifecycle adapter may manage persistent pre-rendered markup; complex Import markup is not rewritten into generic HTML strings.
- Metadata, Import, Settings, Review Sessions, scope lifecycle, document packages, source editing, media, and document actions retain their current domain owners.
- No service endpoint, request payload, package selection, index selection, scope lifecycle, import, metadata, or review-session behavior changes.

### Permanent UI Workbench

- Admin owns the local inspection host, registry discovery, comparison controls, and any temporary audit-output reading.
- App-neutral workbench support lives outside `shared/frontend/`; only behavior proven across real app packs belongs under `shared/ui-workbench/`.
- Docs Viewer owns its specimen registry, fixture data, injected clients, and adapters under its test/workbench support.
- Each specimen imports the real production renderer, component, and styles. No production markup, modal body, list, field, status, or action row is copied into a demo implementation.
- The Docs Viewer pack exposes normal, empty, selected, validation, busy, success, failure, and long-content states where the production workflow supports them.
- The pack supports desktop and narrow viewport review plus light and dark presentation without writing source, calling mutation services, or requiring an external workspace.
- A lightweight permanent check proves every registered specimen mounts without a browser error. It does not approve appearance or replace focused production behavior checks.
- Selected browser-assisted screenshots are tracked centrally under `admin-app/app/assets/ui-workbench/screenshots/<app>/`, not embedded in documents or treated as exhaustive visual baselines.
- `var/admin/ui-workbench/` is reserved for untracked scratch captures, computed-style comparisons, measurements, exports, and generated review reports.
- Studio specimens and Studio UI corrections remain a separate delivery.

### Control Equivalence

- The workbench groups related lists, fields, statuses, and action rows for comparison without assuming that similar appearance means identical behavior.
- Selection cardinality, focus model, keyboard navigation, state semantics, loading/empty/error behavior, actions, and data ownership determine whether controls implement the same idea.
- Same-contract controls should converge on one production primitive where the result is simpler.
- Different-contract controls may share tokens or presentation while retaining separate behavior.
- The exact specimen and implementation inventory remains code-owned.

- [ ] Confirm the shared lifecycle, presentation variants, action order, status-clearing rule, nested-control behavior, and mobile/theme matrix before changing code.
- [ ] Confirm each difference is unintended drift rather than a workflow requirement.

- [ ] Supply fixture data, injected clients, and callbacks that reach every agreed modal state without a real service mutation, source write, package workspace, or staged import.

### DMC-P2 — Establish One Lifecycle Owner

- [ ] Extract the reusable lifecycle behavior currently embedded in the ephemeral management modal shell without adding a second modal framework.

### DMC-P3 — Migrate Persistent Management Modals

- [ ] Move Metadata and Settings onto the shared lifecycle while retaining their existing field, save, warning, and focus targets.
- [ ] Preserve Metadata parent-picker Escape precedence and navigation-row focus restoration.
- [ ] Move Import only after the simpler persistent modals are stable.
- [ ] Preserve Import initialization, file and collection states, busy cancellation policy, terminal Cancel-to-Close transition, internal scrolling, and manage-action focus return.
- [ ] Remove superseded lifecycle callbacks, listeners, state, and selectors rather than keeping compatibility paths.
- [ ] Keep the normal, validation, busy, terminal, and long-content specimens current through each migration.

### DMC-P4 — Normalize Styling, Feedback, And Equivalent Controls

- [ ] Define every supported card-size modifier, including a real compact variant, and remove meaningless modifiers.
- [ ] Give document-style modal bodies and action areas clear scroll and footer ownership.
- [ ] Standardize action order, wrapping, disabled/busy appearance, focus treatment, and status-state attributes across ordinary and Import buttons.
- [ ] Add the missing Review Sessions list, row, selection, metadata, empty, and status styles through the modal stylesheet owner.
- [ ] Use the DMC-P0 implementation matrix and workbench comparison to consolidate only controls with the same meaningful contract.
- [ ] Reuse the current shared Record List, Selectable List, Search List, or File Picker only where its exact production contract fits; do not force modal-local lists into an unsuitable universal component.
- [ ] Reuse current design tokens and remove duplicate modal-only rules where ownership becomes clear; do not consolidate unrelated Docs Viewer styling.
- [ ] Apply the agreed stale-status clearing behavior without hiding terminal results.

### DMC-P5 — Verify, Retain, And Close

- [ ] Prove every registered Docs Viewer specimen still mounts through production modules and styles without network writes or browser errors.
- [ ] Run the focused shared modal lifecycle module smoke and the smallest affected workflow/module checks.
- [ ] Retain package, import, metadata, settings, scope lifecycle, and review-session request/result checks only where they protect durable behavior; do not duplicate ordinary click choreography.
- [ ] Inspect every modal family through the workbench at desktop and approximately `390 × 844` in light and dark presentation with keyboard and pointer input.
- [ ] Check initial focus, forward/reverse Tab wrapping, nested Escape, backdrop and Cancel, focus restoration, page-scroll restoration, action order, disabled/busy state, stale-status clearing, long content, and horizontal overflow.
- [ ] Run one representative live integration path for each durable browser/service boundary rather than every visual state.
- [ ] Confirm the workbench can be revisited after this delivery and that adding a materially changed Docs Viewer modal/control requires a specimen update.
- [ ] Use browser-assisted screenshots where helpful, save selected reviewed captures centrally with filenames identifying specimen, state, theme, and viewport, and update or remove stale assets without embedding them in documents.
- [ ] Update the smallest durable modal/runtime owner, run `git diff --check`, and close only when no current modal remains on a separate lifecycle contract.

## Verification Boundary

Permanent automated evidence should protect:

- workbench registry validity, production imports, and specimen boot;
- shared modal lifecycle behavior;
- workflow request/response and ready/busy agreement;
- module wiring and management-only ownership;
- explicit nested-control behavior where it affects modal closure.

Permanent workbench review should make these states directly reachable:

- visual spacing and hierarchy;
- button order and tactile feel;
- focus timing and indication;
- backdrop, scroll, and pointer feel;
- light/dark presentation;
- narrow-screen and long-content fit;
- related-control comparison.

Do not add exhaustive pixel-approved screenshot baselines, exhaustive markup snapshots, wording assertions, one permanent browser script per modal, or fixtures that duplicate production markup.

## Explicitly Outside This Delivery

- Changes to modal workflow purpose, domain validation, service endpoints, request or result schemas, writes, or action placement.
- A generic cross-app modal framework, third-party dialog dependency, or compatibility layer.
- Recreating the retired Admin UI Catalogue or maintaining parallel demo components, CSS, or documentation.
- Public Docs Viewer behavior or assets.
- A broad Docs Viewer redesign or consolidation of unrelated route, package, source-editor, or shared-component CSS.
- New modals, new management capabilities, native-dialog conversion, animation, theming systems, or speculative action variants.
- Index Multiple Selection consumer changes or partial Returned Packages behavior.
- A Studio specimen pack, Studio UI audit, or Studio remediation; those require a separate delivery.
- Foundational manage-route theme-token restoration or first-pass dark-mode repair; those belong to Docs Viewer Manage Theme Restoration.
- Automatic calendar scheduling, release gating, pixel approval, or screenshot-diff infrastructure.

## Completion Gates

- every meaningful current Docs Viewer modal state is directly inspectable through a permanent production-backed workbench specimen;
- the Admin host, shared workbench support, Docs Viewer pack, production UI, and temporary-output boundaries remain explicit;
- every current management modal uses one lifecycle owner;
- every supported modal size and workflow-local component has intentional styling;
- same-contract controls converge where doing so reduces duplication, while different-contract controls retain truthful behavior;
- action, focus, Escape, backdrop, scroll, status, busy, and focus-return behavior is consistent except where a documented workflow rule requires a difference;
- Import, Metadata, Settings, Review Sessions, scope lifecycle, packages, source editing, media, and action confirmations retain their domain behavior;
- focused automated evidence and the complete tactile matrix pass;
- obsolete lifecycle and styling duplication is removed without aliases;
- the retained workbench imports production implementations and contains no parallel component library or permanent visual baseline.
