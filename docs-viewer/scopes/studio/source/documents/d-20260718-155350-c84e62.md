# Share Document Packages

## Status

Shipped. The repo-integrated Docs Viewer owns the fixed profiles, package workspace adapter, direct service endpoints, checked-selection Prepare Action, Review package Action, whole-return validation, Docs Review publication, and whole-package Docs Import. The duplicate package browser routes, inspector endpoint and UI, narrow summary/hierarchy review and apply paths, generic Data Sharing source tree, Analytics package routes, and tag-package workflows are retired.

## Outcome

The repo-integrated Docs Viewer owns its supported JSON and JSONL document-package workflows. It presents a small set of fixed document capabilities rather than routing document behavior through a generic Data Sharing application, domain registry, and adapter framework.

Core Docs Viewer retains its existing static HTML export. JSON package preparation, returned-package review, and collection import are optional integrated capabilities and are not part of the portable core contract.

The generic Data Sharing feature does not survive this move. Its useful document behavior becomes document sharing; its tag-package behavior is retired without replacement.

## Document Capabilities

Docs Viewer explicitly owns the profiles it supports. The ownership cutover covers the currently enabled compact document-content and document-tree profiles. A profile selects known Docs Viewer behavior; it is not a declaration-only extension point for arbitrary domains.

Docs Viewer preserves two document-package workflows under distinct browser composition:

- **Prepare package** in the manage Actions menu uses the checked index documents and active scope, gathers package options once in a compact modal, then creates one package through `POST /docs/packages/prepare`.
- **Review package** in the manage Actions menu shows only validated return-import packages for the active scope in a **File name** and **Documents** table, prepares one complete package for Docs Review, and opens the read-only review in a new tab. A matching existing review is reused without overwrite.

Returned records are atomic package input. Prepare has already fixed the selected set, so there is no returned-row selection, filtering, skip, or partial apply. If any returned record is missing, extra, duplicated, or invalid, the package is rejected and regenerated externally as a whole.

[Document Sharing UI Redesign](/docs/?scope=studio&doc=d-20260720-230623-c7e921) moved package preparation onto checked index documents and review preparation into compact manage Actions. [Review Package UI Redesign Delivery](/docs/?scope=studio&doc=d-20260722-204025-15ea54) records the full-review cutover and package-route retirement.

The compact package Actions use the Docs Viewer management presentation owners. The retired route-only package stylesheet was deleted; broader management restyling remains separate.

## Future Profile Boundary

`document-full-source` remains a potential later feature owned by Docs Viewer. It is not an enabled profile and is not part of the ownership cutover. Existing lower-level source-content normalization does not establish a live package contract; any future source-faithful profile needs its own explicit Docs Viewer feature, format, validation, and tests.

## Ownership

- Docs Viewer owns the document profiles, browser workflows, local API orchestration, package I/O, safe external-workspace paths, metadata, schemas, and document-specific package helpers under `docs-viewer/config/document-packages/` and `docs-viewer/services/docs_document_packages/`.
- No transitional `docs-viewer/data-sharing/` bucket, generic adapter registry, or cross-app document bridge remains.
- Tag Data Sharing was retired rather than moved. Future tag assistance remains manual editing or a separately designed advisory mechanism.
- Existing external workspace paths and serialized `data_sharing_*` package identifiers remain compatibility-frozen storage and format contracts; changing them requires an explicit storage or format migration.
- The repo-integrated Docs Viewer may consume Studio-owned catalogue and tag projections for semantic references through a direct integration module. That one-to-one connection is not Data Sharing and is not part of portable core.
- Docs Viewer source/write/rebuild services and Docs Import remain the only canonical document mutation paths.
- Docs Review may remain an isolated validated-package reading surface; it does not become a generic package application.

## Boundaries

- No generic cross-domain adapter registry, capability projection, or Analytics-hosted gateway.
- No tag package preparation, returned workflow, automated reconciliation, or replacement sharing mechanism.
- No promise that every package supports a returned path or canonical apply.
- No requirement for direct ChatGPT-created JSON or JSONL imports to carry Data Sharing provenance.
- No change to ordinary HTML export, Publish, or static snapshot ownership.
- No subscription or packaging decision is made here; the architecture merely leaves the JSON package capability outside core.

## Current Position

The ownership and browser cutovers are complete. Docs Viewer hosts the checked-selection Prepare Action, the Review package Action, and the retained package APIs; there is no separate package browser route or inspector endpoint. Structurally invalid, missing, extra, or duplicate records block review and whole-package Import. Analytics retains only its current tag tools until [Tag Management In Studio](/docs/?scope=studio&doc=d-20260718-155350-a2d49c) completes the separate app-retirement dependency.

## External Workspace

Package artifacts remain outside the repository under `$DOTLINEFORM_PROJECTS_BASE_DIR/data-sharing/`. `exports/`, `meta/`, `import-staging/`, and `import-preview/` keep their existing meanings and serialized package identifiers. The path name is a frozen storage contract, not evidence of a live Data Sharing application or repository subsystem. `docs-viewer/services/docs_document_packages/workspace.py` is the only package-workspace owner.
