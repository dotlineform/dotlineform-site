# Share Document Packages

## Outcome

The repo-integrated Docs Viewer owns its supported JSON and JSONL document-package workflows. It presents a small set of fixed document capabilities rather than routing document behavior through a generic Data Sharing application, domain registry, and adapter framework.

Core Docs Viewer retains its existing static HTML export. JSON package preparation, returned-package review, and collection import are optional integrated capabilities and are not part of the portable core contract.

The generic Data Sharing feature does not survive this move. Its useful document behavior becomes document sharing; its tag-package behavior is retired without replacement.

## Document Capabilities

Docs Viewer explicitly owns the profiles it supports. The ownership cutover covers the currently enabled compact document-content and document-tree profiles. A profile selects known Docs Viewer behavior; it is not a declaration-only extension point for arbitrary domains.

The ownership cutover preserves the current two-workflow shape under Docs Viewer ownership:

- **Prepare** at `/docs/packages/prepare/` selects a scope, profile, and complete document set, then creates one package.
- **Returned Packages** at `/docs/packages/returned/` selects a staged package, validates its complete prepared record set, and offers the supported Docs Review or canonical handoff.

Returned rows are for inspection only. Prepare has already fixed the selected set, so there is no returned-row selection or partial apply. If any returned record is missing, extra, duplicated, or invalid, the package is rejected and regenerated externally as a whole.

The dedicated routes are the intended durable product shape. They already contain workflow-local modal interactions where those are useful, so the ownership cutover does not squeeze either workflow into an enclosing Docs Viewer modal. Any later proposal to replace or combine the routes requires a separate product decision after the migrated workflows can be reviewed in place.

Document-package presentation also keeps a dedicated stylesheet during the cutover. Consolidation, token alignment, or broader restyling is deferred to a separate Docs Viewer-wide styling review rather than mixed into this ownership migration.

## Future Profile Boundary

`document-full-source` remains a potential later feature owned by Docs Viewer. It is not an enabled profile and is not part of the ownership cutover. Existing lower-level source-content normalization does not establish a live package contract; any future source-faithful profile needs its own explicit Docs Viewer feature, format, validation, and tests.

## Ownership

- Docs Viewer absorbs the documents adapter, document profile configuration, browser workflow, local API orchestration, and document-specific package helpers.
- Move that code directly into a Docs Viewer-owned integrated-only module boundary. Do not create a transitional `docs-viewer/data-sharing/` bucket or preserve generic Data Sharing naming after document ownership is explicit.
- Retire tag Data Sharing rather than moving it. Future tag assistance remains manual editing or a separately designed advisory mechanism.
- With no second package owner, move package I/O, safe external-workspace paths, metadata, and schema helpers required by documents into Docs Viewer ownership.
- Preserve current external workspace paths and serialized package identifiers during the initial code cutover; changing them requires an explicit storage or format migration.
- The repo-integrated Docs Viewer may consume Studio-owned catalogue and tag projections for semantic references through a direct integration module. That one-to-one connection is not Data Sharing and is not part of portable core.
- Docs Viewer source/write/rebuild services and Docs Import remain the only canonical document mutation paths.
- Docs Review may remain an isolated validated-package reading surface; it does not become a generic package application.

## Boundaries

- No generic cross-domain adapter registry, capability projection, or Analytics-hosted gateway.
- No tag package preparation, returned workflow, automated reconciliation, or replacement sharing mechanism.
- No promise that every package supports a returned path or canonical apply.
- No requirement for direct ChatGPT-created JSON or JSONL imports to carry Data Sharing provenance.
- No change to ordinary HTML export, Publish, or static snapshot ownership.
- No subscription or packaging decision is made here; the architecture merely leaves the JSON package capability outside core.

## Current Position

DSO-1 and DSO-2 are shipped. Docs Viewer now owns fixed document profiles, direct `/docs/packages/*` service endpoints, atomic returned-package validation, the package workspace/metadata boundary, and the two dedicated Prepare and Returned Packages browser routes. Those routes use Docs Viewer-owned modules, modals, and package CSS without Analytics transport, config, runtime, app/domain state, or returned-row selection. The old Analytics routes, generic registry, and tag behavior remain only until the separately gated DSO-3 retirement; they are not dependencies of the new Docs Viewer workflows.
