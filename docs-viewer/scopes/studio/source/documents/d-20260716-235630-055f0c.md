# External-Local Scope Media Layout

## Outcome

This precursor is now expressed by the shipped scope-first layout. External-local scopes use the same root-relative source/published separation as repo-backed local scopes:

```text
scopes/<scope>/
  source/
    documents/*.md
    media/mermaid/*.mmd     # when configured
  published/
    documents/by-id/*.json
    search/index.json
    media/
      img/
      svg/
      files/
```

For external published media this is:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/scopes/<scope>/published/media/
```

The browser contract remains `/docs/media/<scope>/<type>/<filename>` for locally served configured media types.

For every configured local scope, the registered local `img` and `files` directories are materialized even when they contain no media. They are part of the visible workspace contract, not virtual paths that appear only on first write.

## Ownership

External describes the storage location, not a different ownership model. Ordinary imported media and authored figures are managed by Docs Viewer in both local scope types:

- create establishes one scope root plus required source and registered local media directories;
- local service startup restores any missing configured media directories;
- rename atomically moves the enclosing scope root, including published media;
- delete removes the manifest-owned scope root, including media.

Config loading remains read-only. Materialization belongs to local-service startup and confirmed scope creation, where a failure to establish the configured directories is visible rather than deferred until an import. Repo-backed empty directories use tracked marker files so the structure also survives a clean clone.

Generated docs, search data, and published media remain separate from canonical source beneath the scope's `published/` branch.

The former `docs-viewer/scopes/<scope>/source/media/<published-type>/` paths are not compatibility lookups. Existing published bytes must be moved manually before adopting this boundary; canonical build-source directories such as `media/mermaid/` remain beneath source.

## Separate Asset-Collection Boundary

[External Asset Collections](/docs/?scope=studio&doc=d-20260714-194217-b2bdd7) inventory user-owned files that are not ordinary scope media. Their roots stay outside the registered scope root, are not named or derived by scope lifecycle, and are never moved or deleted merely because the associated scope is renamed or deleted.

## Code Owners

- `docs_scope_config.py` derives omitted-location local media beneath the scope's published media root and validates explicit independent destinations.
- `docs_media_storage.py`, `docs_import_media.py`, and `docs_import_markdown_package.py` resolve and materialize imports through the common location adapter.
- `docs_scope_rename.py` moves one external scope root; media moves with that container.
- scope deletion removes the manifest-owned scope root.

[Source Organisation](/docs/?scope=studio&doc=d-20260423-000000-8fd731), [Scope Lifecycle](/docs/?scope=studio&doc=d-20260515-000000-f43740), and [Media And Asset Handling](/docs/?scope=studio&doc=d-20260514-184303-7914e2) own the durable current contracts.
