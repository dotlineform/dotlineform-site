# Introducing JavaScript Configuration to the Repository

> Research doc: not indicative of current implementation.

## Purpose

This guide outlines a practical approach to introducing project-wide JavaScript configuration.

The main aims are to:

- make JavaScript behaviour consistent across the repository;
- give VS Code, Codex, and other tools a shared understanding of the project;
- distinguish browser code from Node-based scripts;
- exclude generated and vendored files from analysis;
- support JSDoc-based type information;
- provide repeatable validation commands;
- introduce stricter checking gradually rather than all at once.

The main configuration files considered here are:

```text
jsconfig.json
eslint.config.js
package.json
```

A formatter configuration can be added separately if required.

---

## 1. Why configuration matters

Without project configuration, tools may interpret individual files differently.

For example:

- one JavaScript file may use `// @ts-check`;
- another may not be checked;
- browser globals such as `window` may be recognised in one context but not another;
- Node globals such as `process` may be incorrectly assumed to exist in browser code;
- generated files may be linted unnecessarily;
- different developers may run different validation commands.

Project configuration establishes a shared definition of the repository.

It tells tools:

```text
which files belong to the project
which JavaScript version is expected
which module system is used
which globals are valid
which folders are generated or external
which rules are errors
which checks define a valid change
```

This improves consistency for both humans and coding agents.

---

## 2. The role of each configuration file

### `jsconfig.json`

Defines how VS Code and the TypeScript language service interpret JavaScript files.

It can control:

- project scope;
- JavaScript version;
- module format;
- import resolution;
- path aliases;
- JSDoc-based type checking;
- excluded directories.

### `eslint.config.js`

Defines code-quality and correctness rules.

It can control:

- unused variables;
- undefined names;
- equality rules;
- browser versus Node environments;
- file-specific rules;
- excluded folders;
- suspicious patterns.

### `package.json`

Provides standard commands for developers and agents.

It can define:

```bash
npm run lint
npm run lint:fix
npm run check:js
npm test
npm run check
```

This avoids relying on undocumented command sequences.

---

## 3. Recommended starting principle

Introduce configuration conservatively.

The first objective should be to establish:

- file scope;
- exclusions;
- runtime environments;
- basic correctness rules;
- standard commands.

Do not begin with maximum strictness.

A very strict initial configuration can create hundreds of warnings that are mostly mechanical. This can obscure real defects and lead to broad automated changes that are difficult to review.

A staged approach is safer:

```text
scope and exclusions
        ↓
basic linting
        ↓
selective JSDoc
        ↓
selective type checking
        ↓
broader project checks
```

---

## 4. Introducing `jsconfig.json`

A `jsconfig.json` file tells VS Code that the repository contains a defined JavaScript project.

A conservative starting point might be:

```json
{
  "compilerOptions": {
    "allowJs": true,
    "checkJs": false,
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler"
  },
  "include": [
    "assets/**/*.js",
    "scripts/**/*.js"
  ],
  "exclude": [
    "_site",
    "node_modules",
    "vendor",
    "assets/generated"
  ]
}
```

This configuration:

- recognises JavaScript files as part of one project;
- assumes modern JavaScript;
- treats files as ES modules;
- limits the project to selected source folders;
- excludes build output and external code;
- does not yet enable project-wide type checking.

---

## 5. Why start with `checkJs: false`

Setting:

```json
"checkJs": false
```

means that VS Code can still use JSDoc for autocomplete, hover information, navigation, and local inference without reporting type errors across every JavaScript file.

This is useful during the initial adoption stage.

Individual files can opt into checking with:

```js
// @ts-check
```

Example:

```js
// @ts-check

/**
 * @param {number} a
 * @param {number} b
 * @returns {number}
 */
export function add(a, b) {
  return a + b;
}
```

An invalid call may then be reported:

```js
add("1", 2);
```

This allows checking to be introduced module by module.

Later, when enough important files have accurate JSDoc, the project can move to:

```json
"checkJs": true
```

## 6. Recommended `jsconfig.json` rollout

### Stage 1: project definition only

```json
{
  "compilerOptions": {
    "allowJs": true,
    "checkJs": false,
    "target": "ES2022",
    "module": "ESNext"
  },
  "include": [
    "assets/**/*.js",
    "scripts/**/*.js"
  ],
  "exclude": [
    "_site",
    "node_modules",
    "vendor"
  ]
}
```

This gives tools a shared project boundary without introducing type-checking failures.

### Stage 2: selective file checking

Add:

```js
// @ts-check
```

to stable modules with good JSDoc.

Suitable candidates include:

- media URL utilities;
- shared data-loading modules;
- search-index code;
- route configuration;
- tag and dimension models.

### Stage 3: broader checking

Once type information is reliable:

```json
{
  "compilerOptions": {
    "allowJs": true,
    "checkJs": true,
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": false
  }
}
```

The initial project-wide mode should usually keep:

```json
"strict": false
```

until the volume and quality of reported issues are understood.

### Stage 4: increase strictness selectively

Possible later options include:

```json
{
  "compilerOptions": {
    "checkJs": true,
    "strict": true,
    "noImplicitAny": false
  }
}
```

This would require a deliberate migration and should not be introduced as a casual cleanup.

## 7. Include and exclude rules

The project should explicitly define which files are source code and which are not.

Example:

```json
{
  "include": [
    "assets/js/**/*.js",
    "scripts/**/*.js",
    "tests/**/*.js"
  ],
  "exclude": [
    "_site",
    "node_modules",
    "vendor",
    "assets/generated",
    "coverage",
    "tmp"
  ]
}
```

Common exclusions include:

```text
_site/
node_modules/
vendor/
coverage/
dist/
build/
generated output
third-party libraries
temporary files
```

These exclusions matter because generated and third-party files can:

- produce large numbers of irrelevant warnings;
- contain styles that do not match the repository;
- waste agent context;
- make validation slower;
- create pressure to edit files that should not be changed manually.

---

## 8. Browser and Node environments

A repository may contain more than one JavaScript runtime.

For example:

```text
assets/js/
    browser code

scripts/
    Node-based build or maintenance scripts

tests/
    test-runner environment
```

These environments should not be treated as interchangeable.

Browser code may use:

```js
window
document
fetch
localStorage
```

Node scripts may use:

```js
process
Buffer
__dirname
fs
path
```

Tests may use:

```js
describe
test
expect
beforeEach
```

Configuration should represent these differences explicitly.

## 9. Introducing ESLint

Modern ESLint projects commonly use a flat configuration file:

```text
eslint.config.js
```

A conservative starting point could be:

```js
import js from "@eslint/js";
import globals from "globals";

export default [
  {
    ignores: [
      "_site/**",
      "node_modules/**",
      "vendor/**",
      "assets/generated/**",
      "coverage/**",
    ],
  },

  js.configs.recommended,

  {
    files: ["assets/js/**/*.js"],

    languageOptions: {
      ecmaVersion: "latest",
      sourceType: "module",
      globals: globals.browser,
    },

    rules: {
      "eqeqeq": ["error", "always"],
      "prefer-const": "error",
      "no-unused-vars": [
        "error",
        {
          argsIgnorePattern: "^_",
          varsIgnorePattern: "^_",
        },
      ],
    },
  },

  {
    files: ["scripts/**/*.js", "*.config.js"],

    languageOptions: {
      ecmaVersion: "latest",
      sourceType: "module",
      globals: globals.node,
    },
  },
];
```

This configuration:

- ignores generated and external folders;
- enables ESLint’s recommended rules;
- treats browser files as browser code;
- treats scripts and config files as Node code;
- adds a small number of useful consistency rules.

---

## 10. Why environment-specific configuration matters

Without environment separation, tools may fail in two directions.

### False errors

Browser code may be incorrectly flagged:

```js
document.querySelector("#app");
```

because `document` is not recognised.

### Missed errors

Node-specific code may accidentally enter browser modules:

```js
const config = process.env.APP_CONFIG;
```

If `process` is incorrectly accepted everywhere, the error may not be detected until runtime.

Environment-specific configuration makes module boundaries clearer.

This also helps coding agents understand where code is intended to run.

---

## 11. Recommended initial ESLint rules

A useful starting set is:

```js
rules: {
  "eqeqeq": ["error", "always"],
  "prefer-const": "error",
  "no-unused-vars": [
    "error",
    {
      argsIgnorePattern: "^_",
      varsIgnorePattern: "^_",
    },
  ],
}
```

These rules help identify:

- accidental loose equality;
- variables that should not be reassigned;
- unused variables and parameters;
- abandoned code paths;
- misleading imports.

Avoid adding a very large style rule set initially.

Rules should primarily target correctness and clarity before cosmetic uniformity.

## 12. Example repository-specific ESLint structure

A more detailed configuration could distinguish browser modules, management code, build scripts, and tests:

```js
import js from "@eslint/js";
import globals from "globals";

export default [
  {
    ignores: [
      "_site/**",
      "node_modules/**",
      "vendor/**",
      "coverage/**",
      "assets/generated/**",
    ],
  },

  js.configs.recommended,

  {
    files: ["assets/js/**/*.js"],

    languageOptions: {
      ecmaVersion: "latest",
      sourceType: "module",
      globals: globals.browser,
    },

    rules: {
      "eqeqeq": ["error", "always"],
      "prefer-const": "error",
      "no-unused-vars": [
        "error",
        {
          argsIgnorePattern: "^_",
          varsIgnorePattern: "^_",
        },
      ],
    },
  },

  {
    files: ["scripts/**/*.js"],

    languageOptions: {
      ecmaVersion: "latest",
      sourceType: "module",
      globals: globals.node,
    },
  },

  {
    files: ["tests/**/*.js", "**/*.test.js"],

    languageOptions: {
      ecmaVersion: "latest",
      sourceType: "module",
      globals: {
        ...globals.node,
        describe: "readonly",
        test: "readonly",
        expect: "readonly",
        beforeEach: "readonly",
        afterEach: "readonly",
      },
    },
  },
];
```

The exact test globals should match the actual test framework.

## 13. Package configuration

The required packages might be installed with:

```bash
npm install --save-dev eslint @eslint/js globals
```

A minimal `package.json` script section could be:

```json
{
  "scripts": {
    "lint": "eslint .",
    "lint:fix": "eslint . --fix"
  }
}
```

Later, JSDoc-based checking can be added through TypeScript:

```bash
npm install --save-dev typescript
```

Then:

```json
{
  "scripts": {
    "lint": "eslint .",
    "lint:fix": "eslint . --fix",
    "check:js": "tsc --noEmit -p jsconfig.json",
    "check": "npm run lint && npm run check:js && npm test"
  }
}
```

This gives developers and coding agents a standard command:

```bash
npm run check
```

---

## 14. One command should define completion

A coding agent benefits from a clear validation target.

Instead of relying on several undocumented commands, provide one standard command:

```bash
npm run check
```

That command might run:

```text
ESLint
JSDoc-based type checking
JavaScript tests
site build checks
```

Example:

```json
{
  "scripts": {
    "lint": "eslint .",
    "check:js": "tsc --noEmit -p jsconfig.json",
    "test": "node --test",
    "build:site": "bundle exec jekyll build",
    "check": "npm run lint && npm run check:js && npm test && npm run build:site"
  }
}
```

The exact command should reflect the repository’s actual tooling.

## 15. Configuration comments

JSON does not officially support comments.

Therefore, `jsconfig.json` should remain concise and self-explanatory.

For more complex decisions, use a nearby Markdown file such as:

```text
docs/development/javascript-configuration.md
```

That file can explain:

- why checking is currently selective;
- which folders are excluded;
- why browser and Node files are separated;
- which strictness options are deferred;
- how to add new file groups.

The ESLint flat config is JavaScript, so limited comments are possible:

```js
{
  files: ["assets/js/**/*.js"],

  // Public site code runs in the browser and must not depend on Node globals.
  languageOptions: {
    globals: globals.browser,
  },
}
```

Comments should explain decisions, not restate syntax.

## 16. Configuration and JSDoc

JSDoc comments can exist without `jsconfig.json`.

Example:

```js
/**
 * @param {string} workId
 * @returns {Promise<WorkRecord>}
 */
export async function loadWork(workId) {
  // ...
}
```

However, project configuration makes the treatment of those comments more consistent.

It tells VS Code and other tools:

- which JavaScript files belong together;
- how imports should be resolved;
- whether JSDoc types should be checked;
- which language features are expected;
- which folders should be ignored.

The relationship is:

```text
JSDoc
    describes functions, objects, and contracts

jsconfig.json
    defines how the project interprets JavaScript and JSDoc

ESLint
    enforces code-quality and environment rules

package.json
    provides standard commands
```

## 17. Configuration and Codex

Project configuration helps Codex because it provides an explicit machine-readable description of the repository.

Codex can inspect:

```text
jsconfig.json
eslint.config.js
package.json
```

and infer:

- the expected module format;
- whether code is browser or Node based;
- which folders are generated;
- which lint rules matter;
- whether JSDoc types are checked;
- which command validates a change.

This reduces guesswork.

It also gives Codex a reliable completion process:

```bash
npm run check
```

Without configuration, Codex may have to infer different commands and environments from the code itself.

## 18. Recommended staged introduction

### Stage 1: map the JavaScript codebase

Identify:

- browser code;
- Node scripts;
- tests;
- generated files;
- vendored files;
- archived code;
- build output.

Do not configure linting before these boundaries are understood.

### Stage 2: add `jsconfig.json`

Start with:

```json
"checkJs": false
```

Define:

- target;
- module format;
- included folders;
- excluded folders.

### Stage 3: add basic ESLint

Enable:

- recommended rules;
- browser and Node environments;
- unused-variable checking;
- strict equality;
- `prefer-const`.

### Stage 4: add standard package scripts

At minimum:

```json
{
  "scripts": {
    "lint": "eslint .",
    "lint:fix": "eslint . --fix"
  }
}
```

### Stage 5: introduce JSDoc selectively

Prioritise:

- exported functions;
- shared data structures;
- configuration types;
- subsystem boundaries;
- non-obvious rules.

### Stage 6: enable file-level type checking

Use:

```js
// @ts-check
```

on stable modules.

### Stage 7: add project-wide checking

Move to:

```json
"checkJs": true
```

only after reviewing the likely error volume.

### Stage 8: consider stricter rules

Increase strictness only where it improves correctness and remains maintainable.

## 19. Suggested first implementation

A sensible first change would include:

```text
jsconfig.json
eslint.config.js
package.json scripts
documentation of exclusions and environments
```

It should not include:

- broad behavioural rewrites;
- conversion to TypeScript;
- comments on every function;
- maximum strictness;
- mass auto-fixing without review.

Example scope:

```markdown
Introduce baseline JavaScript project configuration.

Add:

- `jsconfig.json` with explicit source folders and exclusions;
- ESLint flat configuration;
- separate browser and Node environments;
- conservative correctness rules;
- standard `npm run lint` and `npm run lint:fix` commands.

Keep project-wide JSDoc type checking disabled initially.

Do not change runtime behaviour.
```

## 20. Suggested Codex task

```markdown
Introduce conservative JavaScript project configuration.

Requirements:

1. Add `jsconfig.json`.
2. Include the active JavaScript source and script folders.
3. Exclude generated output, vendored code, build output, dependencies, and
   temporary folders.
4. Use modern ECMAScript and ES modules where consistent with the current
   repository.
5. Keep project-wide `checkJs` disabled initially.
6. Add an ESLint flat configuration.
7. Separate browser code, Node scripts, and tests where applicable.
8. Enable ESLint recommended rules plus a small set of correctness-oriented
   rules.
9. Add package scripts for linting and automatic fixes.
10. Document the configuration and standard validation commands.

Do not:

- convert files to TypeScript;
- enable maximum strictness;
- add JSDoc to every function;
- change runtime behaviour;
- lint generated or vendored files.

Run the configured checks and report any existing issues separately from
configuration changes.
```

## 21. Review checklist

Before accepting the configuration, check:

- Are all active JavaScript source folders included?
- Are generated and vendored folders excluded?
- Are browser and Node environments separated?
- Does the module setting match the current code?
- Does the target JavaScript version match supported browsers and tools?
- Are test globals configured correctly?
- Are lint rules focused on correctness rather than stylistic churn?
- Is project-wide type checking still appropriately conservative?
- Are standard commands documented?
- Does `npm run lint` complete successfully?
- Are automatic fixes reviewed rather than accepted blindly?
- Does the configuration avoid changing runtime behaviour?

---

## 22. Recommended initial policy

```markdown
## JavaScript configuration policy

- JavaScript source folders and runtime environments must be declared
  explicitly.
- Browser code, Node scripts, and tests should use separate configuration
  blocks where their globals differ.
- Generated, vendored, build-output, dependency, and temporary folders must
  be excluded from analysis.
- ESLint should initially enforce correctness and clarity rather than broad
  stylistic rules.
- JSDoc type checking should be introduced gradually.
- Stable modules may opt into checking with `// @ts-check`.
- Project-wide `checkJs` should be enabled only after the existing error
  volume has been reviewed.
- Standard validation commands must be exposed through `package.json`.
- Configuration changes should not alter runtime behaviour.
```

## Conclusion

Configuration is primarily about consistency, but its value is broader than visual uniformity.

It creates a shared machine-readable definition of the JavaScript project:

```text
project scope
+ runtime environments
+ module format
+ exclusions
+ lint rules
+ JSDoc interpretation
+ validation commands
```

This helps VS Code, Codex, developers, and automated checks interpret the repository in the same way.

The recommended approach is to establish boundaries and conservative checks first, then introduce JSDoc-based type checking gradually.
