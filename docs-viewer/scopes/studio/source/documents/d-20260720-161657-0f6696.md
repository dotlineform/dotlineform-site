# Site Overview

## What Site Is

Site is the public dotlineform website. It is a tracked static artifact served directly from `site/`; there is no deploy-time build or copy step.

The catalogue experience is centred on `/series/`. `/` redirects there, Series organise Works, and selected Work and detail views continue through fixed query-driven route shells.

Public Moments, Analysis, and Library are Docs Viewer scopes. They share site navigation and presentation where appropriate but do not become catalogue runtime modules.

## Public Catalogue Surface

- `/series/` — Works home, initially grouped by Series; selecting a Series shows its ordered Works.
- `/works/` — flat Works list; `?work=<work_id>` selects Work media, metadata, details, prose, and Series navigation.
- `/work-details/?detail=<detail_uid>` — selected nested Work detail and sibling context.
- `/recent/` — recent publication snapshots.
- `/catalogue/search/` — public Work and Series search.
- `/about/` and `404.html` — small supporting pages.

A Series containing one Work links directly to that Work rather than adding an intermediate Series view.

## Execution Path

```text
canonical catalogue source in Studio
  -> catalogue services validate and build
  -> tracked public JSON and media under site/assets/
  -> fixed HTML route shell under site/
  -> one route ES module
  -> narrow shared helpers and components
  -> rendered catalogue view
```

The runtime is browser-side JavaScript over static files. It has no public application server and no browser write path.

## Ownership

### Public Runtime

- route HTML owns stable markup, boot data attributes, and the catalogue module entrypoint;
- `site/assets/js/catalogue/routes/` owns route state, payload loading, and orchestration;
- `site/assets/js/catalogue/components/` owns reusable catalogue rendering;
- `site/assets/js/catalogue/navigation/` owns keyboard, swipe, and Series navigation behaviour;
- `site/assets/js/catalogue/shared/` owns URL, fetch, text, thumbnail, and local-storage helpers;
- site CSS owns the shell and public catalogue presentation.

### Source And Build

Studio owns canonical Works, Series, details, prose, validation, generation, media publication, and build planning. Generated public payloads belong in `site/` because Site consumes them, even though Studio services produce them.

### Public Docs

Docs Viewer owns the `/moments/`, `/analysis/`, and `/library/` reader runtimes and their generated scope payloads. Site owns the shared public chrome and host presentation they consume.

## Extension Method

For a public catalogue change:

1. decide whether it changes route state, a list-wide projection, a selected record, media, or presentation;
2. update the narrow route/component/helper owner;
3. update Studio source/build code only when the public data contract changes;
4. keep URL construction in the catalogue URL helper;
5. run the smallest route, payload, validation, and visual checks that prove the result.

Do not introduce a public server, per-record HTML pages, duplicated URL construction, or browser access to canonical source merely to simplify one view.

## Known Weak Spots

- Shared head, header, navigation, and footer markup is repeated across tracked route shells.
- `main.css` still mixes global shell rules with older catalogue route styles while `catalogue.css` owns newer components.
- `/series/` is publicly labelled Works, so filenames and product vocabulary do not always align.
- Route context is split between query parameters and local storage; restoration rules need care when changing navigation.
- Some generated catalogue artifact families are retained even when the current public route modules do not read them directly.
