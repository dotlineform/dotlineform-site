# Site Review And Cleanup

## Purpose

Make the public catalogue easy and safe to change again after its large JavaScript refactor and a period of limited Site work.

The catalogue works, but routing and restoration rules are split across URLs and local storage, CSS ownership is divided unevenly, repeated static shells can drift, and some generated artifacts or older runtime paths may no longer have clear consumers. These are one maintenance problem: the current Site contract is harder to see and verify than it should be.

## Result

- Current public behaviour is recorded before cleanup begins.
- Route, query, persistence, redirect, and link ownership is explicit.
- Global shell, catalogue component, and route-specific CSS have clear homes.
- Repeated static-shell markup is either aligned or given a justified maintenance method.
- Dead runtime, styles, generated artifacts, and tests are removed only with usage evidence.
- Focused checks protect durable route and data boundaries without turning presentation details into brittle contracts.

The baseline review will recommend one or more independently complete cleanup deliveries. Likely areas are routing and state, then CSS and shell ownership, but the review decides whether those are separate outcomes or one safe change set.

## Boundary

- Preserve current public catalogue behaviour unless the review confirms a defect or an obsolete path.
- Keep `site/` as the tracked deploy artifact; do not introduce a build, template, framework, or server merely to remove repetition.
- Do not turn maintenance cleanup into a visual redesign or new catalogue feature.
- Do not redesign canonical catalogue data or Studio generation unless consumer evidence exposes a separate bounded problem.
- Public Docs Viewer route internals remain under Docs Viewer. Only genuinely shared site-shell and navigation boundaries belong here.
- Update durable Site documents with shipped current state; keep review evidence and delivery sequencing in this feature.

The feature is proposed. No runtime cleanup begins until the baseline review is accepted and its first cleanup delivery is explicitly approved.
