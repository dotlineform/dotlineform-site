# DSO-0 — Behavior And Dependency Findings

## Review Gate

This is the temporary audit record for [Document Sharing Ownership Cutover Delivery](/docs/?scope=studio&doc=d-20260720-172819-93d6b2). It records the DSO-0 findings; code and runtime configuration remain unchanged.

The cutover has no discovery blocker. DSO-1 should not begin until this document's target surface and contract corrections have been reviewed.

## Headline Findings

- The live document capability has two prepare profiles: `document-content` and export-only `document-tree`. `document-full-source` is not an enabled profile.
- Most document behavior already lives in Docs Viewer services. Analytics supplies the route shell/API gateway, while `data-sharing/` supplies generic dispatch, a thin documents adapter, tag behavior, workspace paths, and returned metadata lookup.
- Trusted export metadata already stores `selected_doc_ids`, so complete-set validation can be added without changing the external package format or workspace location.
- Current returned parsing does not compare the returned identity set with `selected_doc_ids`. Missing and extra records are not detected as a package-level mismatch; duplicate and malformed identities are warnings or skipped rows in some paths.
- The current browser renders a returned-row selector, but its review and apply requests omit `record_indices`. The visible selection therefore does not control those actions; they operate on all parsed rows.
- Docs Import, Docs Review, management capability reporting, and staged-media services also import the generic Data Sharing workspace helper. That helper must move and be retargeted before `data-sharing/` can be deleted.
- The Data Sharing schema directory is empty apart from `.gitkeep`, and `services/package_io.py` only exposes its unused path. There is no generic schema implementation to migrate.

## Current Prepare Contract

| Profile | Current behavior | Return path |
| --- | --- | --- |
| `document-content` | JSONL by default; JSON supported. Exports document identity, relationships, summaries, headings, and rendered-derived Markdown or plain text. Content is capped at 50,000 characters per document with paragraph-boundary truncation. | Supported |
| `document-tree` | JSON-only nested document IDs and titles for the selected subtree. | Export-only |

Both profiles use explicit document IDs, declare descendant and non-viewable inclusion, preserve source order, and reject unknown selected IDs. The current engine expands descendants automatically for `document-tree`; `document-content` exports the explicit IDs supplied by the route. The route also supports select-all. `document-content` is compact rendered-derived content; it is not source-faithful Markdown.

The prepare result reports `ok`, dry-run/write state, `export_id`, scope/profile/format, output paths, selected/exported IDs, counts, warnings/errors, and whether output was written.

## Package Identity And Workspace

The initial cutover must preserve these physical paths:

| Artifact | Current and retained path |
| --- | --- |
| Outbound package and context sidecar | `$DOTLINEFORM_PROJECTS_BASE_DIR/data-sharing/exports/` |
| Manually returned package staging | `$DOTLINEFORM_PROJECTS_BASE_DIR/data-sharing/import-staging/` |
| Markdown review evidence and validated Docs Review projections | `$DOTLINEFORM_PROJECTS_BASE_DIR/data-sharing/import-preview/` |
| Trusted internal export metadata | `$DOTLINEFORM_PROJECTS_BASE_DIR/data-sharing/meta/` |

The initial cutover must also preserve current serialized identifiers even where they retain Data Sharing terminology:

- export IDs: `ds_YYYYMMDDTHHMMSSZ`;
- returned package schema: `data_sharing_returned_package_v1`;
- internal metadata schema: `data_sharing_export_meta_v1`;
- JSONL routing row: `record_type: data_sharing_header`.

JSON packages carry an envelope with `export_id` and `records`. JSONL packages carry the routing row first. Prepare also writes an external context sidecar and a trusted internal metadata sidecar. The internal sidecar records profile, scope, format, return eligibility, selected IDs, source update values, counts, and config checksum.

## Current Returned Workflow

1. The service lists `.json` and `.jsonl` files under `import-staging/`, excludes sidecars, resolves `export_id`, and requires matching trusted internal metadata.
2. Export-only or unknown profiles are blocked before becoming actionable.
3. Inspect parses all records, normalizes document fields, annotates current source state, and returns counts, issues, records, and review rows.
4. The current review menu offers:
   - **Content** — validates and atomically publishes an immutable text-only package for Docs Review;
   - **Summaries** — writes Markdown review evidence under `import-preview/`;
   - **Hierarchy** — writes the same form of Markdown review evidence with hierarchy-oriented status copy.
5. Apply offers summary or hierarchy preflight, confirmation, source writes, and targeted Docs Viewer rebuilds.
6. Docs Import separately consumes supported staged document packages through its collection planning/apply boundary.

Summary and hierarchy apply accept `record_indices`; omission means all records. The browser omits the field. Content publication intentionally validates the full staged file. Current narrow apply treats unchanged or missing candidate fields as skips/no-ops and prevents writes when it finds a missing canonical target, but duplicate or malformed rows can be skipped while otherwise valid rows remain actionable.

## Atomic Return Correction

The accepted product contract is stricter than current behavior. DSO-1 must add one shared package-level validator before inspect, review, publication, apply, or Docs Import handoff can become actionable.

The validator should require:

- trusted metadata for a return-enabled profile;
- a non-empty `selected_doc_ids` list in that metadata;
- exactly one base-valid returned record for every prepared document ID;
- no returned ID outside the prepared set;
- no duplicate or missing document ID;
- valid package format, profile, scope, routing identity, and required base fields.

Prepared and returned ID **sets** must match; row order is not identity and may change. Optional action fields may remain sparse: an omitted supported field or an unchanged value is a valid no-op, while a present invalid value blocks that action for the whole package. The cutover should not invent a universal source-freshness policy; existing action-specific current-source checks remain authoritative.

`record_indices`, selected-record reports, and `selectable` review-row semantics should be removed from browser requests, service requests, configuration, results, and tests. Inspection can still return ordered rows and row diagnostics, but the only actionable unit is the package.

The external guidance for `document-content` should explicitly require one returned record per prepared ID. Its existing sparse-field guidance can remain.

## Current Dependency Trace

```text
Analytics prepare/returned route shells
  -> Analytics config, transport, route state, modal, activity, CSS
  -> /analytics/api/data-sharing/*
  -> Analytics Data Sharing API and gateway
  -> generic registry + workflow dispatch
  -> thin documents adapter
  -> Docs Viewer export/returned/package services
  -> Docs Viewer source, write/rebuild, Docs Import, and Docs Review owners

Shared path + metadata helpers
  -> document packages
  -> Docs Import and staged media
  -> Docs Review sessions/packages
  -> Docs Viewer management capability reporting
```

The target removes the middle three generic layers:

```text
Docs Viewer package routes
  -> fixed Docs Viewer package client
  -> direct document-package service
  -> existing Docs Viewer, Docs Import, and Docs Review owners
```

## Move Or Absorb

| Current owner | DSO action | Target owner |
| --- | --- | --- |
| `data-sharing/adapters/documents/config/prepare-profiles.json` | Move without adding profiles | `docs-viewer/config/document-packages/profiles.json` |
| Thin documents adapter wrappers and family dispatcher | Absorb behavior; do not preserve adapter abstractions | `docs-viewer/services/docs_document_packages/` |
| `docs-viewer/services/docs_data_sharing/`, `docs_export*`, and `docs_returned_import*` | Re-home and remove adapter/subset terminology | `docs-viewer/services/docs_document_packages/` |
| `data-sharing/services/paths.py` | Move first and retarget all Docs Viewer/Import/Review consumers | `docs_document_packages.workspace` |
| `data-sharing/services/returned_metadata.py` | Move and make document-specific | `docs_document_packages.metadata` |
| `docs_import_data_sharing_*` and the compact/full-source content normalizer | Keep under Docs Import, rename package boundary, and call the Docs Viewer metadata/parser owner | Docs Import document-package modules |
| Docs Review package/session services | Keep in place; retarget workspace imports only | Existing Docs Review owners |
| Analytics document browser modules and route HTML | Move useful document-only behavior, remove domain/app/row-selection state, and replace Analytics dependencies | `docs-viewer/runtime/js/packages/` and Docs Viewer route shells |
| Data Sharing CSS in `analytics.css` | Extract only the required route presentation | Docs Viewer-owned package CSS |
| Document activity emission | Retarget route/endpoint context and log source | Docs Viewer document-package service |

This is a final integrated-only Docs Viewer boundary. Do not create `docs-viewer/data-sharing/`, adapter aliases, or an Analytics proxy.

## Retire Rather Than Move

- all `data-sharing/adapters/tags/` code, tag profiles, returned parsing/apply, and tag package tests;
- the tag records in `data-sharing/config/adapters.json`;
- generic registry, registry schema, dispatch, and workflow modules under `data-sharing/services/` and `data-sharing/workflows/`;
- unused `data-sharing/services/package_io.py` and the empty `data-sharing/schemas/` directory;
- Analytics Data Sharing server gateway, resolver, endpoint composition, and neutral route constants;
- Analytics app/domain selection, generic capability projection, tag-specific browser branches, returned-row selection UI, and Data Sharing route shells after the document UI has moved;
- Analytics home/config links, route checks, smoke coverage, runner labels, and active activity-contract entries that describe the retired routes;
- current generic/tag tests after equivalent document contracts are protected under Docs Viewer.

Retiring active route/activity configuration does not require deleting historical activity records. Any display metadata still needed to render old records should be classified as historical data, not left as an active route or capability.

## Proposed Frozen Docs Viewer Surface

### Browser Routes

| Workflow | Route |
| --- | --- |
| Prepare | `/docs/packages/prepare/` |
| Returned Packages | `/docs/packages/returned/` |

### Same-Origin Service Endpoints

| Method | Endpoint | Responsibility |
| --- | --- | --- |
| `GET` | `/docs/packages/config` | Fixed browser-safe scopes, profiles, actions, and workspace availability |
| `GET` | `/docs/packages/documents` | Selectable documents for one `scope` |
| `GET` | `/docs/packages/returned` | Actionable and blocked document packages from staging |
| `POST` | `/docs/packages/prepare` | Dry-run or write one selected document package |
| `POST` | `/docs/packages/context` | Validate or save external profile context |
| `POST` | `/docs/packages/returned/inspect` | Parse and validate the complete returned package for inspection |
| `POST` | `/docs/packages/returned/review` | Produce current Content, Summaries, or Hierarchy review result for the complete package |
| `POST` | `/docs/packages/returned/apply` | Preflight or confirm one supported whole-package apply action |

Do not add a package-specific health endpoint. The existing `/health` and `/capabilities` endpoints should report service and `document_packages` availability.

Requests should use direct document terms: `scope`, `doc_ids`, `select_all`, `profile_id`, `target_format`, `content_format`, `staged_filename`, `review_action`, `apply_action`, `dry_run`, and `confirm`. They should not accept `app`, `data_domain`, `adapter_id`, generic `operation`, nested adapter selection, or `record_indices`.

The old Analytics pages and APIs are removed at cutover with no redirect, compatibility endpoint, or dual route.

## Fixture And Baseline Evidence

There is no checked-in canonical document-package fixture corpus. Current Python tests synthesize source documents, JSON/JSONL packages, and trusted metadata in temporary workspaces. The Analytics browser smoke stubs the Data Sharing API rather than exercising a real returned package through the full service.

DSO-0 baseline on 2026-07-20:

- 67 focused Docs Viewer export, returned parsing, review publication, summary/hierarchy apply, and package-content tests passed;
- 39 Analytics API, generic registry/dispatch, adapter, and tag prepare tests passed;
- tests ran with isolated writable `DOTLINEFORM_PROJECTS_BASE_DIR` roots.

DSO-1/DSO-4 should establish a small document-owned fixture matrix covering:

- `document-content` JSONL and JSON plus matching metadata/context;
- export-only `document-tree`;
- valid exact-set return with reordered rows;
- missing, extra, duplicate, malformed, or mismatched records blocking the package;
- sparse valid action fields and unchanged no-ops;
- Docs Review publication and summary/hierarchy preflight/confirmed apply.

## Review Decisions

Please review these proposed freezes before DSO-1:

1. accept `/docs/packages/prepare/` and `/docs/packages/returned/` as the two browser routes;
2. accept the eight direct endpoints above and the absence of a package health endpoint;
3. accept exact set equality with order-insensitive identity and action-specific sparse fields;
4. accept `docs-viewer/config/document-packages/` and `docs-viewer/services/docs_document_packages/` as the final integrated-only owners;
5. accept creation of a small document-owned package fixture matrix during implementation;
6. keep source-freshness handling action-specific during this cutover.

No DSO-1 implementation has started.
