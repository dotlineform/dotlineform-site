# Docs Viewer Core Definition Analysis Delivery

## Outcome

Produce an evidence-backed **Docs Viewer Core Definition Baseline v1** that is precise enough to support transition architecture and the first implementation delivery without pretending the boundary is immutable.

The current [Standalone Docs Viewer Concept](/docs/?scope=studio&doc=d-20260718-105957-c72b2a) contains a useful core hypothesis. It is not yet an authoritative definition derived from the current application, its consumers, and its open proposals.

## Scope

This is a main-owned analytical delivery. It inspects the current repository and records conclusions; it does not extract modules, change runtime behavior, create the long-lived refactor worktree, or package a standalone application.

The analysis must answer:

- Which user-visible and internal capabilities constitute Docs Viewer today?
- Which responsibilities are required by both integrated and standalone compositions?
- Which responsibilities are integrated-only, standalone-only, adjacent-application-owned, or incidental repository machinery?
- What inputs, outputs, state, services, filesystem roles, configuration, and process assumptions does each responsibility require?
- Where do current adapters provide a real seam, and where do they hide unresolved ownership?
- Which open or ready roadmap proposals depend on, challenge, or extend the candidate boundary?
- What evidence would prove that the proposed core can serve both compositions without becoming a generic framework?

## Method

1. Inventory material reader, management, scope, source, build, search, media, import, export, service, and configuration capabilities.
2. Trace their current browser, Python, generated-data, filesystem, environment, and adjacent-application dependencies from authoritative code and config.
3. Map current consumers and composition-specific registrations, including public, repository, R2, Data Sharing, Docs Review, Studio, and external-local behavior.
4. Classify each responsibility as core, integrated-only, standalone-only, adjacent, or unresolved, with evidence and rationale.
5. Test the candidate boundary against current integrated behavior, the standalone product boundary, and open roadmap proposals.
6. Map current tests to durable capability and composition contracts using the sibling [Testing](/docs/?scope=studio&doc=d-20260718-123734-6d84c1) rules.
7. Identify contradictions, responsibilities that must move together, and decisions that Transition Architecture must resolve.

Use temporary inventories or dependency scans when detailed evidence must survive across sessions. Keep them as disposable child notes and promote only stable conclusions into the definition baseline.

## Provisional Finding: Import Boundary

This initial finding predates the full capability inventory. Treat it as an evidence-backed classification for the analysis to verify, not as an accepted Core Definition Baseline.

Ordinary direct import is a core Docs Viewer responsibility. It includes confined staged-source listing, classification of core-owned formats, preview and conversion, create-only document creation, associated media handling, and rebuild. Current core candidates are HTML, DOCX, Markdown, plain text, and a Markdown directory containing one document plus its assets.

Reviewed document-package import is integrated-only. Recognition of trusted JSON or JSONL packages, whole-package planning and apply, collision decisions, result reporting, and Docs Review or Data Sharing context must be registered by the integrated composition. Package adapters may consume package-neutral core primitives such as `ImportContent` and per-document plan/apply without moving package semantics into core.

There remains one Import action and workflow surface. The integrated application adds a document-package import contribution; the standalone application composes only ordinary document import. Do not create parallel core and full Import implementations, scatter a standalone-mode flag through the workflow, or import an excluded package integration and merely hide its controls.

Staged-source classification is service-owned. Core classifies only formats it supports. The integrated composition may register its package recognizer before listing or dispatch, and the browser consumes the resulting format classification rather than sniffing file contents. When that recognizer is absent, package candidates remain unsupported and direct requests must not bypass the boundary.

Current code provides useful seams but does not yet satisfy this dependency direction:

- staged listing already accepts additional `registered_source_formats`;
- ordinary and package imports share package-neutral document planning and apply primitives;
- `docs_import_source_service.py` still imports package recognition, collection planning and apply, and Data Sharing workspace paths directly;
- the management controller lazy-loads Import, but the loaded browser entry imports the collection controller eagerly and the modal shell assumes package controls;
- the service launcher imports document-package and Docs Review modules at startup.

Transition Architecture must therefore define an explicit staging-location input, a concrete optional import-format registration and dispatch seam, separate ordinary and integrated browser entry composition, and dependency checks proving that the standalone distribution neither imports nor serves document-package or Docs Review modules. Integrated verification must preserve the current combined Import behavior.

Terminology must remain explicit: a Markdown package is one ordinary document plus assets and is core; a reviewed document package is a multi-document integration and is not core.

## Defined Deliverables

The completed analysis must provide:

- a capability and responsibility inventory with current owners;
- a composition matrix showing core, integrated-only, standalone-only, adjacent, and unresolved classifications;
- a dependency and environment-assumption map;
- explicit core inclusion and exclusion criteria;
- named composition and extension seams justified by current consumers;
- an assessment of affected open and ready roadmap proposals;
- proof scenarios for the integrated and standalone compositions;
- an initial test map classifying durable, owner-coupled, obsolete, and missing evidence;
- a concise Core Definition Baseline v1 as a durable sibling document;
- a list of unresolved transition questions handed to Core Refactor Transition Architecture.

## Completion Gates

- Every material current capability has an evidence-backed classification or is explicitly unresolved.
- Core responsibilities have named inputs, outputs, owners, and allowed environment assumptions.
- Integrated-only and adjacent responsibilities have an explicit composition or ownership reason for exclusion.
- The standalone boundary can be expressed without copying shared behavior or adding a global mode flag.
- Open roadmap proposals have been assessed as compatible, constraining, newly dependent, or in need of reshaping.
- Proof scenarios are concrete enough to test later implementation deliveries.
- Durable contracts have current evidence or an explicitly named evidence gap.
- Remaining uncertainty is bounded and does not prevent Transition Architecture from selecting a first safe slice.
- The Core Refactor parent and concept link to the accepted baseline.

## Adjustment Contract

Baseline v1 is a decision-enabling definition, not a permanent freeze. Implementation may reveal hidden coupling, incorrect classifications, or a better ownership boundary.

Any proposed adjustment must record:

- the new evidence or implementation constraint;
- the previous and proposed classification or boundary;
- affected compositions, consumers, open proposals, and active deliveries;
- the effect on transition order, verification, and completed invariants;
- the updated proof that the revised boundary remains coherent.

Make the change in the main-owned Core Definition document and propagate its consequences through Transition Architecture and affected delivery/proposal documents. Do not let the definition drift silently inside feature-branch code. If an adjustment invalidates an active delivery, pause and reshape that delivery before continuing.

## Status

Proposed. The planning baseline must be committed before this analysis becomes active. No core definition baseline, transition architecture, implementation delivery, or long-lived refactor worktree exists yet.
