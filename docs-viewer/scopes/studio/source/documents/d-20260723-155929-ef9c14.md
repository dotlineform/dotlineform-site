# UI Workbench Guide

## Purpose And Current Boundary

Use the Workbench to reach production-backed UI states quickly, inspect them under controlled conditions, and surface rendering, lifecycle, browser, and presentation problems before constructing a complete workflow.

The Workbench does not prove that a real service, payload, navigation path, persistence operation, or side effect reaches the state correctly. Focused behavior checks and representative end-to-end or live routes remain required where those are the risk. See [UI Workbench Concept](/docs/?scope=studio&doc=d-20260723-093229-dd63ab) for ownership and design boundaries.

The first app pack is Docs Viewer. It provides exact-name specimen selection and named comparison recipes. Random pairing is not a review method.

A production module is eligible only when a current product workflow composes it or the approved delivery is adding that composition. If the Workbench exposes an unwired or abandoned surface, stop reviewing its appearance and resolve whether to retire it or approve it as product work.

## Open The Workbench

Use `/admin/ui-workbench/`. It is the only Workbench route.

The normal composition is:

`Admin host → app-owned pack entrypoint → Admin-owned isolated frame → production-backed specimen`

Docs Viewer owns its registry, fixtures, adapters, and production imports. Admin loads that code and narrowly serves its local dependencies; it does not proxy Docs Viewer workflow APIs. Each specimen renders in an isolated frame so modal focus, scroll, and listeners do not interfere with the host or another specimen.

## Development Workflow

1. Define the production owner, current composition path, affected states, and verification boundary in the delivery.
2. Add or update the affected app specimen in the same implementation slice.
3. Open `/admin/ui-workbench/` and select the exact named specimen to iterate without a real write or external workspace.
4. Review the relevant specimens across light/dark and desktop/narrow presentation.
5. If the delivery defines a useful comparison, state its question, counterpart, dimensions, and sequential or side-by-side review mode.
6. Run focused behavior checks for the interaction contract being changed.
7. Run the existing end-to-end check or representative live route where services, payloads, persistence, navigation, or side effects are the risk.
8. Keep the permanent registry boot check green so every specimen continues to load through production modules without browser errors or writes.

Workbench observation alone is not completion evidence.

## Select A Specimen

Start from the delivery question, not from the controls:

1. Search for the exact named specimen that exercises the changed contract. Its label includes family and state as context.
2. Review light and dark presentation where theme tokens or state colours matter.
3. Review desktop and narrow presentation where sizing, wrapping, scrolling, or action access matters.

Most reviews should use one specimen at a time. A new or materially changed modal should define:

- the normal and exceptional states that require specimens;
- the production renderer and injected dependencies;
- the focused behavior checks that retain authority;
- the representative live path, if one remains necessary;
- any meaningful comparison, including the exact counterpart and question.

## Compare Only To Answer A Question

Comparison is an optional diagnostic, not the coverage mechanism. The registry supplies state reachability; a delivery supplies the reason to compare.

| review mode | use it for |
| --- | --- |
| sequential | initial focus, Tab and Shift-Tab, Escape precedence, backdrop cancellation, focus return, scroll restoration, state transitions, and narrow-screen interaction |
| side by side | width, offset, spacing, typography, field structure, status treatment, action order, disabled/busy presentation, body scrolling, and footer ownership |

Side-by-side review reduces reliance on visual memory and holds theme and viewport conditions constant. It does not add lifecycle verification because only one specimen can meaningfully own focus and receive interaction at a time.

Useful comparison questions include:

- Do Metadata and Settings normal states use the same field and action-row presentation where their contracts match?
- Do Settings and Import busy states distinguish disabled actions and valid cancellation policy?
- Do Settings and Import failure states present status and terminal actions consistently?

Do not compare merely because two specimens share a broad registry group or look superficially similar. Classify selection, focus, navigation, state, data ownership, actions, and workflow responsibility first.

For a new or changed modal, record one of:

- no comparison required;
- sequential review against named specimens for named behavior questions;
- side-by-side review against one named specimen for named presentation questions.

Choose the named review recipe. It selects the exact pair, shows the question and dimensions, and either presents both panels or exposes sequential A/B steps. If no recipe answers the delivery question, record `no comparison required` or add a focused recipe with the affected specimens; do not improvise a random pair.

## Evidence Responsibilities

| evidence | responsibility |
| --- | --- |
| Workbench | make controlled UI states cheap to reach, compare, and interact with; surface problems early |
| focused behavior checks | prove focus, keyboard, cleanup, scroll, cancellation, and workflow-owned state contracts |
| end-to-end checks and representative live routes | prove real services and application wiring reach the correct state and perform the correct effects |
| permanent registry boot check | prove every registered specimen imports and mounts without browser errors or writes |

Use the smallest evidence that protects the actual boundary. Do not replace focused checks with screenshots or duplicate complete workflow choreography in the Workbench.

## Screenshots And Scratch Output

The live production-backed specimen is the current reference. A screenshot records one review moment.

- Keep selected reviewed screenshots under `admin-app/app/assets/ui-workbench/screenshots/<app>/`.
- Name a retained capture by specimen, state, theme, and viewport.
- Review, replace, or remove a capture when its specimen materially changes.
- Keep scratch captures, measurements, exports, and generated reports under `var/admin/ui-workbench/`.
- Do not embed screenshot inventories in documents or treat captures as pixel-approved release gates.

## Maintenance Triggers

- A delivery adding or materially changing a modal, list, field, status region, action row, menu, or shared control updates its app pack in the same slice.
- A specimen whose current product composition cannot be identified is removed or separately approved; a runtime source path alone is not product ownership.
- A new or changed modal explicitly records whether comparison is useful and, if so, the named counterpart, question, dimensions, and review mode.
- A shared production component change reviews every registered consumer specimen.
- A major UI refactor or app-shell change begins with a complete relevant-pack review.
- A periodic maintenance review may revisit all packs to identify drift that no single delivery exposed.

The Workbench should make these reviews cheap. It does not introduce an automated calendar, release gate, or screenshot approval system.
