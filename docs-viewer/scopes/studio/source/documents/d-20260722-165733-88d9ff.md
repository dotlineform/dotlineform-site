# Missing-Summary Package Preparation Delivery

## Status

**Active. MSP-P3 is complete; MSP-P4 awaits explicit approval.**

This delivery captures four related changes:

1. a missing-summary filter and live target count in the **Prepare package** modal;
2. an **Include non-viewable documents** package choice, on by default;
3. a generic **Select all** transition over the complete manage index;
4. removal of the manage-index visibility filter and the separate **Show** viewability Action, followed by an audit and retirement of their unused dedicated workflow.

The index selection owner assembles checked ids; the Package Prepare workflow interprets package-specific filters; **Edit metadata** owns changes to a document's viewability. Public projection continues to enforce viewability independently. MSP-P3 has shipped the complete manage index and generic bulk-selection transition; stop for explicit approval before removing the separate **Show** workflow in MSP-P4.

## Intended Outcome

A user can always see and select every source document in the manage index, including non-viewable documents. They can optionally add descendants, limit the **Document content** package to documents whose current source metadata has no summary, and choose whether the package includes non-viewable documents. Before confirmation, the modal shows how many documents the current choices are expected to prepare.

For the scope-wide case, the intended path is:

1. choose **Select** in the manage index;
2. choose **Select all**;
3. choose **Prepare package** from **Actions**;
4. use the **Document content** profile;
5. enable **Only documents missing summaries**;
6. leave **Include non-viewable documents** on, or turn it off to exclude them;
7. confirm the displayed total and prepare the package.

The raw checked count is only the index-selection starting point. The modal derives one final effective target after descendant expansion, both package filters, and any configured profile limit. That effective target is sent as explicit document ids; **Select all** does not switch Package Prepare to the service's separate `select_all` request mode.

## Current Starting Point

- The package engine already accepts `missing_summary_only` and skips a requested document with reason `has_summary` when its normalized source `summary` is non-empty.
- Both current profiles declare `supports_missing_summary_only: false`; the browser-safe profile projection does not expose the support or default flags.
- The direct Prepare service currently passes `missing_summary_only=None`, so a browser request cannot activate the engine filter.
- The package-document feed already includes source summary text, parent ids, and selection eligibility. The browser can therefore calculate an immediate count from the same loaded snapshot used for descendant expansion.
- None of the 512 current source documents across Studio, Library, Analysis and its sub-scope, Moments, Processing, and external Notes uses document-level `published` front matter. That older pipeline-inclusion field was removed from the Docs source model in May 2026, then unintentionally returned inside the July package source projection and its synthetic tests.
- The content profile declares `include_non_viewable: true`, but the package-document feed currently marks non-viewable documents as unselectable. The service treats the profile value as fixed rather than accepting a per-request choice.
- The index selection owner supports enter, toggle, Shift-range, clear, exit, and reload reconciliation, but no bulk membership transition.
- Manage mode currently defaults to showing non-viewable documents but exposes a **Show non-viewable** scope control that can remove them from the index and selection population.
- The separate **Show** Action is not a simple metadata shortcut: it can include non-viewable ancestors and optionally descendants through the bulk viewability endpoint. **Edit metadata** changes only the active document's own `viewable` value.
- Package Prepare sends explicit ids with `select_all: false` and keeps checkbox selection unchanged after cancel, success, or failure.
- The current export engine counts documents removed by selection filters as `skipped` outside `selected`. A large filtered-out population can therefore produce a `skipped` count larger than `selected`, even though those documents were deliberately excluded before preparation. This delivery replaces that reporting model for the direct Prepare workflow.
- Both profiles currently set `limits.max_documents` and `limits.max_total_chars` to `null`. The content profile separately truncates each document's content at 50,000 characters.

## Proposed Modal Contract

- Show **Only documents missing summaries** immediately below **Include descendants**.
- Show it only when the selected profile explicitly supports the filter. The **Document content** profile supports it; the **Document tree** profile does not.
- Leave it unchecked by default so existing Package Prepare behavior does not change silently.
- Define a missing summary as empty or whitespace-only source `summary` metadata. Body content, headings, generated proposal fields, and published search text do not affect this decision.
- Show **Include non-viewable documents** below the missing-summary choice for the **Document content** profile and leave it checked by default. The tree profile keeps non-viewable documents included without offering this filter.
- Calculate in this order: checked-id eligibility, optional descendant expansion, non-viewable filtering when requested, missing-summary filtering, then any configured maximum document count.
- Display `Total documents to be prepared: N` after the choices and recalculate it whenever the profile or any checkbox changes.
- Describe deliberate filter exclusions before submission, for example `98 documents excluded because they already have summaries.` These exclusions do not become result `skipped` documents because they are outside the effective target.
- When non-viewable documents are included, disclose their count, for example `Includes 4 non-viewable documents.` When they are excluded, report how many were excluded before submission.
- Remove the initial `Prepare N checked documents` message. The raw checked count is not a package result and becomes irrelevant once modal choices alter the target.
- If the total is zero, disable **Prepare package** and explain which active filters excluded every targeted document.
- Keep per-profile option state. Switching profiles must not leak an unsupported filter choice into the tree profile.
- Send only the final effective target ids, plus explicit `missing_summary_only` and `include_non_viewable` choices so the service can revalidate those ids against current source. The service never broadens the snapshot by adding a document the modal excluded.
- Treat the live total as the result for the loaded source snapshot. The service remains authoritative if document summaries or hierarchy change before submission.

## Result Contract

The existing result order remains:

1. `selected`
2. `exported`
3. `failed`
4. `skipped`
5. `truncated`

`selected` is the effective post-modal target and must equal the displayed **Total documents to be prepared** for the submitted snapshot. It is not the number of manually checked index rows.

The primary result invariant is:

`selected = exported + failed + skipped`

`truncated` remains a subset of `exported`, so `truncated <= exported`. Each failed or skipped count is therefore bounded by `selected` and can never be inflated by documents deliberately excluded in the modal.

For example, 100 checked documents may expand to 120 with descendants and reduce to 10 after modal filters. The modal reports **Total documents to be prepared: 10** and sends those 10 ids. The expected result is `selected: 10`, `exported: 10`, `failed: 0`, and `skipped: 0`; the 110 deliberate exclusions remain pre-submit information only.

The service revalidates the ten submitted targets without broadening them. If one no longer exists or no longer meets an active filter, a possible result is `selected: 10`, `exported: 9`, and `skipped: 1`. Record-building errors contribute to `failed` instead. Trusted written-package metadata and returned-package validation must continue to describe the exact prepared record set rather than the earlier raw checkbox population.

## Proposed Select-All Contract

- Add **Select all** to the active manage-index selection controls alongside the selected count, **Clear**, and **Done**.
- Use one pure bulk transition owned by `docs-viewer-index-selection.js`; do not reconstruct selection by reading checkbox DOM state.
- Select every document in the canonical manage-index population, including non-viewable documents and rows inside collapsed branches.
- Replace the selected-id set with the eligible-id set and clear the Shift-range anchor. Disable **Select all** when every eligible document is already checked.
- Keep selection browser-only and preserve the existing lifecycle: same-scope reload prunes ineligible ids, while leaving the owning scope, management context, or `index-tree` clears selection.
- A document's public viewability is not selection eligibility. Changing `viewable` does not prune that document from the checked set.
- Do not make the index aware of summaries or Package Prepare profiles. The generic control selects eligible documents; the modal owns the missing-summary filter.
- Do not change Action targeting, drag/drop, active-document highlighting, link navigation, or other singular controls.

## Manage Viewability Contract

- Manage mode always includes every source document in the index. Remove the **Show non-viewable** scope control, its browser state, visibility filtering, event handling, and selection reconciliation branch.
- Retain the non-viewable icon as the index indication for a document whose own `viewable` value is false.
- Public Docs Viewer continues to exclude a non-viewable document and the descendants gated by a non-viewable ancestor.
- Remove the separate **Show** Action and control. **Edit metadata** becomes the only UI for changing the active document's own **non-viewable** checkbox.
- Do not migrate automatic ancestor or descendant changes into **Edit metadata**. Its save remains one-document metadata editing.
- Explain in the metadata modal or its durable documentation that a non-viewable ancestor also prevents a document appearing publicly. Clearing one document's own checkbox does not override its ancestors.
- Retire the current ancestor confirmation and optional descendant choice with the **Show** Action rather than retaining an invisible parallel workflow.
- Audit the dedicated single and bulk viewability clients, endpoints, planners, routes, capability/config entries, tests, and documentation. Remove each item with no remaining real consumer, without aliases. If the audit finds an active consumer, document it and stop for a decision before retaining or changing that contract.
- Preserve `viewable` handling in the ordinary metadata mutation and in public build/runtime projection. This cleanup removes duplicate controls and dead transport, not the canonical metadata field or its public gate.

`viewable: false` is a public-display gate, not an external-sharing prohibition. The package checkbox is an explicit composition choice; a future requirement that some documents must never leave the local workspace would require a separate policy rather than overloading `viewable`.

## Confirmed Legacy Cleanup

Document-level `published` is not part of the current Docs source contract. Remove it from `DocumentPackageSourceRecord`, source-context loading, export-document projection, the browser package feed, and synthetic fixtures/tests. Package selectability must not depend on it, and no source migration is required because the current corpus contains no instances.

This retirement is narrowly about document front matter and package records. Scope configuration, generated-document locations, public projections, publication commands, and other artifact paths that use the word `published` are separate active concepts and remain unchanged.

## Confirmed Package-Size Decision

Do not add or enforce a document-count or total-character package limit. The visible effective total and explicit confirmation are sufficient for the current workflow. Retain the existing 50,000-character per-document content truncation, and reconsider broader limits only if an observed scale or external-tool constraint creates a real need.
