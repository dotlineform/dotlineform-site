# Runtime Dependencies

## Authorities

- `requirements.txt` — pinned Python package baseline.
- feature-local `package.json` and `package-lock.json` pairs — direct and resolved Node dependency authority.
- `.codex/setup.sh` — cloud bootstrap and optional system-package installation.
- `.devcontainer/Dockerfile` / `devcontainer.json` — Codespaces image and environment.
- owning scripts — exact external commands a workflow invokes.

Read those files for current versions and exhaustive package lists. This page explains dependency classes and change method.

## Python Baseline

The pinned environment currently supports:

- spreadsheet/catalogue I/O;
- HTML parsing and sanitization for Docs Import;
- raster conversion for document/media import;
- Python Markdown rendering for docs/catalogue prose;
- pytest collection.

All packages in `requirements.txt` are installed for local/cloud parity even when a specific task uses only part of the stack. “Feature-specific use” is not the same as “optional install” once pinned in the baseline.

## Node Package Roots

Node dependencies are feature-local rather than collected into a repository-root package. Each package root owns its own manifest, lockfile, commands, and ignored `node_modules/` output.

The current package root is `docs-viewer/build/mermaid/`. It pins `@mermaid-js/mermaid-cli` for repository-side SVG generation; it is not a browser or public-route dependency. Install its locked tree with:

```bash
npm ci --prefix docs-viewer/build/mermaid
```

Do not install workflow executables globally or treat a globally available command as the dependency contract. `package.json` owns direct dependencies, `package-lock.json` owns the resolved graph, and the owning Python service invokes the package-root executable.

[Document Diagrams](/docs/?scope=studio&doc=d-20260719-123719-fb7565) owns Mermaid's producer registration, scope opt-in, and reader boundary.

## External Tools

- `ffmpeg` and `heif-convert` are workflow-specific media tools.
- `sips` is a macOS-only convenience used by some local image paths.
- Git/GitHub CLI and compiler/build packages support bootstrap/development rather than app runtime.

Docs/search/static validation does not require media tools. A media workflow must check its actual command dependencies and source roots before running.

## Runtime Boundaries

Public preview, Studio, Admin, Analytics, and Docs Viewer are separate Python-backed services/runners. Sharing one interpreter/dependency set does not merge their routes or ownership. `bin/local-all` supervises siblings; it is not a combined app server.

## Local Versus Cloud

- Local commands follow the configured Miniconda interpreter convention.
- Cloud/Codespaces bootstrap creates/reuses `.venv` and installs `requirements.txt`.
- The current Python bootstrap does not install feature-local Node package roots; an environment that renders Mermaid must run the package root's `npm ci` command explicitly.
- Codespaces includes media tools in the image; Codex setup installs them only when requested.
- Private/external source roots and publishing credentials remain environment/secret-store concerns, not dependency setup.

## Change Method

### Python Package

1. Prove the owning workflow needs it and an existing dependency/standard library is insufficient.
2. Pin it in `requirements.txt`.
3. Update bootstrap only if installation needs system support.
4. Run `pip check` plus the focused consumer test.
5. Update this page only if a dependency class or bootstrap expectation changed.

### System Tool

Document/check it in the owning workflow and add it to cloud/container setup only when that environment must run the workflow. Do not make all tasks depend on media tooling.

### Node Package

1. Give the workflow a focused package root rather than adding unrelated packages to another feature.
2. Pin direct dependencies exactly and update the adjacent lockfile through npm.
3. Keep `node_modules/` ignored and install reproducibly with `npm ci --prefix <package-root>`.
4. Register executable behaviour in code; package presence alone does not grant a scope or route capability.
5. Run the package-root installation check plus the focused consumer tests.
6. Update local/cloud bootstrap only when those environments must execute the workflow.

## Weak Spots

- One requirements file installs feature-specific packages everywhere and will grow without periodic review.
- Local Miniconda and cloud `.venv` may use different Python patch/system-library versions.
- Command-line dependencies are discovered from scripts rather than one machine-readable registry.
- Node dependencies have feature-local manifests, so setup must inventory those roots rather than assuming one repository-level install.
- A successful bootstrap does not prove private media/workspace availability or publish credentials.
