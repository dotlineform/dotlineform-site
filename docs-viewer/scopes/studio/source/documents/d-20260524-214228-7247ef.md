# Source Tree Ownership

## Repository Shape

The repository contains several local apps, shared domain services, and one tracked public deploy tree. A runner may supervise them together, but it does not merge their ownership.

| Root | Owns |
| --- | --- |
| `studio/` | catalogue authoring app, catalogue/media services, canonical catalogue data, generated Studio read models, Studio/public-catalogue tests |
| `admin-app/` | cross-repo checks, audits, activity, testing operations, reports, and their local app |
| `analytics-app/` | tag and Data Sharing browser/API workflows plus future analysis tools |
| `data-sharing/` | headless adapter registry, package contracts, prepare/review/import/export workflows |
| `docs-viewer/` | Docs Viewer source, runtime, service, config, local published payloads, import/export, and tests |
| `shared/` | app-neutral frontend modules and styles used by more than one local app |
| `site/` | tracked static deploy root: public shells, runtime assets, generated public data, and public thumbnails |
| `site-tools/` | public-tree validation and durable site-level tooling config |
| `var/` | ignored reports, logs, caches, staging, and other local working output |

Machine-owned projects, catalogue media staging, Docs Viewer external scopes, and Data Sharing packages live below `$DOTLINEFORM_PROJECTS_BASE_DIR`, outside the repository.

## Studio Boundary

| Studio path | Responsibility |
| --- | --- |
| `studio/app/frontend/` | browser shell, route templates/scripts, shared route helpers, checked browser config |
| `studio/app/server/` | loopback server, HTTP routing, runtime config projection, API adapters |
| `studio/app/assets/` | Studio-only CSS and static assets |
| `studio/services/catalogue/` | catalogue validation, mutation, lookup, build, publication, cleanup, import, and reports |
| `studio/services/media/` | derivative generation and remote media transport |
| `studio/data/canonical/` | authoritative catalogue metadata and optional catalogue Markdown |
| `studio/data/config/` | checked Studio/domain policy |
| `studio/data/generated/` | rebuildable Local Studio read models |
| `studio/shared/` | Python helpers shared by Studio-owned services and commands |
| `studio/tests/` | Studio and public-catalogue contracts |

Studio-generated public payloads still belong in `site/` because the public runtime owns their consumption. Generator location does not determine artifact ownership.

## App Boundaries

### Admin

Admin owns `/admin/...` and `/admin/api/...`, the checks/report producers under `admin-app/checks/`, commands such as `run_checks.py`, app tests, and ignored outputs under `var/admin/`.

Do not reintroduce retired Studio or standalone aliases for Admin routes.

### Analytics And Data Sharing

Analytics owns `/analytics/...` and `/analytics/api/...`, its frontend/server/config/tests, and tag-domain canonical data. `data-sharing/` remains a headless domain boundary used by Analytics rather than becoming browser code inside the app.

Do not restore old `/studio/analytics/...` or `/studio/data-sharing/...` compatibility paths.

### Docs Viewer

Docs Viewer owns `docs-viewer/source/`, runtime, static assets, shells, config, services, launcher, tests, and local published payloads under `docs-viewer/published/`.

Public read-only installs consume generated payloads under `site/assets/data/docs/`. Local Studio may link to the Docs Viewer service but does not own `/docs/`, its management APIs, or Docs Review.

## Shared Frontend Rule

`shared/frontend/` is for production UI behavior with more than one real app consumer.

Shared modules must not call an app-specific API or assume one route shell. Each app keeps transport, data adapters, commit behavior, and route orchestration with its owner.

Do not move code into `shared/` merely because it looks reusable. Reuse should be demonstrated by consumers with the same contract.

## Source, Generated, And Working Data

```text
checked config + canonical source
  -> owning app/service
  -> generated local or public read model
  -> runtime consumer

temporary inputs, logs, reports, and caches
  -> var/ or external workspace
```

- Canonical source is edited and validated.
- Generated output is rebuildable even when checked in.
- Public generated output lives under `site/`.
- App-local generated output lives with its app.
- Reports and run artifacts do not become documentation source.

## Compatibility Rule

Do not preserve old ownership through copied files, aliases, broad fallback reads, or static-serving shims.

In particular, do not restore former `site/assets/studio/`, `_docs_catalogue/`, root check/test folders, old Studio-owned Analytics/Admin routes, or alternate Docs Viewer source homes. A narrow wrapper is acceptable only when it delegates clearly to the current owner and does not create a second data or route contract.

## How To Place New Work

Ask in order:

1. Which app or headless domain owns the user-visible capability?
2. Is this canonical source, checked policy, generated output, or local working data?
3. Which runtime consumes it?
4. Does sharing represent a proven same contract or only hoped-for reuse?
5. Which tests and documentation owner will make the boundary discoverable?

The [Projection Contract](/docs/?scope=studio&doc=d-20260523-222005-1085b2) covers cross-boundary source-to-output checks. App overviews and code are the authority for exact current subdirectories.
