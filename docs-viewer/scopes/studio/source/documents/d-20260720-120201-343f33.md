# Diagram Detail View Delivery

## Status And Outcome

This is a **proposed sibling delivery** under Diagrams. It does not need a separate concept document.

A reader can use a small overlaid diagram-detail control to open a rendered diagram in a new browser tab for easier inspection and browser zoom. The same visible action works for both diagram forms while preserving their different storage contracts:

- a published SVG diagram opens its stable served SVG URL;
- a successfully rendered inline Mermaid diagram opens a temporary standalone SVG created from its in-memory SVG DOM.

The control is labelled **Open diagram in new tab** for assistive technology and uses an expand/open-in-new visual treatment. The diagram body itself remains available for ordinary selection and any diagram-owned links rather than becoming one large click target.

This delivery owns only the detail-view affordance. The shipped Mermaid producer remains responsible for canonical `.mmd` and published SVG, while [Inline Mermaid Rendering](/docs/?scope=studio&doc=d-20260720-102658-7de33d) remains responsible for turning a local Markdown fence into an in-memory SVG.

## Settled Placement And Visibility

The control is an HTML element outside the `<svg>` or persistent `<img>`. A shared HTML frame contains the existing diagram surface and one native new-tab link. The detail adapter may wrap or register the diagram, but it must not insert the control into Mermaid-generated SVG, include it in temporary standalone SVG serialization, or make the diagram body the trigger.

Representative ownership shape:

```html
<div class="docsViewer__diagramFrame">
  <div class="docsViewer__diagramViewport">
    <!-- existing persistent image or adapter-owned inline SVG host -->
  </div>
  <a class="docsViewer__diagramDetailControl"
     href="..."
     target="_blank"
     rel="noopener"
     aria-label="Open diagram in new tab">…</a>
</div>
```

The frame is the positioning container. The diagram viewport keeps its normal size and owns any horizontal scrolling. The compact control is absolutely positioned over the frame's top-right corner with a sufficient stacking level and its own quiet contrasting background. It remains pinned while a wide diagram scrolls underneath. Only the control's small box intercepts pointer input; the rest of the diagram remains selectable and any diagram-owned links remain operable.

The control is permanently present and never creates a toolbar row, reserved top gap, or layout shift. On hover-capable fine-pointer devices it is visible but muted at rest. It becomes fully unmuted when any of these is true:

- the diagram frame is hovered;
- the control itself is hovered;
- the control has keyboard focus, or the frame matches `:focus-within`.

The direct control-hover rule is explicit rather than relying only on frame hover. On touch or other non-hover input, the control remains fully unmuted because there is no discoverable hover state. Reveal styling changes opacity or color only; it must not toggle `display`, `visibility`, dimensions, positioning, padding, or document flow. A visible focus ring remains independent of the muted treatment, and reduced-motion preferences suppress any nonessential opacity transition.

This accepts the possibility that the compact control temporarily overlays a small amount of diagram content. That bounded overlap is preferable to permanently shrinking the diagram, leaving unexplained whitespace, or moving the document when interaction begins.

## Delivery Boundary

In scope:

- one shared rendered-diagram frame and absolutely overlaid detail-control contract;
- persistent SVG diagrams in ordinary rendered document content;
- inline Mermaid diagrams after their browser render succeeds;
- native link and keyboard behavior, a concise accessible label, focus visibility, stable overlay geometry, and responsive placement;
- stable served-URL navigation for persistent SVG;
- temporary Blob URL or equivalent standalone-SVG navigation for inline Mermaid;
- cleanup of temporary browser resources when document content is replaced;
- focused public, local, navigation-remount, accessibility, and failure checks.

Out of scope:

- an in-page lightbox, modal, pan-and-zoom canvas, custom zoom toolbar, or fullscreen viewer;
- making the whole diagram clickable or intercepting links inside a diagram;
- a layout-flow toolbar, reserved empty control gutter, or hover/focus behavior that shifts document content;
- rasterizing a diagram to WebP, PNG, or another image format;
- persisting inline Mermaid output, allocating media identities, or adding it to Scope Media inventory;
- changing Mermaid authoring, rendering, sanitization, publication, source opening, or export behavior;
- adding the control to arbitrary icons, decorative SVG, report/detail embeds, source mode, or exported static HTML without an explicit host decision.

## Delivery Checkpoints

### DDV-0 — Eligibility And Host Contract

- [ ] Decide the smallest explicit marker or payload/runtime contract that distinguishes supported document diagrams from ordinary images and decorative SVG.
- [ ] Confirm the same host shape can contain the current published-SVG `<img>` and the proposed inline Mermaid `<svg>` without changing either source contract.
- [ ] Keep the detail control as an HTML sibling outside the SVG/image surface so standalone SVG serialization never includes UI chrome.
- [ ] Confirm absolute top-right placement changes no diagram or document geometry and leaves only the compact control box above diagram content.
- [ ] Confirm persistent SVG detail viewing is appropriate on local, external-local, and public document routes.
- [ ] Keep reports, review/detail surfaces, embedded documents, source mode, and exports excluded unless their host context is explicitly added later.
- [ ] Stop if reliable persistent-diagram recognition would require a general media registry or broad Markdown-output refactor.

### DDV-1 — Persistent SVG Detail Control

- [ ] Decorate each eligible published SVG diagram after an ordinary document mount with one absolutely overlaid top-right **Open diagram in new tab** control.
- [ ] Open the current stable served SVG URL directly; preserve absolute and same-origin configured media locations without provider-specific branching.
- [ ] Use native keyboard-capable link behavior, visible focus, an accessible label, and safe new-tab relationship attributes.
- [ ] Keep the control visibly muted at rest for fine-pointer hover devices, fully unmute it on frame hover, its own hover, or focus, and keep it fully unmuted when hover is unavailable.
- [ ] Change only paint properties during interaction; never add/remove the control or change spacing, dimensions, positioning, or document flow.
- [ ] Avoid duplicate controls across navigation, repeated mounts, or management-mode projection.
- [ ] Verify the existing Configuration And Extension Points and Docs Import diagrams as proof fixtures.

### DDV-2 — Inline Mermaid Detail Target

- [ ] Depend on a successfully rendered inline Mermaid host rather than parsing the Markdown fence again.
- [ ] Serialize only the adapter-owned rendered SVG into a standalone `image/svg+xml` browser resource; do not accept arbitrary page SVG.
- [ ] Preserve the diagram's responsive `viewBox`, visible content, title/description accessibility metadata, and required standalone presentation.
- [ ] Open the temporary target from direct user activation and manage its lifetime without leaking one object URL per remount.
- [ ] Leave the rendered inline diagram usable when detail-target creation or opening fails.

DDV-2 may ship only after inline Mermaid rendering provides the shared host contract. DDV-1 can ship independently.

### DDV-3 — Integration And Closeout

- [ ] Prove multiple persistent and inline diagrams receive one control each and keep distinct targets.
- [ ] Prove public routes add no inline Mermaid runtime while published SVG detail links continue to work.
- [ ] Check narrow width, light/dark themes, rest/frame-hover/direct-control-hover/focus/touch treatments, zero layout shift, keyboard and screen-reader labelling, browser zoom, navigation remount, and failed-open behavior manually.
- [ ] Add focused module/route contracts without testing ordinary visual choreography.
- [ ] Update Document Diagrams and the Diagrams feature parent with the shipped contract; do not add export promises.

## Completion Evidence

- A published Mermaid-backed SVG opens its stable configured SVG URL in a new tab from the overlaid control.
- A local inline Mermaid diagram opens a visually equivalent temporary standalone SVG without writing media or source files.
- The two forms share one accessible control and diagram-host contract while retaining distinct target creation.
- Diagram-owned links remain operable because the diagram body is not the detail trigger.
- Rest, hover, direct control hover, focus, and touch states change no diagram or document geometry and leave no unexplained control gap.
- Navigation and repeated mounts create no duplicate controls or abandoned temporary URLs.
- Public and external-local routes do not acquire the inline Mermaid renderer through this feature.
