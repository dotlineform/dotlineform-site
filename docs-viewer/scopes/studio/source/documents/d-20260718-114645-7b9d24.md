# Docs Viewer Core Refactor

## Purpose

Extract and prove an explicit Docs Viewer core while keeping the refactor's planning, decisions, delivery state, working notes, and handoffs readable through the stable `main` Docs Viewer even when the feature worktree does not build or start.

This root-level parent is both the refactor feature router and its development control plane. The [Docs Viewer Roadmap](/docs/?scope=studio&doc=d-20260714-234030-1e0af6) owns its scheduling. [Standalone Docs Viewer](/docs/?scope=studio&doc=d-20260511-120352-90ffa6) is the subsequent product feature that consumes the extracted core, while its [concept](/docs/?scope=studio&doc=d-20260718-105957-c72b2a) currently describes the shared target composition.

## Feature Boundary

This feature owns the behavior-preserving extraction of the core, explicit integrated and standalone composition seams, and recomposition of the existing integrated application. Each bounded refactor slice must also assess the constraints it imposes on open Docs Viewer proposals; those proposals are design inputs as well as parallel work. This feature does not own packaging or releasing the standalone application; those belong to Standalone Docs Viewer after the core boundary is executable.

## Configuration And Runtime Ownership

The current configuration path is a concrete Core Refactor input rather than a permanent application contract. Scope registry parsing, external-workspace availability, process environment bootstrap, and operational filesystem access are coupled during import. `DOCS_SCOPE_CONFIGS` and `DOCUMENT_SOURCE_ROOTS` are mutable process globals refreshed by several services and tests, while a targeted repository-backed build can still resolve unrelated external-local scopes.

This feature may gather the related cleanup into one or more bounded deliveries when transition evidence supports the boundary:

- separate declarative scope-registry parsing from provider availability and operational access checks;
- make each composition root own its environment, logical roots, configured scope registry, and optional integrations explicitly;
- resolve and validate only the scopes or capabilities required by the active operation, while real external-scope operations continue to fail loudly when their workspace is unavailable;
- replace cross-module mutation of import-time scope maps with explicit registry or context ownership;
- remove test collection's dependency on a user's real external workspace and preserve isolated fixture roots as test infrastructure;
- keep targeted builders targeted while giving deliberate all-scope validation its own explicit command.

The run-owned workspace used by the Docs check profile is bounded validation hygiene. It prevents a recurring false failure but does not implement this architectural cleanup or weaken real external-workspace validation.

## Documentation Ownership

- Author the canonical refactor workspace documents in the `studio` scope on `main`.
- Read and manage them through the stable Docs Viewer; the feature worktree's Docs Viewer is never a prerequisite for coordination.
- Bring committed workspace updates from `main` into the feature branch through the normal main-to-feature integration rhythm.
- Do not maintain parallel authoritative copies of the same planning document on both branches.
- Return durable architectural conclusions to the feature concept or the eventual current-system owner; retire temporary notes at delivery closeout.

A local-external scope is appropriate only for private, machine-specific, or disposable material that should not be repository history. Any decision or handoff needed to continue the shared refactor belongs back in this main-owned workspace.
