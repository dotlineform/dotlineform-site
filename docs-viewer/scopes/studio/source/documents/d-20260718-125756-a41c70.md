# Docs Viewer Source Documentation And Linting Pilot Delivery

## Outcome

Establish and prove the smallest useful source-documentation and linting conventions needed for new or materially changed Docs Viewer core code.

This is a bounded pilot, not repository-wide backfill. It begins after Core Definition Baseline v1 identifies representative boundaries and completes before the long-lived Core Refactor worktree is created.

## Scope

- Select a small set of representative JavaScript and Python Docs Viewer boundaries from the accepted core definition.
- Inventory the active Docs Viewer source roots, delivery environments, generated/vendor exclusions, and current check owners before configuring tools.
- Agree practical JSDoc and Python docstring conventions from those real modules.
- Document contracts, intent, non-obvious data shapes, lifecycle, side effects, and failure behavior without commenting obvious code.
- Add Docs Viewer-owned, pinned Ruff and ESLint development tooling with explicit, reviewable configuration.
- Begin with correctness, unused-code, and other low-noise rules appropriate to the selected surfaces.
- Provide short project-local commands that lint the pilot area and can later target new or changed Core Refactor files.
- Resolve genuine findings inside the selected pilot surfaces and record any justified local suppressions.
- Update stable repository guidance only after the examples and commands have proved useful.

## Decisions

- Current code and canonical configuration remain exact behavior authority. Source comments explain and route; tests and schemas provide executable evidence; accepted architecture and delivery documents own intended change.
- The pilot toolchain is owned by Docs Viewer rather than a new repository-wide JavaScript project. It may target both `docs-viewer/runtime/js/` and site-owned public/shared Docs Viewer modules through explicit paths.
- The existing Mermaid renderer package remains isolated and does not acquire repository lint dependencies.
- `jsconfig.json`, TypeScript, and `// @ts-check` are not initial pilot requirements. The delivery includes one bounded evaluation and persists them only if the selected surface proves useful, low-noise checking.
- Normal completion remains compatible with the existing focused `run_checks.py` profiles. The pilot does not introduce one repository-wide `npm run check`.
- Java/Processing, unrelated applications, semantic retrieval infrastructure, generated reference pages, formatting, and broad automatic fixes remain separate outcomes.

## Planned Tooling

Required when the delivery becomes active:

- one exact Ruff version in a Python development-dependency owner;
- exact ESLint, `@eslint/js`, and `globals` versions in a private Docs Viewer-owned JavaScript tooling package and lockfile;
- focused commands that run from the repository checkout without relying on global Ruff or ESLint installations.

Conditional:

- pin TypeScript only if the selective JSDoc type-checking evaluation is retained;
- add no JSDoc coverage plugin by default; syntax or tag validation must prove useful before becoming another dependency.

The exact versions and file locations are selected at the tooling checkpoint from the supported Python, Node, and direct-browser module environments. Do not copy version, target, module-resolution, or global settings from background research without verifying them against the application.

## Representative Surfaces

The pilot must include enough variety to test the conventions without becoming a backfill project:

- one exported browser/module contract with a non-trivial object or callback shape;
- one Python service or domain boundary with meaningful inputs, outputs, or failure behavior;
- one builder, filesystem, configuration, or process boundary with side effects;
- one current example where documentation reveals that simplification or splitting is preferable to a longer comment.

Choose the exact modules after Core Definition Analysis so the pilot documents likely future boundaries rather than fossilizing accidental current structure.

## Adoption Rules

- Require the proven conventions for new or materially changed exported/public APIs in Core Refactor deliveries.
- Use types for types and documentation for meaning; do not repeat signatures in prose.
- Treat lint as focused feedback, not as authority for mechanical redesign.
- Review each failure or suppression in the context of the current contract.
- Keep behavioral edits separate from broad formatting where practical.
- Do not add compatibility code merely to satisfy a lint rule or documentation tool.
- Use canonical repository terminology rather than adding synonyms for agent retrieval.
- Update affected source documentation in the same change as a documented contract.
- Require each material documented claim to have a current code/config owner and executable evidence or an explicitly named evidence gap.

## Delivery State

**Current state:** proposed; no source-documentation, lint-tool installation, JavaScript project configuration, or Core Refactor implementation is active.

**Next checkpoint:** accept Core Definition Baseline v1, then complete SDL-P0 before installing tools or changing representative modules.

## Ordered Steps

### [ ] SDL-P0 — Bound The Pilot And Its Tool Owners

- Select the exact representative JavaScript and Python boundaries from the accepted Core Definition Baseline.
- Confirm the active browser-module roots, Python roots, workbench/test context, and generated, vendor, retired, and unrelated exclusions.
- Record the supported direct-browser JavaScript environment, Python interpreter target, and existing check-profile owner.
- Choose the smallest Docs Viewer-owned configuration, dependency, lockfile, and command locations without reusing the Mermaid renderer package or creating a repository-wide package boundary.

**Checkpoint:** the selected surfaces and tool owners are narrow, reproducible, and sufficient to exercise every representative-surface requirement before any dependency is installed.

### [ ] SDL-P1 — Prove The Source-Documentation Conventions

- Add concise module responsibility documentation where ownership is not clear from the path and name.
- Add JSDoc to the selected exported JavaScript contract, including one useful shared object or callback shape.
- Add Python type hints where needed, then add concise docstrings to the selected service/domain and side-effect boundaries.
- Verify purpose, inputs, outputs, mutation, failure behavior, lifecycle, invariants, ordering, privacy, and exclusions only where they materially apply.
- Review any function that needs a long explanation and classify whether a small same-outcome simplification or a separate follow-up is the correct response.

**Checkpoint:** reviewed examples improve real code re-entry without restating signatures, inventing contracts, or documenting accidental structure.

### [ ] SDL-P2 — Pin And Configure Focused Linting

- Install and pin Ruff through the chosen Python development-dependency owner.
- Create the private Docs Viewer JavaScript tooling package and lockfile; install and pin ESLint, `@eslint/js`, and `globals`.
- Configure only the selected Python surfaces and the real Docs Viewer browser-module contexts, with explicit generated, vendor, retired, and unrelated exclusions.
- Begin with correctness, undefined-name, unused-code, and similarly low-noise rules; do not enable docstring coverage, formatting, or broad automatic fixes.
- Add short direct commands and capture the initial findings in temporary delivery evidence rather than this durable document.

**Checkpoint:** both linters run reproducibly against only the agreed surfaces, and their initial findings are classified before source fixes begin.

### [ ] SDL-P3 — Evaluate Selective JSDoc Type Checking

- Begin without a repository or Docs Viewer `jsconfig.json`.
- Evaluate one stable, well-documented JavaScript module with file-level or equivalently bounded TypeScript/JSDoc checking.
- Compare useful contract errors with false positives, annotation overhead, browser-module compatibility, and maintenance cost.
- If the result is useful and low-noise, pin TypeScript and persist the smallest Docs Viewer-owned configuration and command limited to the proven surface.
- If the result is not useful, remove the experiment and explicitly defer `jsconfig.json`, TypeScript, and `checkJs` without blocking the lint/documentation pilot.

**Checkpoint:** the delivery has one evidence-backed documentation-only or selectively type-checked JavaScript convention; project-wide `checkJs` remains out of scope.

### [ ] SDL-P4 — Resolve Findings And Join Normal Checks

- Fix genuine lint or documentation defects only inside the selected surfaces.
- Keep suppressions rare, local, and justified; remove misleading unused imports or abandoned references only after exact-use verification.
- Add or adjust focused executable evidence only when a newly documented material rule lacks adequate current coverage.
- Make the direct lint commands able to target new or materially changed Core Refactor files.
- Add the proven focused lint commands to the smallest appropriate `run_checks.py` profile without replacing its existing Python, build, or browser evidence.

**Checkpoint:** the direct commands and owning check profile pass, source behavior remains unchanged except for separately reviewed genuine fixes, and no unrelated cleanup entered the diff.

### [ ] SDL-P5 — Stabilize Guidance And Close The Pilot

- Record only the proven JSDoc, docstring, type-hint, lint, suppression, and command rules in stable repository guidance.
- Keep background research labelled as non-authoritative and leave generated reference documentation for a later outcome.
- Confirm new or materially changed Core Refactor APIs have a clear adoption rule.
- Run the declared focused checks and `git diff --check`, report generated follow-through, and close only when every completion gate is true.

**Checkpoint:** the pilot leaves a small reproducible toolchain, useful reviewed examples, and a low-noise rule for subsequent Core Refactor deliveries.

## Out Of Scope

- Repository-wide JSDoc or docstring backfill
- Repository-wide zero-warning lint baselines
- A root JavaScript package, repository-wide `jsconfig.json`, or repository-wide `checkJs`
- Missing-documentation enforcement across historical code
- Broad reformatting
- Formatter adoption or broad automatic-fix batches
- Generated source-reference documentation
- Semantic retrieval infrastructure or agent-retrieval benchmarking
- Complexity-driven mechanical rewrites
- Processing or unrelated application adoption
- Changes to the isolated Mermaid renderer toolchain
- Expansion of rule sets beyond what the pilot proves

Those remain separate outcomes under the owning repository feature.

## Completion Gates

- The chosen JavaScript and Python conventions are demonstrated by useful reviewed examples.
- Ruff and ESLint dependencies, lockfiles, configurations, and direct commands are Docs Viewer-owned, pinned, and reproducible.
- Configured source roots and runtime contexts match the actual application and exclude generated, vendor, retired, and unrelated sources.
- Focused project-local lint commands pass on the representative surfaces.
- The selective JSDoc type-checking evaluation ends with an explicit retained or deferred decision; `jsconfig.json` is not required for completion.
- New and materially changed Core Refactor code has a clear, low-noise adoption rule.
- Material source-documentation claims agree with current code/config and executable evidence or name the remaining evidence gap.
- Genuine pilot findings are fixed or explicitly classified; suppressions are local and justified.
- The proven commands are available through the smallest appropriate existing check-profile owner.
- Stable repository guidance and check documentation reflect only the proven rules and commands.
- No repository-wide backfill, formatting churn, or unrelated cleanup has entered the delivery.
