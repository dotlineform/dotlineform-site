# Scope Media Management

## Purpose

Give Docs Viewer authors one truthful view of the media owned by a scope, including referenced, unreferenced, missing, ready, and build-source media, without creating a document per item or a second external-asset identity system.

Physical location does not determine ownership. Scope media may live beneath the local scope root, in an independently registered repository destination, or in R2 while retaining one logical scope/type/identity contract.

- [Concept](/docs/?scope=studio&doc=d-20260720-100017-e094d8) — product model, inventory surface options, UI sketches, lifecycle direction, storage flexibility, and unresolved decisions.

- [Docs Viewer Roadmap](/docs/?scope=studio&doc=d-20260714-234030-1e0af6) — suggested order and dependencies.

## Boundary

- Scope configuration remains the ownership and location authority.
- The existing provider-independent media inventory is the starting point.
- The first useful surface is local and read-only.
- Unreferenced is an inspection state, not permission to delete.
- Storage-provider choices remain hidden from ordinary authoring actions.
- Arbitrary Processing project source, user workspaces, and independently owned file trees are not Docs Viewer media merely because an inventory would be useful.

The feature is proposed. The first delivery deliberately begins with a report-host fit decision so the concept can be refined before a new management surface or lifecycle contract is built.
