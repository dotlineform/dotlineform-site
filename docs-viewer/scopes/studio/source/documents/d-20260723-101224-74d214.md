# Docs Viewer Manage Theme Restoration Delivery

## Status And Outcome

**Complete. DMT-P0 through DMT-P2 are shipped.**

Restore one complete light and dark theme contract across the integrated Docs Viewer manage route and every current overlay. Close this delivery before beginning Docs Viewer Modal Consistency.

## Current State

`site/docs-viewer/static/css/docs-viewer-theme.css` is now the public-safe raw palette owner for public, manage, review, and future standalone hosts. It defines the same namespaced semantic roles for light and dark themes. `docs-viewer.css` maps those roles to the existing component-facing viewer tokens; manage CSS no longer carries a duplicate palette or styles against management capability.

Public-site `main.css` still owns the surrounding site palette. Its coincident values are not a Docs Viewer adapter or a second viewer palette. Installed public routes load the shared viewer theme before the base viewer stylesheet, while manage and review receive that same site-owned asset through the explicit local-service route map. The shells no longer project the retired `data-allow-management` attribute.

Base viewer, report, manage, Import, and source-editor components now consume the shared semantic interaction and status roles. Theme-sensitive host fallback chains, component-local status palettes, component-level light/dark selectors, and literal modal overlays have been removed. Active, focus, disabled, busy, success, warning, danger, overlay, panel, field, menu, and list states therefore change through the shared palette rather than through workflow-specific rules. The diagram canvas intentionally remains a neutral white surface.

Live manage-route inspection after restarting the local service confirms the shared stylesheet loads with both theme blocks, the obsolete management attribute is absent, and the canvas, surface, text, muted, focus, selection, disabled, overlay, and status roles resolve in both themes. Body, sidebar, search, buttons, document, menus, Settings, Import, source editing, native fields, and native `color-scheme` values switch between the expected palettes. The checks did not submit a workflow, and the original light selection and rendered-document view were restored after inspection.

DMT-P2 retains focused module evidence for normalization, synchronized toggle state, storage projection, and blocked-storage fallback. The real manage route proves the shared roles reach computed viewer, panel, document, modal, field, selection, and action styles in both themes and survive reload. The public read-only route proves Library resolves the same namespaced palette without management controls or a management styling attribute.

Desktop browser review covered Import, Metadata, Settings, New scope, and Review package with pointer opening and visible keyboard focus. A `420 × 820` review covered the same persistent, dynamic, and package modal families. That review found and corrected the standard card's narrow-height calculation so a tall Metadata card now remains inside the viewport and scrolls internally. Public Library remains visually intact in dark mode. No workflow was submitted, and both reviewed routes were restored to light mode.

## Accepted Contract

### Theme Authority

- `html[data-theme="light"|"dark"]` and the existing `theme` storage key remain authoritative.
- The existing toggle changes the attribute, its accessible state, and persisted selection through one owner.
- One always-loaded, Docs Viewer-owned theme map supplies namespaced viewer tokens to both public and manage routes.
- Public site CSS retains ownership of the surrounding public page. Manage CSS retains management-only structure and workflow styling; public does not import it.
- The route configuration and capability response remain the only management-availability authority.
- Do not restore `data-allow-management` as a second runtime or styling contract.

### Token Model

- Define one semantic Docs Viewer token vocabulary for canvas, raised and subtle surfaces, primary and muted text, borders, links, focus, selection, overlay, disabled, busy, success, warning, and danger states.
- Components consume semantic tokens; they do not select light/dark values or introduce workflow-local theme palettes.
- Shared light and dark theme blocks assign values to the same semantic roles for public, manage, and a future standalone host.
- Do not maintain duplicate public and manage copies of the raw Docs Viewer palette. Host separation comes from namespaced tokens, selector scope, and stylesheet ownership.
- A deliberate host override maps or replaces the relevant namespaced token explicitly; it does not fork the complete palette.
- Palette values remain deliberately replaceable. A later adjustment from the current stark black-and-white dark palette must not require component or workflow changes.
- This delivery requires readable contrast and distinguishable states, not a final aesthetic palette.

### Surface Coverage

- Body, viewer root, top controls, index panel, document content, info panel, menus, buttons, fields, native controls, source editor, and status surfaces are readable in both themes.
- Current dynamic and persistent modal cards, backdrops, fields, lists, statuses, and actions consume the same viewer tokens.
- Import, Metadata, Settings, Review Sessions, confirmation, choice, notice, package, scope, media, and source workflows retain their behavior.
- Semantic success, warning, danger, disabled, busy, selected, focus, link, and visited states remain distinguishable without introducing a new palette system.
- Public Docs Viewer retains its site-owned page behavior while using the shared Docs Viewer palette inside the viewer.
