# Index Multiple Selection Architecture

## Architectural Outcome

Multiple selection should be one manage-only checked-document set consumed by action and drag/drop boundaries. It must remain independent from the route's singular selected doc and must not turn action definitions into a new view, workflow, or plugin system.

## Vocabulary

- **Selected doc** means the one document currently rendered by the route. Its row alone has the active highlight and `aria-current="page"` state.
- **Selection** or **selected docs** means the zero-or-more document ids represented by checked boxes while selection mode is active.
- **Checked** is the visible projection of selection membership. Checked rows do not acquire a second row highlight.

The checkbox selection is an unordered batch target set. It has no primary document. Range anchor, focus, context row, and displayed document are separate interaction state.

## Shipped Foundation

`docs-viewer/runtime/js/management/docs-viewer-action-definitions.js` already owns:

- code-owned action ids
- scope, active-document, and selection targets
- primary, all, and exactly-one selection policies
- pure target resolution and disabled reasons

Current runtime still supplies a one-document selection context, so this foundation preserves existing behavior. That current input is not the future checkbox model: controls for the displayed doc should resolve the active document, while batch consumers should resolve the checked set. A primary-selection policy must not become hidden state inside checkbox selection.

[Runtime Module Ownership](/docs/?scope=studio&doc=d-20260605-125108-1d1ef3) is the durable current-state authority.

## Long-Term Capability Direction

The drag/drop discussion exposed a Docs Viewer-wide concern larger than selection: an operation may be placed by the view registry, targeted by action definitions, enabled by route features, and authorized by live backend capabilities.

The long-term aim is a coherent model across toolbar, Actions-menu, context-menu, and selection consumers, with each decision still owned in the correct layer:

- the view registry owns known views, controls, placement, and eligibility
- action definitions own operation identity, target, and cardinality
- route features enable known frontend behavior
- live backend capabilities state real service authority
- focused controllers own workflow and state

The Control Surfaces feature now owns that horizontal work. This architecture owns only the checkbox-selection and group-move boundaries that consume it.

## Proposed Selection State

A focused manage-only owner should hold:

- selected ids
- range-anchor id
- toggle, visible-range, clear, exit, and prune behavior
- stable checkbox and selected-count projection in current visible tree order

`selectedDocument.selectedDocId` continues to identify the document rendered by the route. It is not repurposed as the selection store.

```text
checkbox interaction
  -> focused selection owner updates local state
  -> checkbox and count projection reflect selected ids
  -> batch action resolver receives selected ids
  -> a committed action invokes its existing workflow owner

document-link navigation
  -> selectedDocument changes the displayed doc
  -> one row receives the active-document highlight
  -> checkbox selection remains unchanged
```

The interaction controller recognizes checkbox toggle, checkbox Shift-range, Clear, and Done. It delegates all selection transitions. Link navigation, row focus, and context-menu opening do not mutate the selected ids.

The shared sidebar may accept a narrow post-render checkbox projection callback but does not own manage-only selection state. Checkbox `checked` values are controlled projections of the state owner; actions must never reconstruct selection by scraping the DOM.

## Assumed Runtime

This architecture assumes the index view has a registered control host and successful moves can project canonical server results into the local model and rendered tree.

Group movement consumes those existing contracts. This feature changes the move target from one dragged document to either one unchecked dragged document or the checked set; it does not own the general control-host or tree-projection architecture.

## Group Move Boundary

Group drag/drop is a later complete delivery:

1. normalize and de-duplicate the checkbox-selected ids
2. reduce checked descendants beneath checked ancestors to effective move roots
3. reject a destination inside any moved subtree
4. validate every requested change before writing
5. apply one mutation plan
6. rebuild docs and search once
7. return canonical moved records and apply one local model/tree projection while keeping the displayed document unchanged

The existing move path should become plural directly rather than retain parallel singular and plural contracts. A one-item selection continues through the plural path.

Browser and service normalization must express the same domain rule, but server validation remains authoritative for writes.

## Ownership Map

| responsibility | owner |
| --- | --- |
| action targeting and cardinality | current `docs-viewer-action-definitions.js` |
| selection state, checkbox projection, and selected count | proposed focused manage-only selection module |
| DOM event recognition | current management interactions owner |
| committed move model/tree projection | focused shared tree projection owner |
| effective move roots and destination checks | drag/drop domain helper |
| workflow, busy state, cache invalidation, and results | management action controller |
| transport | management client |
| validated source writes and coordinated rebuild | source mutation service |

## Extension Method

New consumers reference the existing action id and declare a complete target/cardinality rule. They keep their own normalization, confirmation, workflow, service, and result behavior.

Do not broaden an action to `all` merely because selection exists. Each multi-document consumer is a separate delivery when its complete behavior is known.

## Known Weak Spots

- Existing `primary` selection policies describe current one-document action resolution and must not leak into the checkbox model.
- Reading checkbox DOM state inside each consumer would create competing selection authorities.
- DOM-only movement without the matching index-model update would create two tree authorities.
- Routine full index refresh after a bounded move resets unrelated interaction context and makes checkbox preservation unnecessarily expensive.
- Parent-plus-descendant moves can flatten a subtree if effective roots are not normalized consistently.
- Duplicate targeting rules across toolbar, Actions-menu, and context-menu renderers would make identical actions disagree.

The feature parent owns its delivery boundaries. This proposed architecture is not an implementation checklist.
