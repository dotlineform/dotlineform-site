# Docs Viewer Manage Theme Restoration

## Purpose

Restore complete, readable light and dark presentation across the integrated Docs Viewer manage route before beginning [Docs Viewer Modal Consistency](/docs/?scope=studio&doc=d-20260723-091111-aaa2d6).

This is a regression repair and prerequisite, not a redesign or a cross-app modal project.

## Current State

Complete. Public, manage, and review routes load one public-safe Docs Viewer theme map, and the obsolete management-attribute styling dependency has been removed. Base viewer, report, manage, Import, and source-editor components consume the same semantic surface, interaction, overlay, disabled, and status roles without workflow-local palettes or component-level theme selectors.

Permanent module and route evidence covers normalization, synchronized toggle projection, reload persistence, computed manage/modal styles, and the public read-only boundary. Desktop and `420 × 820` browser review covers the current persistent, dynamic, and package modal families with pointer and keyboard focus. The retained Docs Viewer smoke profile passes `16/16`.

## Confirmed Cause

Before this repair, the manage shell correctly read and toggled `html[data-theme]`, but the viewer theme tokens did not:

- `docs-viewer-manage.css` defined its light and dark tokens only for `.docsViewer[data-allow-management="true"]`;
- the route-production shells no longer had a runtime owner for that attribute;
- management availability now comes from the route configuration and capability flow.

Dark mode therefore produced empty `--docs-viewer-theme-*` values while viewer and modal surfaces retained light colors.

The mismatch was introduced when the route-production migration replaced the server-rendered shell that projected management availability into the attribute with a static shell. The route/capability migration completed, but the theme selector retained the retired DOM dependency.

Public Docs Viewer retained its surrounding site-owned palette. It now consumes the same Docs Viewer-owned namespaced palette inside the viewer. Studio remained comparison evidence only; no Studio CSS or component implementation was copied.

## Contract

- `html[data-theme]` and the existing `theme` storage key remain the one user theme selection.
- One Docs Viewer-owned, namespaced light/dark palette supplies both public and manage viewer surfaces. Do not duplicate its raw color assignments to preserve host separation.
- Components consume semantic tokens for canvas, surfaces, text, borders, links, focus, selection, overlays, and status states rather than theme-specific color values.
- Light and dark palettes assign values to that stable token vocabulary. Either palette can be tuned later without changing component ownership or selectors.
- Public site CSS continues to own the surrounding page, header, footer, and other site components. Manage CSS continues to own management-only structure and workflow surfaces; public does not load manage CSS.
- A host may deliberately override a namespaced Docs Viewer token, but the boundary is an explicit token mapping rather than a copied palette.
- The same Docs Viewer palette can provide defaults to a future standalone package without importing public-site or manage-workflow styling.
- Styling does not use a stale DOM attribute as a parallel management-capability channel.
- Existing viewer, toolbar, index, document, info, menu, form, source-editor, Import, Settings, Metadata, and modal surfaces consume the same Docs Viewer theme tokens.
- Native controls receive the matching `color-scheme` without mixing it with light component colors.
- Public Docs Viewer keeps its current site-owned page behavior while its viewer consumes the shared Docs Viewer palette.
- Studio remains comparison evidence only; this repair does not copy Studio CSS or component code.
- Management endpoints, capabilities, workflows, payloads, writes, validation, and action placement do not change.

## Boundary

This repair may retire the obsolete `data-allow-management` styling dependency, establish the semantic token contract, and correct current theme-token consumers. It does not need to perfect the current dark palette, establish the UI Workbench, converge modal lifecycle ownership, consolidate controls, or broadly restyle modal structure.

Those concerns remain with [Docs Viewer Modal Consistency](/docs/?scope=studio&doc=d-20260723-091111-aaa2d6), which starts from the restored theme baseline.
