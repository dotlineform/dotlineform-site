# Scope-First Storage Cutover Working Notes

Temporary sibling for detailed delivery evidence that should not overload the durable feature or delivery. The clean pre-delivery snapshot below was captured before this note was created. Re-run the complete preflight immediately before SFL-4; that later snapshot is authoritative for mutation. Offer this note for retirement at delivery closeout after stable conclusions have reached their durable owners.

## SFL-1 Frozen Contracts

The cutover preserves document and sub-scope ids, parent relationships, Markdown media tokens, served media references, public document and search paths, R2 keys, payload schemas, management service routes, public routes, and exposure policy. Public projections and R2 destinations remain independently registered; they are not descendants of the local scope root.

Repository manage payloads deliberately change their direct browser data URLs from `/docs-viewer/published/...` to `/docs-viewer/scopes/<scope>/published/...`. The external-local Notes service URLs remain logical `/docs/...` endpoints. The semantic-reference target lookup is the other deliberate URL change: it moves from `/docs-viewer/published/semantic-references/target-lookup.json` to `/docs-viewer/data/generated/semantic-references/target-lookup.json` without changing its schema or behavior.

## Config And Resolver Contract

SFL-2 replaces `docs_scopes_v2` with one accepted `docs_scopes_v3` shape. Each scope has one direct artifact location named `scope_root`:

```json
{
  "scope_id": "studio",
  "scope_type": "local",
  "scope_root": {
    "provider": "repository",
    "path": "docs-viewer/scopes/studio"
  },
  "source": {
    "build_media": {}
  },
  "published": {
    "media": {}
  },
  "public_projection": null,
  "sub_scopes": []
}
```

The root provider is `repository` or `external_local`; no R2 scope root is introduced. The resolver owns these fixed descendants and consumers do not repeat them in config:

| Artifact | Root-relative location |
|---|---|
| source container | `source/` |
| canonical documents | `source/documents/` |
| source build media | configured relative paths beneath `source/`, currently `media/mermaid/` |
| sub-scope documents | `source/sub-scopes/<sub-scope>/documents/` |
| working documents | `published/documents/` |
| document references | `published/documents/references/` |
| working search | `published/search/index.json` |
| sub-scope working documents | `published/documents/sub-scopes/<sub-scope>/` |
| sub-scope working search | `published/search/sub-scopes/<sub-scope>/index.json` |
| locally held published media | `published/media/<type>/` |

`source.documents_path`, `source.sub_scopes_path`, `published.documents.location`, and `published.search.location` are removed. Build-media records keep only their source-relative path, producer, and output type. Media records remain per type: an omitted `location` means the fixed scope-local `published/media/<type>/` descendant, while a repository public projection or R2 media type keeps its explicit independent location. This avoids repeating the scope-root provider and path for local media. Sub-scope local paths are derived from the parent root; a sub-scope may retain only its identity, title, and independently registered public projection.

Validation rejects unsupported providers, absolute or escaping relative paths, source/published overlap, wrong fixed artifact classes, duplicate scope roots, unregistered build inputs, media/reference mismatches, recursive sub-scopes, and any second local canonical location. Production accepts only the new schema after cutover; no compatibility loader or old-path fallback is added.

## Structural Attribution

- `.DS_Store` is untracked filesystem metadata. It is neither migrated nor included in byte verification.
- Tracked `.gitkeep` markers for configured local published-media branches move with those branches.
- Empty source `sub-scopes/` markers for scopes without configured sub-scopes are obsolete under config-driven materialization and are removed during migration.
- `source/analysis/sub-scopes/tags/sub-scopes/.gitkeep` represents unsupported recursion and is removed during migration.
- Studio and Notes contain registered Mermaid inputs. Library, Analysis, and Processing also configure Mermaid build input even where the clean snapshot contains no authored input. Moments has no Mermaid build input.
- Processing's local files branch contains substantive ZIP, XLSX, and CSV bytes. Moving its Docs Viewer scope does not change the separate Processing project or its toolchain.
- Studio, Processing, and Notes direct-published local media must be preserved byte-for-byte before any registered producer is rerun.

## Migration Failure Boundary

SFL-4 begins with a clean worktree, stopped writers, and a fresh write-free inventory using the same exclusions. It refuses a missing source, occupied destination, symlink, collision, changed inventory, inaccessible external root, or changed public/R2 registration.

Repository-owned tracked content moves with Git-aware operations. Notes is copied into a temporary sibling staging container on the same external filesystem, mapped to the new artifact classes, and verified by relative path and bytes before one atomic rename to `scopes/notes`. The old Notes roots remain canonical until the config/code switch; staging is never readable as a second scope.

Before the switch, failure discards staging and leaves old roots canonical. After the switch but before removal, recovery is either a full config/code reversion to the verified old roots or a fix-forward at the new root; production never gains a dual reader. Reproducible documents, search, references, Mermaid output, and the semantic lookup are rebuilt only after consumers point at the new locations.

Old roots may be removed only after all of these gates pass:

- `docs_scopes_v3` validation and focused resolver, lifecycle, builder, media, route, and external-local tests;
- repository manage document/search reads at their new direct URLs and real Notes reads through its unchanged service URLs;
- local media and Mermaid verification, canonical source/media inventory comparison, and clean reproducible rebuilds;
- semantic picker behavior at the new registered data URL;
- unchanged public routes, projection inventories, served media references, and R2 registrations;
- regenerated manifest/runtime config with no retired production path literals;
- verified destination counts and hashes, no staging residue, and no compatibility path or duplicate canonical byte set.

Only then are the old repository role-first trees, the three old Notes role roots, and `docs-viewer/published/semantic-references/` removed. No top-level `docs-viewer/source/` or `docs-viewer/published/` remains after the complete cutover.
