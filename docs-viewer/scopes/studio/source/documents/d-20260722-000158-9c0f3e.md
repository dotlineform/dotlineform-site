# Index Multiple Selection Consumer Assessment

## Decision Boundary

The checkbox-selection foundation is shipped. No action consumes the checked set yet, and this assessment does not approve or implement one.

Each consumer needs a separate delivery because usefulness, hierarchy semantics, confirmation, atomicity, failure recovery, and browser projection differ. Existing singular and scope actions must never change target merely because one or more checkboxes are selected. An approved plural workflow receives an explicit action id and label such as **Export selected…**, **Move selected…**, or **Delete selected…**.

The selection owner supplies the raw checked document ids. Each consumer owns any derived interpretation:

- **Prepare package** may offer an include-descendants choice.
- **Move selected…** may reduce checked descendants beneath checked ancestors to effective move roots.
- **Export selected…** must decide exact documents, descendants, media, and output shape.
- **Delete selected…** must define parent-and-descendant overlap, preview, and atomicity.

There is no universal effective selection and no general batch-action framework.

## Consumer Assessment

| Candidate | Selection relationship | Usefulness | Blast radius | Current disposition |
| --- | --- | --- | --- | --- |
| **Prepare package** | The checked ids become the package input; a compact workflow owns profile, format, descendants, and confirmation. | High. It removes the duplicate document picker and uses an existing package request shape that already accepts explicit document ids. | Medium. Canonical documents are not mutated, but the browser composition and current route retirement need a complete cutover. | Strongest first candidate. |
| **Export selected…** | A new selection action exports an explicitly defined document set. The current scope-wide **Export** action remains unchanged. | Medium to high. Read-only output is a natural use of an assembled set. | Medium. Descendant, media, format, filename, and one-versus-many artifact semantics are not yet defined. | Early candidate after its product semantics are settled. |
| **Move selected…** | A canonical button or Actions-menu command owns the workflow; dragging a checked document is an optional alternate invocation of the same action. | High. It removes repeated reparenting and provides an accessible non-drag invocation. | High. Effective roots, destination validation, atomic source mutation, coordinated rebuild, local tree projection, and recovery must agree. | High-value candidate, but not a small extension of drag/drop. |
| **Delete selected…** | A new destructive selection action previews and confirms the complete requested change. The current singular **Delete** remains unchanged. | Medium. Useful for deliberate cleanup, but less frequent than preparation, export, or movement. | Very high. Parent/descendant overlap, deletion consequences, atomicity, rebuild, partial-failure prevention, and recovery all matter. | Defer until safer consumers have proven the action boundary. |
| **Import** | Ordinary Import creates documents in the active scope; checked existing documents are not plural input. | Low as a selection consumer because the meaning is unclear. | High if overloaded to replace or mutate selected documents, and incompatible with the ordinary create-only import contract. | Ruled out. A future **Import under selected parent…** would be a distinct exactly-one action with its own need and delivery. |
| **Returned packages** | None. It opens a scope-owned inbox and operates on one complete prepared package. | Useful as a scope workflow, not as a selection consumer. | Selection would violate the whole-package validation and apply contract. | Excluded. ## Current Actions That Stay Unchanged

- Singular document actions continue to resolve the displayed document or their explicit invocation document: **Delete**, **Move**, **Copy link**, **Copy subtree**, **Edit metadata**, **Show**, **Open**, **Open in VS Code**, **New child**, **New sibling**, and source-document actions.
- Scope actions remain scope actions: **Import**, current **Export**, **Publish**, **Rebuild**, **Settings**, **New**, show-non-viewable, and scope or sub-scope lifecycle actions.
- **Returned packages** remains scope navigation into the whole-package inbox.

A later delivery may add a distinct plural counterpart. It must not make one of these actions conditional on hidden checkbox state.

## Provisional Order

This is a value-to-risk ordering, not an approved roadmap:

1. **Prepare package** — strongest reuse of an existing explicit-id workflow with no canonical document mutation.
2. **Export selected…** — read-only, once its document/media/output semantics are explicit.
3. **Move selected…** — high daily value but a materially larger hierarchy and mutation delivery; button action is canonical and drag is an alternate gesture.
4. **Delete selected…** — last because it combines hierarchy semantics with destructive mutation.

Move may be chosen ahead of Export if its practical value justifies accepting the larger first delivery. Import and Returned packages do not enter this ordering.

## Delivery Gate For Any Approved Consumer

Before implementation, its delivery must define:

- explicit action id, label, placement, cardinality, and disabled reasons;
- whether it consumes raw checked ids or a named consumer-owned projection;
- exact descendant, ancestor, eligibility, and cross-scope rules;
- preview, confirmation, payload, service authority, and atomicity;
- success projection or reload, cache invalidation, and resulting checkbox state;
- failure reporting and authoritative recovery without partial success ambiguity;
- public, review, integrated-only, and portable-core boundaries;
- focused pure-state, service, route, action-target, and browser evidence.
