# Cloud Environments

## Authorities

- `.codex/setup.sh` — shared bootstrap entrypoint.
- `.devcontainer/` — Codespaces base image, packages, editor settings, and environment.
- `requirements.txt` — pinned Python packages.
- platform settings — agent network access, environment variables, and secrets.

Platform behavior can change independently of this repo; verify it in the current environment rather than preserving service assumptions here.

## Bootstrap Flow

```text
repo checkout
  -> optional apt/system packages
  -> create/reuse .venv
  -> install requirements.txt
  -> optionally persist venv PATH/SITE_PYTHON/PYTHON_BIN
  -> print environment diagnostics
```

Setup does not build docs/catalogue/search, validate the site, run profiles, or publish anything.

```bash
bash .codex/setup.sh
```

## Setup Flags

- `SETUP_SKIP_APT=1` — skip system-package installation.
- `FORCE_APT_PACKAGES=1` — install even when baseline commands appear present.
- `SETUP_INSTALL_MEDIA_PACKAGES=1` — require/install `ffmpeg` and libheif tools.
- `SETUP_PERSIST_AGENT_ENV=0` — do not append the venv environment block to `~/.bashrc`.

The script is the exact behavior authority; keep this list only because these flags are its operator interface.

## Codespaces

The devcontainer uses an Ubuntu base, installs Python/build/GitHub/media tooling, delegates post-create to `.codex/setup.sh`, and selects the repo `.venv` interpreter.

`devcontainer.json` owns current container environment values, including projects/media roots and srcset concurrency. They are development defaults, not portable production config.

## Runtime Modes

### Normal Docs/Code Work

Use baseline setup, then `.venv/bin/python` for checks.

### Media Work

Enable/install media packages and verify the required external source root exists. Dependencies cannot create private media.

### Publish-Sensitive Work

Bootstrap first, then run explicit validation such as `bin/site-validate` and the focused check profile. Setup success is not deploy evidence.

## Environment And Secrets

- `DOTLINEFORM_PROJECTS_BASE_DIR` is required for external Data Sharing/Docs scope/media workflows that use it.
- Other workspace/concurrency defaults are owned by current runner/container config.
- R2 credentials belong in the platform secret store and must never be committed or written into docs output/logs.
- `.env.local` is local-machine configuration; cloud sessions use platform environment/secrets.

Use the minimum network access required. Dependency installation and agent web access are separate platform concerns.

## Verification

```bash
.venv/bin/python -m pip check
.venv/bin/python admin-app/commands/run_checks.py --list
.venv/bin/python admin-app/commands/run_checks.py --profile quick
SITE_PYTHON="$PWD/.venv/bin/python" bin/site-validate
```

Use dry-run generators by default unless the task explicitly owns writes.

## Weak Spots

- Cached environments can retain old packages/setup behavior; reset cache after material bootstrap changes.
- Cloud/local system libraries and Python patch versions can differ.
- Persisting an absolute repo venv path into `.bashrc` assumes the checkout path remains stable.
- Container environment defaults can look like application config even though they exist only for that workspace.
- Broad agent network access increases prompt-injection/exfiltration risk and should not be enabled by habit.
