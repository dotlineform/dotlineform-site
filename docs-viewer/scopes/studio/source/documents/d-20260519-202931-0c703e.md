# Catalogue Build And Lookup Refresh

## Two Different Derived Outputs

Catalogue source writes can affect two families of derived data:

- **Studio lookups** make search, focused editors, and related-record summaries fast.
- **Public builds** produce site JSON, pages, search entries, thumbnails, and staged primary media.

They share canonical source but have different planners and failure boundaries. A successful source save can therefore report lookup refresh, public build, and media outcomes separately.

## Studio Lookup Refresh

Generated lookup files live under `studio/data/generated/catalogue-lookup/`. Browser-safe paths are registered in `studio/app/frontend/config/studio-config.json`; focused Work and detail reads are service projections rather than generated per-record files.

After a canonical write:

1. the catalogue field registry determines whether changed fields affect `studio-lookup`;
2. serializer dependencies in `studio/services/catalogue/catalogue_lookup.py` identify the affected payloads;
3. `catalogue_lookup_refresh.py` writes a single record, a targeted related set, or the full lookup family;
4. the mutation response reports the chosen refresh mode.

A Work change can affect Work search, its focused projection, Series member summaries, and detail context. A Series change can affect Series search, its focused projection, and related Work summaries. That cross-record embedding is why the planner cannot assume one source record means one generated file.

Bulk, import, create, delete, and mixed dependency changes may use a full refresh when a narrower dependency set is not proven. Full refresh is a safe fallback, not the desired default for every mutation.

## Scoped Public Build

`POST /studio/api/catalogue/build-preview` plans without writing. `build-apply` executes the same validated scope.

The planner can select:

- focused Work or Series public JSON and pages;
- affected aggregate indexes and catalogue search;
- local Work and detail media derivatives;
- media-only regeneration when no source or public metadata write is required.

`studio/services/catalogue/catalogue_build_scopes.py` owns record scope. `catalogue_build_field_plan.py` maps changed fields to artifact families. `catalogue_build_media.py` owns source-image resolution and local derivative work. `catalogue_build_commands.py` owns generator/search command shapes. `catalogue_json_build.py` is the CLI and orchestration boundary.

Draft Work and Series source saves do not create public output. Publication requirements are checked before a public scope can run.

## Media Boundary

Local build media is not remote publication:

- source images resolve below the configured projects root;
- primary and thumbnail derivatives are generated below `$DOTLINEFORM_PROJECTS_BASE_DIR/catalogue/media/`;
- public thumbnails are copied into repo-owned asset folders;
- primary variants remain staged for the separate R2 publishing workflow;
- scoped build does not advance a confirmed media version.

The Work editor may compose source save, public build, and remote media publication into one user action, but each result remains independently visible and retryable.

## Extension And Weak Spots

- Add lookup dependency knowledge beside the serializer that consumes the field, not in another hand-maintained inventory.
- Add public artifact impact through the catalogue field registry and build planner.
- Keep targeted writes targeted; only global validation, import, or genuinely cross-record changes should normalize the whole corpus by default.
- The distinction between generated list files and service-only focused projections is easy to blur. `studio-config.json`, `studio_catalogue_api.py`, and `catalogue_lookup.py` together define the current read boundary.
- The wider catalogue CLI still supports Moment generation and thumbnail maintenance, but there is no active Moment editor route or Moment mutation API in Local Studio. Do not infer an app capability from a generator option.

Tests for lookup planning, field-registry verification, build scopes, and media plans are the exact behavioral authority.
