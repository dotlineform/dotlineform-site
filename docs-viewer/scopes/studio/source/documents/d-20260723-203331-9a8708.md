# Improving Repository Context for Coding Agents

> Research doc: not indicative of current implementation.

## Purpose

This guide explains how coding agents use repository context, the difference between grep-based and semantic retrieval, and how documentation, type information, linting, and tests can make a codebase easier for agents and humans to understand.

The proposed additions are:

- JSDoc for JavaScript
- Python docstrings
- Python type hints
- Ruff for Python linting
- ESLint for JavaScript linting

These additions should improve context evaluation, but they do so in different ways.

---

## 1. What `grep` is

`grep` is a command-line text-search tool. It searches files for literal text or patterns and returns matching lines.

A recursive search might look like this:

```bash
grep -R "make_work_images" .
```

This searches the current directory and its subdirectories for the exact text `make_work_images`.

Modern development tools often use `ripgrep`, invoked with `rg`, because it is fast and works well with source-code repositories:

```bash
rg "media_prefix"
rg "thumb-480"
rg "function.*thumbnail" --glob "*.js"
```

Typical useful forms include:

```bash
# Search the whole repository
rg "series_ids"

# Search selected folders
rg "work_id" scripts assets

# Restrict the search to JavaScript
rg "thumbnail" --glob "*.js"

# Search for several alternatives
rg "thumbnail|thumb|image.*size|srcset"

# List files containing a match
rg -l "media_base"

# Show surrounding lines
rg -n -C 3 "has_details"
```

`grep` and `ripgrep` do not understand the meaning of the code. They find matching text.

---

## 2. Grep-based context in coding agents

A coding agent cannot normally place an entire large repository into its model context at once. It must select relevant material.

With grep-based retrieval, the agent builds context by repeatedly:

1. interpreting the request;
2. generating likely search terms;
3. searching the repository;
4. opening matching files;
5. following imports, callers, configuration, tests, and related symbols;
6. refining its searches.

The process can be represented as:

```text
Natural-language request
        ↓
Model infers likely concepts and vocabulary
        ↓
Exact repository searches
        ↓
Relevant files and code sections
        ↓
More precise searches
        ↓
Working context for the task
```

For example, a request such as:

> Where is the work thumbnail path generated?

might lead to:

```bash
rg "thumb-480"
rg "media_prefix"
rg "thumbnail"
rg "srcset"
```

Once the agent discovers the actual identifier `media_base`, it can perform a more precise search:

```bash
rg "media_base"
```

This is an important distinction:

- the **model** can reason semantically about the prompt;
- the **retrieval operation** can still be an exact text search.

A repository does not necessarily need a separate embeddings index for an agent to handle vague language. The model can translate vague language into several candidate searches, inspect the results, learn the repository vocabulary, and search again.

---

## 3. Why grep-based retrieval works well for code

Source code contains many exact identifiers:

- function names;
- class names;
- module names;
- filenames;
- import paths;
- route paths;
- configuration keys;
- JSON properties;
- command names;
- error messages;
- test names.

For example:

```bash
rg "series_ids"
```

could locate:

- the schema definition;
- data-generation code;
- JSON output;
- validation;
- UI consumers;
- tests;
- migration logic.

Exact search is especially strong when the task is to:

- find every use of an identifier;
- locate the source of a specific error message;
- identify callers of a function;
- trace a configuration value;
- confirm whether an old code path still exists;
- find all files that depend on a field or route.

It is fast, deterministic, inexpensive, and easy to verify.

---

## 4. Limitations of grep-based retrieval

Exact search works only when the search vocabulary overlaps with the repository vocabulary.

Suppose the request is:

> Where does the site decide whether a work has extra metadata?

The implementation might use:

```json
{
  "has_details": true,
  "details_count": 4
}
```

A search for `extra metadata` may return nothing because those words never appear in the code.

The agent must infer better search terms:

```bash
rg "details|metadata|has_.*detail|count"
```

This is often successful, but it becomes harder when:

- names are obscure;
- the prompt is conceptual;
- documentation and implementation use different language;
- similar behaviour is distributed across multiple modules;
- there are many abandoned or duplicate implementations;
- domain rules are implied rather than stated.

---

## 5. Semantic retrieval

Semantic retrieval searches by meaning rather than exact wording.

A semantic query such as:

> logic controlling whether a work has additional information

might retrieve code involving:

- `has_details`;
- `details_count`;
- metadata completeness;
- work-detail rendering;
- conditional display rules.

This can happen even when none of the query wording appears literally in the source.

Semantic retrieval systems commonly:

1. split files into chunks;
2. convert each chunk into a numerical representation called an embedding;
3. convert the query into the same representation;
4. retrieve chunks that appear conceptually similar.

Semantic retrieval is useful for discovery when the user does not know the repository vocabulary.

It is less reliable for proving completeness. A semantic search may identify several highly relevant references but still miss an exact caller or an unusual use of a symbol.

---

## 6. The best approach is hybrid

Grep-based and semantic retrieval solve different problems.

| Task | Exact search | Semantic retrieval |
|---|---:|---:|
| Find every use of `media_prefix` | Excellent | Unnecessary |
| Locate a specific error message | Excellent | Usually unnecessary |
| Find logic related to image fallback behaviour | Requires guessed terms | Useful |
| Discover code when the vocabulary is unknown | Limited | Strong |
| Confirm all callers of an exact function | Strong | May be incomplete |
| Find conceptually similar code with different names | Weak | Stronger |
| Verify a configuration key has been removed | Excellent | Weak |
| Locate architectural intent | Depends on documentation wording | Useful |

A strong agent workflow is:

```text
Semantic interpretation or semantic retrieval
        ↓
Likely modules, concepts, and identifiers
        ↓
Exact grep/ripgrep searches
        ↓
Definitions, callers, imports, tests, and configuration
        ↓
Verified implementation context
```

Semantic techniques are strongest for **discovery**.

Exact search is strongest for **verification and completeness**.

## 7. How repository documentation improves context

Documentation does not automatically create a semantic-search system. It does, however, add useful semantic clues and searchable vocabulary.

Compare an undocumented function:

```js
export function makeUrl(id, size, config) {
  return `${config.base}/${id}-${size}.webp`;
}
```

with a documented function:

```js
/**
 * Builds the URL for a generated work-image derivative.
 *
 * The function selects only the URL. It does not create the derivative.
 * The returned URL may be site-relative or use the remote media host,
 * depending on the supplied media configuration.
 *
 * @param {string} workId - Canonical work identifier.
 * @param {800 | 1200 | 1600 | 2400} width - Requested derivative width.
 * @param {MediaConfig} config - Local or remote media configuration.
 * @returns {string} Absolute or site-relative image URL.
 */
export function buildWorkImageUrl(workId, width, config) {
  return `${config.base}/${workId}-primary-${width}.webp`;
}
```

The second version adds searchable concepts such as:

- generated derivative;
- work image;
- local or remote media;
- URL selection;
- canonical work identifier.

A vague request such as:

> Find the runtime code that chooses remote primary images.

now has much more textual overlap with the implementation.

## 8. JSDoc for JavaScript

JSDoc adds descriptions, parameter types, return types, object structures, and domain rules directly beside JavaScript code.

Official JSDoc documentation describes it as a way to document JavaScript APIs through comments attached to modules, classes, functions, parameters, and other symbols.

### 8.1 Function contract

```js
/**
 * Returns the ordered work IDs for a series.
 *
 * The order is editorial and must not be replaced with lexical work-ID order.
 *
 * @param {string} seriesId - Canonical series identifier.
 * @param {SeriesIndex} index - Loaded series index.
 * @returns {string[]} Ordered work identifiers.
 * @throws {Error} If the series does not exist in the index.
 */
export function getSeriesWorkIds(seriesId, index) {
  const series = index.series[seriesId];

  if (!series) {
    throw new Error(`Unknown series: ${seriesId}`);
  }

  return series.work_ids;
}
```

This explains a non-obvious rule: ordering is editorial rather than lexical.

### 8.2 Object structure

```js
/**
 * @typedef {object} MediaConfig
 * @property {string} mediaBase
 *   Empty for local assets, or an absolute remote-media origin.
 * @property {string} mediaPrefix
 *   Path prefix below the origin.
 * @property {boolean} useRemotePrimaries
 *   Whether primary derivatives are loaded from remote storage.
 */
```

The type can then be reused:

```js
/**
 * @param {MediaConfig} config
 * @returns {string}
 */
export function getMediaRoot(config) {
  return `${config.mediaBase}${config.mediaPrefix}`;
}
```

### 8.3 Constrained values

```js
/**
 * @typedef {0 | 1 | 2 | 3 | 4 | 5} DimensionScore
 */

/**
 * @typedef {"unipolar" | "bipolar" | "axis2d" | "vector"} DimensionShape
 */
```

This makes the permitted values explicit.

### 8.4 Module-level description

```js
/**
 * Work image URL and derivative-selection utilities.
 *
 * This module maps canonical work IDs and derivative widths to asset URLs.
 * It does not generate, resize, upload, or validate image files.
 *
 * @module work-image-urls
 */
```

A concise module comment can prevent an agent from confusing runtime URL selection with the offline image-generation pipeline.

### 8.5 When JSDoc is most useful

Use it primarily for:

- exported functions;
- public classes and methods;
- complex object shapes;
- domain-specific values;
- non-obvious return values;
- side effects;
- exceptional cases;
- important invariants;
- boundaries between subsystems.

Avoid comments that merely restate obvious syntax:

```js
/**
 * Adds two numbers.
 */
function add(a, b) {
  return a + b;
}
```

Excessive low-value comments increase retrieval noise.

## 9. Python docstrings and type hints

Python type hints provide machine-readable structural information. Docstrings add purpose, domain meaning, side effects, invariants, and exceptions.

Python’s type-hint syntax is designed to support static analysis, editor assistance, and refactoring. Python’s docstring conventions describe the high-level structure of documentation attached to public modules, classes, functions, and methods.

### 9.1 Typed function with a domain-focused docstring

```python
from collections.abc import Mapping, Sequence


def build_work_record(
    row: Mapping[str, object],
    series_ids: Sequence[str],
) -> dict[str, object]:
    """Convert a validated source row into a canonical work record.

    Null values are retained because null means that a field has not been
    supplied. Private notes must never be included in public JSON output.

    Args:
        row: Validated source fields for one work.
        series_ids: Canonical series identifiers in editorial order.

    Returns:
        A canonical record suitable for the per-work JSON output.

    Raises:
        ValueError: If the work identifier is missing or invalid.
    """
    work_id = str(row.get("work_id", "")).strip()

    if not work_id:
        raise ValueError("work_id is required")

    return {
        "work_id": work_id,
        "series_ids": list(series_ids),
        "title": row.get("title"),
        "year": row.get("year"),
    }
```

The annotations describe the data flow. The docstring records the rules that matter.

### 9.2 Typed domain model

```python
from dataclasses import dataclass
from typing import Literal, TypeAlias

DimensionScore: TypeAlias = Literal[0, 1, 2, 3, 4, 5]


@dataclass(frozen=True)
class WorkSummary:
    """Lightweight work metadata used in the generated search index."""

    work_id: str
    title: str | None
    year: int | None
    series_ids: tuple[str, ...]
    details_count: int
    has_details: bool
```

This gives the agent a compact statement of the intended data model.

### 9.3 Explicit semantic distinction

```python
def include_score_in_statistics(score: int | None) -> bool:
    """Return whether a dimension score contributes to statistics.

    ``None`` means unscored and is excluded. Zero means explicitly absent
    and is included.
    """
    return score is not None
```

The null-versus-zero distinction is exactly the kind of rule an agent may otherwise misunderstand.

### 9.4 Module-level docstring

```python
"""Generate public work JSON from canonical source records.

This module owns validation, canonical field ordering, checksums, and public
output filtering. It does not generate image derivatives or upload media.
"""
```

This makes responsibility boundaries searchable and explicit.

### 9.5 Where docstrings add the most value

Use docstrings for:

- public modules;
- public functions and classes;
- pipeline stages;
- data transformations;
- non-obvious validation;
- privacy rules;
- canonical-versus-derived distinctions;
- ordering rules;
- null semantics;
- side effects;
- filesystem or network operations.

Private helpers with obvious behaviour often need only a descriptive name and accurate types.

## 10. How type information helps agent context

Type information reduces the number of interpretations an agent must consider.

Without types:

```python
def generate(value, config):
    ...
```

The agent must infer:

- what `value` is;
- what `config` contains;
- what the function returns;
- whether it mutates anything;
- whether it can return `None`.

With types:

```python
def generate_series_index(
    series: list[SeriesRecord],
    output_dir: Path,
    *,
    dry_run: bool = False,
) -> GenerationReport:
    ...
```

The likely role of the function is much clearer before the body is even read.

Types are also strong grep targets:

```bash
rg "GenerationReport"
rg "SeriesRecord"
rg "MediaConfig"
```

They help agents follow data through the repository.

## 11. Ruff for Python linting

Ruff improves context indirectly by making Python code more consistent and reducing misleading noise.

It can help identify:

- unused imports;
- undefined names;
- accidental shadowing;
- unreachable or suspicious code;
- inconsistent import organisation;
- obsolete helpers;
- unsafe exception patterns;
- style inconsistencies;
- documentation issues, when selected.

Ruff can be configured in `pyproject.toml`, `ruff.toml`, or `.ruff.toml`.

### 11.1 Conservative starting configuration

```toml
[tool.ruff]
target-version = "py312"
line-length = 100
extend-exclude = [
  ".venv",
  "_site",
  "vendor",
]

[tool.ruff.lint]
select = [
  "E",    # pycodestyle errors
  "F",    # Pyflakes
  "I",    # import sorting
  "UP",   # Python syntax upgrades
  "B",    # likely bugs and design problems
  "SIM",  # simplifications
]
ignore = [
  "E501", # allow the formatter/editor to manage line wrapping initially
]

[tool.ruff.format]
quote-style = "double"
indent-style = "space"
docstring-code-format = true
```

This is intentionally a baseline rather than an aggressively strict configuration.

### 11.2 Commands

```bash
# Check the repository
ruff check .

# Apply safe automatic fixes
ruff check . --fix

# Format Python
ruff format .

# Verify formatting without modifying files
ruff format . --check
```

### 11.3 Optional docstring enforcement later

Once existing public APIs are documented consistently, docstring rules can be introduced selectively:

```toml
[tool.ruff.lint]
extend-select = [
  "D",
]

[tool.ruff.lint.pydocstyle]
convention = "google"
```

Do not enable strict docstring enforcement before deciding:

- which modules are public;
- which functions require documentation;
- which docstring convention the project will use;
- which generated or legacy folders should be excluded.

Otherwise, the initial lint migration can become a large mechanical exercise with little semantic benefit.

## 12. ESLint for JavaScript linting

ESLint also improves context indirectly. It reduces ambiguity, unused code, accidental globals, and inconsistent patterns.

Modern ESLint projects can use a flat configuration file such as:

```text
eslint.config.js
```

### 12.1 Conservative starting configuration

```js
import js from "@eslint/js";

export default [
  {
    ignores: [
      "_site/**",
      "vendor/**",
      "node_modules/**",
      "assets/generated/**",
    ],
  },

  js.configs.recommended,

  {
    files: ["**/*.js"],

    languageOptions: {
      ecmaVersion: "latest",
      sourceType: "module",
      globals: {
        window: "readonly",
        document: "readonly",
        fetch: "readonly",
        URL: "readonly",
        console: "readonly",
      },
    },

    rules: {
      "no-unused-vars": [
        "error",
        {
          argsIgnorePattern: "^_",
          varsIgnorePattern: "^_",
        },
      ],
      "no-constant-condition": [
        "error",
        {
          checkLoops: false,
        },
      ],
      "prefer-const": "error",
      "eqeqeq": ["error", "always"],
    },
  },
];
```

The exact global list should reflect the runtime environments in the repository. Browser scripts, Node scripts, service workers, and tests may require separate configuration blocks.

### 12.2 Commands

```bash
# Check JavaScript
npx eslint .

# Apply available fixes
npx eslint . --fix
```

### 12.3 Separate browser and Node contexts

```js
import js from "@eslint/js";
import globals from "globals";

export default [
  js.configs.recommended,

  {
    files: ["assets/**/*.js"],
    languageOptions: {
      sourceType: "module",
      globals: globals.browser,
    },
  },

  {
    files: ["scripts/**/*.js", "*.config.js"],
    languageOptions: {
      sourceType: "module",
      globals: globals.node,
    },
  },
];
```

Separating runtime contexts helps prevent an agent from assuming that browser-only globals are available in build scripts, or that Node APIs are available in public site code.

## 13. What linting contributes to context quality

Linting does not explain the domain by itself.

Its value is mainly that it reduces false leads.

For example, an unused import can make an agent believe that a module still participates in a workflow:

```js
import { buildLegacyIndex } from "./legacy-index.js";
```

If the import is unused, ESLint can expose it. Removing it makes the actual architecture clearer.

Likewise, an abandoned Python helper may look authoritative even though it is never called. Ruff can help identify surrounding unused imports and inconsistent code, while exact reference searches can confirm whether the helper is active:

```bash
rg "build_legacy_index"
```

Linting improves the repository’s signal-to-noise ratio.

## 14. Tests are also semantic context

Descriptively named tests can provide some of the strongest context in a repository because they express executable rules.

```python
def test_private_notes_are_excluded_from_public_work_json() -> None:
    ...
```

```python
def test_null_dimension_score_is_excluded_from_statistics() -> None:
    ...
```

```js
test("series navigation loops from the final work to the first", () => {
  // ...
});
```

```js
test("direct work URLs do not show series navigation", () => {
  // ...
});
```

These names are:

- searchable;
- precise;
- tied to behaviour;
- less likely than comments to drift unnoticed.

A useful hierarchy of evidence is:

```text
Passing tests and schemas
        ↓
Types and executable validation
        ↓
Current implementation
        ↓
Docstrings and JSDoc
        ↓
Architecture documents
        ↓
Historical comments and notes
```

This does not mean comments are unimportant. It means agents should verify important claims against executable evidence where possible.

## 15. Architecture and schema documents

Small, focused Markdown documents can provide broader context that code comments cannot.

Useful documents might include:

```text
docs/
├── architecture/
│   ├── data-flow.md
│   ├── media-pipeline.md
│   ├── search-pipeline.md
│   └── repository-boundaries.md
├── schemas/
│   ├── works-json.md
│   ├── series-json.md
│   └── dimensions.md
└── development/
    ├── testing.md
    ├── linting.md
    └── generated-files.md
```

A short boundary statement can prevent major misunderstandings:

```markdown
## Image responsibilities

- `processing/` creates source artwork and offline visual assets.
- `scripts/images/` creates image derivatives.
- `assets/js/media/` selects derivative URLs at runtime.
- R2 stores primary derivatives and attachments.
- Repository-local thumbnails remain part of the static site build.
```

This gives a vague prompt several searchable routes into the implementation.

## 16. Documentation principles for agent-friendly code

### 16.1 Document intent, not syntax

Weak:

```js
/**
 * Set hasDetails to true if detailsCount is greater than zero.
 */
const hasDetails = detailsCount > 0;
```

Better:

```js
/**
 * Whether the lightweight index should link to a detailed work record.
 *
 * A value of zero means that no optional detail fields survived public
 * filtering.
 */
const hasDetails = detailsCount > 0;
```

### 16.2 Record boundaries

```python
"""Read canonical work records and write generated search indexes.

This package must not modify source records.
"""
```

### 16.3 Record invariants

```js
/**
 * Work IDs are stable strings. Never parse them as numbers because leading
 * zeroes and future non-numeric prefixes are permitted.
 */
```

### 16.4 Record exclusions

```python
"""Return a public work record.

The output excludes ``notes_private`` and internal storage metadata.
"""
```

### 16.5 Explain ordering

```js
/**
 * Preserve the input order. It represents editorial sequencing rather than
 * an automatically sortable property.
 */
```

### 16.6 Avoid historical narration in active code

Weak:

```js
// We used to do this through Liquid, but later moved it into JavaScript...
```

Better:

```js
// Runtime route resolution belongs to the JavaScript shell.
```

Put historical decisions in an architecture decision record or changelog rather than active implementation comments.

## 17. Recommended source-file structure

A predictable structure helps both agents and humans:

```text
module-level purpose
imports
types and constants
public functions
private helpers
entry point or exports
```

Example:

```js
/**
 * Search result shaping for works and series.
 *
 * Applies ranking-independent rules after raw matches have been produced.
 */

/** @typedef {import("./types.js").SearchResult} SearchResult */

/**
 * Collapse subordinate work results when an exact series match dominates.
 *
 * @param {SearchResult[]} results
 * @returns {SearchResult[]}
 */
export function shapeSearchResults(results) {
  return collapseSubordinateWorks(results);
}

function collapseSubordinateWorks(results) {
  // ...
}
```

The public behaviour is encountered before the implementation details.

## 18. Suggested validation commands for agents

A repository can define a small set of standard checks:

```bash
# Python
ruff check .
ruff format . --check
python -m pytest

# JavaScript
npx eslint .
npm test

# Site or generated-output validation
bundle exec jekyll build
python scripts/validate_generated_json.py
```

A single wrapper command is even more useful:

```bash
./scripts/check.sh
```

Example:

```bash
#!/usr/bin/env bash
set -euo pipefail

ruff check .
ruff format . --check
python -m pytest

npx eslint .
npm test

bundle exec jekyll build
```

This gives coding agents a clear definition of “finished”.

## 19. Suggested staged rollout

Introducing all rules at maximum strictness in one change is likely to produce a large amount of low-value cleanup.

### Stage 1: establish boundaries

1. Identify generated, vendored, build-output, and archived directories.
2. Exclude them from linting where appropriate.
3. Document the main repository subsystems.
4. Add standard lint and test commands.

### Stage 2: introduce conservative linting

1. Add Ruff with core error, import, bug, and upgrade rules.
2. Add ESLint’s recommended rules.
3. Fix clear defects and unused code.
4. Avoid broad auto-fixes without reviewing the diff.

### Stage 3: add semantic documentation

1. Add module-level descriptions to important files.
2. Document exported JavaScript functions with JSDoc.
3. Add type hints to Python public functions.
4. Add Python docstrings for domain rules and pipeline stages.
5. Define reusable JavaScript object types with `@typedef`.

### Stage 4: strengthen executable contracts

1. Add tests for important domain rules.
2. Add JSON Schema or validation models where appropriate.
3. Add checks for privacy exclusions and generated-output structure.
4. Add golden-output tests for stable generated artifacts.

### Stage 5: increase strictness selectively

1. Introduce stricter rules only where they improve correctness.
2. Enable docstring enforcement after the documentation convention is stable.
3. Split browser, Node, test, and generated-code ESLint configurations.
4. Consider static type checking for Python as a separate later step.

## 20. Relative value for context evaluation

| Addition | Direct semantic value | Structural value | Noise reduction |
|---|---:|---:|---:|
| Clear names and module boundaries | Very high | Very high | High |
| Python type hints | High | Very high | Medium |
| Python docstrings | Very high when selective | Medium | Medium |
| JSDoc contracts and typedefs | Very high when selective | High | Medium |
| Ruff | Low | High | High |
| ESLint | Low | High | High |
| Descriptive tests | Very high | Very high | High |
| Schemas and validation | High | Very high | High |
| Small architecture documents | Very high | High | Medium |
| Comments on obvious syntax | Low | Low | Negative if excessive |

The proposed tools should therefore help, but not equally:

- **JSDoc, docstrings, and types** provide additional meaning and searchable vocabulary.
- **Ruff and ESLint** improve consistency and reduce misleading code.
- **Tests and schemas** provide executable evidence.
- **Architecture documents** explain system-wide intent and boundaries.

## 21. Important limitations

### Documentation can become stale

Incorrect documentation is worse than missing documentation because an agent may treat it as authoritative.

Documentation should be updated in the same change as the code it describes.

### Linting does not prove architecture

A repository can pass all lint checks while still containing:

- poor module boundaries;
- duplicated implementations;
- misleading names;
- weak tests;
- undocumented business rules.

Linting is a foundation, not a substitute for design.

### Semantic retrieval does not prove completeness

A semantic result can look highly relevant while missing an unusual caller or edge case.

Use exact searches to verify:

```bash
rg "functionName"
rg "config_key"
rg "specific error text"
```

### More context is not always better

Large docstrings, repeated explanations, and historical comments can crowd out the code that matters.

The target should be:

```text
concise purpose
+ explicit contracts
+ domain rules
+ accurate types
+ executable tests
```

## 22. Recommended repository policy

A concise policy could be:

```markdown
## Code documentation and quality policy

- Public Python modules, classes, and functions should use type hints.
- Public Python APIs should have concise docstrings where purpose, domain
  rules, side effects, or exceptions are not obvious.
- Exported JavaScript functions and shared object structures should use
  JSDoc.
- Module comments should state responsibility boundaries rather than repeat
  implementation details.
- Important domain rules should be represented by tests or validation, not
  comments alone.
- Ruff and ESLint should run before changes are considered complete.
- Generated, vendored, and build-output directories should be explicitly
  identified and excluded where appropriate.
- Documentation must be changed alongside the behaviour it describes.
```

---

## 23. Overall conclusion

Introducing JSDoc, Python docstrings, Python type hints, Ruff, and ESLint should make vague development prompts easier for coding agents to resolve.

They will not independently create a semantic retrieval system. Instead, they improve the material available to both semantic reasoning and exact search:

```text
descriptive names
+ clear module boundaries
+ types
+ concise documentation
+ linting
+ tests
+ schemas
+ architecture notes
```

For coding agents:

- semantic reasoning or semantic retrieval identifies likely concepts;
- documentation supplies useful vocabulary and intent;
- types expose data structures and contracts;
- grep confirms exact definitions and references;
- linting reduces false leads;
- tests and schemas verify behaviour.

The strongest repository context is therefore not produced by a single tool. It comes from combining semantic clues with precise, verifiable structure.
