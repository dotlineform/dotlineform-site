# Tag Source Data

## Canonical Owners

All four sources live under `analytics-app/data/canonical/`.

| Source | Owns | Why separate |
| --- | --- | --- |
| `tag-registry.json` | canonical tag identity, group, label, description, allowed-group policy | vocabulary changes without rewriting assignments |
| `tag-aliases.json` | shorthand keys and canonical tag targets | alias curation can evolve independently of canonical identity |
| `tag-assignments.json` | series tags and nested per-work override deltas | assignment state changes independently of vocabulary |
| `tag-groups.json` | ordered descriptive taxonomy prose | explanatory content is not registry or assignment state |

The exact schemas and limits are enforced by `tag_services/tag_source_model.py`; the JSON files are current data, not documentation examples.

## Relationship Model

```text
tag-groups descriptions
        |
tag-registry canonical IDs
        ^
tag-aliases shorthand targets
        |
tag-assignments series rows
        -> nested work override deltas
        -> public series/work membership indexes
```

Durable rules:

- registry IDs use `<group>:<slug>` and the group must be allowed;
- alias targets must be canonical tags and may not duplicate a group within one alias;
- a work override belongs under one series row, not in a global work table;
- effective work tags combine series tags with the override delta;
- an override that duplicates a series tag is removed during normalization;
- series/work membership comes from public catalogue indexes and remains externally owned.

## Write Boundary

The browser reads through `/analytics/api/*` and writes through allowlisted operations. Mutation planners validate cross-file effects before `tag_write_transactions.py` atomically replaces one or more canonical files.

`tag-groups.json` is currently read-only through the app. Registry, aliases, and assignments are writable.

## Change Method

- vocabulary change: start with Registry/Aliases planners and inspect assignment effects;
- assignment change: preserve series-first ownership and validate catalogue membership;
- schema change: update loaders, normalization, all mutation/import paths, audits, and focused tests;
- catalogue membership change: use the catalogue/Analytics synchronization path, then run the narrow consistency check.

## Weak Spots

- cross-file referential integrity is enforced in services/audits rather than by one transactional database;
- Analytics assignment validity depends on generated public catalogue indexes;
- aliases can preserve historical selection context in assignments, creating a subtle dependency even after canonical IDs are resolved;
- hand-editing JSON bypasses planners and activity records.

[Site Consistency Audit](/docs/?scope=studio&doc=d-20260331-000000-cdc571) is the cross-boundary check. Focused schema behaviour lives in the tag service tests.
