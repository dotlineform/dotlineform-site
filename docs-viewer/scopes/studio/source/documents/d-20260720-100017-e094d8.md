# Scope Media Management Concept

## Problem

Docs Viewer can publish an image or file into the active scope without creating a wrapper document. That leaves the correct remaining problem: the scope can own media that is referenced, unreferenced, missing, generated from retained source, or stored remotely, but authors have no ordinary surface for seeing that state or making later lifecycle decisions.

The code already has a provider-independent inventory over configured published media and registered build-media source. It can associate published identities with document references, retain unreferenced owned items, and report references whose published bytes are missing. This should become the foundation of the feature rather than adding an independently named asset root or association registry.

## Mental Model

![Scope configuration, registered storage, build sources, and document references feed a provider-independent media inventory and its manage-only inspection surface.]([[media:docs/studio/svg/scope-media-management-model.svg]])

The stable identity is:

```text
scope + media type + relative identity
```

Provider, filesystem root, R2 prefix, and served URL are resolved from scope configuration. A media item does not need a document wrapper, and an owned item does not need to be currently referenced.

## Invariants

- Scope ownership is logical and configuration-backed; it is not inferred from whether bytes are local or remote.
- Every published identity resolves to exactly one configured destination.
- An inventory is derived current state, not a second per-file registry that must be synchronized after every change.
- Source, published output, and consumer references remain distinct.
- Missing, referenced, and unreferenced are evidence states. None independently authorizes mutation.
- Replacement and deletion require provider-aware preview, reference impact, and explicit confirmation.
- No normal author action asks the user to choose repository, external-local, or R2.
- Arbitrary project working files remain with their project owner. Docs Viewer manages only media deliberately brought into its scope contract.

## First Useful Surface

The initial product should be one manage-only **Scope Media** inventory, not one document per media item and not one report document per scope.

Illustrative wireframe:

```text

│ Scope [Processing ▾]  Type [All ▾]  State [All ▾]  [Search…]  [Refresh]  │
│ 40 items · 4.1 MB · 17 referenced · 23 unreferenced · 0 missing          │
```

The summary figures are a useful orientation, not a health score. An unreferenced item may be intentional, and a build source normally has no direct reader reference.

![An author opens the Scope Media surface, which requests a current inventory from the local Docs service and returns provider-independent rows for inspection.]([[media:docs/studio/svg/scope-media-inventory-workflow.svg]])

## Can The Existing Report Mechanism Host It?

### What Already Fits

The current report mechanism is a credible first host because it already provides:

- a manage-only access boundary inside the normal document pane;
- one report document that can select another configured scope through report-prefixed route state;
- an allowlisted metadata registry and lazy report-module loading;
- report-owned filtering, sorting, row rendering, empty/error states, and shared table controls;
- local management-service reads without exposing them to public entrypoints;
- existing examples that request fresh server-side audit/config data.

A read-only inventory table therefore does not require a dashboard framework or a new application route.

### What Does Not Fit Automatically

The report mechanism is deliberately document-level and inspection-oriented. Adding a new report currently requires coordinated registry metadata, an allowlisted JavaScript loader, a report-service method, and usually a focused local endpoint. It is not a schema-driven table plugin.

The main product weakness is discoverability and scope context: a single Studio-owned report can inspect Processing or another scope through `report_scope`, but opening a Studio document to manage the active Processing scope may feel indirect. A scope action can deep-link to the report with the active scope selected, but that navigation needs a brief UI trial.

Reports also should not become the host for replacement, deletion, moving, or other confirmed lifecycle workflows. Those actions need capability projection, preview/apply services, busy state, and explicit result handling. If lifecycle actions become important, a dedicated scope-level **Media** view or focused management component may be the better owner.

### Surface Decision Gate

Use the existing report mechanism for the first delivery only if a small fit assessment confirms all of the following:

1. one report document can deep-link from any manageable scope without confusing scope ownership;
2. the current report layout remains usable for the expected row count and path lengths;
3. R2 listing latency and authentication failures can produce contained report states;
4. refresh is explicit and does not imply a background cache or job;
5. the report remains read-only apart from navigation, copying a reference, and opening a served item.

If one of those conditions fails, create a separate development need for a reusable scope-management view host. Do not widen the report runtime into a general workflow framework inside this feature delivery.

## Alternative UI Hosts

| host | strengths | costs and risks |
| --- | --- | --- |
| Existing report-backed document | Reuses access isolation, route state, tables, service reads, and normal Docs navigation. | Studio-owned host may feel indirect for another active scope; intentionally unsuitable for mutations. |
| Scope action opening a modal or panel | Starts from the active scope and can stay close to the object of action. | Dense inventories, deep links, sorting, responsive tables, and history are awkward in a modal. |
| Dedicated manage-mode Media view | Natural long-term home for inventory and lifecycle actions. | New view/controller/navigation contract and a broader runtime-development need before the first value is delivered. |

The provisional preference is the existing report for read-only inventory, with a scope action that opens it already filtered. A dedicated view should be justified by the lifecycle workflow, not created in anticipation of it.

## Simple Future Lifecycle

Lifecycle actions are later outcomes, but the inventory should leave room for this sequence:

![A future lifecycle action previews references, requires explicit confirmation, revalidates provider state, applies one exact mutation, and refreshes the inventory.]([[media:docs/studio/svg/scope-media-lifecycle-workflow.svg]])

Automatic orphan deletion, reference-counted cleanup, rollback copies, and silent R2 removal are outside the intended model.

## Nested Organisation

The location adapters and logical-token model can represent confined relative identities, but source-editor publication and the current local media route still assume one flat filename. Supporting paths such as these would make scope media substantially easier to manage:

```text
docs/processing/files/iching/source-data.zip
docs/processing/img/iching/reference/hexagram-01.webp
```

Nested identity support should remain one scope/type namespace. It does not require a collection registry. The complete authoring, route, token, inventory, collision, and containment boundary must change together before nested paths are exposed.

## More Flexible Storage

Current configuration chooses one provider/location per scope media type. That is simple but coarse when the same delivery type contains both modest and very large assets.

Use this decision rule:

- add a **media type** when rendering, sanitization, build, MIME, or delivery behavior differs;
- add a **named destination within a type** when only storage, size, provenance, or retention policy differs.

A possible later logical shape is:

```text
docs/processing/files/default/project-notes.xlsx
docs/processing/files/large/iching-source.zip
docs/processing/files/archive/original-bundle.zip
```

`default`, `large`, and `archive` would be stable configured identity prefixes, not provider names. Each prefix would resolve to exactly one location, and the authoring workflow would return the resulting logical reference without presenting an R2 selector.

Do not route an unqualified identity by searching several providers. Do not rely only on a mutable size threshold, because later rendering must still resolve previously published bytes deterministically. A per-object registry is a fallback only if real requirements cannot be represented by stable prefix routing.

## Boundaries

- No external asset root or scope-to-collection association.
- No arbitrary filesystem browser or inventory of Processing project source.
- No public inventory surface.
- No digital asset manager, tagging system, upload history, OCR, thumbnails, archive expansion, or background indexing.
- No automatic deletion of unreferenced media.
- No nested authoring or multiple destinations until their complete contracts are separately accepted.
- No report-framework generalization without a second concrete report need or a failed surface-fit gate.

## Questions Before Implementation

- Should the first inventory include references from configured sub-scopes, and how should those document identities be labelled?
- Is a Studio-owned report deep-link acceptable when inspecting another active scope?
- What inventory size or R2 response time makes explicit refresh insufficient?
- Which columns belong in the first payload versus being derived in the browser?
- Should open/download be omitted for build source and unavailable providers rather than represented as a disabled action?
- Does nested organization need author-selected folders, code-owned prefixes, or both?
- Is there a real same-type case requiring multiple destinations, or are new handling types such as video the next actual need?
