# Index Multiple Selection Delivery

## Outcome

The manage-mode tree index has an explicit selection mode with row checkboxes, Shift-range selection, a selected count, **Clear**, and **Done**.

Checkbox selection is browser-only state independent from the displayed document. It does not change link navigation, the active-row highlight, current single-document controls, drag/drop, or backend mutation contracts.

This is the complete visible selection foundation. Group movement is a later delivery after this interaction is proven.

## Selection Contract

- A registered manage-only **Select** control appears only for the `index-tree` view.
- Entering selection mode reveals a checkbox gutter immediately left of every eligible document node.
- The gutter and checkbox are siblings of the node. They do not move or narrow the node, wrap its title, replace its status slot, or begin drag/drop.
- The selection owner holds selected document ids, the range anchor, and whether selection mode is active. Consumers never reconstruct selection by reading checkbox DOM state.
- Clicking a checkbox changes only selection membership. Shift-click selects the inclusive visible range from the stable anchor in current tree order.
- The checkbox is the complete checked-state indicator. Only the displayed document keeps the existing active-row highlight and `aria-current="page"`.
- Document links remain ordinary links. Navigation, native modifier-click, row focus, context-menu opening, and expansion changes do not alter selected ids.
- The index control reports the count and exposes **Clear** and **Done**. **Clear** empties the set and remains in selection mode; **Done** exits and clears it.
- Selection changes perform no request, write, rebuild, busy state, or cache invalidation.

## Action-Target Isolation

Feed the checked ids into the existing code-owned action context, but do not let that silently retarget current single-document controls.

Audit every current action definition that names `selection`. Until a complete multi-document workflow is delivered, each existing toolbar, Actions-menu, context-menu, and drag action must continue to resolve the displayed document or its explicit invocation document exactly as it does before this delivery. No destructive or mutation action becomes plural here.

The later group-movement delivery may make the move action consume the checked set. It must not recover a hidden primary selection or broaden another action as a shortcut.

## Reload And Transition Contract

| event | selection result |
| --- | --- |
| Initial boot, browser reload, scope change, loss of management context, or leaving `index-tree` | Start or return with selection mode off and an empty set. Do not persist selection in storage or across scopes. |
| Document-link navigation, history navigation within the same scope, focus, context menu, expansion, Search, or Recent | Preserve selection mode, selected ids, and range anchor. These do not replace the index population. |
| Same-scope index reload after create/import, source save, metadata change, manual rebuild, settings change, viewability change, or committed-move recovery | Preserve selected ids that remain eligible in the refreshed manage index; prune missing or newly ineligible ids; retain the anchor only while its document remains eligible. |
| Successful local single-document move | Preserve the selection exactly. The existing committed local projection remains the normal path and does not reload the index. |
| Delete followed by index reload | Remove the deleted or otherwise missing ids and preserve every remaining eligible id. Stay in selection mode even if the set becomes empty. |
| **Clear** or **Done** | **Clear** empties membership and anchor; **Done** also exits selection mode. |

A Shift-click whose previous anchor is no longer in the current visible range behaves as an ordinary checkbox toggle and establishes the new anchor. Reload reconciliation is one selection-owner operation, not case-specific state spread through every mutation workflow.

## Not In Scope

- group drag/drop, plural move payloads, effective move roots, destination validation, writes, rebuilds, or local multi-move projection;
- [Document Sharing UI Redesign](/docs/?scope=studio&doc=d-20260720-230623-c7e921) or another multi-document consumer;
- batch delete, viewability, metadata, copy, export, or a general batch-action framework;
- modifier-click selection, keyboard range extension, direct toggle shortcuts, lasso, touch drag/drop, sibling reordering, or cross-scope selection;
- selection persistence in local storage, session storage, URLs, packages, or backend state;
- public, review, Search, Recent, graph, or mobile selection UI;
- a broader index, toolbar, or Docs Viewer visual redesign.

## Done When

- manage `index-tree` alone exposes the registered selection control and stable checkbox gutter;
- checkbox state, selected count, range anchor, and selection mode have one focused browser-state owner;
- checkbox click and Shift-click change only the checked set, while document navigation and active-row semantics remain independent;
- **Clear**, **Done**, scope/view exit, and reload reconciliation follow the explicit lifecycle contract;
- same-scope reloads preserve eligible checked ids and prune missing or ineligible ids without storage persistence;
- all existing single-document controls and current drag/drop retain their pre-delivery targets and behavior;
- selecting documents performs no network request, mutation, rebuild, cache invalidation, or busy transition;
- focused automated evidence covers durable state, targeting, DOM wiring, and route boundaries, with visual and keyboard feel left to user review;
- [Docs Viewer Runtime](/docs/?scope=studio&doc=d-20260331-000000-c313fd) or [Runtime Module Ownership](/docs/?scope=studio&doc=d-20260605-125108-1d1ef3) records the shipped owner before this delivery closes.
