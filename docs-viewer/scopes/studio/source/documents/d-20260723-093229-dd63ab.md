# UI Workbench Concept

## Status And Relationship

**Proposed as cross-app infrastructure; the first Docs Viewer pack is delivered.** This concept originates with [Docs Viewer Modal Consistency](/docs/?scope=studio&doc=d-20260723-091111-aaa2d6).

The intended workbench is cross-app infrastructure. Admin hosts the inspection surface, each app owns its specimens and production code, and Studio participation requires a separate bounded delivery. The Docs Viewer delivery must not silently become a Studio UI audit.

The [UI Workbench Guide](/docs/?scope=studio&doc=d-20260723-155929-ef9c14) owns route choice, the delivery workflow, specimen selection, comparison practice, verification, screenshots, and maintenance triggers. This concept retains purpose, ownership, and design boundaries.

## Development Role

The workbench shortens the UI feedback loop. It makes production-backed validation, busy, failure, long-content, theme, viewport, and related-control states immediately reachable without first constructing the complete workflow. This supports rapid before-and-after lifecycle and presentation review, but it does not replace the existing test stack.

| evidence | responsibility |
| --- | --- |
| workbench | make UI states cheap to reach, compare, and interact with; surface rendering, browser, lifecycle, and tactile problems early |
| focused behavior checks | prove interaction contracts such as focus handling, Escape precedence, cleanup, scroll restoration, and workflow-owned state transitions |
| end-to-end checks and representative production routes | prove real services, payloads, persistence, navigation, and side effects reach the correct UI state |

Fixtures make the production implementation renderable and exercisable in a controlled state. They do not prove that the real workflow reaches that state correctly. The concrete benefit is earlier, broader UI feedback before paying the setup and execution cost of complete integration paths.

## Intention

Make complete, comparable UI states available without navigating every real workflow, preparing external data, or causing service writes.

The workbench is a living maintenance fixture:

- developers can review related controls and modal states together;
- implementation differences become visible before they become conventions;
- new capabilities can be checked against established interaction principles;
- periodic reviews can detect accumulated drift;
- production routes remain the final integration authority.

It is lighter than a design system and more concrete than screenshots or prose. It does not define product workflows.

## Principles

### Production-Backed

Specimens import the real production renderers, components, and styles. Fixture adapters provide data, injected clients, callbacks, and state transitions only.

Do not copy production markup or maintain workbench-specific versions of buttons, fields, lists, modal bodies, status regions, or action rows.

If a workflow cannot render a useful state without performing its service operation, treat that as evidence that rendering and orchestration ownership may need a focused seam. Do not reproduce the UI in the workbench to avoid correcting that seam.

### Federated Ownership

| Responsibility | Owner |
| --- | --- |
| workbench index, navigation, comparison controls, and audit entrypoint | Admin |
| app-neutral specimen contract and genuinely reused harness behavior | shared support only after two real packs prove the contract; none is currently required |
| Docs Viewer modal registry, fixture state, and adapters | Docs Viewer test/workbench support |
| Studio registry, fixture state, and adapters | a separate Studio delivery |
| production components and styles | their current app or `shared/frontend/` owner |
| tracked reference screenshots | `admin-app/app/assets/ui-workbench/screenshots/<app>/` |
| untracked exports and generated comparison reports | `var/admin/ui-workbench/` |

`shared/frontend/` remains the home for proven production components used by real apps. Workbench code does not move there merely because it can display several apps.

Each app owns its pack code, fixtures, adapters, and production imports. Admin serves the only Workbench route plus the narrow local asset surface needed to load those app-owned packs and isolate specimens. This does not move production ownership to Admin or proxy sibling workflow APIs.

### Semantic Before Visual

Controls that look similar are not automatically one component. Compare their contracts first:

| Dimension | Questions |
| --- | --- |
| selection | none, single, multiple, or hierarchical |
| focus | native order, roving focus, active descendant, or checked control |
| navigation | Tab, arrows, type-ahead, Enter, Space, or double-click |
| state | selected, checked, current, expanded, disabled, or busy |
| data states | loading, empty, filtered-empty, warning, or error |
| actions | row activation, separate toolbar, inline command, or form submission |
| ownership | shared primitive, app component, modal-local renderer, or workflow controller |

If two controls have the same meaningful contract, prefer one production primitive. If their semantics differ, share only the applicable tokens or presentation rules.

## Specimen Contract

The exact registry schema belongs in code. Each specimen should identify enough to support discovery and maintenance:

- stable specimen id and app/family grouping;
- production renderer or component owner;
- fixture state and injected dependencies;
- normal, empty, selected, validation, busy, success, failure, and long-content variants where applicable;
- supported theme and viewport review;
- intended initial focus and keyboard interaction;
- whether a representative live workflow still needs separate integration verification.

The registry is the current inventory. Do not duplicate it as a long-lived table in documentation.

Family, state, and app are registry metadata and searchable context, not cascading navigation. The default review opens one exact named specimen. Optional named recipes record a review question, exact A and B specimens, dimensions, and either sequential or side-by-side presentation; arbitrary pair selection is not part of the contract.

## Historical Boundary

A former Admin-hosted UI Catalogue was retired after maintaining parallel demo documents, JavaScript, CSS, and assets. Do not restore that model.

This workbench differs in one decisive way: its specimens exercise production implementations. Admin provides an inspection surface, not a second component library.

## Non-Goals

- A universal cross-app UI framework or forced visual identity.
- A replacement for app-owned routes, state, adapters, services, or integration checks.
- A permanent screenshot or pixel-diff regression suite.
- Screenshot images embedded in documents or maintained as a document catalogue.
- A gallery of copied markup.
- Automatic promotion of similar controls into `shared/frontend/`.
- Studio remediation inside the Docs Viewer modal delivery.

## First And Later Consumers

The [Docs Viewer Modal Consistency Delivery](/docs/?scope=studio&doc=d-20260723-091112-5e1b62) supplied the first production-backed modal pack and uses it to correct Docs Viewer lifecycle and styling drift.

A later Studio delivery should register Studio's real modals and repeated controls, then use the same semantic comparison process. Only proven same-contract behavior should move into shared production components.
