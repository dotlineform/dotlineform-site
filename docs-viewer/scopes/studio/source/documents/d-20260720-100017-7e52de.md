# Scope Media Inventory Surface Delivery

## Status And Purpose

This is a **proposed** delivery used to test feasibility and blast radius. It is not implementation-ready and should be reshaped if the [concept](/docs/?scope=studio&doc=d-20260720-100017-e094d8) changes or the report-host assessment exposes a separate development need.

## Outcome

One local, read-only Scope Media surface shows the current provider-independent inventory for a selected configured scope, including source/published role, media type, relative identity, provider, size, served reference, referencing documents, and missing-reference state.

The outcome is complete only when the surface is easy to reach with the intended scope selected, public routes remain isolated, and current inventory failures produce contained actionable states.

## First Decision: Report Host Fit

Before implementation, use a small code-and-UI assessment to decide whether the existing report mechanism is an adequate host.

Accept the report host when it can:

- use one Studio-owned local report document for all configured scopes;
- open from another scope with `report_scope` already selected;
- render useful path-heavy rows responsively through existing report structure and styles;
- request a fresh inventory from the local service and present R2 latency/authentication failures safely;
- remain inspection-only.

If the assessment fails, stop this delivery. Record a separate bounded development need for a scope-level management view host, then revise this delivery rather than adding workflow responsibilities to reports.

## Likely Blast Radius

This table identifies owners to inspect; it is not an instruction to change every listed file.

| boundary | likely current owners | possible delivery pressure |
| --- | --- | --- |
| Inventory model | `docs_media_inventory.py`, `docs_scope_config.py`, artifact location adapters | Include complete same-scope and sub-scope references, normalized status and summary counts, and contained provider warnings without adding a per-file registry. |
| Local read API | `docs_management_routes.py`, `docs_management_read_service.py`, `docs_management_service.py`, service capability projection | Add one scope-validated GET request and compact response contract; keep credentials, absolute paths, and provider clients server-side. |
| Report metadata and host document | `docs-viewer/config/reports/reports.json`, Studio report source documents | Register one local report id and create one report host document rather than one document per scope or media item. |
| Manage report runtime | `docs-viewer-reports.js`, `docs-viewer-report-service.js`, a focused new report module | Allowlist lazy loading, scope selection, refresh, filtering, sorting, row/detail rendering, copy-reference, and safe open/download links. |
| Existing report composition | `docs-viewer-management-document-reports.js`, shared document controller hooks | Prefer no change; change only if active-scope deep-linking cannot be expressed through current context and report-prefixed URL state. |
| Presentation | `site/docs-viewer/static/css/docs-viewer-reports.css` | Reuse shared controls and responsive list/table rules; add media-specific layout only where path/role/status columns need it. |
| Assurance | `test_docs_media_inventory.py`, management read/route tests, report module smoke, public runtime boundary tests | Prove reference attribution, local/R2 responses, traversal and credential isolation, missing/unreferenced distinction, loader registration, and public exclusion. |
| Durable docs | Reports, Media And Asset Handling, Source Organisation, Docs Images And Assets | Transfer only shipped workflow, extension method, and weak spots at closeout. |

## Potential Later Blast Radius, Not This Delivery

The concept deliberately exposes larger follow-on areas that must not be pulled into the read-only inventory delivery:

| later outcome | likely wider owners |
| --- | --- |
| Nested media identities | staged-media planning/apply, local media routes, Markdown/token rendering, collision checks, artifact adapters, inventory reference parsing, tests and author docs |
| Multiple destinations within one type | scope config schema/policy, served-reference resolution, publication planning, inventory aggregation, build producers, public projection, migration and byte verification |
| Replace/delete lifecycle | capability projection, preview/apply services, reference-impact model, exact provider mutation, R2 cleanup policy, activity, busy/result UI, lifecycle docs and tests |
| Dedicated scope Media view | manage navigation, view/controller lifecycle, route state, layout, service adapter, capability host, module/bootstrap tests |

Finding one of these useful does not widen this delivery.

## Proposed Checkpoints

### SMI-0 — Surface Fit And Payload Contract

- [ ] Confirm report-backed-document navigation from Studio, Processing, Notes, and a public scope viewed through local management.
- [ ] Sketch the smallest inventory response from current `DocsMediaInventory` data and identify any missing sub-scope or provider-warning evidence.
- [ ] Try representative path lengths and row counts in the existing report layout.
- [ ] Decide **use current report host** or **stop and create separate view-host need**.

Stop after this checkpoint until the concept and delivery boundary are accepted.

### SMI-1 — Provider-Independent Read Service

- [ ] Add one scope-validated read contract over the existing inventory owner.
- [ ] Keep absolute roots, credentials, raw exceptions, and remote clients out of the response.
- [ ] Return referenced, unreferenced, missing, source, and published states without declaring deletion eligibility.
- [ ] Cover repository, external-local, R2, unavailable-provider, and sub-scope-reference behavior at the service layer.

### SMI-2 — Local Inventory Surface

- [ ] Register and mount one local report module if SMI-0 accepted the report host.
- [ ] Add scope, type, role/status, and text filtering with explicit refresh.
- [ ] Show summary counts and inspectable item/reference detail.
- [ ] Offer only safe read actions such as copy logical reference and open/download when supported.
- [ ] Keep the surface unavailable on public routes and free of mutation controls.

### SMI-3 — Integration And Closeout

- [ ] Verify loader/registry/service agreement and direct deep-linking to a selected scope.
- [ ] Run focused inventory, API, module, and public-isolation checks.
- [ ] Manually review dense, empty, missing, R2-error, narrow-screen, and long-path states.
- [ ] Transfer shipped behavior to the durable report/media owners and close or reshape the feature documents.

## In Scope

- Existing configured scope media and build-media source.
- One selected scope at a time, with all configured scopes available in local management.
- Provider-independent inventory and current document-reference correlation.
- Explicit refresh, summary, filters, row detail, reference copy, and safe open/download.
- Report-host assessment and a stop decision when the fit is poor.

## Not In Scope

- External collection roots or arbitrary filesystem inventory.
- New media types, nested authoring, multiple storage destinations, or byte migration.
- Replace, rename, move, delete, upload history, or automatic orphan cleanup.
- Background jobs, caches, hashing policy, thumbnails, OCR, content indexing, or archive expansion.
- Public report promotion.
- A generic report plugin, schema-driven table framework, dashboard, or new management route unless SMI-0 first identifies that as a separate need.

## Completion Evidence

- The same logical inventory response works for filesystem and R2-backed types.
- Referenced, unreferenced, build-source, and missing states are distinguishable and tested.
- A report deep-link opens with the intended scope selected, or the delivery stops with an accepted separate host need.
- Public runtime/config tests prove the inventory loader, service URL, and data are not projected publicly.
- Manual UI review confirms the result remains readable at realistic density and path length.
- [Reports](/docs/?scope=studio&doc=d-20260513-105343-6de36b), [Media And Asset Handling](/docs/?scope=studio&doc=d-20260514-184303-7914e2), and [Source Organisation](/docs/?scope=studio&doc=d-20260423-000000-8fd731) receive only the stable shipped contract.

No implementation begins from this document until SMI-0 and the concept are accepted.
