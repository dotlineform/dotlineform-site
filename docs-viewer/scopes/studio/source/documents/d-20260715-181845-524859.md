# Document Diagrams Concept

## Purpose

High-level diagrams make execution paths, ownership, and public/manage/server boundaries easier to understand than prose or text art.

## Decision

Use Mermaid as the default source language for authored architecture and workflow diagrams in any Docs Viewer scope.

Mermaid source is the durable authority. A repository-owned Docs media producer renders configured `.mmd` source to static SVG. Docs Viewer displays the result as an ordinary image and does not parse Mermaid or load a diagram runtime in the browser.

Finished SVG and Mermaid source share **Add image…** but take different service paths. A staged `.svg` is sanitized and published as finished vector media; a staged `.mmd` is retained as canonical build source and passed through the registered Mermaid producer before its SVG token is returned. An unprocessed `.mmd` beside a manually imported SVG is not a supported workflow.

## Viewer Author Workflow

1. Open the target document's source editor and choose **Add image…**, not global **Import**.
2. Select a staged `.mmd` file and enter useful diagram alt text.
3. The action copies the canonical source into the current scope's configured `media/mermaid/` location and invokes the Mermaid producer for that source.
4. After successful SVG publication and byte verification, the action inserts this form at the current selection or caret:

<pre><code>![Diagram alt text](&#91;&#91;media:docs/&lt;scope&gt;/svg/&lt;diagram-name&gt;.svg&#93;&#93;)</code></pre>

5. **Rebuild doc** commits the dirty Markdown buffer and its targeted document payload. Returning to the displayed document shows the published SVG.

A render or publication failure inserts no token. The interactive action renders the selected source immediately; a later full Docs build runs the same producer over all configured Mermaid source so the derived SVG remains reproducible.

## Editing Boundary

Mermaid CLI is the repository renderer, not the human editing interface. This feature does not surface or edit raw `.mmd` inside Docs Viewer. The document source editor remains a Markdown editor; it does not host an embedded Mermaid editor or live diagram preview.

Human authors create diagrams in an external Mermaid-aware editor and save the initial `.mmd` into staging for **Add image…**. A browser editor still requires an explicit save back to a local file; a local-file editor such as VS Code with Mermaid support is the direct-save fit. Codex and repository authors can edit canonical `.mmd` directly.

External-editor preview is authoring assistance. The repository-pinned Mermaid version and producer output remain authoritative.

## Ongoing Edit Workflow

After initial intake, the scope-owned `.mmd` is edited in place rather than staged again:

1. view the rendered document and open Document Info;
2. under **Diagrams**, choose the diagram-title link corresponding to the rendered SVG to open its Mermaid source in VS Code;
3. save the `.mmd` in its scope-owned `media/mermaid/` location;
4. Docs Watcher detects the configured Mermaid-source change and runs the producer for that identity;
5. reload the displayed document to see the updated SVG at its existing token.

Rendered Document Info lists only scope-confined Mermaid sources that correspond to the saved document's SVG tokens and registered `mermaid` to `svg` build relationship. A standalone SVG has no canonical `.mmd` and does not appear. Multiple diagrams become separate labelled links; no toolbar action, caret inference, arbitrary path, or source picker is required.

These links are manage-only and expose logical identities rather than physical paths. The server revalidates the document reference, configured same-basename Mermaid source, containment, and file existence before delegating to VS Code. In Markdown source mode the info surface remains the semantic-token picker and does not show diagram links.

The Markdown token does not change, so a normal diagram edit does not require **Add image…**, **Rebuild doc**, or a docs-search rebuild.

## Repository And Codex Workflow

Repository authors and Codex do not drive **Add image…** or use staging. They:

1. write the canonical `.mmd` directly to the scope's configured `media/mermaid/` source location;
2. add the SVG media token directly to the owning Markdown document;
3. run the targeted Docs builder for the owning document:

```bash
$HOME/miniconda3/bin/python3 docs-viewer/build/build_docs.py --scope <scope> --only-doc-ids <doc-id> --write
```

The Mermaid delivery extends targeted builds so they render and publish Mermaid outputs referenced by the selected document before rendering its payload. The normal targeted search build follows when the document prose changed. A full Docs build continues to process all configured Mermaid source. No separate diagram-conversion CLI is required.

## Repository Build Contract

- Own the Node boundary beneath `docs-viewer/build/mermaid/`: exact dependency and scripts in `package.json`, resolved dependency graph in `package-lock.json`, and shared render settings in `mermaid-config.json`.
- Pin `@mermaid-js/mermaid-cli` exactly to `11.16.0`; invoke that local executable and never an unversioned global `mmdc` installation.
- Register Mermaid in each participating scope's `source.build_media` configuration.
- Read canonical `.mmd` files from the configured `media/mermaid/` source location.
- Render each source to same-basename SVG directly in the configured published `svg` location.
- Support targeted rendering for **Add image…** and Mermaid outputs referenced by `--only-doc-ids`, plus complete rendering during a full Docs build, through the same producer.
- Extend Docs Watcher beyond Markdown documents to configured Mermaid source and run only the changed diagram identity on save.
- Render with the neutral theme, white background, strict security, `htmlLabels: false`, and an explicit 1200 by 800 browser viewport. Preserve the SVG `viewBox` and do not add Markdown dimensions by default.
- Require useful Mermaid `accTitle` and `accDescr` source plus useful Markdown alt text. Disabling HTML labels keeps generated text in ordinary SVG elements that survive the shared sanitizer.
- Report invalid source or a failed render as a build failure rather than silently retaining or accepting an unrelated SVG.

The [Mermaid CLI](https://github.com/mermaid-js/mermaid-cli) is the upstream command authority. The pinned repository version, producer, and render configuration become the project authority.

## Reader Boundary

Readers receive static SVG. Mermaid JavaScript, Node, npm, Puppeteer, source files, and build commands do not ship to Docs Viewer routes or the public site.

## Suggested Repository Convention

Use the existing configured source and published-media roots:

```text
<scope source root>/media/mermaid/<diagram-name>.mmd
<configured published svg root>/<diagram-name>.svg
```

- For a repo-backed scope the source maps to `docs-viewer/scopes/<scope>/source/media/mermaid/`.
- The producer writes the SVG to the scope's configured `svg` location. Current public scopes keep this class in the repository even when raster or download media uses R2.

Markdown uses the same logical token in every scope: <code>&#91;&#91;media:docs/&lt;scope&gt;/svg/&lt;diagram-name&gt;.svg&#93;&#93;</code>.

Additional conventions:

- use lowercase kebab-case names describing the diagram, not the rendering tool;
- keep one `.mmd` source and one `.svg` output with the same basename;
- do not edit the generated SVG by hand;
- render with the repository's explicit background, theme, dimensions, and accessibility options;
- include useful surrounding prose and alt text; the SVG is not the sole statement of the document's meaning;
- if two scopes need the same diagram, each scope owns its own `.mmd` source and derived SVG rather than depending on another scope's media.

## Invariants

- Markdown remains readable and editable without a graphical renderer.
- Every canonical `.mmd` source has one build path to its derived SVG.
- A broken diagram fails visibly at the media-producer boundary.
- Ordinary documents do not load a visualisation dependency they do not use.
- Public routes receive no build-time renderer or source file by accident.
- The supported tool version and material render options are pinned, making generated results reproducible enough to review through source control and the publication gate.
