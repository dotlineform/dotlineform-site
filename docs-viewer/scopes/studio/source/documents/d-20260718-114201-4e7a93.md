# Docs Viewer Core Refactor Working Method

## Purpose

Run the core extraction without turning the normal checkout, Local Studio, watchers, real external-local documents, or planning surface into an unfinished development environment. This page owns the cross-environment and concurrent-work rules. The [Core Refactor parent](/docs/?scope=studio&doc=d-20260718-114645-7b9d24) owns navigation; [Git](/docs/?scope=studio&doc=d-20260718-124244-3f7a92), [Testing](/docs/?scope=studio&doc=d-20260718-123734-6d84c1), and the [Delivery Overview](/docs/?scope=studio&doc=d-20260718-120304-2c6f81) own their detailed methods.

## Two-Way Change Assessment

`main` and the refactor constrain each other. Treat impact assessment as reciprocal rather than asking only whether incoming feature work will conflict with the refactor.

| direction | required question |
| --- | --- |
| bounded feature on `main` → refactor | Does this feature touch the current extraction boundary, change the intended core, or add a dependency the refactor would immediately need to move? |
| bounded refactor slice → open features | Does this slice constrain, reorder, simplify, complicate, or invalidate any open or ready feature proposal? |

The active refactor delivery must name its current code, configuration, generated-contract, and runtime ownership boundary. That gives other work a bounded go, coordinate, or wait decision rather than imposing a repository-wide freeze.

Before closing each refactor checkpoint:

1. inspect the open and ready Docs Viewer proposals that depend on responsibilities touched by the slice;
2. identify changes to extension points, configuration ownership, service APIs, browser state, generated schemas, or composition eligibility;
3. classify each affected proposal as unaffected, coordination required, newly dependent or reordered, or materially reshaped;
4. update the owning proposal or roadmap when the constraint is real rather than leaving it implicit;
5. record why the boundary still supports the known product need, or surface a decision before closing the checkpoint.

Open proposals are design evidence, not instructions to build speculative generality into the core. Preserve their known requirements where they are credible, but prefer an explicit roadmap constraint or proposal revision over a generic extension system created for hypothetical compatibility.

## Environment Separation

| resource | stable environment | refactor environment |
| --- | --- | --- |
| checkout | current worktree on `main` | sibling worktree on `feature/docs-viewer-core` |
| purpose | normal work and current-product reference | extraction, standalone composition, and packaging experiments |
| ports and browser origin | current Local Studio values | dedicated non-overlapping port range |
| environment file | current machine-local `.env.local` | worktree-specific untracked `.env.local` |
| application data, logs, and caches | current roots | isolated refactor roots |
| external workspace base | current configured projects base | dedicated contained test base |
| repository source and published output | current checkout | refactor worktree |

Do not point both environments' watchers or management services at the same external-local scope roots. Keep remote/public writes disabled in the refactor environment or direct them to explicitly isolated test targets.

## Scope And Output Ownership

| scope or output | refactor rule | integration rule |
| --- | --- | --- |
| refactor planning, decisions, and handoffs | author canonically in `studio` on `main` | merge committed main updates into the feature branch |
| other repository-backed scope source | branch-owned files may diverge | merge source changes through Git |
| repository-local published/search output | each watcher updates only its worktree | resolve source/config first, then regenerate |
| real external-local source and media | stable environment has exclusive write ownership | no Git merge; the integrated application continues to read the current external data |
| refactor external-local source and media | use a seeded copy, fixture, or dedicated test root | disposable unless a source change is deliberately promoted through review |
| remote/public storage | disable writes or use isolated test targets | never use production targets for refactor validation |

Repository-backed documents can move in parallel, but refactor coordination documents have one owner: `main`. Merge committed workspace and ordinary documentation changes into the feature branch and do not edit an authoritative planning document independently on both branches. Generated payloads are not the merge authority: settle the canonical source and configuration first and rebuild the target branch rather than hand-merging generated JSON.

External-local data is not branch-aware. There is no automatic bidirectional synchronization between the stable and refactor roots. Prove any scope-layout migration in preview against a fresh copy before a separate, backed-up production cutover.

## Documentation Control Plane

Keep canonical planning, architecture, delivery state, working notes, and handoffs under the main-owned [Core Refactor parent](/docs/?scope=studio&doc=d-20260718-114645-7b9d24). Read and manage them with the stable Docs Viewer. This remains available if the feature branch has broken the Docs Viewer route, builders, configuration, or startup.

The refactor Docs Viewer is an optional test surface, not a documentation dependency. When it is operational, run it on an isolated port to compare feature behavior with the stable application. Its failure must not prevent reading the current plan, recording a decision, or handing off the next checkpoint.

Worktree-specific runtime configuration still belongs in an untracked `.env.local` with non-overlapping ports and an isolated `DOTLINEFORM_PROJECTS_BASE_DIR`. Define a named refactor launcher or profile when implementation starts so this separation does not rely on a remembered environment recipe.

## Concrete Starting Point

Work remains on `main` through the analytical starting sequence:

1. settle and commit the intended planning baseline without sweeping in unrelated dirty files;
2. complete the [Core Definition Analysis Delivery](/docs/?scope=studio&doc=d-20260718-121833-9fa2d4) and accept an adjustable Core Definition Baseline v1;
3. write the Core Refactor Transition Architecture under the main-owned parent;
4. use that evidence to define one independently complete first implementation delivery;
5. pass the creation gate in [Git](/docs/?scope=studio&doc=d-20260718-124244-3f7a92) and activate the worktree only when implementation is ready to begin.

No long-lived branch or worktree is needed during planning. The [Delivery Overview](/docs/?scope=studio&doc=d-20260718-120304-2c6f81) owns the transition sequence, discovery handling, and delivery-quality gates.

## Git Maintenance

Codex may handle dirty-state inspection, branch and worktree mechanics, focused commits, main/refactor synchronization, generated-output accounting, conflict resolution, integration preparation, recovery, and closeout when explicitly asked to activate or advance that work.

The user retains product and architecture decisions, delivery scope, discovered-work classification, proposal constraints, migration decisions, and final integration approval. Git delegation does not silently authorize commits, pushes, merges, destructive cleanup, or external migration.

Keep reports decision-sized:

```text
main: clean / current
refactor: 3 commits ahead; 1 main commit not yet integrated
active delivery: application-root contract
conflicts: none
next safe action: merge main into refactor
```

The sibling [Git](/docs/?scope=studio&doc=d-20260718-124244-3f7a92) page owns the detailed creation gate, activation, maintenance, integration, recovery, and closeout rules.

## Delivery And Evidence

The [Delivery Overview](/docs/?scope=studio&doc=d-20260718-120304-2c6f81) determines whether discovered work belongs in the active slice, precedes it, or leaves the refactor. [Testing](/docs/?scope=studio&doc=d-20260718-123734-6d84c1) determines which contracts survive, move once with an owner, or retire. Do not recreate their detailed rules here.

## Guardrails

- Never attach the refactor watcher to the real external-local source root while the stable watcher or management service can write there.
- Never treat an isolated external copy as a second live source of truth.
- Never hand-merge generated payloads before resolving their source and configuration.
- Never make production remote writes from packaging or standalone experiments.
- Never close a refactor checkpoint that silently constrains an open feature proposal; record the impact or resolve the decision first.
