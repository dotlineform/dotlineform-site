# Local Tree Move Projection Delivery

## Outcome

A successful existing single-document drag/drop updates the browser's index model and rendered tree from the server's committed move result without refetching or redrawing the complete index.

The displayed doc, unrelated nodes, expansion state, and other local interaction state remain intact. This is a shipped improvement to the current drag/drop workflow in its own right.

## Completion

The delivery is shipped:

- the committed `doc_id` and `parent_id` response drives one focused shared model and DOM projector;
- old and new child collections are reconciled and sorted with the existing comparator;
- the mounted `<li>` subtree is detached and reinserted without replacing its local node state;
- affected parent chrome, the destination trail, displayed ancestry, and document info are re-projected;
- the moved payload plus search and recent caches are invalidated without an eager read;
- a reconciliation failure reports the committed-state mismatch and performs one explicit selected-document index recovery;
- focused model, DOM identity, workflow, server-response, public-boundary, syntax, and site-manifest checks pass;
- a real manage-route drag/drop proves the runtime composition seam sends one move request, preserves the URL and displayed document, retains mounted node identities, and performs no post-commit `index-tree` read.

## Replaced Behavior

The move service already validated the requested parent, wrote source, rebuilt affected docs/search data, and returned the committed `doc_id` and `parent_id`. The browser then called its general index reload path, refetched `index-tree`, rebuilt every index map, cleared the navigation DOM, and changed the route to the moved document.

That broad refresh was implementation convenience rather than a requirement of the mutation. The shipped projection removes it from the normal successful move path.

## In Scope

- keep the server-first validation, source write, rebuild, and committed response boundary;
- use the committed move record as the browser projection input after success;
- update the moved document's parent in the browser index model;
- update and deterministically sort the old and new parent child collections;
- detach and reinsert the existing rendered `<li>` subtree when it is currently mounted rather than recreating unrelated rows;
- update old/new parent child-list and expand/collapse chrome when either gains or loses its first/last child;
- expand only the destination trail needed to reveal the committed result while preserving unrelated expansion state;
- keep the displayed document unchanged and refresh its breadcrumb/info projection only when the moved ancestry affects it;
- invalidate affected document, search, and recent caches without eagerly reloading them;
- preserve stable node identity so local state within unaffected and moved subtrees survives;
- add focused model, renderer/projection, action-workflow, service-response, and route-boundary evidence.

The browser applies the patch only after the server reports a successful committed mutation. This is not optimistic UI and needs no rollback path.

## Normal Execution Path

```text
single drag intent
  -> server validates destination
  -> source write + targeted docs/search rebuild
  -> canonical move record returned
  -> browser patches document and children models
  -> existing subtree moves to its sorted destination
  -> affected parent chrome and displayed metadata re-project
```

No successful move performs an index read or complete sidebar render.

If the committed result cannot be reconciled with the current browser model, report the projection failure and perform one explicit authoritative index recovery. Recovery is not the normal success path and must not hide a reproducible projection bug.

## Not In Scope

- checkbox selection, selection-mode controls, selected counts, or group drag/drop;
- plural service payloads, effective selected roots, or multi-document validation;
- local projection for create, delete, rename, metadata, viewability, import, scope lifecycle, or external file changes;
- an all-mutations patch framework, client/server revision protocol, background synchronization, transactional browser store, or speculative concurrency machinery;
- optimistic movement before server commit;
- visual redesign of the index or drag/drop target behavior;
- compatibility paths that run both local projection and a routine full refresh.

Those boundaries keep this as one current single-move improvement rather than a general incremental-update project.

## Code Authority To Inspect

- management drag/drop interaction and action workflow under `docs-viewer/runtime/js/management/`;
- shared document-index state, tree ordering, sidebar rendering, route state, and app-session owners under `site/docs-viewer/runtime/js/shared/`;
- move planning and committed mutation response under `docs-viewer/services/`;
- focused drag/drop, action-target, tree, management mutation, and route/runtime tests under `docs-viewer/tests/`.

These are responsibility areas, not a permanent file inventory. Extract one focused projection owner if the current modules cannot hold the complete model/DOM update coherently.

## Done When

- the current single-document move retains server validation, atomic source writing, and targeted rebuild behavior;
- the successful response contains the canonical records needed for local projection;
- the browser model reflects the new parent and deterministic sibling order;
- the existing rendered subtree moves without replacing unrelated nodes;
- old/new parent controls correctly reflect their resulting child state;
- the displayed doc and unrelated expansion state do not change because another node moved;
- breadcrumb/info state updates when the displayed doc belongs to the moved subtree;
- affected payload/search/recent caches are invalidated without an eager index reload;
- the normal successful path neither reads `index-tree` nor runs the complete sidebar renderer;
- explicit recovery is covered separately from normal success;
- focused evidence passes and the durable owner below describes the shipped execution path.
