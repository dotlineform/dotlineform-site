# Tag Editor

## Capability

`/analytics/series-tag-editor/?series=<series_id>` edits one series assignment and its per-work override deltas.

Series tags are inherited by every work in the series. A work row stores only additional tags; the service removes duplicates of inherited series tags. Empty work overrides are removed unless the workflow explicitly keeps the row.

## Execution Path

```text
series query ID
  -> load public series/work indexes
  -> load registry, aliases, groups, assignments
  -> build series + work editor state
  -> tag/weight/alias-assisted changes
  -> one Save action
     -> direct allowlisted API writes when safe
     -> otherwise stage one versioned offline series snapshot
```

The header/media area is catalogue context. Canonical writes are limited to Analytics assignments.

## Save Modes

### Direct

When the Local Analytics service is reachable and the series has no staged offline row, Save posts the changed series row and work deltas. Multiple writes share one initiating activity context. The server validates IDs, weights, membership, inheritance, and canonical source before atomic replacement.

### Offline

When the service is unavailable—or the same series already has staged work—Save records a base snapshot plus staged row in browser storage. It does not silently switch back to direct writes while unresolved staged work exists. Reconciliation happens through [Series Tags](/docs/?scope=studio&doc=d-20260331-000000-974bf0).

If a direct save fails, the editor reports the failure and requires another explicit Save to stage offline.

## Durable Rules

- series ID is required and slug-safe; work IDs use the catalogue identity contract;
- manual weights are the fixed values enforced by `tag_source_model.py`;
- one tag ID appears at most once in a row;
- aliases help resolve input and may preserve historical context, but canonical tag ID remains authoritative;
- effective work tags are derived, not stored as a complete copy.

## Code Map

- page/context boot: `series-tag-editor-page.js`
- editor coordinator: `analytics-tag-editor.js`
- domain/state/render/suggestion modules: `analytics-tag-editor-*`
- save policy: `analytics-tag-editor-save-controller.js`, `analytics-tag-editor-save.js`
- offline format: `tag-assignments-offline*.js`
- server planner: `tag_assignment_service.py`

The route root follows shared Analytics ready/busy state; selected-work and editor/empty modes are route-level state, not persistence authority.

## Weak Spots

- The editor coordinates catalogue context, tag search, inheritance, weights, direct writes, and offline staging; module splits reduce file size but not conceptual breadth.
- Public catalogue indexes are a generated dependency and can disagree with canonical assignment state.
- Offline and direct modes deliberately gate each other, so save-mode bugs can strand valid staged work or bypass conflict review.
