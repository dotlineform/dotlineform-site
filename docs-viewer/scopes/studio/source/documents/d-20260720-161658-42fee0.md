# Public Catalogue Design

## Product Model

The public catalogue is a quiet, media-led way to browse dotlineform Works.

- **Works** are the primary objects readers view.
- **Series** organise related Works and provide their order.
- **Work details** remain nested parts of one Work rather than independent catalogue entries.
- **Recent** and **search** are alternate entrances into the same canonical Work and Series routes.

`/series/` is the Works home. It shows Series groupings first and the ordered Works inside one Series after selection. A Series containing one Work links straight to that Work.

## Navigation Model

- Top-level navigation moves between Works and the public Docs Viewer surfaces.
- A selected Series supplies Work ordering and previous/next context.
- Work links preserve Series context when it is useful for navigation back to the group.
- A detail links back to its parent Work and preserves relevant Work/Series context.
- Required return paths are visible links; browser history is supplementary.

Public URL construction belongs to the shared catalogue URL helper. Route modules should not invent their own path or query conventions.

## Presentation Model

- The site shell stays visually restrained so Work media and catalogue information dominate.
- Indexes offer list and image-grid views over the same records.
- Year and title sorting change presentation, not catalogue authority.
- Selected Works combine primary media, compact metadata, optional downloads/links, detail groups, and optional prose.
- Selected details retain the same media-and-metadata language as their parent Work.
- Light and dark themes share one token-led presentation.
- Content measure, image sizing, wrapping, and toolbar layout adapt to narrow screens without creating a separate mobile product.

View, sort, and page preferences may persist in local storage. Selected records and navigation context belong in URLs when they must survive a direct link.

## Accessibility Model

- Route shells retain semantic headings even when a visual heading is unnecessary.
- View and sort controls expose pressed state and useful labels.
- Keyboard focus remains visible.
- Images carry meaningful alternative text derived from catalogue titles.
- Navigation controls use links for destinations and buttons for local view state.
- Visual/manual review remains necessary for responsive fit, reading order, contrast, and media presentation.

## Frontend Ownership

- `site/assets/css/main.css` owns tokens, base typography, global layout, shared site chrome, and older route-specific rules.
- `site/assets/css/catalogue.css` owns the newer public catalogue components used across index, Work, and detail routes.
- `site/assets/js/catalogue/routes/` owns page-level data and interaction state.
- `site/assets/js/catalogue/components/` owns reusable rendering primitives.
- `site/assets/js/catalogue/navigation/` owns catalogue traversal behaviour.
- `site/assets/js/catalogue/shared/` owns route-neutral helpers within the public catalogue runtime.

Place a rule or behaviour with the narrowest owner that describes it. Do not move code into a shared layer until more than one real consumer uses the same contract.

## Change Rule

When changing the public experience, preserve the distinction between:

- catalogue meaning, owned by canonical source and projections;
- route/navigation state, owned by URL and route modules;
- reusable rendering, owned by components;
- site-wide presentation, owned by global CSS;
- catalogue-specific presentation, owned by catalogue CSS.

Keep this document at the level of product and design invariants. Exact selectors, dimensions, labels, and module APIs belong to code and focused tests.

## Known Weak Spots

- Global and catalogue-specific CSS ownership is only partly separated.
- Repeated static shell markup makes site-wide visual changes a copy-together operation.
- The Works product name and `/series/` implementation name expose historical structure.
- Persisted view/page state and URL-carried navigation state are not yet one completely explicit restoration model.
