# Document Diagrams

## Purpose

Docs Viewer uses Mermaid for authored architecture and workflow diagrams that benefit from durable, reviewable source.

Canonical `.mmd` is build source. The registered Docs media producer renders it to a same-basename sanitized SVG, and documents display that SVG as ordinary media. In this shipped static-media workflow, readers never parse Mermaid or load Node, npm, Puppeteer, Mermaid JavaScript, or source files.

Finished SVG and Mermaid source share the source editor's **Add image…** action but keep different contracts. A staged `.svg` is already-finished vector media; a staged `.mmd` is retained as canonical source and rendered before its SVG token is returned.

That static-media workflow is shipped and remains the only supported path for public and external-local scopes. [Inline Mermaid Rendering](/docs/?scope=studio&doc=d-20260720-102658-b55348) is the shipped repo-backed local-scope alternative: a `scope_type: local` reader renders a Mermaid fence whose source remains directly in canonical Markdown. It creates no `.mmd` media, SVG output, or export conversion.

## Dependency And Registration

The feature-local Node package root is `docs-viewer/build/mermaid/`:

- `package.json` pins the direct Mermaid CLI dependency;
- `package-lock.json` is the resolved dependency authority;
- `mermaid-config.json` owns shared renderer settings;
- `node_modules/` is local, ignored installation output.

Install the locked dependency tree from the repository root:

```bash
npm ci --prefix docs-viewer/build/mermaid
```

[Runtime Dependencies](/docs/?scope=studio&doc=d-20260423-204839-111501) owns Node package-root policy. [Local Setup Toolchain](/docs/?scope=studio&doc=d-20260519-210908-2f8f00) owns the supported Node/npm baseline and environment checks.

Registration has two layers:

1. `docs-viewer/build/docs_builder/media_builds.py` allowlists the code-owned `mermaid` producer implemented by `docs-viewer/services/docs_mermaid_media.py`.
2. Each participating scope opts in through reciprocal source and published-media records in `docs-viewer/config/scopes/docs_scopes.json`.

```json
"source": {
  "build_media": {
    "mermaid": {
      "path": "media/mermaid",
      "producer": "mermaid",
      "publishes_to": "svg"
    }
  }
}
```

The same scope's published `svg` record must declare:

```json
"build_inputs": ["mermaid"]
```

Create `<scope source root>/media/mermaid/` as the visible canonical-source location. Sub-scopes do not inherit their parent's registration; configure one separately only when it independently owns diagrams and an SVG publication contract.

## Author Workflow

For initial intake through Docs Viewer:

1. save an accessible `.mmd` as a direct child of shared import staging;
2. open the target document's Markdown source editor and choose **Add image…**;
3. select the Mermaid source and provide useful Markdown alt text;
4. allow the service to render in temporary storage, publish and verify canonical source plus SVG, and insert the SVG token;
5. review the dirty Markdown buffer and choose **Rebuild doc**.

A render or publication failure inserts no token and writes no document source. Staged input remains in place after success.

For later edits, view the rendered document and open Document Info. Its manage-only **Diagrams** section lists verified diagram-title links. Choosing one delegates the logical source identity to VS Code without exposing a physical path. Save the canonical `.mmd`; Docs Watcher renders only that changed identity. The Markdown token and search index do not change.

Public routes never show these source links. Markdown source mode retains its semantic-reference interface rather than becoming a Mermaid editor.

## Repository Workflow

Repository authors and Codex can work directly:

1. write canonical source to `<scope source root>/media/mermaid/<diagram-name>.mmd`;
2. add the same-basename SVG token to the owning Markdown document;
3. run the targeted document build:

```bash
$HOME/miniconda3/bin/python3 docs-viewer/build/build_docs.py --scope <scope> --only-doc-ids <doc-id> --write
```

The producer runs before the selected document payload is rendered. A full Docs build renders every configured Mermaid identity in the scope. The watcher, staged intake, targeted builds, and full builds all call the same registered producer.

## Source And Publication Contract

Use lowercase kebab-case identities:

```text
<scope source root>/media/mermaid/<diagram-name>.mmd
<configured published svg root>/<diagram-name>.svg
```

Markdown uses the scope's ordinary SVG media token:

<pre><code>![Useful diagram alt text](&#91;&#91;media:docs/&lt;scope&gt;/svg/&lt;diagram-name&gt;.svg&#93;&#93;)</code></pre>

Each scope owns its source and derived SVG. Do not share canonical source across scopes, edit generated SVG by hand, or register an unversioned global `mmdc` executable.

The repository-pinned renderer uses a neutral theme, white background, strict security, `htmlLabels: false`, and an explicit 1200 by 800 browser viewport. Every source requires non-empty Mermaid `accTitle` and `accDescr`; every Markdown reference requires useful alt text. The shared SVG sanitizer must preserve a responsive `viewBox`, accessibility metadata, and visible content before publication succeeds.

## Code Authority And Extension

- dependency and renderer settings: `docs-viewer/build/mermaid/`
- producer implementation: `docs-viewer/services/docs_mermaid_media.py`
- producer registry and full/targeted invocation: `docs-viewer/build/docs_builder/media_builds.py`
- scope activation and locations: `docs-viewer/config/scopes/docs_scopes.json`
- staged author intake: `docs-viewer/services/docs_staged_media_service.py`
- rendered source discovery/opening: `docs-viewer/services/docs_diagram_source_service.py`
- changed-identity rebuild: `docs-viewer/services/docs_live_rebuild_watcher.py`
- focused contracts: `docs-viewer/tests/python/test_docs_mermaid_media.py`, `test_docs_staged_media_service.py`, `test_docs_diagram_source_service.py`, and `test_docs_live_rebuild_watcher.py`

To add another build-source type, register one code-owned producer, configure its confined source path and published target, add the reciprocal published `build_inputs` declaration, and test full builds, targeted builds, intake, watcher invalidation, publication verification, and reader-runtime exclusion together.

## Inline Local Workflow

In any configured repo-backed `local` scope, author an accessible fenced `mermaid` block directly in the document when no static/public consumer needs rendered bytes. The generated payload retains `<pre><code class="language-mermaid">`; after the ordinary document mount, the manage reader finds those fences, lazily loads the checked Mermaid `11.16.0` browser asset once for the application session, and replaces each successful fence with an inline SVG host. Several fences render sequentially. Navigation invalidates late results, and a failed fence retains its source with a concise visible status while technical detail stays in the console.

The exact active `scope_type` is the eligibility rule. `public` and `local_external` readers leave the fence as readable code and do not load the adapter or browser asset. A local document without a Mermaid fence also loads no Mermaid asset. Static HTML export retains the generated code block without a runtime bundle or generated SVG; the Markdown document-content transform retains the `mermaid` fence language and source.

Runtime authority is `site/docs-viewer/runtime/js/shared/docs-viewer-inline-mermaid.js`, composed only by the manage entrypoint and invoked through the ordinary document controller. Shared content CSS owns the responsive neutral host and failure treatment. [Inline Mermaid Rendering Delivery](/docs/?scope=studio&doc=d-20260720-102658-7de33d) records the shipped dependency, scope, runtime, export, and browser evidence. [Diagram Detail View](/docs/?scope=studio&doc=d-20260720-120201-343f33) separately owns any future control for inspecting a detailed diagram.
