# Generated Payloads Report

This document describes a possible future report for [Admin Checks Operational Reports](/docs/?scope=studio&doc=d-20260610-225919-188d54).

## Summary

Report id: `generated-payloads`

Purpose: inventory generated JSON payload size, count, and shape evidence for selected target surfaces.

Primary review question:

```text
Which generated payloads are large, broad, or structurally complex for the selected target?
```

## Inputs

- selected scope, family, area, and route filters from the run manifest
- generated roots allowlisted in checks config
- target-map route and area data where generated roots clearly support one route or area

Possible options:

| Option | Default | Purpose |
| --- | --- | --- |
| `limit` | `20` | Maximum payloads shown per Markdown section. |
| `max_sample_keys` | `20` | Maximum top-level keys kept per payload. |

## Output

Artifacts:

```text
var/admin/checks/<run-id>/generated-payloads/
  report.json
  report.md
  report.csv
```

Required JSON fields:

- payload file count and total bytes
- counts and bytes grouped by generated root
- largest payloads
- JSON validity, top-level type, top-level keys, and array item counts
- per-payload `path`
- per-payload `bytes`
- per-payload `top_level_type`
- per-payload `top_level_key_count`
- per-payload `array_item_count`
- per-payload `target_match`

The stable join key for payload consumers should be repo-relative `path`.

## Markdown Shape

The Markdown should show payloads that are likely to matter during review.

Sections:

- summary counts
- largest payloads
- payloads with broad top-level shapes
- invalid JSON payloads

Example:

```text
Largest payloads
Payload                           Size KB  Type    Items
--------------------------------  -------  ------  -----
docs/studio/by-id/dev-home.json       184  object      -
```

Likely target layers:

- scope
- area
- route where generated payload roots clearly support one route

## Calculation Method

The report should read generated JSON files only from allowlisted generated roots.
It should not rebuild payloads or infer app behavior from generated content.
For public scopes, it should inspect local published output, not public snapshots, unless a later option explicitly asks for public-output comparison.

## Verification

- fixture generated roots with valid and invalid JSON
- bounded largest-payload ordering tests
- safe generated-root allowlist validation
- Markdown output with no wide Markdown tables

## Non-Goals

- no rebuild of generated payloads
- no schema migration or cleanup
- no full schema validation in v1
- no Markdown source-document scan
