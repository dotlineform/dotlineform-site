# Package Return Review

## Boundary

This workflow takes a returned document package from `import-staging/`, proves its prepare-run identity through [Export Metadata](/docs/?scope=studio&doc=d-20260627-113800-55dd63), validates every returned row, and prepares one complete read-only Docs Review package.

It does not mutate canonical documents. The normal Docs Viewer **Import** Action is the only returned-package apply path.

## Execution Path

```text
staged filename
  -> export_id + internal metadata
  -> profile/format/scope validation
  -> parse all returned rows
  -> validate content and package-local hierarchy
  -> immutable validated Docs Review projection
  -> optional later whole-package Docs Viewer Import
```

The review list treats the staged package as stable for the modal interaction: the records found by its scan are the records prepared for review. It does not add changed-since-list tokens or snapshot reconciliation.

## Docs Review Projection

For a supported `document-content` package, validates the complete staged file and atomically publishes an immutable folder under `import-preview/<metadata-derived-id>/` with:

- `docs_review_validated_package_v1` manifest/provenance/diagnostics;
- one safe Markdown source projection per valid returned record;
- generated read-only Docs Review output.

Allowed projected front matter is limited to title, parent, summary, and viewable. Package-local cycles/unsafe identities are fatal; a parent outside a compact returned selection is removed from the projection and recorded as a warning.

The projection is rendered-derived text only. It is not registered as a normal Docs Viewer scope and does not mutate canonical source. Docs Review has no Import control or query-string handoff. To accept a returned package, return to Docs Viewer and choose the normal **Import** Action; it discovers the trusted staged package independently.

## Atomic Return Contract

Prepare fixes the package document set. Returned rows are inspection-only: the direct document-package service rejects `record_indices`, and review, publication, and Docs Import operate on the complete package.

Trusted export metadata must contain `selected_doc_ids`. Returned identities must equal that prepared set without regard to row order. Missing, extra, duplicate, malformed, or base-invalid rows block every downstream action; the external service must regenerate the package as a whole.

Docs Import plans each valid record as a create or overwrite and asks for one package-level confirmation. A malformed record, missing parent, or hierarchy cycle blocks the package; there is no record skip or partial-selection path. Returned content, summary, parent, and viewability values use the normalized import contract, while omitted fields preserve current canonical values on overwrite.

## Assurance And Weak Spots

- Metadata/profile identity is required before parsing; no row-shape fallback.
- Source writes have no document-package backup artifact; use Git/external recovery.
- Import confirmation revalidates the package identity and server-planned create/overwrite actions; the browser does not author target actions.
- Compact content can omit source-only constructs and media; inspect the generated review before confirming canonical import.
- The lower-level full-source normalizer is not an enabled returned workflow.

Exact blockers and field normalization belong to `review_sources.py`, `docs_import_document_package_content.py`, `docs_import_collection_plan.py`, `docs_import_collection_apply.py`, and their tests.
