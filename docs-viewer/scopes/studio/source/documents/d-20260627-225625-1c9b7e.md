# Package Return Review

## Boundary

This workflow takes a returned document package from `import-staging/`, proves its prepare-run identity through [Export Metadata](/docs/?scope=studio&doc=d-20260627-113800-55dd63), parses candidate rows, and produces evidence or a narrow confirmed mutation.

It is not a general document importer and does not infer intent from filenames or fields.

## Execution Path

```text
staged filename
  -> export_id + internal metadata
  -> profile/format/scope validation
  -> parse all returned rows
  -> compare with current Docs Viewer source
  -> normalized review rows + issues
  -> one of:
       Markdown review evidence
       immutable validated Docs Review projection
       dry-run/confirmed summary or hierarchy apply
```

Returns may be sparse. Rich export context need not be echoed; an action reads only its candidate field. Omitted body content means no content change for narrow apply.

## Review Outputs

### Record Review

Writes a focused Markdown snapshot under `import-preview/` for the complete validated package. It is evidence, never apply input.

### Content / Docs Review Projection

For a supported `document-content` package, validates the complete staged file and atomically publishes an immutable folder under `import-preview/<metadata-derived-id>/` with:

- `docs_review_validated_package_v1` manifest/provenance/diagnostics;
- one safe Markdown source projection per valid returned record;
- generated read-only Docs Review output.

Allowed projected front matter is limited to title, parent, summary, and viewable. Package-local cycles/unsafe identities are fatal; a parent outside a compact returned selection is removed from the projection and recorded as a warning.

The projection is rendered-derived text only. It is not registered as a normal Docs Viewer scope and does not mutate canonical source. Docs Import may later consume the associated staged package through an explicit collection plan.

## Narrow Apply Actions

- `summary_apply` considers returned `summary` only.
- `hierarchy_apply` considers returned `parent_id` only.

Both reparse the complete staged input, validate current targets, plan without writing, require confirmation, then write through Docs Viewer source/rebuild helpers. Unknown/missing targets and invalid/self parents block the focused action.

They do not apply body content, create documents, create missing parents, or accept unknown fields.

## Atomic Return Contract

Prepare fixes the package document set. Returned rows are inspection-only: the direct document-package service rejects `record_indices`, and review, publication, apply, and Docs Import handoff operate on the complete package.

Trusted export metadata must contain `selected_doc_ids`. Returned identities must equal that prepared set without regard to row order. Missing, extra, duplicate, malformed, or base-invalid rows block every downstream action; the external service must regenerate the package as a whole.

Supported action fields may remain sparse. An omitted field or unchanged valid value is a no-op, while an invalid present value blocks that whole action. Existing action-specific current-source checks remain authoritative.

## Assurance And Weak Spots

- Metadata/profile identity is required before parsing; no row-shape fallback.
- Source writes have no document-package backup artifact; use Git/external recovery.
- Apply-time parsing prevents an old preview object becoming write authority, but source freshness still needs action-specific comparison.
- Compact content can omit source-only constructs and media; replacing canonical content is a later explicit Docs Import decision.
- The lower-level full-source normalizer is not an enabled returned workflow.

Exact skip/warning/error counts and field normalization belong to `docs_document_packages/apply_*`, `review.py`, `review_sources.py`, and their tests.
