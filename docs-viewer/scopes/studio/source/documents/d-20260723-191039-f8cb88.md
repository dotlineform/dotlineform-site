# Public Docs Viewer Mobile Experience

## Purpose

Make the live public Docs Viewer genuinely usable on mobile. This is a public-reader feature, not a claim that the Python-backed Admin or local management applications should run on a phone.

## Problem

Public Docs Viewer routes are real web surfaces, but their scope identity, document index, search, reading area, information controls, and return paths were composed primarily around desktop space. Showing a complete scope on a narrow screen is an information-architecture and interaction problem, not just a CSS breakpoint.

The local UI Workbench's `390 × 844` condition remains useful for constrained-layout stress. It does not prove public routing, touch use, browser chrome, navigation continuity, or actual-device behaviour.

## Direction

- Review representative public scopes and the tasks people must complete on a phone before choosing a navigation pattern.
- Decide how scope identity, document hierarchy, search, current-document context, information, and return navigation remain understandable when the index cannot stay beside the document.
- Include long documents, deep trees, diagrams, media, links, theme, zoom, orientation, touch targets, safe areas, and browser back/forward behaviour.
- Preserve the public read-only boundary and existing desktop experience.
- Verify the checked public deploy artifact in representative mobile browsers and devices; do not treat a resized desktop management app as equivalent evidence.

## Boundary

This feature owns investigation and a coherent mobile reader outcome. It does not add mobile support to local Admin, management, Import, Docs Review, or Python service workflows. It does not yet approve a drawer, bottom navigation, scope switcher, application rewrite, or implementation delivery.

## Planning

[Docs Viewer Roadmap](/docs/?scope=studio&doc=d-20260714-234030-1e0af6) owns scheduling. Create a delivery only after the review establishes the smallest complete public-mobile outcome.
