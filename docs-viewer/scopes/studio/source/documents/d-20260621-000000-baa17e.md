# Documents Adapter Bridge

## Role

This is a temporary DSO-1 compatibility bridge for the existing Analytics browser routes. It translates legacy Data Sharing requests into the direct Docs Viewer document-package service and restores only the response annotations the old browser expects.

The bridge does not own document selection, package creation, returned-package review, apply, source, or collection import. Those behaviors now live under `docs-viewer/services/docs_document_packages/`, while Docs Import owns create/overwrite/skip into a configured collection. DSO-3 deletes this bridge with the old routes; it is not a compatibility commitment.

## Structure

```text
adapter.py             handler wiring
context.py             scope/dependency/path validation
prepare.py             legacy prepare delegation
returned.py            legacy list/review/apply delegation
families/documents.py  request/response translation only
       |
docs-viewer/services/docs_document_packages/
  direct service, source context, package, validation, review, apply, write/rebuild
```

## Source And Scope

The direct service uses `scope`. The bridge translates legacy `selection.docs_scope` while the old UI remains. Selectable records and export content come from source Markdown/front matter through Docs Viewer source/rendering context.

Generated tree, by-id, search, Recent, report, or metadata payloads are never fallback canonical input.

## Prepare

`docs-viewer/config/document-packages/profiles.json` chooses the supported record shape, content/container formats, selection rules, field mappings, context, limits, and returned-import eligibility. The bridge converts legacy `config_id` and nested selection into direct `profile_id`, `doc_ids`, and `select_all` fields.

## Returned Path

The Docs Viewer document-package service:

- lists staged files through trusted export metadata;
- parses returned rows against the metadata-selected profile;
- produces normalized review rows/issues;
- writes focused Markdown review evidence;
- for supported content packages, can publish an immutable validated Docs Review folder;
- previews/applies the configured summary or hierarchy action through Docs Viewer writes and focused docs/search rebuild follow-through.

The validated review folder has `docs_review_validated_package_v1`, a safe metadata-derived ID, source Markdown projection, validation diagnostics, and generated read-only output. It is not a registered Docs Viewer scope and does not mutate canonical documents.

## Apply Boundary

Current actions are intentionally narrow:

- `summary_apply` updates `summary` only;
- `hierarchy_apply` updates `parent_id` only.

Both reparse the complete staged input, validate exact package identity and current source, support dry-run, require confirmation for writes, and use Docs Viewer write/rebuild dependencies. They do not accept returned-row selection, create missing documents, or apply returned body content.

## Known Weak Spots

- `document-content` is reconstructed from rendered-derived Markdown/plain text, so source-only syntax/media constructs may be absent from the review projection.
- `docs_import_document_package_content.py` still recognizes a lower-level `document-full-source` contract even though no enabled returned-import profile owns it.
- The old browser still depends on generic route/config/response shapes until DSO-2.
- Scope fallback exists only in the legacy translation path; direct requests require an explicit configured scope.
- Data Sharing apply and Docs Import are adjacent mutation paths; keep narrow field update distinct from collection import.

## Extension Rule

Do not add document behavior to this bridge. New document-package behavior belongs to an explicit Docs Viewer profile/service contract, and generated Docs Viewer payloads must not become canonical inputs.
