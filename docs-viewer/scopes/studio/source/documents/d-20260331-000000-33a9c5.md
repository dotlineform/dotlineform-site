# Public Catalogue Data Flow

## Route Map

| Public route | Reads | Why |
| --- | --- | --- |
| `/series/` | `series_index.json` | list/grid of Series and their primary thumbnails |
| `/series/?series=<series_id>` | `series_index.json`, `works_index.json` | ordered Work cards for the selected Series |
| `/works/` | `works_index.json`, `series_index.json` | flat Work list and Series labels |
| `/works/?work=<work_id>` | focused Work JSON, `series_index.json` | rich Work content, details, and Series navigation |
| `/work-details/?detail=<detail_uid>` | parent Work JSON | selected detail and sibling/section context |
| `/recent/` | `recent_index.json`, `series_index.json` | publication snapshots and canonical destinations |
| `/catalogue/search/` | catalogue search index | flat search-oriented records |

The exact public route modules live under `site/assets/js/catalogue/routes/`. Static shells live under `site/series/`, `site/works/`, and `site/work-details/`.

## Execution Flow

```text
canonical catalogue source
  -> scoped catalogue generator
  -> aggregate indexes + focused record JSON
  -> optional catalogue search build
  -> static route shell
  -> route module selects query state and fetches the required payloads
```

The browser never treats a public payload as writable source. Studio mutates canonical source; the build projects that source into public runtime shapes.

The generated per-Series JSON family is not a current `/series/` route read. Do not infer a runtime dependency from an artifact being present beneath `site/assets/series/index/`.

## Detail Ownership

There is no public global detail index. `detail_uid` identifies the parent Work, and the Work payload carries sections plus nested details. This keeps ordering and navigation with the aggregate that owns them and avoids a second payload family that must stay synchronized.

## Moments Boundary

`/moments/` is a public Docs Viewer scope. It reads the Moments docs index, by-ID documents, and Moments search payloads—not catalogue Moment records.

The navigation places Works and Moments beside one another, but their source, build, route runtime, and management boundaries are different. The catalogue generator or older docs must not be used to infer a Local Studio Moment capability.

## Change Rule

When a public route needs new data, first decide whether it is:

- list-wide context for an aggregate index;
- selected-record context for a focused payload;
- detail context already owned by the Work aggregate;
- search-only text for the search index;
- or data that belongs to another scope entirely.

Update the serializer, field registry, route module, and focused tests for that boundary. Avoid adding the field to every payload for convenience.
