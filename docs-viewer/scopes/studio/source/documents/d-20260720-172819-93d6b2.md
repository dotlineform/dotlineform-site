# Document Sharing Ownership Cutover Delivery

## Status And Purpose

This is an **active behaviour-preserving ownership cutover**. DSO-1 moved the working document-package service into the repo-integrated Docs Viewer, DSO-2 moved the two browser workflows onto dedicated Docs Viewer routes, and DSO-3 retired the generic Data Sharing subsystem and its Analytics package surfaces. DSO-4 verification and durable-document closeout remain separately gated.

## Outcome

Docs Viewer exclusively owns document-package preparation and returned-package workflows. The generic Data Sharing subsystem, its Analytics routes, and its tag-package behavior no longer exist.

The result remains an integrated-only Docs Viewer capability. It is not part of portable core or Standalone Docs Viewer.

## Product Contract To Preserve

- Prepare selects the complete document set and writes the supported JSON or JSONL package.
- A returned package must contain the complete prepared set with valid identities and content.
- Returned rows are inspection-only. There is no second selection, `record_indices`, partial review, partial publication, or partial apply.
- Missing, extra, duplicate, or invalid returned records block the whole package. The external service regenerates it as a whole.
- Valid unchanged records may produce no-op results; invalid records are not skipped into an otherwise successful package.
- Docs Review publication and supported summary, hierarchy, or Docs Import handoffs keep their current authority boundaries.
- Existing package schema identifiers, profile behavior, and external workspace locations remain stable during this code-ownership cutover.

## Target Ownership

| concern | target owner |
| --- | --- |
| Prepare and Returned Packages browser routes | Docs Viewer local service |
| Browser workflow and presentation | dedicated Docs Viewer route modules, shells, workflow-local modals, and package CSS |
| Fixed document profile configuration | `docs-viewer/config/` |
| Document selection, packaging, validation, review, projection, and supported apply | one integrated-only Docs Viewer document-package service boundary |
| Package I/O, metadata, schemas, and safe workspace paths still required by documents | Docs Viewer document-package services |
| Canonical document reads, writes, rebuilds, collection import, and review publication | existing Docs Viewer, Docs Import, and Docs Review owners |

Do not create a `docs-viewer/data-sharing/` bucket. Package route and module names should describe document packages directly.

#### DSO-1 Shipped Boundary

- Fixed profiles now live at `docs-viewer/config/document-packages/profiles.json`.
- Document selection, export, returned-package parsing, exact-set validation, review, publication, apply, metadata, activity, and workspace resolution now live behind `docs-viewer/services/docs_document_packages/` and `docs_document_package_routes.py`.
- The Docs Viewer local service owns the eight accepted `/docs/packages/*` endpoints. They require management mode and same-origin local access and report availability through `capabilities.document_packages`.
- Direct requests use document terms. Generic app/domain/adapter/operation fields, nested selection, and `record_indices` are rejected.
- Trusted `selected_doc_ids` and returned IDs must be equal as sets. Order may change; missing, extra, duplicate, malformed, or base-invalid records block the package before review, publication, apply, or Docs Import handoff.
- Docs Import, Docs Review, staged media, and management capabilities now import the Docs Viewer document-package workspace and metadata owners. The physical `$DOTLINEFORM_PROJECTS_BASE_DIR/data-sharing/` paths and serialized `data_sharing_*` identifiers remain unchanged.
- The existing Analytics document routes temporarily call the new owner through a thin response/request bridge. The DSO-2 routes do not use that bridge; it remains only for deletion with the old Analytics routes in DSO-3.
- Focused DSO-1 verification covers direct routes/contracts, profile preparation, reordered exact-set returns, package rejection, Docs Review materialization, narrow whole-package apply, Docs Import collection planning/apply, and the temporary Analytics bridge. The standard Docs profile passed 355 Python checks plus Studio docs and search builds.

- [x] Preserve the current workflow, broad layout, and useful workflow-local modal interactions while replacing Analytics transport, config, modal-framework, and route-state dependencies with narrow Docs Viewer owners.

#### DSO-2 Shipped Boundary

- `docs-viewer/shell/docs-viewer-package-prepare.html` and `docs-viewer-package-returned.html` are the two management-only route shells served directly by `bin/docs-viewer` at the accepted trailing-slash browser routes.
- `docs-viewer/runtime/js/packages/` now owns one same-origin client, route entries, document/row rendering helpers, and the workflow-local modal frame. The modules call only the direct `/docs/packages/*` service endpoints and do not load shared Analytics runtime files.
- Prepare presents fixed Docs Viewer scopes and profiles, JSON/JSONL and content-format choices, hierarchical document selection, descendant selection, filter/select-all/clear actions, package preparation, result details, and the profile-context editor. It has no app, domain, adapter, or generic operation state.
- Returned Packages presents a scope, actionable staged packages, blocked-package evidence, automatic whole-package inspection, read-only returned document rows, the fixed Docs Review actions, and the fixed whole-package apply actions. It has no row checkboxes, selected-record state, `record_indices`, or partial action payloads.
- The dedicated `docs-viewer-packages.css` preserves the broad two-workflow panel/list/action shape and responsive behavior without importing or consolidating Analytics CSS.
- Focused DSO-2 verification covers management-only shell hosting, package runtime/static ownership, browser-safe profile context, direct route contracts, and absence of Analytics paths and partial-selection fields. JavaScript and Python syntax checks passed; all 15 focused Python checks passed when the localhost route check was allowed to bind; a live isolated Docs Viewer check loaded 271 Studio documents, opened the context modal, booted the Returned route, and reported no browser console errors; and the standard Docs profile passed 356 Python checks plus Studio docs and search builds.

Prepare records use the server-generated export ID; returned summary and hierarchy applies use the staged filename. The former Analytics/tag activity actions are no longer active contract entries.

## In Scope

- Documents-only JSON and JSONL prepare profiles already supported by the repository.
- Existing returned-package validation, review evidence, Docs Review publication, and supported whole-package apply/import handoffs.
- Direct Docs Viewer routes, APIs, modules, configuration, capability reporting, activity, and focused tests.
- Complete retirement of tag Data Sharing and the now-unnecessary generic subsystem.

## Not In Scope

- An enclosing package modal, toolbar action, combined package screen, or visual redesign. Existing workflow-local modals may remain within the dedicated routes.
- Consolidating document-package CSS into the general Docs Viewer styling system before a Docs Viewer-wide styling review.
- New package profiles, returned formats, canonical mutations, or tag-sharing replacement.
- Changing package schema identifiers or migrating `$DOTLINEFORM_PROJECTS_BASE_DIR/data-sharing/` during this delivery.
- Changing Docs Import collection decisions, Docs Review editing, or ordinary static HTML export.
- Docs Viewer Core Refactor, Standalone Docs Viewer, subscriptions, or product packaging.

## Completion Evidence

- `bin/docs-viewer` alone hosts both document-package workflows and their APIs.
- Current prepare fixtures produce equivalent package content and metadata through the new owner.
- Returned packages are accepted or rejected as complete sets; no partial-selection state survives in UI, requests, services, or config.
- Analytics and the repository source tree contain no live Data Sharing route, registry, adapter, tag-package, or document-package implementation.
- No compatibility path or cross-app proxy preserves the former document routes.
- [Share Document Packages](/docs/?scope=studio&doc=d-20260718-155350-c84e62) and current durable owners receive the shipped boundary before closeout.
