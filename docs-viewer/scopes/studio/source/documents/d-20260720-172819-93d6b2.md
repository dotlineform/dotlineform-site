# Document Sharing Ownership Cutover Delivery

## Status And Purpose

This is a **proposed behaviour-preserving ownership cutover**. It moves the working document-package capability into the repo-integrated Docs Viewer before any larger UI simplification.

## Outcome

Docs Viewer exclusively owns document-package preparation and returned-package workflows. The generic Data Sharing subsystem, its Analytics routes, and its tag-package behavior no longer exist.

The result remains an integrated-only Docs Viewer capability. It is not part of portable core or Standalone Docs Viewer.

## Product Contract To Preserve

- Prepare selects the complete document set and writes the supported JSON or JSONL package.
- A returned package must contain the complete prepared set with valid identities and content.
- Returned rows are inspection-only. There is no second selection, `record_indices`, partial review, partial publication, or partial apply.
- Missing, extra, duplicate, or invalid returned records block the whole package. The external service regenerates it as a whole.
- Valid unchanged records may produce no-op results; invalid records are not skipped into an otherwise successful package.
- Docs Review publication and supported summary, hierarchy, or Docs Import handoffs keep their current authority boundaries.
- Existing package schema identifiers, profile behavior, and external workspace locations remain stable during this code-ownership cutover.

## Target Ownership

| concern | target owner |
| --- | --- |
| Prepare and Returned Packages browser routes | Docs Viewer local service |
| Browser workflow and presentation | `docs-viewer/runtime/` and Docs Viewer-owned CSS/shells |
| Fixed document profile configuration | `docs-viewer/config/` |
| Document selection, packaging, validation, review, projection, and supported apply | one integrated-only Docs Viewer document-package service boundary |
| Package I/O, metadata, schemas, and safe workspace paths still required by documents | Docs Viewer document-package services |
| Canonical document reads, writes, rebuilds, collection import, and review publication | existing Docs Viewer, Docs Import, and Docs Review owners |

Do not create a `docs-viewer/data-sharing/` bucket. Package route and module names should describe document packages directly.

## Proposed Checkpoints

### DSO-0 — Freeze Behavior And Dependencies

- [x] Record current prepare profiles, returned-package validation, review/publication/apply actions, package fixtures, workspace paths, and observable results.
- [x] Map the document-only calls currently split across Analytics, `data-sharing/`, and Docs Viewer services.
- [x] Identify tag-only and generic registry code for retirement rather than migration.
- [x] Confirm the exact Docs Viewer route and endpoint names before adding them.

[DSO-0 — Behavior And Dependency Findings](/docs/?scope=studio&doc=d-20260720-174842-4c2f6a) is the accepted temporary audit record. DSO-1 is ready to begin when separately requested; no implementation has started.

### DSO-1 — Direct Docs Viewer Service Ownership

- [ ] Move document profile configuration and document-package services into a final Docs Viewer-owned integrated boundary.
- [ ] Add fixed document-package capabilities and same-origin endpoints to the Docs Viewer service without a cross-domain registry.
- [ ] Call Docs Viewer source, write, rebuild, import, and review owners directly.
- [ ] Enforce whole-package returned validation and remove partial-record request semantics.
- [ ] Move document-only package I/O, metadata, schema, path, and activity helpers out of generic Data Sharing ownership.

### DSO-2 — Prepare And Returned Routes

- [ ] Move the two browser workflows into Docs Viewer-owned route shells, modules, and CSS.
- [ ] Preserve the current workflow and broad layout while replacing Analytics transport, config, modal, route-state, and style dependencies with narrow Docs Viewer owners.
- [ ] Remove app/domain selection from Prepare and row selection from Returned Packages.
- [ ] Keep route entrypoints thin enough that a later Docs Viewer action or modal can reuse the workflow without another service move.
- [ ] Do not import, proxy, or mount Analytics runtime files from Docs Viewer.

### DSO-3 — Retire Generic Data Sharing

- [ ] Remove tag adapter code, package profiles, dispatch records, returned-package behavior, and tag Data Sharing tests without migrating them.
- [ ] Remove the generic adapter registry, workflow dispatch, and remaining repository `data-sharing/` code after document-required helpers have moved.
- [ ] Remove Analytics Data Sharing routes, API composition, config projection, frontend modules, CSS, and route-specific tests.
- [ ] Retain canonical tag editing and validation tests under Studio ownership; do not replace tag packages with another automated exchange path.
- [ ] Do not add compatibility aliases, redirects, proxies, or dual document routes.

### DSO-4 — Verification And Documentation

- [ ] Run package/profile/service tests against representative JSON and JSONL exports and returned packages.
- [ ] Prove prepare, complete-set validation, Docs Review publication, and supported whole-package apply with Analytics stopped.
- [ ] Prove an invalid returned record blocks every downstream action and no record-selection controls or payload fields remain.
- [ ] Prove Analytics exposes no Data Sharing route or document/tag package capability.
- [ ] Update durable Docs Viewer, Docs Review, Docs Import, Analytics retirement, source-tree, runner, endpoint, and external-workspace documentation.
- [ ] Inventory former Data Sharing documents for explicit closeout; do not delete them without confirmation.

## In Scope

- Documents-only JSON and JSONL prepare profiles already supported by the repository.
- Existing returned-package validation, review evidence, Docs Review publication, and supported whole-package apply/import handoffs.
- Direct Docs Viewer routes, APIs, modules, configuration, capability reporting, activity, and focused tests.
- Complete retirement of tag Data Sharing and the now-unnecessary generic subsystem.

## Not In Scope

- A modal, toolbar action, combined package screen, or visual redesign.
- New package profiles, returned formats, canonical mutations, or tag-sharing replacement.
- Changing package schema identifiers or migrating `$DOTLINEFORM_PROJECTS_BASE_DIR/data-sharing/` during this delivery.
- Changing Docs Import collection decisions, Docs Review editing, or ordinary static HTML export.
- Docs Viewer Core Refactor, Standalone Docs Viewer, subscriptions, or product packaging.

## Completion Evidence

- `bin/docs-viewer` alone hosts both document-package workflows and their APIs.
- Current prepare fixtures produce equivalent package content and metadata through the new owner.
- Returned packages are accepted or rejected as complete sets; no partial-selection state survives in UI, requests, services, or config.
- Analytics and the repository source tree contain no live Data Sharing route, registry, adapter, tag-package, or document-package implementation.
- No compatibility path or cross-app proxy preserves the former document routes.
- [Share Document Packages](/docs/?scope=studio&doc=d-20260718-155350-c84e62) and current durable owners receive the shipped boundary before closeout.

The DSO-0 findings and exact cutover surface were accepted on 2026-07-20. DSO-1 is the next checkpoint but has not started.
