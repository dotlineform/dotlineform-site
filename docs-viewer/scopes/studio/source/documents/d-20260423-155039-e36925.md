# Docs Images And Assets

Use this guide to choose how a document should refer to an image, download, diagram, or interactive asset.

## Choose The Outcome

| goal | use |
| --- | --- |
| Show an existing public-scope image or file. | A `media` token pointing at its R2 object. |
| Show an existing asset in a local or external-local scope. | A `/docs/media/<scope>/...` link. |
| Add a staged image, Mermaid source, or download to the document being edited. | Source editor **Add image…** or **Add file…**; it publishes the required output and inserts a logical reference. |
| Import the contents of HTML, Markdown, text, or a trusted documents package. | Docs Import; it interprets the supported content and writes document source. |
| Add a reproducible authored diagram. | Stage `.mmd` for **Add image…**, or author canonical Mermaid source directly in the configured scope. |
| Keep a small code-owned visual with the explanation. | Inline safe SVG or HTML in the Markdown source. |
| Run a self-contained interactive example. | Published `html` media with an `html-media` sandbox token. |

## Documents And Media Are Separate

Global Docs Import lists only document sources: HTML, Markdown, text, Markdown packages, and trusted document collections. It interprets those sources and creates or overwrites canonical document content.

Ordinary image and downloadable files are media, not document sources. Add them from an active source editor:

```text
example.json -> Add file -> published bytes + link inserted into the current buffer
```

A trusted Data Sharing documents package also uses JSON/JSONL, but it appears in Docs Import only when its package identity and schema are recognized:

```text
trusted documents package -> collection plan -> several canonical documents
```

CSV, TSV, ZIP, ordinary JSON/JSONL, PDF, and supported Office files can be added as opaque downloads. Their contents are not converted into document content. [Media And Asset Handling](/docs/?scope=studio&doc=d-20260514-184303-7914e2) owns the complete capability distinction and code pointers.

## Link Existing Public Media

Public scopes use literal media tokens so source does not hardcode the shared media origin.

Image:

<pre><code>![Example](&#91;&#91;media:docs/library/img/example.jpg&#93;&#93;)</code></pre>

SVG image:

<pre><code>![Diagram](&#91;&#91;media:docs/library/svg/diagram.svg&#93;&#93;)</code></pre>

Download:

<pre><code>[Download data](&#91;&#91;media:docs/library/files/example.json&#93;&#93;)</code></pre>

An image token may include positive integer dimensions:

<pre><code>![Example](&#91;&#91;media:docs/library/img/example.jpg width=800 height=600&#93;&#93;)</code></pre>

To show a literal token in documentation, encode each bracket as <code>&amp;#91;</code> or <code>&amp;#93;</code> inside a raw HTML code element. Token expansion runs before Markdown rendering, so backticks and fenced code blocks do not protect token text.

The docs builder replaces the token with `<media_base>/docs/<scope>/<class>/<filename>` before rendering. The token does not upload the object or prove that it exists.

For a manually staged public asset, use the dry-run-first [Publish Media To R2](/docs/?scope=studio&doc=d-20260508-093841-3e27b1) command. Prefer a new filename when a replacement must appear immediately because Docs media has no cache-version contract.

## Link Existing Local Media

Repo-backed and external-local scopes use the confined local media route:

```md
![Example](/docs/media/studio/img/example.png)
![Diagram](/docs/media/studio/svg/diagram.svg)
[Download data](/docs/media/studio/files/example.json)
```

Place the file at the configured published location. Repo-backed local scopes normally use `docs-viewer/scopes/<scope>/published/media/<type>/`; external-local scopes use `$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/scopes/<scope>/published/media/<type>/`. The local service resolves the URL through scope configuration and rejects traversal and symlink escapes.

Do not put local design documents or working references in `site/`; that tree is the public deploy artifact.

## Add A New Image Or Download

1. Put the source file in `$DOTLINEFORM_PROJECTS_BASE_DIR/data-sharing/import-staging/`.
2. Open the target document's Markdown source editor.
3. Choose **Add image…** or **Add file…**, select the staged file, and enter useful alt text or a link label.
4. Choose **Add image** or **Add file**. New or byte-identical media proceeds directly; changed bytes using the same identity prompt **Replace** or **Cancel**.
5. Review the inserted Markdown, then use **Rebuild doc** to commit the source edit.

The target scope decides what happens to the bytes: configured public media can publish to R2, while repository and external-local media publish to their configured roots. The editor never asks the author to choose a provider.

Publication completes and is byte-verified before the logical Markdown reference is inserted. The action never creates or replaces a document, and it leaves the staged file in place. Cancelling or abandoning the later source edit can therefore leave unreferenced media, but cannot commit a link to failed publication.

For Mermaid `.mmd`, **Add image…** renders in temporary storage first, then writes and verifies the canonical source under the scope's configured `media/mermaid/` root and its same-basename published SVG. The inserted Markdown always references the SVG. After intake, use the diagram-title link in rendered Document Info's **Diagrams** section to edit the canonical source; Docs Watcher republishes only that changed identity, so the existing token and document source do not change.

The write-free publication preview is internal. Authors see another modal only when changed bytes require replacement confirmation or SVG sanitization removed unsafe content that should be reviewed.

Markdown package imports are different: one Markdown source becomes one document, package images are converted to readable WebP outputs, and supported attachments are copied and linked. HTML and Markdown imports can also extract raster data URLs written as Markdown images. [Docs Import](/docs/?scope=studio&doc=d-20260424-000000-fc8b5d) owns those operator workflows.

## SVG, Mermaid, And Interactive Assets

Use **Add image…** for new SVG. It sanitizes the staged SVG and publishes it through the scope's configured `svg` media class before inserting an image reference. Docs Import uses the same sanitizer and class when it extracts inline SVG from imported HTML. Historical SVG tokens beneath `img` and manually authored inline SVG remain supported; they are not migrated automatically.

Use Mermaid when the diagram should remain reproducible and editable as source. Canonical `.mmd` is build media rather than reader media; the registered producer renders, sanitizes, publishes, and byte-verifies the SVG. Readers load only that static SVG and never receive Mermaid, Node, or Puppeteer runtime code.

HTML is stored once at the scope's configured published `html` root, regardless of whether it is interactive. Embed it with a logical token such as:

<pre><code>&#91;&#91;html-media:docs/library/html/example.html height=546&#93;&#93;</code></pre>

The renderer resolves that identity through scope configuration and emits a lazy iframe with `sandbox="allow-scripts"`. HTML is not served through `/docs/media/` and role-marked interactive companions are not selected as ordinary document sources.
