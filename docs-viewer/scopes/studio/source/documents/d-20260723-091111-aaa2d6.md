# Docs Viewer Modal Consistency

## Purpose

Make Docs Viewer modals behave and feel like one product while keeping each workflow's domain state, validation, requests, and results with its existing owner.

This is a maintenance feature, not a new modal framework or a general visual redesign.

## Dependency

[Docs Viewer Manage Theme Restoration](/docs/?scope=studio&doc=d-20260723-101224-e62173) closes first. It owns the manage route's theme-token regression and initial dark presentation across current overlays. This feature then uses that working baseline to converge modal lifecycle, presentation, and implementation ownership.

## Current Boundary

Docs Viewer currently has three modal composition paths:

- the shared ephemeral management shell used by scope lifecycle, document packages, copy, creation, source-editor, media, confirmation, choice, and notice workflows;
- persistent Metadata, Import, and Settings markup with a separate lifecycle controller;
- a separately rendered Review Sessions modal.

They reuse many class names, but they do not share one complete lifecycle or presentation contract. Current drift includes sizing, action order, scroll locking, disabled and status treatment, focus indication, and Review Sessions styling.

## Direction

- Keep one Docs Viewer-owned modal lifecycle contract for initial focus, focus containment, Escape and backdrop handling, page-scroll locking, focus restoration, and keyboard-focus indication.
- Let that lifecycle manage both dynamically rendered modal frames and persistent workflow-owned markup.
- Keep explicit compact, standard, document, import, and wide presentation variants. Consistency does not require identical dimensions or initial-focus targets.
- Standardize action order, disabled treatment, status states, stale-status clearing, body/footer overflow ownership, responsive fit, and light/dark presentation.
- Treat the restored manage theme tokens as an input; do not reopen foundational route theming inside modal-consistency checkpoints.
- Keep domain workflow bodies and state local. The shared modal owner must not learn about scopes, packages, imports, review sessions, metadata, media, or service endpoints.
- Contribute a permanent Docs Viewer specimen pack to the federated UI Workbench so every meaningful modal state can be inspected without executing the real workflow.
- Use the workbench to classify apparently equivalent controls by interaction contract before sharing implementations. Studio participation and remediation remain a separate delivery.
- Use focused automated evidence for the shared lifecycle and browser integration boundaries. Use temporary or manual browser inspection for visual fit, focus feel, pointer behaviour, copy, and narrow-screen ergonomics.

## Outcome

The feature is complete when every current Docs Viewer modal uses the same lifecycle owner, deliberate style variants cover every modal family, workflow-local behavior is preserved, every meaningful modal state is available through the permanent production-backed workbench pack, and focused plus tactile evidence covers desktop, narrow-screen, light, dark, keyboard, pointer, long-content, busy, validation, and terminal-result states.

The exact modal inventory remains code-owned. Do not turn this feature page into a permanent selector or workflow catalogue.
