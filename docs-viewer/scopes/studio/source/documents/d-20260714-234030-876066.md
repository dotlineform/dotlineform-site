# Index Multiple Selection Concept

## Problem

The manage-mode index currently displays and highlights one selected doc at a time. Reparenting several related documents repeats the same interaction and rebuild work, while future multi-document actions would each be tempted to invent their own target rules.

Multiple selection should be a separate checkbox set. It must not overload the selected-doc highlight, link navigation, or browser modifier-click behavior.

The original drag/drop idea also exposed a wider need: controls in the toolbar, Actions menu, context menu, and selection surface should share a coherent understanding of what an action targets and which capability makes it available. That long-term direction is part of the feature concept even though it should not be implemented as one enormous drag/drop delivery.

## Desired Interaction

- **Selected doc** in the singular means the one document currently displayed. Exactly one row keeps the existing active highlight and `aria-current="page"` state.
- **Selection** or **selected docs** in the plural means the zero-or-more documents whose checkboxes are checked.
- A manage-only **Select** control in the index-view toolbar enters explicit selection mode and reveals row checkboxes.
- Selection mode reveals a checkbox gutter immediately to the left of each document node. The checkbox is a sibling of the node, not content inside its draggable or link surface.
- Revealing the gutter must not move or narrow the node, wrap its title, expand the index panel, or consume space reserved for a node status icon. Checkbox interaction never begins the existing single-document drag/drop; the document node keeps its current drag behavior.
- Document titles remain ordinary links. Clicking or opening a title changes the displayed doc and its single highlighted row without changing any checked boxes. Native browser modifier-click behavior remains available.
- Clicking a checkbox changes only membership of the selection. Shift-clicking checkboxes selects one visible contiguous range from a stable anchor.
- Checked rows receive no additional row-selection highlight; the checkbox is their complete visible selection state.
- The index toolbar shows the selected count and provides **Clear** and **Done**. Leaving selection mode clears the selection so no invisible selection survives.
- The checkbox set has no primary selection. Single-document controls continue to follow the displayed doc or an explicitly owned context target.
- Navigation, incidental row clicks, focus, and context-menu opening do not change a carefully assembled checkbox selection.
- Selection changes remain immediate browser state. Network, mutation, rebuild, and busy state begin only when an action is committed.
- A committed move updates the local tree without a routine full-index refetch, preserving the displayed doc, unrelated nodes, expansion state, and checkbox elements.

Potential consumers now include package preparation, selected export, selected movement, and selected deletion. [Index Multiple Selection Consumer Assessment](/docs/?scope=studio&doc=d-20260722-000158-9c0f3e) compares their value and blast radius before any one receives a delivery. If movement is approved, checked ancestors and descendants remain checked while the move treats only the highest checked ancestors as effective roots so existing subtrees are not flattened.

## Product Boundaries

- Manage-mode `index-tree` only; no public, review, search, recent, graph, or mobile selection contract.
- No modifier-click selection, lasso, touch drag/drop, cross-scope moves, or sibling reordering.
- No implicit selection of descendants.
- No separate selected-row styling or hidden primary checkbox selection.
- No general batch-action framework.
- Destructive or mutation actions stay single-document until their complete multi-document workflow is separately designed.
- Toolbar, Actions-menu, and context-menu controls may place an action differently, but the same action id must resolve the same targets.
- View placement, action targeting, route enablement, and backend authority remain distinct even when they participate in one coherent capability model.

Keyboard range extension, direct toggle shortcuts, batch delete, and other consumers wait for a demonstrated need.

## Delivery

The action-target foundation is already shipped and has its durable owner. This concept assumes a registered index-view control host and a committed local tree-projection boundary are available.

The visible selection foundation is shipped without changing drag/drop or backend mutation contracts. Every consumer remains a separate delivery after an explicit choice.

## Consumer Decisions

The shipped lifecycle contract answers reload preservation and pruning. Consumer-specific questions now belong to [Index Multiple Selection Consumer Assessment](/docs/?scope=studio&doc=d-20260722-000158-9c0f3e) and the delivery chosen from it. Group drag, if delivered, begins only from a checked document; dragging an unchecked document remains the current single-node action.
