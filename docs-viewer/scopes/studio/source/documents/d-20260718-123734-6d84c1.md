# Testing

## Purpose

Test the Core Refactor without turning continuous updates to tests of the old implementation into a second refactor.

The repository [Testing](/docs/?scope=studio&doc=d-20260501-174746-efd581) and [Test Contract Discipline](/docs/?scope=studio&doc=d-20260627-212121-7cf7de) policies still apply. This page adds only the working rules needed by this refactor.

## Rule

Preserve durable behavior, safety, and composition contracts—not the machinery that currently produces them. Test completed delivery boundaries rather than every temporary module arrangement inside a delivery.

## Baseline

Core Definition Analysis must produce an initial test map alongside the capability map:

- record the relevant tests and profiles that currently pass;
- identify existing failures without widening the analysis to repair them;
- map evidence to capabilities and stable contracts rather than file layout;
- identify critical contracts with no useful current evidence.

Do this once as baseline evidence. Do not rerun or rewrite the whole suite after every internal move.

## Existing Test Treatment

| test classification | treatment |
| --- | --- |
| durable product, data, security, or composition contract | preserve |
| valid contract coupled to its current owner | migrate once when that ownership moves |
| implementation-detail assertion | replace with boundary evidence or retire |
| legacy or accidental behavior | retire after confirming it is not required |
| missing critical boundary | add the smallest focused contract test |

A failing historical test is not automatic authority for production behavior. Confirm the contract before changing code. Never add a compatibility alias solely to preserve an old test import, fixture, or constructor shape.

## Per Delivery

Before implementation, each delivery names:

- tests expected to remain unchanged;
- tests that move or are rewritten with an owner;
- tests made obsolete by the completed boundary;
- missing evidence that must be added;
- focused checks required for completion and integration.

During implementation, run syntax/import checks and the narrow tests for the responsibility being changed. Intermediate work-in-progress commits do not need to satisfy every historical suite, but they are not mergeable checkpoints. The complete delivery must leave its declared evidence green and the integrated application working.

If an unexpected test fails, first classify whether it protects the active contract, reveals a separate regression, or encodes retired structure. Do not repair unrelated brittle tests inside the delivery without explicitly widening or splitting the work.

## Stable Test Seams

Prefer evidence that survives module movement:

- pure model, planner, parser, and service inputs and outputs;
- scope, path, write, and security confinement;
- configuration validation and browser-safe projection;
- generated document and search contracts;
- management API requests, responses, and mutation boundaries;
- integrated and standalone composition boot;
- public, manage, review, and asset-authority boundaries.

Avoid permanent tests for temporary import paths, coordinator wiring being replaced, UI choreography, current modal timing, or byte-for-byte snapshots of the entire application.

## Cadence

| point | evidence |
| --- | --- |
| inner development loop | syntax/import check plus the smallest affected contract test |
| ownership checkpoint | affected model, service, config, or generated-data contracts |
| completed delivery | declared delivery evidence plus integrated composition proof |
| merge into `main` | narrowest relevant profile and affected route/service boundary |
| major core milestone | broader integrated parity and isolated standalone-composition proof |
| standalone release work | packaged clean-environment, configuration, upgrade, and external-local proof |

Broad Docs Viewer profiles run at meaningful integration points, not reflexively after every moved function.

## Side-By-Side Proof

Use the stable `main` application as a reference for agreed contracts, not as a byte-for-byte oracle. Compare representative API responses, generated shapes, capability projections, and workflows at integration points. Use permanent browser smoke only when route boot, module loading, assets, or browser/service agreement is the actual boundary; leave visual and tactile behavior to manual checks.

## Status

Proposed. Core Definition Analysis must supply the initial test map. No refactor-specific replacement suite or broad new test harness has been approved.
