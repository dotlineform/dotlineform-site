# Recommended JSDoc Approach for the Repository

> Research doc: not indicative of current implementation.

## Purpose

This guide explains how JSDoc comments relate to JavaScript syntax and recommends a practical way to introduce JSDoc into the repository.

The objective is not to add comments everywhere. It is to make important JavaScript interfaces, data structures, responsibilities, and domain rules easier for:

- developers to understand;
- VS Code to inspect;
- coding agents to retrieve and interpret;
- linting and documentation tools to validate;
- future maintainers to change safely.

---

## 1. What `/** ... */` means

JavaScript has block-comment syntax:

```js
/*
 * This is an ordinary block comment.
 */
```

A JSDoc comment uses:

```js
/**
 * This is a documentation comment.
 */
```

The JavaScript engine does not assign any special meaning to the extra `*`.

To JavaScript, both forms are comments and are ignored at runtime:

```text
/*   JavaScript block-comment syntax
/**  documentation-comment convention recognised by tools
```

The distinction exists because external tools recognise comments beginning with `/**` as structured documentation.

These tools can include:

- JSDoc;
- VS Code;
- the TypeScript language service;
- documentation generators;
- ESLint plugins;
- coding agents;
- static-analysis tools.

For example:

```js
/*
 * Ordinary implementation comment.
 */
function resetCache() {
  // ...
}
```

```js
/**
 * Clears the in-memory work index without deleting generated files.
 */
function resetWorkIndexCache() {
  // ...
}
```

Both are valid JavaScript comments. The second can be associated with the function as documentation.

---

## 2. What JSDoc provides

JSDoc provides a standard structure for documenting JavaScript.

Common tags include:

```js
@param
@returns
@throws
@typedef
@property
@template
@deprecated
@example
@module
```

A typical function comment might be:

```js
/**
 * Builds the URL for an existing work-image derivative.
 *
 * This function selects a URL. It does not generate or upload image files.
 *
 * @param {string} workId - Canonical work identifier.
 * @param {800 | 1200 | 1600 | 2400} width - Required derivative width.
 * @param {MediaConfig} config - Local or remote media configuration.
 * @returns {string} Absolute or site-relative image URL.
 */
export function buildWorkImageUrl(workId, width, config) {
  return `${config.mediaBase}${config.mediaPrefix}/${workId}-primary-${width}.webp`;
}
```

JSDoc-aware tools can use this information to provide:

- hover documentation;
- parameter hints;
- type checking;
- autocomplete;
- navigation;
- generated API documentation;
- more useful repository search results.

---

## 3. What JSDoc does not provide

JSDoc does not automatically discover the important meaning of the code.

It does not independently know that:

```text
This function selects an existing derivative.
It does not generate the image.
```

That information must be supplied by:

- the developer;
- a coding agent inspecting the implementation;
- tests;
- architecture documentation;
- domain knowledge.

An editor may generate a mechanical skeleton:

```js
/**
 *
 * @param {*} workId
 * @param {*} width
 * @param {*} config
 */
```

However, that skeleton does not explain:

- the function’s responsibility;
- the meaning of each parameter;
- allowed values;
- side effects;
- invariants;
- privacy rules;
- ordering rules;
- what the function deliberately does not do.

The useful part of JSDoc is therefore authored semantic content, not merely the comment markers.

---

## 4. Recommended division of responsibility

A practical division of responsibility is:

```text
JSDoc and editor tooling
    provide the structured comment format

Codex or another coding agent
    drafts documentation from code, tests, and surrounding modules

Developer
    confirms architectural intent and domain rules

Tests and linting
    provide executable checks and consistency
```

Codex can draft JSDoc efficiently, but it should not add generic comments to every function without review.

The important questions are:

- Is the description accurate?
- Does it explain something not obvious from the code?
- Does it identify subsystem boundaries?
- Does it state important constraints?
- Is it likely to remain stable?
- Is the function important enough to document?

## 5. Recommended scope

JSDoc should initially be added selectively.

### Document first

Prioritise:

- exported functions;
- shared utility modules;
- configuration objects;
- public classes and methods;
- callbacks with non-obvious contracts;
- functions that transform canonical data;
- functions with side effects;
- code handling public versus private data;
- code with important ordering behaviour;
- module boundaries;
- complex data structures;
- values with constrained options.

### Usually do not document

Avoid routine JSDoc for:

- obvious one-line helpers;
- simple local variables;
- trivial getters;
- implementation details already made clear by names;
- generated files;
- abandoned or deprecated code that should instead be removed;
- every private helper by default.

Weak example:

```js
/**
 * Returns the title.
 *
 * @param {object} work - The work.
 * @returns {string} The title.
 */
function getTitle(work) {
  return work.title;
}
```

This adds little information.

Better use of documentation:

```js
/**
 * Returns the public display title for a work.
 *
 * Untitled works use the configured fallback label. Private catalogue notes
 * must never be used as display text.
 *
 * @param {WorkRecord} work - Canonical work record.
 * @param {string} fallback - Label used when no public title is available.
 * @returns {string} Public display title.
 */
function getWorkDisplayTitle(work, fallback) {
  return work.title?.trim() || fallback;
}
```

## 6. Recommended comment content

A useful JSDoc comment should explain some combination of the following.

### Purpose

```js
/**
 * Builds the lightweight work entry used by the client-side search index.
 */
```

### Responsibility boundary

```js
/**
 * Resolves derivative URLs at runtime.
 *
 * This module does not create, upload, or validate image files.
 */
```

### Parameter meaning

```js
/**
 * @param {string[]} seriesIds
 *   Canonical series identifiers in editorial order.
 */
```

### Return contract

```js
/**
 * @returns {Promise<WorkRecord | null>}
 *   The public work record, or `null` when the work does not exist.
 */
```

### Side effects

```js
/**
 * Writes the updated route configuration to local storage.
 *
 * @returns {void}
 */
```

### Exceptions

```js
/**
 * @throws {TypeError} If the supplied score is not an integer from 0 to 5.
 */
```

### Invariants

```js
/**
 * Work IDs are stable strings and must never be parsed as numbers.
 */
```

### Semantic distinctions

```js
/**
 * A null score means unscored and is excluded from statistics.
 * Zero means explicitly absent and is included.
 */
```

### Ordering rules

```js
/**
 * Preserves editorial order. Do not sort the returned IDs lexically.
 */
```

### Privacy rules

```js
/**
 * Returns a public work record.
 *
 * Internal notes and storage-location metadata are excluded.
 */
```

## 7. Module-level documentation

Important files should begin with a short statement of responsibility.

```js
/**
 * Work image URL utilities.
 *
 * Maps canonical work IDs and derivative widths to local or remote URLs.
 * Image generation and upload are handled by the offline media pipeline.
 *
 * @module work-image-urls
 */
```

This is particularly useful where several parts of the repository deal with the same general concept.

For example:

```text
processing/
    creates artwork or source visual assets

scripts/images/
    generates derivatives

assets/js/media/
    selects image URLs at runtime

R2
    stores primary derivatives and attachments
```

A module-level comment helps prevent an agent from confusing those responsibilities.

## 8. Shared object types with `@typedef`

JSDoc can define reusable object structures.

```js
/**
 * @typedef {object} MediaConfig
 * @property {string} mediaBase
 *   Empty for repository-local assets or an absolute remote origin.
 * @property {string} mediaPrefix
 *   Path prefix below the origin.
 * @property {boolean} useRemotePrimaries
 *   Whether primary derivatives are served from remote storage.
 */
```

The type can then be reused:

```js
/**
 * Returns the configured media root.
 *
 * @param {MediaConfig} config
 * @returns {string}
 */
export function getMediaRoot(config) {
  return `${config.mediaBase}${config.mediaPrefix}`;
}
```

Another example:

```js
/**
 * @typedef {object} WorkSummary
 * @property {string} work_id
 * @property {string | null} title
 * @property {number | null} year
 * @property {string[]} series_ids
 * @property {number} details_count
 * @property {boolean} has_details
 */
```

This gives developers and agents a searchable declaration of the data model.

---

## 9. Constrained values

JSDoc can record allowed values.

```js
/**
 * @typedef {0 | 1 | 2 | 3 | 4 | 5} DimensionScore
 */
```

```js
/**
 * @typedef {"unipolar" | "bipolar" | "axis2d" | "vector"} DimensionShape
 */
```

```js
/**
 * @param {DimensionScore | null} score
 * @returns {boolean}
 */
export function includeScoreInStatistics(score) {
  return score !== null;
}
```

This helps VS Code and agents distinguish valid values from arbitrary numbers or strings.

## 10. Type checking without converting to TypeScript

JSDoc can provide meaningful type checking while retaining `.js` files.

A file can opt into TypeScript-style checking with:

```js
// @ts-check
```

Example:

```js
// @ts-check

/**
 * @param {number} a
 * @param {number} b
 * @returns {number}
 */
export function add(a, b) {
  return a + b;
}
```

An incorrect call may then be reported by VS Code:

```js
add("1", 2);
```

For project-wide checking, a `jsconfig.json` can be introduced later:

```json
{
  "compilerOptions": {
    "checkJs": true,
    "allowJs": true,
    "module": "ESNext",
    "target": "ES2022",
    "strict": false
  },
  "exclude": [
    "_site",
    "node_modules",
    "vendor",
    "assets/generated"
  ]
}
```

A conservative approach is preferable:

1. use JSDoc without project-wide strict checking;
2. add `// @ts-check` to selected modules;
3. resolve genuine problems;
4. expand gradually;
5. enable broader `checkJs` only when the repository is ready.

## 11. Relationship with ESLint

JSDoc and ESLint solve different problems.

JSDoc contributes:

- intent;
- contracts;
- parameter and return information;
- domain terminology;
- object shapes;
- responsibility boundaries.

ESLint contributes:

- consistency;
- unused-code detection;
- undefined-name detection;
- suspicious-pattern detection;
- clearer module structure;
- reduced repository noise.

Together:

```text
JSDoc
    explains what the code means

ESLint
    helps ensure the code is mechanically coherent
```

An ESLint configuration should not initially require JSDoc on every function. That would encourage low-value comments.

Doc-comment enforcement can be added later for selected public APIs if necessary.

---

## 12. Relationship with tests

Important behaviour should not exist only in comments.

For example:

```js
/**
 * Private notes must never be included in public work JSON.
 */
```

should ideally be supported by a test:

```js
test("public work records exclude private notes", () => {
  const source = {
    work_id: "0001",
    title: "Example",
    notes_private: "Internal only",
  };

  const result = buildPublicWorkRecord(source);

  expect(result).not.toHaveProperty("notes_private");
});
```

A useful rule is:

```text
JSDoc explains the rule.
Tests verify the rule.
```

For especially important data structures, schema validation can add another layer.

## 13. Recommended rollout

### Stage 1: identify important modules

Start with modules that:

- are used widely;
- define shared data structures;
- sit at subsystem boundaries;
- contain domain-specific behaviour;
- are regularly changed by Codex.

Possible initial areas include:

- media URL handling;
- search indexing and result shaping;
- work and series data loading;
- route configuration;
- tag and dimension models;
- generated JSON consumers.

### Stage 2: add module-level comments

Add a concise responsibility statement to each selected file.

```js
/**
 * Client-side work data loading and cache management.
 *
 * Loads generated public JSON. It does not modify canonical source data.
 */
```

### Stage 3: document exported functions

For each exported function, document:

- purpose;
- parameters;
- return value;
- side effects;
- exceptions;
- important rules.

### Stage 4: define shared object types

Add `@typedef` declarations for important structures such as:

- `WorkSummary`;
- `WorkRecord`;
- `SeriesRecord`;
- `MediaConfig`;
- `SearchResult`;
- `RouteConfig`;
- `DimensionDefinition`.

### Stage 5: introduce selective type checking

Use:

```js
// @ts-check
```

in stable, well-documented modules.

Do not enable strict project-wide checking until the initial errors have been assessed.

### Stage 6: connect checks to the normal workflow

A standard validation sequence might be:

```bash
npx eslint .
npm test
```

Later:

```bash
npx tsc --noEmit --allowJs --checkJs
```

or an equivalent project command can be added if JSDoc-based checking becomes part of the repository workflow.

## 14. Suggested Codex task

A useful initial Codex instruction would be:

```markdown
Introduce JSDoc selectively in the JavaScript codebase.

Prioritise:

- module-level responsibility comments;
- exported functions;
- shared configuration and data structures;
- non-obvious domain rules;
- side effects and subsystem boundaries.

Do not add comments that merely restate the implementation.

Use `@typedef` for important shared object structures and reference those
types from function parameters and return values.

Where documentation intent cannot be inferred confidently from code, tests,
or existing architecture documents, flag it for review rather than inventing
a rule.

Do not enable strict project-wide JSDoc enforcement in this change.
```

A more focused first pass might be:

```markdown
Add JSDoc to the media URL and derivative-selection modules.

Make clear which functions:

- select existing image URLs;
- generate image derivatives;
- upload media;
- handle repository-local thumbnails;
- handle remote primary images.

Do not change runtime behaviour.
```

## 15. Review checklist

When reviewing generated JSDoc, check:

- Does the comment explain purpose rather than syntax?
- Are parameter descriptions specific?
- Are nullable values correctly represented?
- Are allowed values constrained where useful?
- Are side effects stated?
- Are important exceptions stated?
- Are privacy rules represented accurately?
- Are ordering rules explicit?
- Does the comment identify what the module does not do?
- Does the documentation agree with tests and current behaviour?
- Is the comment concise enough to remain maintainable?
- Would removing the comment lose useful information?

If the answer to the final question is no, the comment may not be necessary.

---

## 16. Recommended repository policy

```markdown
## JavaScript documentation policy

- Use `/** ... */` for structured JSDoc comments associated with modules,
  types, exported functions, classes, and important interfaces.
- Remember that `/** ... */` remains ordinary JavaScript comment syntax at
  runtime; external tools give it its documentation meaning.
- Document purpose, contracts, side effects, invariants, exclusions, and
  responsibility boundaries.
- Do not add comments that merely restate obvious code.
- Use `@typedef` for shared object structures and constrained values.
- Prefer tests or validation for critical behavioural rules.
- Keep documentation current in the same change as the implementation.
- Introduce JSDoc and type checking gradually rather than enforcing comments
  on every function immediately.
```

---

## Conclusion

The extra `*` in `/** ... */` is not special JavaScript runtime syntax. It is a documentation convention built on ordinary block comments.

JSDoc provides the structure and tooling, but the meaningful description still has to be authored.

The recommended approach is therefore selective:

```text
module responsibilities
+ exported function contracts
+ shared typedefs
+ domain rules
+ selective type checking
+ ESLint
+ tests
```

This should improve both human maintenance and coding-agent context without filling the repository with repetitive comments.
