Factorials grow extremely fast. The table also shows an approximate RAM footprint to hold *n* UHD layers in memory at once (not counting temporary buffers), assuming 32-bit ARGB pixels.

Calculation details: 3840×2160 = 8,294,400 pixels; 4 bytes/pixel → ~31.64 MiB per image (≈ 33.18 MB decimal).

## 2) Runtime & Memory Estimator (UHD)

Rough guidance for sequential blending or averaging. *These are order‑of‑magnitude estimates* — actual performance depends on CPU, JVM heap size, disk speed, and which blend modes/filters you use.

### Time model

For L layers, each composite touches ~L × 8.29M pixels. A modern desktop CPU can loop at anywhere between 80–300 million pixel‑ops/second in Java for simple arithmetic. Using a middle value (≈150 MPix/s), an estimate per composite is:

```
time_per_composite ≈ (L × 8.29M) / 150M  seconds
examples:
  L = 4 → ~0.22 s
  L = 6 → ~0.33 s
  L = 8 → ~0.44 s
```

Auto‑Levels adds one more pass over the image; expect ~10–30% extra.

### Memory model

Minimum live memory while compositing is approximately:

```
RAM_min ≈ (L + overhead) × 31.7 MiB
- L image layers
- +1 output buffer
- +histograms/aux arrays during auto‑levels (tiny)
```

**Rule of thumb:** For 6 UHD layers, aim for at least **2–4 GB** heap to be comfortable when batching many composites. Increase Processing JVM memory with something like -Xmx4G.

Speed tips:

Use Preview scale (10–50%), enable 1‑in‑n stride, set First‑N caps, and prefer AVERAGE strategy when you just need equal contributions. Export FINAL at full res once the settings look right.
