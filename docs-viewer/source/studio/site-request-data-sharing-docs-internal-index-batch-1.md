## Audit Result

The Analytics/Data Sharing API wrappers dispatch through adapter handlers and do not reconstruct docs filesystem paths.

Active generated-artifact dependencies found:

- `docs-viewer/services/docs_data_sharing/package.py`
  - `selectable_document_records(...)` calls `docs_generated_reads.read_generated_docs_index(...)`.
  - `document_selectable_record(...)` consumes `doc_id`, `title`, `parent_id`, `published`, `viewable`, `content_text_length`, and `summary` from those rows.
  - the selectable response reports `source.source = "generated_docs_index"`.
- `docs-viewer/services/docs_export.py`
  - `load_scope_index(...)` reads `assets/data/docs/scopes/<scope>/index.json`.
  - `doc_payload_path(...)` and `load_doc_payload(...)` read generated `by-id/<doc_id>.json`, using `content_url` when present.
  - `build_export(...)` uses flat-index rows for selection, descendant expansion, viewability filtering, missing-summary filtering, metadata, hierarchy fields, and direct field mapping.
  - `source_value(...)` uses generated by-id `content_html` for `headings` and `source_text`.
- `docs-viewer/services/docs_import.py`
  - `load_current_docs_context(...)` reads `assets/data/docs/scopes/<scope>/index.json`.
  - `doc_payload_path(...)` checks generated by-id payload existence from `content_url` or `assets/data/docs/scopes/<scope>/by-id/<doc_id>.json`.
  - returned-package reports include current index load state, current doc existence, viewability, generated payload existence, parent existence, and parent payload existence.
  - preview rendering uses returned-package rows and current-context flags; it does not need generated payload bodies.
- Data Sharing API wrappers
  - `analytics-app/app/server/analytics_app/analytics_data_sharing_api.py`, `data-sharing/data_sharing/services/dispatch.py`, `data-sharing/data_sharing/workflows/prepare.py`, `data-sharing/data_sharing/workflows/review.py`, and `data-sharing/data_sharing/workflows/list_returned.py` are dispatch/orchestration layers only.

No active Data Sharing metadata read was found from public `index-tree.json`, public search payloads, recently-added payloads, or generated payload enumeration as an implicit index.

## Locked Source Record Contract

The new read model is Docs Viewer-owned and is only for Data Sharing document workflows.
The record name for implementation is `DataSharingDocsSourceRecord`.
It is a source-derived projection from scope config plus Docs Viewer source front matter/body rendered through shared Docs Viewer parsing/rendering behavior.

Required record fields:

- `doc_id`
- `scope`
- `title`
- `published`
- `summary`
- `added_date`
- `last_updated`
- `parent_id`
- `parent_title`
- `viewable`
- `ui_status`
- `source_path`
- `viewer_url`
- `content_text_length`

Required helper behavior:

- rendered HTML is available through a helper instead of a stored generated payload dependency
- rendered HTML must use the same Docs Viewer markdown rendering, doc-link rewriting, media token handling, semantic reference handling, and image-title behavior used by docs builds
- source-text extraction and heading extraction must continue to operate on source-rendered HTML
- `published` remains a Data Sharing selectable-record field and must default to true unless source front matter explicitly marks it false
- `source_path` is repo-relative and derived from the configured source root
- `viewer_url` is included because current export metadata can require viewer URL behavior; consumers should only map it where an active export field requires it
- no `generated_from`, `source_scope`, or generated-output provenance field is added in Batch 2 unless a focused implementation test proves the current diagnostics require it

Do not include speculative fields for reports or future tools.
Do not name any file, helper, or response `index.json`.

## Helper API Decision

Add the source metadata implementation under Docs Viewer services.
The preferred module name is `docs-viewer/services/docs_data_sharing/source_metadata.py` because the projection is owned by Docs Viewer but scoped to Data Sharing workflows.

Public helper names for Batch 2:

- `load_data_sharing_docs_source_records(repo_root: Path, scope: str) -> list[DataSharingDocsSourceRecord]`
- `load_data_sharing_docs_source_context(repo_root: Path, scope: str) -> DataSharingDocsSourceContext`
- `render_data_sharing_doc_html(context: DataSharingDocsSourceContext, doc_id: str) -> str`
- `data_sharing_doc_content_text(context: DataSharingDocsSourceContext, doc_id: str) -> str`
- `data_sharing_doc_headings(context: DataSharingDocsSourceContext, doc_id: str) -> list[str]`

`DataSharingDocsSourceContext` should hold records, `records_by_id`, `children_by_parent`, source docs, scope config, and a render cache.
Data Sharing consumers call these helpers and must not reconstruct source roots, generated output roots, by-id paths, search paths, or recently-added paths.

## Scope Behavior

Records are produced for every configured Docs Viewer scope returned by `docs_scope_config.load_docs_scope_configs(repo_root)`.
The implementation must use each scope's configured source root, nested-source setting, viewer base URL, `include_scope_param`, and unresolved-parent policy.
New scopes are covered automatically by config.

Current configured scopes are `studio`, `library`, `analysis`, and `tmp`.
Batch 2 should test at least one public scope and one local/manage scope, and should include a synthetic config-driven scope fixture so coverage is not hard-coded to `library` or `analysis`.

No scope-specific module, repeated `library`/`analysis` branch, generated-publication fallback, or adapter registry `sources.docs_index`/`sources.docs_payload_root` dependency is allowed.

## Error And Stale-Data Behavior

Local Data Sharing calls should fail visibly when canonical source cannot be read or rendered, because generated artifacts are not a valid fallback.

Locked behavior:

- missing configured source root: raise a service error and return a structured failed report at the Data Sharing workflow boundary
- duplicate `doc_id`: raise a service error; do not pick one row
- unresolved `parent_id`: follow the scope config; if unresolved parents are allowed, keep the `parent_id`, leave `parent_title` empty, and report parent existence as false in review context
- missing selected `doc_id`: preserve current export unknown-id errors
- source rendering failure for a selected/exported/reviewed doc: report the doc-level error and do not read generated by-id payloads as fallback
- missing generated public flat indexes, manage/local flat indexes, by-id payloads, search payloads, or recently-added payloads: no warning is emitted because those artifacts are no longer Data Sharing metadata inputs
- stale generated data is irrelevant to Data Sharing metadata and must not influence selectable records, export rows, source-text extraction, or returned-package current-context diagnostics

Returned-package review context should rename generated-payload-specific diagnostics during migration.
`current_payload_missing` and `parent_payload_missing` should become source/renderability diagnostics, preserving user-facing usefulness without implying generated by-id payload ownership.

## Risk Notes

- Rendering source HTML on demand may expose behavior differences from generated by-id payloads if the helper does not reuse the same Docs Viewer renderer path.
- Source-derived ordering must preserve current export selection and descendant expansion behavior closely enough for package diffs to remain understandable.
- Review diagnostics currently talk about generated payloads; those messages need deliberate wording updates when they become source/renderability checks.
- Adapter config fixtures currently describe generated docs paths as active sources even though runtime wrappers do not use them directly.
