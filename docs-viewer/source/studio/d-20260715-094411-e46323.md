# Document Identity Delivery

## Outcome

Every repository-owned Docs Viewer document has an immutable timestamp-code identity and normalized second-resolution `added_date`; every local workflow that creates a genuinely new document allocates the same identity shape without deriving meaning from title or filename.

DI-01 is complete only when existing repository-owned sources and future creation paths share that contract. It is not complete with only a generator or only a source migration.

## Why Now

Docs Viewer is now a capable source editor with title-based navigation, search, source view, and VS Code integration. Recent roadmap cleanup exposed that title-like identities create broad one-off link churn while no longer providing meaningful source discovery.

Doing this before more feature concepts and deliveries accumulate prevents another round of semantic ID drift.

## In Scope

- central immutable ID allocation in the Docs source model;
- second-resolution `added_date` for future documents;
- deterministic normalization of all current repository-owned source dates;
- reviewed old-to-new mapping for repo-owned scopes and sub-scopes;
- matching immutable Markdown filenames;
- rewrites of `doc_id`, `parent_id`, Docs Viewer `doc`/`subdoc` links, configured identity references, and focused fixtures;
- New Document and ordinary genuinely-new import allocation;
- preservation of canonical IDs for overwrite and returned-package update workflows;
- removal of opaque ID fragments from ordinary Docs search relevance while retaining deliberate exact lookup if useful;
- deliberate full docs/search regeneration and stale old-ID removal for affected scopes;
- focused generator, creation, import, mapping, parent, config, build, search, and broken-link evidence;
- promotion of the concept into the durable Document Identity overview.

## Not In Scope

- configured external-local scope migration;
- aliases, redirects, or an old-ID compatibility registry;
- rewriting Git history, completed logs, historical exports, staged packages, or review sessions;
- renaming unrelated media and attachments merely because their filenames contain an old slug;
- globally unique identity enforcement across all repositories;
- using document codes as authorization or secrecy;
- changing title-based index ordering or visible navigation labels.

## Implementation Sequence

1. Add and test the shared timestamp, normalization, validation, and ID-allocation primitives.
2. Produce the complete dry-run migration map and review collisions, provenance methods, source moves, parent changes, configured IDs, and link effects.
3. Cut New Document and genuinely-new ordinary import planning over to allocated IDs while preserving overwrite/update identity.
4. Apply the reviewed repository-owned source/config migration without intermediate generated rebuilds.
5. Update Docs search identity handling and focused fixtures.
6. Rebuild every affected docs and search scope fully, removing stale old-ID payloads.
7. Run parent/config validation, source old-ID scans, broken-links audits, focused service tests, and the narrowest relevant broader check.
8. Promote the concept into durable current documentation and retire the temporary feature, architecture, and delivery documents.

The sequence is one delivery because none of its intermediate states is the desired supported contract. If implementation exposes an independently useful external-scope or historical-artifact outcome, return that to the roadmap rather than widening DI-01.

## Review Artifact

The ignored migration mapping is required evidence, not permanent product configuration. It should make every source identity and inferred timestamp reviewable before apply and remain available through verification and closeout.

## Completion Evidence

- every repository-owned source filename and `doc_id` uses the selected code shape;
- every source `added_date` uses `YYYY-MM-DD HH:MM:SS`;
- every `parent_id`, configured identity, and meaningful Docs Viewer link resolves;
- no authoritative source/config reference uses a migrated old identity;
- New Document and ordinary new-document imports allocate codes independent of title/filename;
- overwrite and returned-package updates preserve canonical identity;
- generated outputs contain no stale old-ID payloads;
- Docs search remains useful without opaque token noise;
- broken-links audits pass for every affected scope;
- the retained Document Identity overview accurately describes the shipped contract;
- temporary roadmap documents are removed once no unresolved work remains.

## Durable Documentation Boundary

[Document Identity Concept](/docs/?scope=studio&doc=d-20260715-094411-2b6e65) becomes the durable feature overview. Exact validators, generator implementation, adapter branches, and migration mechanics remain code/test authority rather than permanent exhaustive documentation.
