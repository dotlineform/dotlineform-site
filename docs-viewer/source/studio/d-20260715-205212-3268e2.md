# Control Surfaces Foundation Delivery

## Outcome

Every existing Docs Viewer app, panel, and hosted-view control has one intentional definition, host, state-projection, and event owner. Existing visible behavior is preserved, obsolete control wiring is removed, and the index panel has a real active-view control mount ready for later features.

This is the feature's complete, implementation-ready foundation delivery.

## In Scope

- establish the control categories and ownership rules in code;
- complete the code-owned definition/projection model for app and view controls across shared, manage, and review entrypoints;
- give viewer, management, index-view, and main-view controls stable hosts;
- keep index collapse/expand and info close as panel chrome projected by panel owners;
- migrate current shell-level viewer controls, management controls, document/view controls, and their live state into the completed model;
- keep toolbar, menu, and context-menu placements aligned through existing action ids and target resolution;
- make displayed-document targets and checkbox-selection targets unambiguous; do not carry a hidden primary document into batch selection;
- replace fixed global-runtime control wiring with host/controller-owned dispatch where the new model makes it obsolete;
- remove or give real meaning to dormant toolbar capability fields; retain no speculative inert contract;
- preserve optional mounts and public/manage/review import boundaries;
- add focused registry, projection, renderer, action-placement, and route/module evidence.

The migration inventory is a working implementation aid. It does not become a durable list of every button.

## Not In Scope

- index selection mode, checkboxes, selected-count UI, range selection, or group drag/drop;
- the index relationship graph or its controls;
- new workflows, action targets, backend endpoints, or capabilities;
- redesigning icons, labels, placement, spacing, or responsive behavior except where necessary to preserve the current UI in the new hosts;
- global registration of report filters, modal actions, picker rows, bookmark rows, search-result controls, tree-node expand/collapse, or other component-local UI;
- user-configurable toolbar order, JSON-defined controls, plugins, or arbitrary module loading;
- compatibility aliases or parallel old/new rendering paths as the completed state.

## Code Authority To Inspect

- shared app-shell, top-bar, viewer-toolbar, index-panel, main-view, info-panel, panel-layout, view-registry, and view-context owners under `site/docs-viewer/runtime/js/shared/`;
- manage-owned hosted-view definitions, action definitions, management action renderers, interaction/controller owners, and source-mode integration under `docs-viewer/runtime/js/management/`;
- route feature and view policy in `docs-viewer/config/routes/` and its shared normalizer;
- focused shared/runtime/management tests and route smokes under `docs-viewer/tests/`.

These are ownership areas, not an instruction to update every module in them. Follow the current code paths and remove only superseded plumbing.

## Main Benefit And Risks

The benefit is a stable extension seam: the next shell or hosted-view control has an obvious registration, host, state, and action path.

The main risk is a broad behavior-preserving migration across public, manage, and review routes. Preserve visible placement and interaction, migrate one surface at a time behind the same completed model, and do not leave mixed ownership at closeout. Optional toolbar routes and management-only imports deserve explicit verification.

## Done When

- every existing shell/panel/view control is registered or explicitly classified as panel chrome;
- every stable control surface is rendered through an intentional host rather than ad hoc insertion;
- eligibility comes from code registration plus current route/app/view/capability projection;
- live hidden, disabled, pressed, busy, label, count, and expanded state is projected by the focused owner rather than encoded in registry definitions;
- toolbar, menu, and context-menu copies of an action resolve the same targets;
- active-document controls follow the displayed doc while future batch controls can consume one independently owned checkbox selection;
- the index-tree view can accept a manage-owned **Select** control without adding another mount or global-runtime DOM branch;
- obsolete direct references, duplicate renderer definitions, inert toolbar fields, and compatibility paths are removed;
- focused pure tests and the narrowest route/module checks pass for public, manage, and review boundaries;
- the durable owner below describes the shipped model and this delivery closes completely.

## Durable Documentation Boundary

The current [Panel Hosts](/docs/?scope=studio&doc=d-20260528-224754-964108) document becomes **Panel Hosts and Control Surfaces** and owns the shipped taxonomy, host model, registration and extension rules, action boundary, and known weak spots.

Do not update separate toolbar inventories, route documents, UI guides, or feature concepts merely because their controls migrated without behavior change. Retire this delivery and the proposed architecture after the durable transfer.
