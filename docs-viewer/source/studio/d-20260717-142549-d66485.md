# Source, Build, Published, And Location Concept

This concept separates semantic roles from physical paths. The complete section below is intended to move intact into the durable Docs Viewer reference documentation when the source and publication model refactor ships.

## Source, Build, Published, Location, And Exposure

### Core Terms

| term | meaning |
| --- | --- |
| **Document** | The conceptual Docs Viewer entity identified by `doc_id`, not any one file or rendered representation. |
| **Source** | The one canonical application/build input retained by Docs Viewer for an artifact that must be transformed. A file is not source merely because it originated elsewhere or sits below a path named `source`. |
| **Build** | A manual or automated transformation from source to a published artifact. |
| **Published** | The one consumer-ready artifact. Published means ready for Docs Viewer to reference, render, serve, or download; it does not mean publicly accessible on the internet. |
| **Location** | The registered physical root and backing provider where a source or published artifact is held: repository, external-local filesystem, or R2. |
| **Public exposure** | Whether an artifact is available through a public route or snapshot. Exposure is independent of artifact role and storage provider. |
| **Rendered view** | The readable or interactive result composed by the viewer from document payloads and referenced published media. It is not the document source. |

`generated` describes provenance: an artifact was produced by a build. It is not the durable umbrella term for consumer-ready artifacts, because images, HTML, files, and other published media may require no generation. A physical directory that is currently named `generated/` may contain published artifacts, but its name does not define their semantic role.

For a normal document, the Markdown file is document source, the Docs Viewer build produces published document payloads, and the viewer composes the rendered document or document view. The document itself is the conceptual entity that persists across those representations.

### Per-Media-Type Contract

The source/build/published decision is made per media type within a scope, such as `img`, `files`, `html`, `video`, or a future build-source type. It is not inferred from a filename suffix alone and is not a single all-or-nothing setting for the scope.

Each media type has this cardinality:

- zero or one registered source location;
- zero or one registered build relationship;
- exactly one registered published location.

If no build is required, there is no Docs Viewer source for that media type. The ready bytes are the published artifact and are stored once at the published location. Import, upload, or manual placement puts the bytes there directly; those operations do not create a second source copy.

If a build is required, there is exactly one source location and exactly one published location. The source is read-only to the build, and the output is written directly to the published location. A source and its output may use the same backing provider, but they remain different artifacts at different registered paths.

Changing a producer from manual to Docs Viewer automation does not change the source or published identity and should not move files. Two producers must not silently claim the same published identity. An externally originated file is provenance, not automatically Docs Viewer source: it becomes source only when Docs Viewer retains it as the canonical input to a registered build.

This cardinality governs media bytes. Docs Viewer document and search payloads may still have separately configured working and public projections; those payload-delivery contracts do not justify source/published duplication for media.

### Roles Do Not Choose Locations

The target role model does not choose a storage provider. Source and published locations are configured independently, and the same workflow uses them through a common location contract.

Current support is narrower:

| artifact role | repository | external-local filesystem | R2 |
| --- | --- | --- | --- |
| Canonical document source | Supported. | Supported. | Not supported. |
| Working document/search build output, currently called generated | Supported. | Supported. | Not supported. |
| Published media | Supported. | Supported. | Supported. |
| Public document/search snapshots | Supported beneath `site/assets/`. | Not supported. | Not supported. |

This table classifies artifacts by semantic role, not by current path name. Ready repository or external-local media may currently sit beneath a scope path named `source`, but it is published media under this model. An R2 upload is likewise a published media artifact, not generated output or canonical source.

The restrictions are consequences of the current implementation rather than inherent workflow requirements: document source and build output are configured as filesystem paths; R2 exists as a special public-media upload mode; and import/publish code branches on storage modes. The target removes that coupling from the main workflows.

All three providers can back source or published roles when the selected location adapter supplies the operations required by that workflow:

| role | repository | external-local filesystem | R2 |
| --- | --- | --- | --- |
| Source | Target-supported when the canonical build input belongs in the repository. | Target-supported when the scope and canonical build input belong in the external workspace. | Target-supported when a canonical build input is intentionally retained in R2 and the workflow's required source operations are available. |
| Published | Target-supported for practical repository-sized artifacts and tracked site delivery. | Target-supported for artifacts owned by an external-local scope. | Target-supported for large, established remote, or otherwise impractical repository artifacts. |

### Location Adapter Contract

Build, import, inventory, and publish orchestration must not contain separate repository, external-local, and R2 workflows. Each asks a shared location boundary to operate on a logical scope, artifact role, media type, and filename identity.

That boundary supplies location-specific helpers for:

- resolving and confining repository paths, external workspace paths, or R2 object keys;
- listing, reading, writing, replacing, deleting, statting, and byte-verifying artifacts;
- acquiring authentication and clients where a remote provider requires them;
- creating a temporary local input or staging an output when a builder requires filesystem paths;
- resolving the served reference or URL independently of physical backing;
- reporting supported capabilities before a workflow starts.

Repository and external-local storage can share filesystem operations while retaining different root and lifecycle policies. R2 supplies the same logical operations through object storage, including any required download/upload staging. Builders and publishers call the common boundary and do not know which adapter fulfilled the request.

Different artifact roles require different capabilities:

| workflow role | required location capabilities |
| --- | --- |
| Canonical managed document source | Enumerate, read, create, replace, rename, delete, and observe change. |
| Media build source | Enumerate, read, byte-verify, and provide a local input when the builder requires one; mutation is required only for an owning management/import workflow. |
| Published media | List, stat, read, write/replace, byte-verify, delete under explicit authority, and resolve a served reference. |
| Document/search build output | Write a consistent artifact set, read current output, remove stale output, and expose it to the configured consumer. |

Configuration may reject a role/location combination when an adapter genuinely lacks a required capability. It must not reject it merely because an existing workflow was written around a different provider. For example, R2 cannot become managed document source simply by accepting a URL: management and watcher semantics also need implementations. Once those operations exist, the document workflow should require no R2-specific branch.

Provider capability is also separate from scope policy. Repo-backed scopes may select repository or R2 per media type regardless of public projection. External-local scopes deliberately keep all owned media in their external folder; that remains a policy constraint even though the common location code can operate on repository and R2 locations elsewhere.

### Docs Viewer Published Workspace

The Docs Viewer-owned local output workspace is a published location, not a statement about how every artifact was produced. Its physical roots are:

```text
docs-viewer/published/docs/<scope>/
docs-viewer/published/search/<scope>/

$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/published/docs/<scope>/
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/published/search/<scope>/
```

These replace the current repository and external-local `generated/` roots. The workspace can contain build-produced document/search payloads and directly placed consumer-ready artifacts without misclassifying the latter as generated. `generated` remains a valid provenance adjective or payload field where it describes an actual build event; it is not the physical workspace role.

Scope configuration is the sole topology authority. Production builders, readers, serving code, validators, lifecycle operations, manifests, and browser-config generation resolve the registered published locations through the shared location boundary. They do not construct, require, or validate a literal `docs-viewer/generated/` or external `generated/` path. After this cutover, changing a published workspace path requires a configuration change, moving the artifacts, and rebuilding where necessary—not a workflow-code change.

Configuration distinguishes the Docs Viewer local published projection from optional public exposure instead of overloading generic `output` and `publish_output` meanings. Public website snapshots remain beneath conventional `site/assets/data/` paths, and R2 objects remain at their configured published locations. Neither is moved merely because the Docs Viewer-owned local workspace is renamed.

Service routes are logical interfaces and do not mirror the directory name. Neutral routes such as `/docs/index-tree`, `/docs/doc`, and `/docs/search` remain valid regardless of backing location. Direct static URLs for repository-published artifacts are derived from configuration. No compatibility path is retained solely for the old physical `generated/` directory.

Public scopes may continue to have a local published management projection plus a separately exposed public website snapshot. That document/search projection contract is distinct from the one-published-location rule for media bytes.

Repository/R2 selection applies to **repo-backed scopes**, regardless of whether a scope has a public projection. Public exposure may affect the serving path, but it does not decide whether repository or R2 is the appropriate backing.

External-local scopes use the same semantic folder structure as repo-backed scopes, but repository/R2 routing does not apply. All source and published media owned by an external-local scope remains physically inside that external scope folder and follows the scope's create, rename, and delete lifecycle.

### Folder And Scope Shape

A repo-backed scope makes documents, build-source media, and sub-scopes visibly distinct:

```text
docs-viewer/source/<scope>/
  documents/
    *.md
  media/
    mermaid/
      *.mmd
  sub-scopes/
    <sub-scope>/
      documents/
        *.md
      media/
        <build-source-type>/
```

`documents/` contains one flat set of canonical Markdown files. It contains neither the document hierarchy nor structural subdirectories. Document hierarchy remains metadata-owned, while `sub-scopes/` explicitly marks nested collection ownership instead of making a sub-scope look like a folder within the parent's document collection. A sub-scope repeats the same `documents/`, `media/`, and `sub-scopes/` shape as needed.

Source `media/` contains only canonical inputs to registered builds. Directly usable media does not gain a redundant source copy. Its published root is registered separately, for example:

```text
site/assets/data/docs/scopes/<scope>/media/<type>/
R2: docs/<scope>/<type>/
```

An external-local scope is a self-contained physical container for both roles:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/source/<scope>/
  documents/                 # document source
  media/
    mermaid/                 # build source, when configured
    img/                     # published media, when configured
    files/                   # published media, when configured
    html/                    # published media, when configured
  sub-scopes/
```

The parent directory happens to be named `source`, but the registered role of each child remains authoritative. Renaming that existing external workspace container is not required to make the role model correct.

Media folders describe stable format or handling boundaries, not changeable behaviour. HTML belongs under `media/html/` whether it is static or interactive; sandboxing, script permission, and interactivity are rendering-code concerns. Files do not move because their behaviour changes.

### Examples

| artifact | source | build | published artifact |
| --- | --- | --- | --- |
| Markdown document | Canonical `.md`. | Docs Viewer document build. | Document/search payload consumed by the viewer or public route. |
| Mermaid diagram | `media/mermaid/diagram.mmd`. | Mermaid to SVG, initially manual and later optionally Docs Viewer-owned. | `media/img/diagram.svg` at the configured `img` published location. |
| JPEG or WebP image | None. | None. | The image placed once at the configured `img` published location. |
| Ready HTML companion | None. | None. | The HTML placed once at the configured `html` published location; sandbox behaviour is separate. |
| PDF, Word file, or archive | None. | None. | The file placed once at the configured `files` published location. |
| Video requiring transcoding | The retained canonical master. | Registered transcode. | The delivery video at the configured `video` published location. |
| Ready video | None. | None. | The delivery video placed directly at the configured `video` published location. |
| Processing or visualization output | Retained project/build input only when Docs Viewer owns the build. | Registered export when one exists. | The resulting HTML, image, video, or bundle at that output type's published location. |

Mermaid therefore does not retain a source SVG when Docs Viewer can reproduce it. During a manual phase, the author renders the `.mmd` and places the SVG directly in the configured published `img` location. A later Docs Viewer builder performs that same build-and-write step without changing either path.

### Ownership And Inventory

A scope owns artifacts through its registered roots, not solely through containment beneath `docs-viewer/source/<scope>/`. Registered roots may include document/build source, repository-published media, R2 prefixes, external-local media, document/search payloads, and public projections.

The inventory records enough information to make that ownership explicit:

- logical scope, media type, and filename identity;
- semantic role: source or published;
- registered location and backing provider;
- producer/build relationship where one exists;
- document references and served path;
- byte evidence needed for migration or cleanup.

An owned artifact need not currently be referenced by a document. Conversely, a document reference alone does not prove that Docs Viewer owns the target bytes. External Asset Collections remain independently owned roots even when Docs Viewer inventories or references them.

### Repository And R2 Policy

Storage is a practical, per-media-type decision rather than a purist classification by suffix:

| scope and media type | source | build | published location |
| --- | --- | --- | --- |
| Analysis and Library `img` | None for ready images. | None. | Repository-first at `site/assets/data/docs/scopes/<scope>/media/img/`. |
| Moments `img` | None for ready images. | None. | Existing R2 `docs/moments/img/` contract. |
| Repo-backed `files` such as PDF, Word, and archives | None for ready files. | None. | R2 unless a specific media-type decision says otherwise. |
| Ready HTML and SVG of practical repository size | None. | None. | The configured repository-published `html` or `img` root. |
| Mermaid | Repository or external-local `media/mermaid/`. | Manual first; optionally Docs Viewer-owned later. | The configured published `img` root. |
| Large video, audio, datasets, or similar media | Only when a registered build retains an input. | Optional by type. | Prefer R2 when repository size is impractical. |
| Any external-local media type | Optional canonical build input within the external scope. | Optional by type. | Within the same external scope folder; never redirected to repository or R2. |

An R2 object that needs no build is not both source and published. It is only the published artifact. Likewise, moving a ready Analysis image from R2 into the repository changes its published location; it does not create or move a source artifact.

Logical scope/type/filename identity stays independent of the provider. A later repository/R2 decision should require only:

1. changing the configured published path/provider for that media type;
2. moving and byte-verifying the affected artifacts;
3. rebuilding dependent document payloads where needed.

Canonical document source should not need rewriting. Removal of the old copy follows inventory and byte verification; unexplained or uniquely held objects are retained until ownership is established.

Existing R2 and cache behaviour remains unchanged. R2 is allowed to cache normally, and any exceptional manual cache clearing remains an operator action. This model adds no cache purge, fingerprint, query-version, or automatic invalidation requirement.

### Build Direction

A full Docs Viewer build follows one direction:

1. inventory canonical document source, registered media build source, and existing published media;
2. run registered media builders from source directly into each configured published location;
3. render document payloads against the published media inventory;
4. publish document and search projections where configured.

The build never writes reproducible output back into source. Media requiring no build bypasses the build stage and is inventoried at its published location. This keeps scope ownership, filesystem structure, and future build automation stable without duplicating artifacts or moving them merely because their behaviour or producer changes.
