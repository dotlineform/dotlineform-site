# External Asset Collections Architecture

## Architectural Outcome

External Asset Collections should extend the existing scope lifecycle, capabilities, local-service, and report boundaries without creating a second document system or a general asset application.

This document records the proposed structure and its known pressure points. Exact current behavior remains authoritative in the linked code and focused tests.

## Ownership And Storage

An enabled scope uses the derived root:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/media/<scope>/
```

- Scope config records capability state and the report document id, not a resolved absolute path.
- The collection is associated with the scope but remains user-owned data.
- Manifest state must distinguish it from lifecycle-owned source and generated files.
- Delete preview names the preserved collection; delete apply never removes it.
- Rename must not silently relocate an enabled collection. Blocking rename is the first safe contract.

Repo-managed attachments in `docs-viewer/source/<scope>/media/` remain separate. Existing `import_media_storage.storage_mode: external_assets` continues to describe where Docs Import materializes attachments for external-local scopes; it does not itself enable collection inventory.

## Execution Path

```text
scope config
  -> management capability projection
  -> report metadata selects scope_assets
  -> local report loader requests confined inventory
  -> local service derives the root and scans metadata
  -> report renders relative paths and reference state
```

The initial scan should be synchronous and metadata-only. It preserves nested relative paths, avoids symlink following, skips minimal platform noise, and reports partial warnings without making unreadable entries fatal.

One report-backed source document exposes the inventory. Asset-specific loading, state, and rendering remain in a focused report owner rather than the shared document controller.

## Local File Boundary

Nested collection links use a separate loopback-service route from the existing flat `/docs/media/<scope>/<img-or-files>/<filename>` import-media route.

The service must:

- require a configured local scope with the capability enabled
- derive and confine the root on the server
- reject absolute paths, traversal, malformed encoding, and symlink escape
- expose relative paths only
- apply `nosniff` and a narrow inline MIME policy
- force executable, interactive, unknown, and ordinary HTML content to download

Public route shells and browser config receive no index, local root, or service URL.

## Extension Method

Use existing owners:

- scope config — enabled state and report document id
- scope lifecycle and manifest — create, rename, delete, and preservation consequences
- management capabilities — service support and per-scope enabled state
- report metadata registry — selection of the `scope_assets` presentation
- focused local service — confined inventory and nested file reads

Filesystem inventory is runtime data, not another registry. Do not add a plugin, action, or general asset registry framework.

## Current Code Pressure Points

- `docs-viewer/services/docs_media_storage.py` couples the derived external media root to external-local import storage.
- `docs-viewer/services/docs_scope_manifest.py` primarily models lifecycle-owned file records.
- `docs-viewer/services/docs_scope_delete.py` and `docs-viewer/services/docs_scope_rename.py` currently give external media different preservation consequences.
- The existing `/docs/media/` contract is flat and classed as `img|files`.
- Project State has a proven metadata traversal, but its walker is mixed with catalogue-specific filtering and cannot be reused directly.

These are inspection starting points, not an exhaustive implementation list. A delivery should update only the owners required for its outcome.

## Known Weak Spots

- The same derived physical root may contain import-managed attachments and user-managed project files; capability and ownership state must remove that ambiguity.
- A derived scope-id path conflicts with preserving files across rename; the first release should block rather than hide a relocation policy.
- Large-tree responsiveness is unknown. Measure Processing and I Ching before adding caches, jobs, hashes, or previews.
- Reference correlation may initially understand only current Docs media and asset routes; any compact authoring token is a separate design decision.

## Delivery Boundary

The feature parent owns its delivery boundaries. The [Docs Viewer Roadmap](/docs/?scope=studio&doc=d-20260714-234030-1e0af6) records only suggested feature order. Each delivery must complete one improvement to this architecture and transfer shipped behavior to the named durable owner.
