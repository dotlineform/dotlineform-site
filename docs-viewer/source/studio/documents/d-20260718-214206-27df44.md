# Mermaid Document Diagrams Delivery

## Outcome

A Docs Viewer author can add a staged Mermaid source through **Add image…**, retain the canonical `.mmd` inside the active scope, and receive a verified SVG token in the current Markdown buffer. In rendered manage mode, Document Info lists the canonical Mermaid sources behind the SVGs being viewed and opens them in VS Code. Repository authors, targeted document builds, full builds, and Docs Watcher use the same registered producer. Readers receive only sanitized static SVG.

The delivery is complete when two useful Studio documents visibly prove the convention:

- [Configuration And Extension Points](/docs/?scope=studio&doc=d-20260512-142027-54b354) receives a diagram of source configuration, executable ownership, generated projection, and runtime capability.
- [Docs Import](/docs/?scope=studio&doc=d-20260424-000000-fc8b5d) receives a diagram of document intake, embedded media, source-editor media, publication, source write, and rebuild boundaries.

## Delivery Boundary

In scope:

- one focused, locked Mermaid CLI dependency beneath `docs-viewer/build/mermaid/`;
- one registered `mermaid` source-media producer publishing same-basename SVG through the configured `svg` class;
- Studio `media/mermaid/` source registration and visible source folder;
- staged `.mmd` intake through **Add image…**, followed by publication verification and token insertion;
- complete rendering during a full Docs build and relevant-identity rendering during targeted builds;
- changed-identity rendering when Docs Watcher observes a configured `.mmd` source;
- manage-only Mermaid source links in rendered Document Info, delegating scope-confined identities to VS Code;
- shared sanitization, accessibility metadata, focused service/build tests, and the two proof diagrams.

Out of scope:

- Mermaid parsing, editing, or live preview inside Docs Viewer;
- a graphical diagram editor or browser Mermaid runtime;
- global `mmdc`, an alternative Docker renderer, or multiple producer choices;
- fingerprints, render caches, incremental dependency graphs, or background queues;
- automatic sharing of one `.mmd` source across scopes;
- migration of historical SVG or inline diagrams;
- D2 support unless a proof diagram exposes a concrete Mermaid limitation.

## Settled Decisions

- Pin `@mermaid-js/mermaid-cli` exactly to `11.16.0` in `docs-viewer/build/mermaid/package.json` and commit the adjacent `package-lock.json`.
- Keep shared Mermaid settings in `docs-viewer/build/mermaid/mermaid-config.json`: neutral theme, strict security, and `htmlLabels: false`.
- Render SVG on a white background with an explicit 1200 by 800 browser viewport, retain a responsive `viewBox`, and add no Markdown dimensions by default.
- Require Mermaid `accTitle` and `accDescr` plus useful Markdown alt text.
- A full build renders every configured Mermaid source. Targeted builds, staged intake, and the watcher render only relevant identities.
- Rendered Document Info lists Mermaid sources derived from the saved document's SVG tokens, configured producer relationship, and existing same-basename `.mmd` files. Source mode remains the semantic-token picker.
- VS Code is the initial external source editor; no separate diagram toolbar action is added.
