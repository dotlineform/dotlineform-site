# Docs Viewer Source And Publication Model Refactor

## Outcome

Docs Viewer uses one coherent source-to-published model before format-specific builders or further media features are added. Source, build, published artifact, storage location, public exposure, and scope ownership are separate contracts; build, import, inventory, and publish workflows are location-neutral. The explicit document, media, and sub-scope folder structure plus repository/external-local/R2 routing follow from that model rather than defining it.

Every configured scope therefore has clear registered roots for canonical document/build source and published media. Documents and sub-scopes have visible boundaries; media that needs building keeps one source; already usable media has no Docs Viewer source and is stored once at its configured published location.

The Docs Viewer-owned repository and external-local output workspaces are renamed from `generated/` to `published/`. Public `site/assets/` snapshots, R2 objects, and logical service routes retain their independent conventional paths.

```text
source/<scope>/
  documents/
    *.md
  media/
    mermaid/
      *.mmd
  sub-scopes/
    <sub-scope>/
      documents/
      media/

<configured published scope root>/
  media/
    img/
    files/
    html/
  sub-scopes/
    <sub-scope>/
      media/
```

Only media types the scope uses need to exist. New source or published types are added when they acquire a real validation, rendering, serving, build, or storage contract.

External-local scopes use the same `documents/`, `media/`, and `sub-scopes/` shape beneath `$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/source/<scope>/`. That physical scope folder contains both source and published roles, so every external-local media type is held beneath its own scope's `media/` directory. Repository/R2 selection does not apply to them, and the parent path name does not make ready media into source.

## Ownership Model

- A scope owns resources through all of its registered roots: document/build source, repository-published media, R2 prefixes, external-local media, document/search payloads, and public projections. Ownership is not restricted to containment beneath a path named `source/`.
- A document is the conceptual entity identified by `doc_id`. Its canonical Markdown file is the **document source**, its built JSON/HTML is the **document payload**, and the readable result is the **rendered document** or **document view**.
- `documents/` contains one flat set of canonical Markdown files. It contains no document hierarchy and no structural subdirectories.
- Source `media/` contains canonical media inputs that require a build, such as Mermaid `.mmd`. Directly usable images, files, HTML, SVG, video, and similar bytes have no Docs Viewer source and belong only in their registered published media root.
- Published media types describe stable formats or handling boundaries, not changeable behaviour. HTML stays under `media/html/` whether it is static or interactive; sandboxing and script execution are code concerns.
- Mermaid source belongs under source `media/mermaid/`. Whether rendered manually or later by Docs Viewer, its SVG goes directly to the published `media/img/` target and is not also retained as source.
- `sub-scopes/` makes nested collection ownership explicit. A sub-scope repeats the same document/media shape without appearing to be a folder inside the parent document collection.
- External Asset Collections remain independently owned roots outside the scope's registered source and published-media trees.

Containment beneath any registered scope root supplies implicit ownership and discoverability. A generated inventory can make that concrete by recording media identity, document references, producer, backing, served path, and unreferenced assets.

## Build Direction

A full Docs Viewer build treats source as read-only:

1. inventory document source, build-source media, and existing published media;
2. run any Docs Viewer-owned format builders directly into the configured published media root;
3. render document payloads against the published media inventory;
4. publish document/search projections where required.

Images, files, HTML, manual renders, and other already usable media have no copy/build stage: import or manual placement writes them directly to the published root. Manual and automated production may target the same published identity, but two producers must never silently claim it.

Build, import, inventory, and publish workflows resolve source and published locations through one common location boundary. Repository paths, external workspace paths, R2 authentication/object keys, temporary local staging, and provider capability checks belong in location helpers; the main workflows do not branch by provider.

## Repository And R2 Policy

For repo-backed scopes, media references use logical scope/type/filename identity independently of physical storage. Repository or R2 backing is selected per media type so changing a decision requires a config path change, moving the bytes, and rebuilding document payloads—not rewriting document source.

The selected backing is the one authoritative and served byte location:

- R2-backed media that needs no build is only the published object at `docs/<scope>/<type>/<filename>`; it is not also source;
- repository-backed media for a public scope is published once beneath `site/assets/data/docs/scopes/<scope>/media/<type>/`;
- repository-backed media without a public projection is stored once beneath its configured published scope root at `media/<type>/`; a current physical path named `generated` does not change that role;
- public projection status affects the repository path that can serve the bytes, but it does not decide whether the type uses repository or R2 backing.

External-local scopes are deliberately simpler: all media types stay inside the external scope folder and follow that scope's create, rename, and delete lifecycle. They are never redirected into the repository or R2 by this policy.

Current decisions:

- repository-first for ordinary Library and Analysis images, stored directly in their tracked public scope media roots;
- keep Moments images in R2 because they are part of the established Catalogue/Moments media system;
- keep downloadable files such as PDF, Word, archives, and similar externally sourced binaries in R2;
- keep directly usable SVG and HTML as single repository-published artifacts, and keep Mermaid `.mmd` as repository build source whose SVG targets the published image root;
- prefer R2 for large video, audio, datasets, or other types whose size makes repository storage impractical;
- keep every external-local media type in its external scope folder regardless of type or size;
- choose R2 for practical size and provenance reasons, not because a suffix belongs there in principle.

R2-backed and repository-published media remains scope-owned through registered roots and inventory. Neither requires a redundant second Docs Viewer copy. Existing stable R2/cache behavior remains unchanged; this feature adds no purge, fingerprint, or versioning requirement.

## Migration Boundary

This refactor includes the consequential source, config, build, and publication changes:

- move every normal scope's Markdown into `documents/`;
- move Analysis Tags into `sub-scopes/tags/documents/`;
- move the Library coincidence-salience HTML asset from its separate site path into `site/assets/data/docs/scopes/library/media/html/`, making the registered published location its only copy;
- inventory actual Docs Viewer R2 objects and their document references before moving or removing anything;
- move referenced Library and Analysis images from R2 directly into `site/assets/data/docs/scopes/<scope>/media/img/` and keep their logical references stable;
- leave Moments images and public downloadable files in R2 unless the inventory exposes a specific exception;
- preserve the same `documents/`, `media/`, and `sub-scopes/` structure for external-local scopes while keeping all of their media under the external source root;
- rename repository and external-local Docs Viewer output workspaces from `generated/` to `published/`, while leaving `site/assets/`, R2, and logical service routes independent;
- make registered config paths the sole topology authority by removing literal generated-root assumptions from validation, lifecycle, serving, manifests, browser config, and builders;
- replace scope-wide media storage modes with explicit type-level source/build/published roles and location mappings;
- update source loading, building, search, imports, watcher behavior, lifecycle, media inventory, serving, and validation to use the new roots without compatibility aliases or duplicate media copies.
