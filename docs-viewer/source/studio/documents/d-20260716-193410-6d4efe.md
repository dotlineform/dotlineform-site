# Copy Subtree To Scope

## Outcome

Docs Viewer can copy one selected document and all of its descendants into another existing non-public scope without changing or endangering the originals.

The result is an ordinary canonical subtree in the target scope. It can be opened, searched, edited, linked, reviewed, or pointed to in a later session. It is not a draft mirror, synchronized branch, restorable backup, or return-import workflow.

## Copy Contract

- The action starts from exactly one document and includes every descendant automatically. It does not depend on checkbox multiple selection.
- The target may be any other configured local or external-local Docs Viewer scope whose canonical source root is available for managed writes. Public scopes are valid sources but are never offered or accepted as targets.
- Every copied document receives a new immutable `doc_id`, matching filename, and one captured copy-time `added_date` and `last_updated`. Repeating the copy therefore creates another independent subtree without overwrite choices.
- The copied parent is placed at the target scope root. Descendant `parent_id` values are remapped through the operation's old-id to new-id map.
- Canonical body and front matter are preserved except for `doc_id`, `added_date`, `last_updated`, `parent_id`, and `viewable`. Viewability follows the non-public target scope's ordinary create default.
- Docs Viewer links between copied documents are remapped to the target scope and new ids. Links outside the copied set, semantic references, media tokens, media URLs, downloads, and interactive assets continue to reference their original targets.
- No media or attachment bytes are copied.
- The source scope is read-only for the operation. No `copied_from` field, synchronization marker, merge path, or permanent source-to-copy relationship is added.

The action uses a write-free preview before confirmation. The preview names the source root, descendant count, target scope, and target root placement. Apply revalidates every planned identity and target path, never overwrites an existing document, writes the complete target set, and rebuilds only the target scope.

## Interaction

**Copy subtree to scope…** is a singular management action in the manage-only index toolbar. The user is acting on a parent/child set visible in the tree, so [Button Placement](/docs/?scope=studio&doc=d-20260716-204013-3be4e1) places the control with the index rather than the rendered document or whole scope. The active tree document supplies the root; its target selector excludes the current scope, every public scope, and scopes that are not currently writable. Eligible targets are sorted by scope id and the first is selected by default; with the current configuration this means `notes`, followed by `processing`. A successful copy opens the new root in the target scope so the independent result is immediately available.

The action model owns the singular source target. The modal owns target-scope selection and confirmation. The server owns descendant expansion, identity allocation, link remapping, validation, source writes, and rebuild authority.

## Current Boundary

Copy Subtree is available from the manage-only index toolbar for an active tree document when at least one other configured non-public scope can accept canonical writes. Ordinary creation still allocates one independent document, Move still changes hierarchy only inside one scope, and Docs Import still creates from external staged content.
