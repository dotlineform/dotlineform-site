# Recent Documents Delivery

## Outcome

Docs Viewer can answer the route-appropriate Recent question accurately: “what is new?” for a catalogue-like collection and “what changed?” for a changing document collection.

One complete delivery makes `last_updated` a consistently maintained full timestamp for future edits, seeds only the bounded historical set needed to populate current edited projections, and replaces the current Recently Added-specific contract with one generic Recent contract. Public Moments receives an added-date projection. Public Library and Analysis, plus every scope displayed in `/docs/`, receive an edited-date projection. There is no Added/Edited toggle or second Recent control.

## Timestamp Contract

- New and changed `last_updated` values use the same `YYYY-MM-DD HH:MM:SS` local timestamp contract as `added_date`.
- Creating a document sets `added_date` and `last_updated` to the same timestamp.
- Only a change to the canonical body, `title`, or `summary` advances `last_updated`. This positive allowlist must be extended deliberately if another field should ever acquire edited-Recent semantics.
- Source-editor rebuild, title/summary metadata changes, imports/overwrites, and other focused mutation paths use the shared source comparison and timestamp owner rather than maintaining the field independently.
- A no-op request, generated-output rebuild, search rebuild, read, preview, or failed mutation does not change it.
- Direct source edits through VS Code, Codex, or another editor must advance the field when body, `title`, or `summary` changes; watcher/check evidence should surface missing timestamp evidence rather than silently rewriting source.
- A one-time Git-history seed walks backward through renames until each edited Recent projection has enough eligible documents to fill its configured limit, then writes the latest body/title/summary author timestamp only to the union of those documents.
- The seed applies context eligibility before choosing its bounded set. With the current limit of 10, one manage and one public edited projection can require at most 20 distinct seeded documents per scope; the history scan may inspect more commits to find them. Public Moments needs no added-date seed, while editorial Moments needs only its manage set.
- The timestamp-only seed commit does not recursively advance the values it is seeding. A currently uncommitted document instead receives its real write-time timestamp.
- Existing date-only documents outside the seed remain unchanged and are ineligible for edited Recent ordering until their next real edit gives them a full timestamp. The delivery does not rewrite the whole corpus or wait for seeded documents to arise naturally.

The main implementation risk is incomplete write-path coverage. Code and tests are authoritative; do not rely on a permanent documentation inventory of every mutation module.

## Recent Projection

- replace `docs_recently_added_v1` and Added-specific paths, feature names, providers, and labels with one accurately named small Recent contract;
- make the public Moments payload order by canonical `added_date`, and make public Library, public Analysis, and every manage/local payload order by canonical `last_updated`, with each payload declaring its `added` or `edited` basis;
- require every route that enables Recent to declare an allowlisted `added` or `edited` basis; do not infer it from publicness, scope type, route path, route id, or scope id;
- allow the same source scope to have different working and published Recent projections: for example, Moments is recently edited in `/docs/` and recently added in `/moments/`;
- compute each limited projection from the complete eligible source set; do not derive a public added list by reordering an already truncated editor list, or vice versa;
- preserve the configured limit, scope/viewability rules, parent context, lazy loading, and `recent-results` host;
- keep one **Recent** control and make the route's code-owned policy select the basis; publicness alone does not imply recently added, and the browser does not expose an Added/Edited toggle or parallel view;
- keep the projection independent from `index-tree.json` and the search payload;
- include newly created documents naturally because creation is both their added date and first update;
- leave Catalogue-owned `/series/` recently added for the same catalogue reason as public Moments, and leave Docs Review without a Recent surface unless a separate review-package need is designed.

## Code Authority To Inspect

- document timestamp and front-matter helpers under `docs-viewer/services/`;
- management creation, source-editor, metadata, placement, import, lifecycle, and coordinated write/rebuild owners;
- Docs watcher/suppression and focused source-validation checks for direct edits;
- Docs builder Recent payload generation, generated-read services, route configuration, scope manifests, and the public publication projection;
- configured-scope provider, Recent controller/view registration, labels, and focused tests.

These are responsibility areas, not an exhaustive update list.
