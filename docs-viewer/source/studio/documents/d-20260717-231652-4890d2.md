# Extract Imported SVG To Published Media Delivery

## Outcome

Standalone and HTML-inline SVG imports produce sanitized `.svg` media at the target scope's configured published `img` location and compact canonical Markdown image references. No imported SVG markup is retained inline.

The delivery reuses the existing import media plan, location adapter, media-before-source ordering, canonical identity allocation, and coordinated rebuild boundaries.

## Delivery Boundary

In scope:

- normalize standalone and HTML-inline SVG into one sanitized media intent;
- deterministic target-document-owned SVG identities and alt text derived from safe import context;
- preview of the resulting token, media target, warnings, and collision state without writing;
- repository, external-local, and R2 publication through the configured `img` location;
- media-before-source apply, byte verification, source token insertion, rebuild follow-through, and bounded failure reporting;
- focused ordinary, overwrite, multi-SVG, unsafe-SVG, provider, collision, and partial-failure tests;
- promotion of the shipped contract into the existing import and media owners.

Out of scope:

- existing-document migration;
- manual source-editor media insertion;
- Markdown package SVG-link interpretation unless it already reaches the same normalized inline-SVG boundary;
- Mermaid source, rendering, freshness checks, or repository tool installation;
- SVG optimization, visual diffing, or content extraction beyond title/alt assistance.

## Code Authority To Inspect

- `docs-viewer/services/docs_html_markdown.py` for SVG parsing, safety warnings, sanitization, and HTML conversion;
- `docs-viewer/services/docs_import_preview.py` for standalone SVG and HTML preview dispatch;
- `docs-viewer/services/docs_import_media.py` for extracted-media planning, deterministic names, tokens, and provider-independent apply;
- `docs-viewer/services/docs_import_document.py` and import service orchestration for media-before-source mutation and rebuild follow-through;
- scope media configuration and artifact-location adapters for published `img` resolution and byte verification;
- focused Docs Import source-format, media, package, collection, and service tests.

These are responsibility areas, not a permanent module inventory.

## Delivery Tracker

**Next checkpoint:** SVG-1 — define one normalized sanitized SVG media plan and prove preview remains write-free.

Complete checkpoints in order. Keep detailed fixtures, generated SVG bodies, and command transcripts outside this delivery.

### [ ] SVG-1 — Normalize Sanitized SVG Media Intent

- [ ] Reuse the current SVG parser/sanitizer while returning sanitized bytes, title/alt assistance, warnings, and media intent instead of final inline Markdown.
- [ ] Normalize standalone SVG and each HTML-inline SVG through the same media-plan shape.
- [ ] Allocate deterministic target-document-owned identities without writing, including overwrite retargeting and stable multi-SVG ordering.
- [ ] Preview the resulting logical `img` token, target provider label, collision state, and sanitization warnings without exposing absolute paths or credentials.

**Checkpoint evidence:** the same safe input and target state produces one deterministic write-free SVG media plan; unsafe content is absent from its candidate bytes.

### [ ] SVG-2 — Publish Then Write Source

- [ ] Publish sanitized SVG bytes through the target scope's configured `img` location adapter and byte-verify the result.
- [ ] Replace standalone or HTML-inline SVG markup with ordinary Markdown image references using the configured logical media prefix.
- [ ] Preserve the established media-before-source ordering so failed publication cannot commit a broken token.
- [ ] Apply existing collision and overwrite authority without silently replacing an owned artifact.

**Checkpoint evidence:** repository, external-local, and R2 fixtures produce the same logical token while only the configured physical location changes.

### [ ] SVG-3 — Integrate Import And Failure Semantics

- [ ] Carry SVG media plans through ordinary import preview/apply and the shared per-document boundary used by collection imports where applicable.
- [ ] Report sanitization, publication, collision, source-write, and rebuild outcomes separately.
- [ ] Ensure partial media publication or source-write failure leaves an explicit result and never claims an unverified token was committed.
- [ ] Keep imported document identity, timestamps, hierarchy, viewability defaults, and rebuild behavior unchanged apart from SVG representation.

**Checkpoint evidence:** an SVG publication failure stops the matching source write, while a later rebuild failure does not erase successfully committed source and media.

### [ ] SVG-4 — Verify, Document, And Close

- [ ] Prove standalone, HTML-inline, multiple, titled, untitled, unsafe, externally referenced, and malformed SVG cases at the lowest stable layers.
- [ ] Prove deterministic naming, overwrite retargeting, collision refusal, provider-neutral tokens, byte verification, and compact resulting Markdown.
- [ ] Prove existing inline-raster extraction and ordinary image/file import behavior remain unchanged.
- [ ] Update the durable import/media owners, rebuild Studio docs/search, run `git diff --check`, and run the focused private-path/secret scan.

**Checkpoint evidence:** imported source contains media tokens rather than SVG markup, every referenced SVG exists at its registered published location, and existing non-SVG import behavior remains green.

## Done When

- standalone and HTML-inline SVG imports publish sanitized `.svg` media and write compact logical image tokens;
- unsafe original SVG bytes are never published or committed to canonical Markdown;
- target scope configuration alone chooses repository, external-local, or R2 placement;
- media publication and byte verification precede source mutation;
- naming, collisions, overwrite retargeting, partial failures, and rebuild results are explicit and tested;
- current raster extraction and non-SVG imports remain unchanged;
- durable import and media documentation describes the shipped behavior.
