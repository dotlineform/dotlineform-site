# Source Editor Add Image And Add File Delivery

## Outcome

The source editor publishes staged raster images, sanitized SVG, or downloads through separate **Add image…** and **Add file…** actions and inserts verified logical references into the active Markdown buffer. Docs Import reuses the SVG foundation only for media embedded in imported document content and no longer creates wrapper documents for standalone media.

## Delivery Boundary

In scope:

- register and serve a distinct configured `svg` media type;
- share provider-neutral publication, byte verification, collision handling, and SVG sanitization;
- add separate source-editor image and file actions bound to the active scope and buffer;
- extract inline SVG during HTML import and replace it with compact image references;
- retire standalone image, SVG, and download wrapper-document imports after their replacement actions ship;
- preserve existing inline-raster, Markdown-package, trusted-collection, and document-import behavior.

Out of scope:

- migration of existing wrapper documents or inline SVG;
- an asset browser, registry, history, automatic cleanup, or rollback system;
- automatic staged-file deletion;
- Mermaid rendering;
- resolving surrounding HTML CSS, external dependencies, or pixel-identical SVG output;
- automated visual diffing.

## Delivery Tracker

**Current state:** planned. Raster and download publication helpers exist, standalone media currently creates wrapper documents, SVG is serialized inline, and the source editor has no media-publication actions.

**Next checkpoint:** MED-1 — establish the shared publication and SVG media foundation.

### [ ] MED-1 — Shared Publication And SVG Foundation

- [ ] Register `svg` with logical, configured physical, serving, inventory, and owned-directory contracts for repository and external-local scopes; add no initial R2 location.
- [ ] Create one shared SVG sanitizer for staged source-editor SVG and HTML-inline SVG, preserving safe internal styling and fragment references while stripping unsafe or external content.
- [ ] Return sanitized bytes, title/alt assistance, validity, and diagnostics rather than inline Markdown.
- [ ] Normalize provider-neutral media intents and byte verification without changing existing raster `img` or opaque `files` behavior.
- [ ] Keep publication write-free until an owning caller applies a confirmed plan.

**Checkpoint evidence:** the same safe SVG produces the same sanitized bytes and logical identity regardless of configured local provider, and no source or media write occurs during planning.

### [ ] MED-2 — Source Editor Add Image And Add File

- [ ] Add focused endpoints that list compatible direct-child staged files and plan/apply publication for the active scope.
- [ ] Add separate **Add image…** and **Add file…** controls to the existing source-editor owner without exposing provider choices.
- [ ] Collect useful image alt text or a download label and preview the logical identity plus **Replace** or **Cancel** on collision.
- [ ] Publish and byte-verify before inserting Markdown at the current selection or caret.
- [ ] Leave the source buffer dirty for the existing **Rebuild doc** commit path and retain staged files after success.
- [ ] Keep document identity, hierarchy, viewability, and timestamps outside media publication.

**Checkpoint evidence:** raster, SVG, and download fixtures publish to their configured locations, insert provider-neutral references only after verified bytes exist, and create no document.

### [ ] MED-3 — HTML SVG Extraction And Wrapper Retirement

- [ ] Extract each inline SVG while converting imported HTML through the shared sanitizer and allocate `<canonical-doc-id>-image-NN.svg` identities in document order.
- [ ] Publish and verify extracted SVG before writing its Markdown image reference; an SVG failure blocks the matching document source write.
- [ ] Remove standalone raster image, SVG, and downloadable-file options from Docs Import and delete their wrapper-preview/apply branches after MED-2 is integrated.
- [ ] Preserve media extraction that belongs to interpreted HTML, Markdown packages, and trusted collections.
- [ ] Leave historical wrapper documents, inline SVG, and surplus unreferenced media unchanged.

**Checkpoint evidence:** HTML import produces verified SVG media and compact tokens, while Docs Import exposes no standalone-media-to-wrapper workflow.

### [ ] MED-4 — Verify, Document, And Close

- [ ] Prove safe, unsafe, internally styled, fragment-referenced, malformed, collision, unchanged-byte, replacement, publication-failure, and source-edit-abandonment cases at the lowest stable layers.
- [ ] Prove separate image/file actions, current-buffer insertion, staged-file retention, and absence of document allocation.
- [ ] Prove HTML-inline SVG extraction, deterministic numbering, media-before-source ordering, and existing non-media document imports.
- [ ] Manually compare representative raster and SVG results before and after insertion/import.
- [ ] Promote shipped contracts into the durable owners, rebuild Studio docs/search, run `git diff --check`, and run the focused private-path/secret scan.

**Checkpoint evidence:** every inserted or imported reference resolves to verified media, standalone media creates no wrapper document, and the durable docs route to authoritative code.

## Done When

- **Add image…** handles raster and sanitized SVG, while **Add file…** handles allowlisted downloads;
- source-editor actions publish before inserting and leave the current buffer dirty;
- `svg` has a distinct configured logical and physical media contract;
- HTML import extracts inline SVG through the shared sanitizer;
- standalone media is no longer a Docs Import document source;
- no media action allocates or replaces a document identity;
- historical inline SVG and wrapper documents remain valid but receive no compatibility machinery or automatic migration;
- existing document-content imports and embedded-media paths remain green.
