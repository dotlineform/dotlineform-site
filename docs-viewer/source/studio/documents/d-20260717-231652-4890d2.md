# Extract Imported SVG To Published Media Delivery

## Outcome

Standalone and HTML-inline SVG imports produce sanitized `.svg` media at the target scope's configured published `svg` location and compact canonical Markdown image references. No imported SVG markup is retained inline.

The delivery reuses the existing import media plan, location adapter, media-before-source ordering, canonical identity allocation, and coordinated rebuild boundaries.

## Delivery Boundary

In scope:

- normalize standalone and HTML-inline SVG into one sanitized media intent;
- route both sources through one shared SVG sanitizer that preserves safe self-contained styling and strips external references;
- register `svg` as a distinct published media type with logical `docs/<scope>/svg/<filename>` identities and separate physical folders;
- validated original basenames for standalone SVG, deterministic `<canonical-doc-id>-image-NN.svg` identities for HTML-inline SVG, and alt text derived from safe import context;
- unchanged write-free document preview with all SVG filename, token, destination, collision, and replacement details kept internal;
- repository publication for repo-backed scopes and contained external-local publication for external-local scopes through their configured `svg` locations;
- media-before-source apply, byte verification, source token insertion, rebuild follow-through, and bounded failure reporting;
- focused ordinary, overwrite, multi-SVG, unsafe-SVG, provider, collision, and partial-failure tests;
- promotion of the shipped contract into the existing import and media owners.

Out of scope:

- existing-document migration;
- manual source-editor media insertion;
- resolving surrounding HTML CSS, fetching external dependencies, reproducing the browser cascade, or emitting speculative appearance warnings;
- imported-media version history, rollback copies, or external source-version management;
- Markdown package SVG-link interpretation unless it already reaches the same normalized inline-SVG boundary;
- Mermaid source, rendering, freshness checks, or repository tool installation;
- SVG optimization, visual diffing, or content extraction beyond title/alt assistance.

## Code Authority To Inspect

- `docs-viewer/services/docs_html_markdown.py` for current SVG parsing/sanitization and HTML conversion, with shared SVG policy moving into a focused sanitizer module;
- `docs-viewer/services/docs_import_preview.py` for standalone SVG and HTML preview dispatch;
- `docs-viewer/services/docs_import_media.py` for extracted-media planning, deterministic names, tokens, and provider-independent apply;
- `docs-viewer/services/docs_import_document.py` and import service orchestration for media-before-source mutation and rebuild follow-through;
- `docs-viewer/services/docs_scope_config.py`, scope configuration, local media serving/lifecycle, and inventory for the new published `svg` type;
- artifact-location adapters for configured `svg` resolution and byte verification;
- focused Docs Import source-format, media, package, collection, and service tests.

These are responsibility areas, not a permanent module inventory.

## Delivery Tracker

**Next checkpoint:** SVG-1 — define one normalized sanitized SVG media plan and keep the ordinary document preview write-free and unchanged.

## Done When

- standalone and HTML-inline SVG imports publish sanitized `.svg` media and write compact logical image tokens;
- `svg` is a distinct configured published media type with separate logical and physical roots from raster `img`;
- both paths use one shared sanitizer, retain safe self-contained styling, strip external references, and do not resolve surrounding HTML CSS;
- standalone publication preserves the validated staged basename while HTML-inline extraction uses deterministic document-owned numbered SVG filenames;
- unsafe original SVG bytes are never published or committed to canonical Markdown;
- target scope-plus-media-type configuration alone chooses placement, initially repository for repo-backed scopes and contained external-local storage for external-local scopes;
- media publication and byte verification precede source mutation;
- naming, collisions, overwrite retargeting, partial failures, and rebuild results are explicit and tested;
- confirmed document re-import replaces stable SVG media without a separate prompt, fallback filename, history, or rollback copy;
- ordinary document preview remains the only preview, with no SVG-specific media details or handling UI;
- smaller re-imports may leave quiet unreferenced SVG files for later provider-independent inventory rather than automatic cleanup;
- possible CSS-dependent appearance changes do not create speculative warnings, and representative results receive before/after visual review;
- import offers one deterministic published-media representation while manually authored inline SVG remains an intentional exception;
- current raster `img` extraction and non-SVG imports remain unchanged;
- durable import and media documentation describes the shipped behavior.
