# External Asset Collections

## Purpose

Associate a local Docs Viewer scope with a useful inventory of user-owned project files without turning every file into a Markdown document or making Docs Viewer own the files.

This is the feature parent. It keeps the broad idea, architecture, and implementation-ready deliveries separate.

## Feature Documents

- [Concept](/docs/?scope=studio&doc=d-20260714-234030-f2b36c) — problem, desired capability, product boundaries, and unresolved choices.
- [Architecture](/docs/?scope=studio&doc=d-20260714-234030-19a064) — proposed ownership, execution boundaries, extension method, code pressure points, and weak spots.
- [Docs Viewer Roadmap](/docs/?scope=studio&doc=d-20260714-234030-1e0af6) — suggested feature order.
- [Association Delivery](/docs/?scope=studio&doc=d-20260714-234030-a63b0b) — proposed configuration and lifecycle boundary for associating an independently owned collection with a scope.

## Status

- Feature: proposed.
- Ordinary external-local scope media is scope-owned in its configured published location; it is not an External Asset Collection.
- The first collection association delivery is proposed again after separating collection identity from scope identity.
- Add another delivery here only when its complete boundary is understood.

Do not add implementation history or duplicate child content here. When the feature ships, current behavior moves to one durable External Asset Collections document and this parent retains only unresolved delivery links.
