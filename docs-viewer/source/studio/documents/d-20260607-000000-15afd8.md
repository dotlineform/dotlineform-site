# Risk Review Method

## Purpose

Risk review chooses a focused improvement that reduces likely breakage, drift, repair cost, or strategic blockage. It is not a permanent scorecard and does not justify speculative refactoring.

Current evidence should come from code/config inspection, focused runtime checks, Admin Checks reports, audits, tests, and measured workflow behavior. The result belongs in a finishable delivery or concrete maintenance action.

## Review Units

Choose the smallest unit that still explains the risk:

- app when architecture or user workflow is the decision;
- route/file family when several modules form one capability;
- file when one coherent owner is under pressure;
- workflow when source, service, build, media, or generated outputs interact;
- cross-app contract when several owners use the same mechanism.

Review frontend and backend together when they form one app workflow.

## Risk Types

### Architectural Fit

Does the current hosting, framework, app boundary, or strategy still fit what the capability is becoming?

### Structural

Are ownership, inputs, dependencies, and change destinations clear enough to avoid broad context and repeated edits?

### Workflow

Are side effects, validation, partial success, generated artifacts, logs, cleanup, and retry behavior visible and controlled?

### Performance And Cost

Is public/runtime work, rebuild work, local service work, or developer navigation broader and more expensive than the value it provides?

## App Emphasis

| App/surface | Review emphasis |
| --- | --- |
| Public site | performance and architectural fit before internal workflow detail |
| Studio | structural and workflow clarity before local performance tuning |
| Analytics/Data Sharing | architectural fit and structure because future capabilities can compound weak boundaries |
| Docs Viewer | structure plus public/local runtime cost, then architecture and workflow |
| Admin | evidence validity, allowlist/path safety, and whether reports produce actionable decisions |

This is emphasis, not a scoring algorithm.

## Evidence Rules

- “Has an owner” is not proof that a file, config key, endpoint, or generated artifact should exist.
- Static evidence should be checked against a runtime or workflow purpose.
- A large file is a navigation signal, not by itself a refactor mandate.
- Generated output needs a consumer or explicit temporary/cleanup contract.
- Browser-visible config should be an intentional projection, not broad pass-through.
- A half-finished improvement usually indicates that the delivery boundary was too large.

## Closeout

A useful review ends with:

1. evidence and the affected owner;
2. one bounded change or an explicit decision to defer;
3. verification that matches the risk;
4. durable architecture/workflow documentation only if the stable boundary changed.

Do not maintain empty per-app risk inventory pages. Use [Admin Checks](/docs/?scope=studio&doc=d-20260608-225607-868570) for repeatable evidence and bounded deliveries for current action.
