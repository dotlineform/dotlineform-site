# Source Editor Add Image And Add File

## Outcome

An author editing Markdown can publish a staged image or downloadable file to the current scope and insert the correct logical reference without manually locating repository or R2 storage.

The source editor exposes two separate actions:

- **Add image…** publishes an `img` artifact and inserts Markdown image syntax with useful alt text.
- **Add file…** publishes a `files` artifact and inserts a labelled download link.

Downloads are uncommon and have different authoring needs from images. The two actions should not be hidden inside a generic media button merely because their lower-level publication code is shared.

## Interaction Boundary

Each action starts from the current source editor and current scope:

1. list only compatible direct-child files from the shared import-staging root;
2. choose one staged file;
3. collect image alt text and optional dimensions, or a download link label;
4. preview the logical identity, configured provider/location label, and any collision or replacement decision;
5. publish and byte-verify through the scope's registered `img` or `files` location;
6. on success, replace the current selection or insert at the caret;
7. leave the source buffer dirty so the existing **Rebuild doc** action remains the canonical source commit point.

Publication precedes token insertion. Abandoning the later source edit may therefore leave an unreferenced scope-owned asset, but it cannot commit a reference to bytes that failed to publish.

## Boundaries

- This is source editing, not standalone Docs Import: it creates no wrapper document and allocates no new document identity.
- The buttons do not expose repository, external-local, or R2 choices. Scope configuration owns the destination.
- **Add image…** initially handles supported raster images and can include sanitized SVG once [Extract Imported SVG To Published Media](/docs/?scope=studio&doc=d-20260717-231652-d6d49e) has established that media contract.
- **Add file…** handles the existing allowlisted opaque-download formats; it does not interpret their contents.
- Existing media needs no republishing action: an author can type or paste its logical token directly.
- Semantic references remain separate because they create registry-backed Work, Series, or Moment links rather than media references.
- The feature adds no asset library, upload history, automatic orphan deletion, or general file browser.

## Questions Before A Delivery

- What is the smallest picker that remains clear when image staging contains many similarly named files?
- Should image alt text be required before insertion, and how should SVG `<title>` seed it?
- Are width and height entered during insertion, derived when reliable, or left as an author edit?
- Which collision choices are appropriate for repository/external-local files versus cacheable R2 objects?
- Should successful publication remove the staged file, retain it, or make that a separate explicit cleanup action?
- Which existing source-editor toolbar owner should host the two buttons without crowding semantic-reference controls?

## Delivery

No delivery is planned yet. The first delivery should be created only after the separate image and file interactions are simple enough to state without recreating Docs Import inside the source editor.
