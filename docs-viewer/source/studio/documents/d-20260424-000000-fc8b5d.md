# Docs Import

Docs Import turns staged source material into canonical Markdown documents and associated media. It runs inside Docs Viewer management; there is no separate import route.

Open the target `/docs/?scope=<scope>` page and choose the Import toolbar action or `Actions` > `Import`.

## Workflow

![Staged document sources and standalone media use separate workflows, while both publish required media before canonical source is written and rebuilt.]([[media:docs/studio/svg/docs-import-media-boundaries.svg]])

The browser presents choices. The local Docs Viewer service resolves staged paths, validates targets, creates the authoritative plan, performs writes, and rebuilds published payloads.

## Stage The Source

Put source material in the shared drop-zone:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/data-sharing/import-staging/
```

Ordinary document sources must be direct children. A Markdown package is a direct-child folder containing one Markdown file plus its local images or attachments. The picker shows only document formats the server currently accepts; standalone raster, SVG, Mermaid `.mmd`, and download sources belong to the source editor's **Add image…** and **Add file…** actions. [Docs Import Architecture](/docs/?scope=studio&doc=d-20260514-164722-3de65e) owns the format-adapter boundary, while [HTML Import Mapping](/docs/?scope=studio&doc=d-20260719-182348-492a48) documents the current semantic conversion rules.

Do not add Docs Viewer front matter to staged Markdown. Import creates the target document front matter and treats the staged Markdown as its body.

## Import Documents

Use `Documents` for Word `.docx`, HTML, Markdown, text, and Markdown packages. Trusted Data Sharing documents use the separate package mode. Standalone raster, SVG, and downloadable files are not Docs Import sources.

1. Select one or more staged sources.
2. Choose the target scope.
3. For HTML, decide whether obvious prompt/meta sections belong in the document.
4. Choose `Import selected`.
5. Inspect the result and open the imported document.

Multi-file runs process the selected files in list order. A failure stops the run; successfully completed earlier imports remain. Docs Viewer refreshes the target index once and selects the final successful document.

HTML can supply the title; Markdown uses its first H1 when present. Word uses the mapped `Title` style and falls back to the humanized staged filename without promoting its first section heading. [Word Import Style Mapping](/docs/?scope=studio&doc=d-20260719-182348-f326fd) owns the exact Word heading rules. The staged name remains source context. A genuinely new document receives an immutable local `doc_id` and matching Markdown filename when the service plans the write.

Every ordinary source import creates a new document with a fresh immutable identity. Importing the same staged file again creates another independent document; the workflow does not scan target documents or offer document overwrite. New documents in public scopes are created non-viewable so they can be checked in management mode before publication.

### Create-Only Document Imports

Ordinary Word, HTML, Markdown, text, and Markdown-package sources are intake material, not update identities. Their staged filenames and titles remain conversion context only. Update an existing canonical document through the source editor or another workflow that explicitly identifies that document; ordinary import never infers an update from a filename or title.

This create-only rule does not remove collision handling from reviewed collections or stable-name assets. Data Sharing collection records carry explicit document identities and retain their overwrite/skip decisions. Role-marked interactive HTML companions retain explicit asset replacement confirmation.

## Import A Reviewed Collection

Use `Data Sharing packages` for a trusted returned documents package. Docs Review can open the modal with its matching staged package selected, but the handoff contains only the package identity; Docs Import resolves the actual staged source again.

The collection path is deliberately plan-first:

1. Choose the package and target scope.
2. Review every record, parent resolution, warning, and blocker.
3. For collisions, choose `Overwrite`, `Skip`, or `Cancel`. `Apply to all` affects only the remaining document collisions.
4. For invalid records, choose `Skip` or `Cancel`; a skip may include a note.
5. Review the complete resolved plan and confirm apply.

The plan covers the whole collection. A package record that creates a document receives a planned local identity; the package identifier remains source provenance and parent references are remapped to the local identities. There is no ordinary subset selection, automatic overwrite, replacement `doc_id`, or `Create as new` path. Missing undeclared parents, hierarchy cycles, unsafe identities, and package-shape errors block confirmation.

On confirmation the service rereads the staged package and recomputes the target plan. If package identity, collisions, hierarchy, or blockers changed, it returns the refreshed plan without writing. Otherwise it applies records in package order and rebuilds once.

Collection apply is synchronous, not transactional. A source-write failure preserves earlier writes and reports later records as not attempted. Generation failure is reported separately and does not undo successful source changes.

## Media And Companion Assets

Media behavior comes from the target scope configuration:

- each logical media type selects its configured published location
- repository, external-local, and R2 locations use the same provider-independent import plan
- the chosen provider must complete and byte-verify the media write before canonical source is committed

Markdown packages convert supported images to readable, 800px-maximum WebP outputs and copy supported attachments unchanged. HTML and Markdown can extract inline raster data URLs, and HTML-inline SVG is sanitized and published as deterministic image media. Word imports pass embedded PNG, JPEG, GIF, and WebP through the same inline-raster publication path; missing alt text receives a deterministic `Word image NN` label, while unsupported, unreadable, or empty images are omitted with warnings. Exact paths, location providers, collision rules, and media limitations belong to [Media Handling](/docs/?scope=studio&doc=d-20260514-184303-7914e2).

A standalone HTML file marked with:

```html
<meta name="dlf:docs-import-role" content="interactive-html">
```

is treated as a companion asset rather than a selectable source. A normal document import copies marked companions directly to the target scope's configured published `html` root after any required overwrite confirmation. Add the resulting `html-media` token to the document manually; import does not decide where the sandboxed iframe belongs.

## Results And Recovery

An ordinary result identifies the created document, target scope, viewer link, media actions, and conversion warnings. A collection result groups created, overwritten, skipped, failed, and not-attempted records and writes a Markdown report under `import-staging/results/`.

The terminal result remains in the modal until `Close`. Reopen the modal to refresh the staged-source list and start another run.

Docs Import is best-effort conversion. Normal prose, headings, lists, links, simple tables, Markdown, and supported media are reliable inputs. Word layout, presentation-heavy HTML, interactive UI, unresolved package links, and unsupported assets may need source editing after import. Conversion HTML and inline media bodies remain inside the local service; the generated canonical Markdown is always checked by the shared Python renderer before a source write.

For implementation boundaries, extension methods, and known structural weak spots, see [Docs Import Architecture](/docs/?scope=studio&doc=d-20260514-164722-3de65e).
