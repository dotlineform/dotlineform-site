# Dated Export Snapshots

## Outcome

Docs Viewer can write a small, browseable snapshot before significant document or roadmap changes without filling Docs Viewer itself with an archive that later needs organisation and retirement.

The snapshot is rendered static HTML on the external projects drive. It is useful as a visual reference or last-resort manual source of text, but it is deliberately not exact canonical source and has no restore, import, editing, synchronization, or return path.

## Snapshot Modes

- **Whole scope:** export the complete manage tree to `<scope> - YYYY-MM-DD`.
- **Selected subtree:** export one selected parent and every descendant to `<parent title> - YYYY-MM-DD`.

Folder labels use the local export date and only the minimum filesystem-safe normalization needed for titles. The snapshot preserves original document ids inside the non-canonical artifact.

If the destination folder does not exist, confirmation creates it. If it already exists, the only choices are **Replace** and **Cancel**. There is no **Save As** or naming workflow, and apply must never wipe an existing folder before explicit replacement confirmation.

Because duplicate or filesystem-equivalent titles can produce the same dated folder name, each snapshot carries a small provenance record with its source scope, optional root `doc_id`, document count, and generation time. A replacement prompt identifies the existing snapshot rather than treating the folder name as identity.

## Boundaries

- Snapshot input is existing generated Docs Viewer payloads, including the manage-visible non-viewable tree; it does not parse or package canonical Markdown.
- Any configured scope shown in `/docs/` may participate when its generated tree and document payloads are readable, regardless of whether its source is public, repo-local, or external-local.
- The export copies no media, download, or interactive-asset bytes. Rendered references retain their existing targets.
- Whole-scope and subtree snapshots share one service contract, destination policy, HTML renderer, and Replace/Cancel boundary.
- Snapshot folders remain user-owned external files. Docs Viewer does not index, list, retain, prune, or manage them as an archive.
- The feature remains separate from Publish, Data Sharing packages, and Copy Subtree To Scope.

## Current Boundary

The current [Export](/docs/?scope=studio&doc=d-20260629-152159-ac1af9) supports one whole repo-backed local scope at a time, always writes to `/docs-export/<scope>/`, and replaces that folder on every apply. It has no subtree mode, dated naming, or preflight replacement choice.

## Delivery

No delivery is planned yet. This enhancement follows the independently useful Copy Subtree feature but does not depend on it.

## Durable Home

[Export](/docs/?scope=studio&doc=d-20260629-152159-ac1af9) will own the shipped snapshot contract and replace its current latest-folder behavior.
