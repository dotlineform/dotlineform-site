# Docs Viewer Source And Publication Model Refactor Delivery

## Outcome

Docs Viewer first gains one role-first source, build, published, location, exposure, and ownership model plus provider-independent workflows. The folder cutover, media-type configuration, builder seams, and file moves implement that model rather than being the feature's primary purpose.

Every configured scope then uses one explicit ownership contract: flat canonical Markdown under `documents/`, build-source media under source `media/`, nested collections under `sub-scopes/`, and directly usable media stored once at its registered published root. Repo-backed scopes select repository or R2 backing per media type independently of whether they also have a public projection; external-local scopes keep every source and published media type inside their external scope folder. Library and Analysis images move directly from R2 into their tracked site scope roots, Moments images and ordinary downloadable files remain in R2, and document references stay stable.

The Docs Viewer-owned repository and external-local output workspaces become `published/` rather than `generated/`. Scope configuration becomes the only topology authority; public `site/assets/`, R2 locations, and logical service routes remain independent.

## Benefits And Risks

Benefits:

- source ownership, document flatness, media identity, and sub-scope boundaries become visible from the filesystem;
- small directly usable assets become reviewable repository-published artifacts while large or externally originated binaries avoid redundant copies;
- manual outputs and future Docs Viewer-owned media builders share one published target contract;
- changing a repo-backed media type between repository and R2 becomes a config-and-file migration rather than a document rewrite;
- build, import, inventory, and publish orchestration use one role-based workflow while location helpers contain filesystem and R2 differences;
- external-local scopes gain the same visible structure without acquiring a repository/R2 decision or losing their self-contained lifecycle.

Risks:

- the cutover touches every source reader, writer, watcher, lifecycle path, builder, import path, and public projection;
- moving public assets can cause broken document renders unless source, config, built payloads, and publication change together;
- R2 contents may include unreferenced, stale, or uniquely held bytes, so remote cleanup requires byte-level evidence and an explicit final target list;
- external-local scopes need the same migration contract without assuming repository paths or silently losing iCloud-backed content.

## Delivery Boundary

In scope:

- the canonical document/build-source workspace and published-media topology;
- flat `documents/`, typed build-source `media/`, namespaced `sub-scopes/`, and registered published media roots for repo-backed and external-local scopes;
- a shared role/location contract and repository, external-local, and R2 adapters that keep provider branching out of the main build, import, inventory, and publish workflows;
- repository and external-local Docs Viewer output roots renamed from `generated/` to `published/`, with configuration replacing literal path assumptions across validation, lifecycle, serving, and browser projection;
- media-type-level repository/R2 storage configuration for repo-backed scopes, with stable logical media identity and no dependency on public projection status;
- external-local media kept entirely beneath each external scope's registered `media/` root;
- source loading, mutation, import, lifecycle, watcher, builder, media inventory, serving, publication, and validation changes required by the cutover;
- migration of current repo and external-local document source;
- migration of Analysis Tags and the Library HTML media asset into their registered document/sub-scope and published-media roots;
- exact Docs Viewer R2 inventory, reference comparison, Library/Analysis image migration, and confirmed redundant-object cleanup;
- focused current-state documentation and removal of superseded paths/contracts without compatibility aliases.

Out of scope:

- implementing the Mermaid renderer or another new format builder;
- new video, audio, Processing, or visualization rendering features;
- redesigning the working Catalogue/Moments R2 media system;
- automatic lifecycle cleanup for independently owned External Asset Collections;
- cache fingerprinting, object versioning, purge automation, or a general R2 asset manager;
- moving ready PDFs, Word files, archives, or other downloads into repository publication merely for structural symmetry.
