# Source Editor Add Image And Add File

## Outcome

An author editing Markdown can publish staged media to the current scope and insert the correct logical reference without creating a wrapper document or choosing a storage provider.

The source editor exposes two distinct actions:

- **Add image…** publishes a supported raster image or sanitized SVG and inserts Markdown image syntax with useful alt text.
- **Add file…** publishes an allowlisted opaque download and inserts a labelled Markdown link.

Docs Import remains for material interpreted as document content: HTML, Markdown, text, trusted document packages, and their embedded media. Standalone image, SVG, and download files belong to these source-editor actions rather than Docs Import.

## Authoring Flow

Each action starts from the active source editor and scope:

1. list only compatible direct-child files from the shared staging root;
2. choose one staged file and collect image alt text or a download label;
3. preview its logical media identity and any existing-media collision;
4. publish and byte-verify through the scope's registered media location;
5. after success, replace the current selection or insert at the caret;
6. leave the source buffer dirty so **Rebuild doc** remains the canonical source-write boundary.

Publication precedes token insertion. Abandoning the later source edit may leave an unreferenced media object, but cannot commit a reference to bytes that failed to publish. Staged-source cleanup remains a separate explicit action.

Collision choices are **Replace** or **Cancel**. The action shows the stable logical identity, not repository, external-local, or R2 details. Replacing media does not replace a document or allocate a document identity.

## SVG And Embedded-Media Contract

`svg` is a distinct published media type. One shared sanitizer handles staged SVG selected through **Add image…** and inline `<svg>` found while Docs Import converts HTML. It preserves safe self-contained styling and fragment references, strips scripts, event handlers, processing instructions, and external references, and returns sanitized bytes plus title/alt assistance and diagnostics.

- A staged standalone SVG retains its validated basename as its media identity.
- SVG extracted from imported HTML uses deterministic `<canonical-doc-id>-image-NN.svg` identities in document order.
- HTML import publishes and verifies extracted SVG before writing Markdown image tokens.
- Manually authored inline SVG remains an expert exception; existing inline SVG and wrapper documents are not migrated automatically.

The same provider-neutral publication boundary serves raster `img`, vector `svg`, and opaque `files`, while their handling and logical roots remain distinct.

## Wrapper-Import Retirement

After both source-editor actions ship, remove standalone raster image, SVG, and downloadable-file sources from Docs Import together with their wrapper-document preview and apply paths. Docs Import continues to extract or publish media that belongs to interpreted document content, including HTML, Markdown packages, and trusted collections.

An author who wants a media-only document can explicitly create a document and then use **Add image…** or **Add file…**. That composition does not justify a second implicit wrapper-document workflow.

## Boundaries

- No asset library, upload history, automatic orphan deletion, version history, or rollback copies.
- No repository/R2 choice in the editor actions; scope configuration owns physical placement.
- Existing published media can still be referenced by typing or pasting its logical token.
- Semantic references remain separate because they create registry-backed Work, Series, or Moment relationships.
- Mermaid rendering remains owned by [Diagrams](/docs/?scope=studio&doc=d-20260715-181845-0158f9), which can later publish through the same `svg` media type.
