# Extract Imported SVG To Published Media

## Outcome

Docs Import stores sanitized SVG once at the target scope's configured published `svg` location and writes a logical media token into canonical Markdown instead of retaining a large inline `<svg>` block.

For a standalone staged `.svg` file, import sanitizes and publishes the file as image media, then creates wrapper Markdown that links to it. For an inline `<svg>` encountered while converting imported HTML, import extracts that element, sanitizes and publishes it, then replaces its markup with an image link. **Extraction** therefore describes only the HTML-inline case; both paths share the same publication contract.

This improves canonical Markdown editability rather than adding SVG editing to Docs Viewer. A separately published `.svg` can be handed to an external editor or ChatGPT and replaced without changing its stable Markdown token, but media browsing, editing, and replacement controls remain separate work.

## Shared Sanitization Flow

```text
standalone .svg file ──┐
                       ├─> shared SVG sanitizer ─> sanitized SVG bytes ─> media plan
HTML inline <svg> ─────┘
```

## Import Contract

- Standalone and HTML-inline SVG pass through one shared SVG sanitizer. Source discovery and media planning remain caller concerns; parsing, safety policy, external-reference removal, and sanitized serialization have one owner.
- SVG is parsed and sanitized before publication. Script blocks, event-handler attributes, and other rejected content never reach the published artifact or canonical Markdown.
- Safe self-contained styling already inside the SVG is preserved, including presentation attributes, element `style` attributes, and internal `<style>` rules. Import does not resolve surrounding HTML CSS, compute inherited page styles, or fetch external stylesheets, fonts, images, or other resources.
- External references are stripped wherever they occur, including element reference attributes, stylesheet imports, external font sources, processing instructions, and CSS `url(...)` values. Same-document fragment references such as `#symbol` and `url(#gradient)` remain available when their target survives sanitization.
- Sanitization is best effort for visual fidelity. Import does not emit speculative warnings about styling that may have depended on the original page; before/after visual review determines whether the published result is acceptable. Malformed SVG and actual sanitization, publication, or source-write failures remain explicit import results. The sanitizer may retain non-user-facing removal diagnostics for tests and troubleshooting.
- A standalone SVG keeps its validated staged basename when published: `diagram.svg` remains `diagram.svg`. An SVG extracted from imported HTML uses the existing document-owned extracted-media convention, `<canonical-doc-id>-image-NN.svg`, numbered from `01` in document order. Overwrite imports retarget these generated HTML-inline identities to the existing canonical `doc_id`.
- The target scope's `svg` configuration selects the physical published location. Import logic does not branch on provider or choose a location from the `.svg` suffix.
- Media publication and byte verification complete before source is written.
- The existing document preview remains write-free and unchanged. SVG filenames, tokens, physical destinations, collision state, and replacement details stay internal; this feature adds no SVG-specific media preview or handling UI. Only actual import failures surface through the existing result boundary.
- Canonical Markdown contains an ordinary image reference such as:

<pre><code>![Diagram](&#91;&#91;media:docs/&lt;scope&gt;/svg/&lt;identity&gt;.svg&#93;&#93;)</code></pre>

- A useful SVG `<title>` supplies the initial image alt text when available. The author can refine that text after import.
- Existing manually authored inline SVG remains valid; this feature changes import conversion, not the general Markdown authoring contract.

## Media Type And Location Decision

`svg` is a separate published media type rather than another suffix inside `img`:

- `img` remains the raster-image contract, used mainly for photographic and other pixel-based media;
- `svg` owns sanitized vector image publication, including imported diagrams and later Mermaid output;
- distinct `media/img/` and `media/svg/` folders make browsing, external editing, sanitization ownership, and future producer relationships clearer;
- the split is justified by stable format and handling differences, not by an assumption that every file extension requires its own media type.

Storage policy is decided for each **scope plus media type**. The initial `svg` policy is repository storage for repo-backed scopes because SVG is generally small, and storage inside the external scope container for external-local scopes because their lifecycle remains self-contained. No scope initially configures `svg` in R2. Configuration remains authoritative, so a later location change does not require canonical Markdown rewriting and must not be implemented as hidden suffix-based routing.

**Planning baseline:** the Docs source, published workspace, and public scope-media roots contain no `.svg` files. The new type therefore starts empty and requires no media migration, compatibility alias, or legacy token handling.

The initial logical and physical shapes are:

```text
docs/<scope>/svg/<filename>

site/assets/data/docs/scopes/<scope>/media/svg/       # public repo-backed scope
docs-viewer/published/docs/<scope>/media/svg/         # local repo-backed scope
<external-scope>/media/svg/                           # external-local scope
```

Downstream Mermaid and source-editor media features must publish SVG through this type, while raster images continue to use `img`.

## Representation Decision

Docs Import does not offer an **Extract** versus **Keep inline** choice. Imported SVG always becomes sanitized published media:

- leaving a standalone `.svg` inline would deliberately expand a separate source file into Markdown;
- retaining HTML-inline SVG would not restore surrounding page CSS that HTML import already discards;
- two canonical representations would multiply preview, overwrite, sanitization, and verification paths;
- the author often cannot judge the better representation until seeing the rendered result.

Manually authored inline SVG remains a deliberate expert exception. If before/after review exposes an unacceptable result, the author can edit or replace the separate SVG, or intentionally substitute inline SVG in canonical Markdown. Add an import-time representation choice only if repeated real examples show that this escape hatch is insufficient.

## Re-import And Versioning

Docs Import maintains the current published SVG representation; it does not manage versions of the external source:

- the existing document-overwrite confirmation is the sole authority needed to replace that document's stable SVG media identities;
- matching published bytes remain `unchanged`, while changed bytes replace the existing object and are byte-verified before source mutation;
- re-import never creates `_2` or another fallback filename, asks for a separate media-overwrite confirmation, or retains history and rollback copies;
- when a later HTML re-import produces fewer SVGs, surplus numbered files may remain as quiet unreferenced media; import does not delete them or emit an orphan warning;
- authors who need SVG source versions keep them outside published Docs media;
- an existing media target during an unrelated new-document import is an ownership anomaly and blocks the import rather than being renamed or replaced.

## Boundaries

In scope:

- standalone SVG source import;
- inline SVG extracted from imported HTML;
- one shared sanitization policy for both sources, including safe internal-style preservation and external-reference removal;
- registration, configuration, serving, inventory, and lifecycle support for the published `svg` media type;
- sanitized SVG serialization, published-media planning, token rewriting, collision handling, and provider-independent apply;
- unchanged write-free document preview, focused apply/failure verification, and before/after rendered-result review.

Out of scope:

- scanning or migrating existing source documents that already contain inline SVG;
- extracting SVG from arbitrary existing Markdown;
- resolving surrounding HTML CSS, reproducing the browser cascade, fetching external dependencies, or guaranteeing pixel-identical output;
- a Mermaid renderer or registered Mermaid build producer;
- source-editor **Add image** or **Add file** actions;
- a general SVG editor, optimizer, formatter, media-replacement UI, or asset registry;
- imported-media version history, rollback copies, or external source-version management;
- automated visual diffing or speculative CSS-dependency warnings.
