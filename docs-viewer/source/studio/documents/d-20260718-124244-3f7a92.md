# Git

## Purpose

Keep Git mechanics safe, visible, and lightweight while the user concentrates on product and architectural decisions. The sibling [Working Method](/docs/?scope=studio&doc=d-20260718-114201-4e7a93) owns environment, scope-data, documentation, and concurrent-change rules.

## Responsibilities

Codex may handle the operational Git work when explicitly asked to activate or advance it:

- inspect dirty state and attribute changes before grouping them;
- create and verify branches and worktrees;
- propose and make focused commits without sweeping in unrelated files;
- keep `main` and the refactor branch current at agreed checkpoints;
- account for watcher-generated output with its source changes;
- inspect and resolve conflicts through the current owners;
- maintain a draft pull request when useful;
- prepare integration, restart, recovery, and closeout steps;
- remove completed worktrees and branches after approval.

The user retains decisions about the core boundary, delivery scope, discovered work, proposal constraints, migrations, product behavior, and final integration. This delegation is not blanket authorization for commits, pushes, merges, migration, deletion, or branch removal; perform those state-changing steps only after the relevant request or approval.

## Status Reporting

At meaningful points, report only the state needed for the next decision:

```text
main: clean / current
refactor: 3 commits ahead; 1 main commit not yet integrated
active delivery: application-root contract
conflicts: none
next safe action: merge main into refactor
```

Include exact commit identities, dirty files, checks, or recovery details when they materially affect the action. Do not turn routine Git maintenance into a running command transcript.

## Branch And Worktree Model

Keep the current checkout on `main` as the stable application. Create a sibling worktree on `feature/docs-viewer-core` only when the first implementation delivery is ready.

A worktree has its own branch, index, and tracked files but shares Git history and objects with the repository. Merge branches and commits, not folders. Bring completed `main` changes into the refactor regularly; feature changes enter `main` only at validated integration points.

Planning and coordination documents remain canonical on `main`. Merge their committed updates into the refactor branch; do not maintain competing authoritative versions.

## Worktree Creation Gate

Create the long-lived branch and worktree only when:

- the main-owned planning baseline is committed and readable through the stable Docs Viewer;
- Core Definition Baseline v1 has been accepted and its remaining uncertainty is bounded;
- the [Docs Viewer Source Documentation And Linting Pilot Delivery](/docs/?scope=studio&doc=d-20260718-125756-a41c70) is complete and its focused commands are available;
- Transition Architecture identifies the smallest safe first slice;
- the first implementation delivery has an exact outcome, current and target owners, checks, and recovery point;
- affected proposals and concurrent work have been assessed in both directions;
- unrelated dirty files on `main` are separated or explicitly accounted for;
- the exact baseline commit, branch name, and sibling path are recorded;
- the main-to-refactor integration rhythm is agreed;
- isolated ports, external data, logs/caches, allowed services, and remote-write policy are decided;
- implementation work is ready to begin immediately.

If pre-delivery analysis needs write-based experiments, use a disposable spike branch or worktree with one explicit question and a discard-or-promote decision. Do not silently turn it into the long-lived refactor branch.

## Activation

After the gate passes:

1. verify the baseline commit and current dirty state;
2. create the branch and sibling worktree;
3. verify its branch, head, and clean initial state before editing;
4. create its untracked `.env.local` and isolated data roots;
5. confirm that it cannot write to real external-local or production remote targets;
6. start only the services needed by the active delivery.

Proposed command shape:

```bash
git worktree add <sibling-path> -b feature/docs-viewer-core <baseline-ref>
```

Resolve and verify both placeholders before execution.

## Maintenance

- Begin a delivery or integration step by checking both worktrees and their divergence.
- Keep commits focused on one completed responsibility or explicit checkpoint.
- Do not use compatibility aliases to make old and new branches appear synchronized.
- Merge `main` into the refactor at controlled intervals while conflict context is fresh.
- Resolve source and configuration first, then regenerate rather than hand-merging generated payloads.
- Treat a conflict as an ownership question before treating it as a text-editing problem.
- Do not make a branch history look clean by hiding unresolved behavior or unrelated user changes.

## Integration

Before feature changes enter `main`:

- verify the exact `main` and feature heads and all uncommitted files;
- complete the delivery's declared [Testing](/docs/?scope=studio&doc=d-20260718-123734-6d84c1) evidence;
- confirm documentation and generated outputs follow their canonical source;
- assess app-data, config, external-scope, and open-proposal effects;
- record the recovery point and affected-service restart procedure.

Stop normal services only when the controlled merge, migration, or restart requires it. Each intermediate feature-to-`main` integration must be independently complete; it is not synchronization for its own sake.

## Closeout

After the Core Refactor is integrated and verified, account for remaining commits and temporary branches, then remove the worktree and feature branch only after explicit approval. Report what was removed, the retained recovery point, and whether any temporary data roots or remote test targets remain.
