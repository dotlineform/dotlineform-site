# Docs Viewer Source Organisation

Each Docs Viewer scope is an independent canonical Markdown collection. Shared reader and management code does not merge scopes into one corpus.

## Source, Build, Published, Location, And Exposure

### Core Terms

| term | meaning |
| --- | --- |
| **Document** | The conceptual entity identified by `doc_id`, not any one file or rendered representation. |
| **Source** | The one canonical application/build input retained by Docs Viewer for an artifact that must be transformed. A file is not source merely because it originated elsewhere or sits below a path named `source`. |
| **Build** | A manual or automated transformation from source to a published artifact. |
| **Published** | The one consumer-ready artifact. Published means ready for Docs Viewer to reference, render, serve, or download; it does not mean publicly accessible on the internet. |
| **Location** | The registered physical root and backing provider where a source or published artifact is held: repository, external-local filesystem, or R2. |
| **Public exposure** | Whether an artifact is available through a public route or snapshot. Exposure is independent of artifact role and storage provider. |
| **Rendered view** | The readable or interactive result composed by the viewer from document payloads and referenced published media. It is not the document source. |

`generated` describes provenance: an artifact was produced by a build. It is not the umbrella term for consumer-ready artifacts because images, HTML, files, and other published media may require no generation.

For a normal document, the Markdown file is document source, the Docs Viewer build produces published document payloads, and the viewer composes the rendered document. The document itself persists across those representations.

### Per-Media-Type Contract

The source/build/published decision is made per configured media type within a scope, such as `img`, `svg`, `files`, `html`, or a future build-source type. It is not one all-or-nothing scope setting. Intake may use a suffix to select an already-defined contract, but the type owns delivery and location rather than merely mirroring that suffix.

Each configured published media type has:

- zero or one registered source location;
- zero or one registered build relationship;
- exactly one registered published location.

If no build is required, there is no Docs Viewer source for that media type. Import, upload, or manual placement writes the ready bytes directly to the published location without creating a second source copy.

If a build is required, one source feeds one published target. Source is read-only to the build, and output is written directly to the published location. Changing a producer from manual to Docs Viewer automation does not change either identity or path; competing producers for one published identity are rejected.

Document and search payloads have separately configured working and public projections. Those payload-delivery contracts do not justify duplicate media bytes.

### Roles Do Not Choose Locations

Roles and locations are configured independently. The same workflow reaches repository, external-local, or R2 artifacts through the common location boundary.

Current configured support is:

| artifact role | repository | external-local filesystem | R2 |
| --- | --- | --- | --- |
| Canonical document source | Supported. | Supported. | Not supported. |
| Media build source | Supported beneath the scope source container. | Supported beneath the external scope container. | Not currently configured. |
| Document/search published workspace | Supported. | Supported. | Not supported. |
| Published media | Supported. | Supported. | Supported. |
| Public document/search projection | Supported beneath `site/assets/data/`. | Not supported. | Not supported. |

Provider capability and scope policy remain separate. Repo-backed scopes may select repository or R2 per media type regardless of public projection. External-local scopes deliberately keep all owned media in their external scope folder.

### Location Adapter Contract

Build, import, inventory, and publish orchestration address a logical scope, artifact role, media type, and confined filename identity. Location helpers own provider-specific details:

- resolving and confining repository paths, external workspace paths, or R2 object keys;
- listing, reading, writing, replacing, deleting, statting, and byte-verifying artifacts;
- acquiring remote authentication and clients;
- temporary local staging when a workflow needs a filesystem path;
- resolving the served reference independently of physical backing;
- reporting required capabilities before mutation.

Repository and external-local providers share filesystem mechanics but retain different root and lifecycle policies. R2 implements the same artifact operations through object storage. A provider's capability does not itself authorize a role/location combination; scope policy and workflow requirements still apply.

### Docs Viewer Published Workspace

The Docs Viewer-owned local payload workspace is a published location:

```text
docs-viewer/published/docs/<scope>/
docs-viewer/published/search/<scope>/

$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/published/docs/<scope>/
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/published/search/<scope>/
```

Scope configuration is the topology authority. Builders, readers, serving code, validators, lifecycle operations, manifests, and browser-config generation resolve registered locations rather than constructing a literal workspace root.

Public website snapshots remain beneath `site/assets/data/`, and R2 objects remain at their configured published locations. Service routes such as `/docs/index-tree`, `/docs/doc`, and `/docs/search` are logical interfaces and do not mirror physical directory names. No compatibility route or filesystem alias exists for the retired `generated/` workspace.

Public scopes may have a local published management projection plus a separately exposed public snapshot. That document/search projection is distinct from the one-published-location rule for media.

### Folder And Scope Shape

A repo-backed source scope makes document source, build-source media, and sub-scopes visible:

```text
docs-viewer/source/<scope>/
  documents/
    *.md
  media/
    mermaid/
      *.mmd
  sub-scopes/
    <sub-scope>/
      documents/
        *.md
      media/
        <build-source-type>/
```

`documents/` is flat. Document hierarchy remains in `parent_id`; structural directories do not define navigation. A sub-scope repeats the same `documents/`, `media/`, and `sub-scopes/` shape while remaining excluded from the parent's normal tree and search.

Directly usable media has no redundant source copy. Its published root is registered separately, for example:

```text
docs-viewer/published/docs/<scope>/media/<type>/
site/assets/data/docs/scopes/<scope>/media/<type>/
R2: docs/<scope>/<type>/
```

An external-local scope is a self-contained physical container:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/source/<scope>/
  documents/                 # document source
  media/
    mermaid/                 # build source, when configured
    img/                     # published media, when configured
    svg/                     # published vector media, when configured
    files/                   # published media, when configured
    html/                    # published media, when configured
  sub-scopes/
```

The container name does not change each child's registered role. External-local media follows the scope's create, rename, and delete lifecycle and is never redirected to repository or R2.

### Examples

| artifact | source | build | published artifact |
| --- | --- | --- | --- |
| Markdown document | Canonical `.md`. | Docs Viewer document build. | Document/search payload consumed by the viewer or public route. |
| Mermaid diagram | `media/mermaid/diagram.mmd`. | Mermaid to SVG through the registered Docs media producer. | `media/svg/diagram.svg` at the configured `svg` published location. |
| JPEG or WebP image | None. | None. | Image placed once at the configured `img` location. |
| Ready SVG image | None. | Sanitization occurs at intake, not as a retained build relationship. | SVG placed once at the configured `svg` location. |
| Ready HTML companion | None. | None. | HTML placed once at the configured `html` location; sandbox behavior is separate. |
| PDF, Word file, or archive | None. | None. | File placed once at the configured `files` location. |
| Video requiring transcoding | Retained canonical master. | Registered transcode. | Delivery video at the configured `video` location. |
| Ready video | None. | None. | Delivery video placed directly at its configured location. |

### Ownership And Inventory

A scope owns artifacts through all of its registered roots, not only containment beneath `docs-viewer/source/<scope>/`. Registered roots may include document/build source, repository-published media, R2 prefixes, external-local media, document/search payloads, and public projections.

Provider-independent inventory records the logical scope/type/filename identity, semantic role, provider, producer relationship, served path, references, and byte evidence. An owned artifact need not currently be referenced. Conversely, a document reference alone does not prove that Docs Viewer owns the target bytes.

External Asset Collections remain independently owned roots even when Docs Viewer inventories or references them.

### Repository And R2 Policy

Storage is a practical per-media-type decision:

| scope and media type | published location |
| --- | --- |
| Library and Analysis `img` | Repository at `site/assets/data/docs/scopes/<scope>/media/img/`. |
| Public-scope `svg` | Repository at `site/assets/data/docs/scopes/<scope>/media/svg/`, independently of raster or download storage. |
| Library `html` | Repository at `site/assets/data/docs/scopes/library/media/html/`. |
| Moments `img` and `files` | Existing R2 `docs/moments/<type>/` contract. |
| Repo-backed downloadable `files` | R2 unless a specific type-level decision says otherwise. |
| Studio and Processing ready local media | Their configured `docs-viewer/published/docs/<scope>/media/<type>/` roots. |
| Mermaid | Source `media/mermaid/`; rendered SVG at the configured published `svg` root. |
| External-local media | Inside the same external scope folder for every type. |

Logical `docs/<scope>/<type>/<filename>` identity stays independent of provider. Moving a media type between repository and R2 requires a config change, byte-verified movement, dependent payload rebuild, and explicit removal of the old copy; it does not require canonical document rewriting.

R2 is chosen for practical size and provenance, not because a suffix inherently belongs there. Existing cache behavior remains unchanged; Docs Viewer adds no automatic cache purge, fingerprint, or remote lifecycle cleanup.

### Build Direction

A full Docs Viewer build follows one direction:

1. inventory canonical document source, registered media build source, and existing published media;
2. run registered media builders from source directly into each configured published location;
3. render document payloads against the published media inventory;
4. publish document and search projections where configured.

The build never writes reproducible output back into source. Media requiring no build bypasses the producer stage and remains at its one published location.

## Other Consumers

Data Sharing and Docs Import read or produce canonical document content through Docs Viewer-owned source and rendering boundaries. Returned-package previews and export artifacts are working data in the external Data Sharing workspace, not configured Docs Viewer scopes or alternative canonical roots.

## Weak Spots

- Flat `documents/` keeps discovery simple but makes `doc_id` and front matter the only structural organization.
- Canonical source, local published payloads, and public snapshots may temporarily disagree until rebuild or Publish completes.
- Scope config is centralized, so manual path edits can affect builders, management reads, media, lifecycle, and public delivery together.
- Directory-level lifecycle ownership is coarser than semantic ownership of every future file below that directory.
- Remote cleanup remains explicit and inventory-backed; there is no reference-counted R2 lifecycle.
- Mermaid is the first registered build-source producer. Each participating scope opts in through reciprocal `source.build_media` and published `svg.build_inputs` configuration; [Document Diagrams](/docs/?scope=studio&doc=d-20260719-123719-fb7565) owns the authoring, dependency, build, and reader boundary.
