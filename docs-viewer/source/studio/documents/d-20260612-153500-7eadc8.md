# GitHub Actions

## Current Workflow

`.github/workflows/public-site.yml` is the checked public-site validation/deployment workflow. The YAML is the authority; GitHub repository variables, Pages settings, runs, artifacts, and live deployment state are external and must be inspected live when they matter.

## Trigger And Jobs

Current triggers:

- `pull_request` when the workflow, `bin/site-validate`, `site/**`, or `site-tools/**` changes;
- manual `workflow_dispatch`.

Current flow:

```text
checkout
  -> Python 3.12
  -> python site-tools/site_validate.py
  -> configure Pages
  -> upload tracked site/ as Pages artifact
  -> evaluate deploy gate
  -> optional deploy-pages job
```

The workflow uploads `site/` directly. There is no site build/copy step.

## Manual Production Gate

Production deployment is intentionally manual. The normal release sequence is:

```text
change checked locally
  -> preview site/ locally
  -> merge/update main
  -> manually dispatch Public site on main
  -> GitHub validates the exact tracked artifact
  -> gated Pages deployment
```

Typical CLI trigger:

```bash
gh workflow run public-site.yml --ref main
```

Inspect the resulting run before treating the site as deployed.

Deployment requires all of:

- ref is `refs/heads/main`;
- repository variable `PUBLIC_SITE_PAGES_DEPLOY_ENABLED` is exactly `true`;
- event is `push` or `workflow_dispatch`;
- validate job succeeded.

Repository variables and Pages source/environment settings are not tracked files. Changing them or triggering a live deployment is an external state change requiring explicit user intent.

The manual dispatch is the deliberate gate between local preview and the live `www` site. Do not add an automatic push trigger unless the release policy is intentionally changed.

## Workflow Cleanup Note

The deploy-gate shell still accepts `push`, but the workflow deliberately has no `push` trigger. That branch is currently unreachable residue, not missing deployment capability.

It can be removed in a focused workflow cleanup so the YAML expresses the manual policy directly. Until then, ordinary pushes do not start this workflow. Whether the live site was actually deployed by a manual run must be checked on GitHub, not inferred from this document.

## Debugging

1. Inspect the current workflow run and failing step.
2. Reproduce validation locally with:

```bash
bin/site-validate
```

3. Separate repo validation failures from GitHub-only permissions, Pages configuration, environment protection, artifact, or deployment failures.
4. Fix the owning source/workflow and rerun only with publishing authority.

## Change Method

- Keep pull-request triggers narrow to public artifact/validator ownership and production deployment manual.
- Pin/update actions deliberately and review their release/migration notes.
- Preserve least-privilege default permissions; grant Pages/id-token only to deploy.
- Keep concurrency/cancellation intentional.
- Test YAML/validation locally where possible, then inspect a real run.
- Update this page only when the workflow architecture, gate, or operator method changes—not for run IDs or current external status.

## Weak Spots

- Pull requests currently configure/upload a Pages artifact even though they cannot deploy; this may be more work/permission surface than validation needs.
- The unreachable `push` branch in the deploy-gate script obscures the deliberate manual-only policy.
- The deployment variable is one gate among external Pages/environment settings and does not prove readiness alone.
- Action versions and GitHub Pages requirements can change; the YAML must be periodically verified against current GitHub behavior.
