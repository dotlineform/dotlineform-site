# Search Build Pipeline

Both search domains follow the same high-level flow:

```text
canonical source -> domain projection -> search builder -> static JSON index -> browser runtime
```

The shared shape is a convention, not a shared build service.

## Catalogue

```bash
$HOME/miniconda3/bin/python3 studio/services/catalogue/search/build_search.py --scope catalogue
$HOME/miniconda3/bin/python3 studio/services/catalogue/search/build_search.py --scope catalogue --write
```

- input: generated public catalogue indexes plus per-work public payloads
- dependency registry: `studio/services/catalogue/search/build_config.json`
- output: `site/assets/data/search/catalogue/index.json`
- runtime: `site/assets/js/catalogue-search.js` and `site/assets/js/search/catalogue-search-runtime.js`

Targeted Catalogue updates use `--only-records kind:id`. Current config is additive-only: a new work or series may be inserted, while changes to an existing record require a full Catalogue search rebuild.

## Docs Viewer

```bash
$HOME/miniconda3/bin/python3 docs-viewer/build/build_search.py --scope studio
$HOME/miniconda3/bin/python3 docs-viewer/build/build_search.py --scope studio --write
```

- input: Markdown front matter from the configured scope source
- scope/output registry: `docs-viewer/config/scopes/docs_scopes.json`
- local/published output: configured per scope
- runtime: shared Docs Viewer search modules under `site/docs-viewer/runtime/js/shared/`

Targeted Docs updates use `--only-doc-ids`. Deleting, renaming, or making a target non-viewable also requires `--remove-missing`. The local watcher calculates affected ids and falls back to a full same-scope build when it cannot prove a safe target set.

## Build Guarantees

- dry-run is the default
- `--write` persists output
- content hashes avoid unnecessary rewrites unless `--force` is supplied
- each builder validates the minimum structure it owns before output
- public publishing paths are explicit config; a local scope does not become public merely because it has a search index

## Change Guide

- source field or record shape: start in the domain builder and its config
- ranking or match behaviour: start in the domain runtime
- scope/output path: start in Docs scope config or Catalogue public-path/build config
- UI timing/copy: start in Catalogue search policy or Docs Viewer config/controller
- write orchestration: inspect the owning catalogue build command or Docs watcher/service

Do not maintain field tables or score tables here. Read the builder/runtime when exact behaviour matters.
