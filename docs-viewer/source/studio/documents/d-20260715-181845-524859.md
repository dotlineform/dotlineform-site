# Document Diagrams Concept

## Why Consider It

High-level diagrams could make execution paths, ownership, and public/manage/server boundaries easier to understand than prose or text art.

## Authored-Diagram Decision

Use Mermaid as the default source language for authored architecture and workflow diagrams in any Docs Viewer scope.

Mermaid source is the durable authority. An author-local command renders it to static SVG before Docs Viewer reads the document; Docs Viewer displays the SVG as an ordinary image and does not parse Mermaid or load a diagram runtime in the browser.

## Two Different Capabilities

### Build-Rendered Figures

A command turns diagram source into SVG or another static asset before publication. Readers receive the result without downloading the rendering library.

This improves runtime independence and export behavior. For the current need it adds an author-local tool and generated-asset ownership, but not a CI or repository package dependency.

### Authored Diagrams

Text in Markdown describes a flowchart, sequence, state, relationship, or architecture map. The diagram changes when the author edits that text.

The [Mermaid CLI](https://github.com/mermaid-js/mermaid-cli) is the upstream authority for command behavior. If the repository later gains a wrapper, its supported version and render options become the project authority.

## Suggested Repository Convention

Use the existing configured source and media roots:

```text
<scope source root>/media/mermaid/<diagram-name>.mmd
<configured published img root>/<diagram-name>.svg
```

- For a repo-backed scope the source maps to `docs-viewer/source/<scope>/media/mermaid/`.
- The published SVG can live in the public site media root, the local Docs Viewer published-media root, or R2 according to the configured `img` location.

Markdown uses the same logical token in every scope: `[[media:docs/<scope>/img/<diagram-name>.svg]]`. The builder resolves that token through the configured `img` publication location and emits the corresponding served URL.

Additional conventions:

- use lowercase kebab-case names describing the diagram, not the rendering tool;
- keep one `.mmd` source and one `.svg` output with the same basename;
- do not edit the generated SVG by hand;
- render with explicit background, theme, dimensions, and other material options once a project command exists;
- include useful surrounding prose and alt text; the SVG is not the sole statement of the document's meaning;
- if two scopes need the same diagram, name one owning source rather than silently forking two `.mmd` files.

## Invariants

- Markdown remains readable and editable without a graphical renderer.
- A broken diagram does not prevent the rest of its document from rendering.
- Ordinary documents do not load a visualisation dependency they do not use.
- Public routes receive no local-only renderer, service, or data source by accident.
- Data-driven visualisations preserve the canonical data owner's authority.
- The semantic graph consumes existing reference and catalogue identities rather than defining parallel nodes or edges.
- Markdown does not become a route for arbitrary JavaScript execution.
- The supported local tool version and material render options are recorded once the first project command exists, and generated results are reproducible enough to review through source control and the publication gate.

## Questions Before A Delivery

1. Which one or two current documents should prove the Mermaid convention?
2. Is a direct local `mmdc` command sufficient initially, or would a small repository wrapper materially improve consistency?
3. Which Mermaid CLI version, theme, sizing, and accessibility defaults are enough for the first diagram?
4. Should a later check detect an SVG older than its `.mmd` source without making rendering part of every docs build?
5. For the semantic graph, what question should the first view answer that the current reference report cannot?
6. Which node and edge types make a bounded useful first graph, and how large is the real dataset?
7. Should the graph begin as a local Docs Viewer panel before any public `/analysis/` promotion?

## Separate Delivery Candidates

- Authored documentation diagrams: prove the `.mmd`-to-SVG convention on one useful diagram, record the local tool/version command, and add only the consistency checks the use case demonstrates it needs.
- Semantic relationship graph: prove one read-only Docs Viewer panel over an explicit generated graph projection, initially limited to the node and edge types needed for a useful index view.

These should remain separate roadmap deliveries. Static Mermaid publishing does not itself justify repository-level JavaScript tooling; the interactive D3 graph may do so later.

When those answers point to one complete result, add an architecture document if needed and promote one bounded delivery into the [Docs Viewer Roadmap](/docs/?scope=studio&doc=d-20260714-234030-1e0af6).
