# Scope-First Storage Cutover Delivery

## Outcome

Every configured Docs Viewer scope uses one explicit repository or external-local scope root containing its canonical `source/` and working `published/` roles. Documents, search, references, source build inputs, and locally held media resolve through that root by artifact class. Scope and sub-scope lifecycle operations, builders, management reads, media publication, and local serving use the same configuration-owned location model. Reproducible application-level data that no scope owns resolves beneath the separate `docs-viewer/data/generated/` boundary.

The cutover preserves document ids, Markdown tokens, management service and public routes, served media references, public document/search projections, and R2 object keys. It intentionally changes the registered manage-only semantic target lookup URL while preserving that artifact's schema and behavior. It removes the old role-first local trees after verified migration and adds no compatibility paths or duplicate canonical locations.

## Why This Is One Delivery

The current config and runtime can describe source, documents, search, and media separately, but create, validation, manifest, rename, delete, fixtures, and some consumers still encode role-first roots. Moving bytes before those owners agree would leave a partially usable scope; retaining both layouts would create the compatibility layer this feature rejects.

This delivery therefore owns one complete configuration, code, and byte cutover. Its checkpoints are implementation order inside that result, not separately shippable storage modes.

## In Scope

- add one explicit local scope-root location for each configured repository or external-local scope;
- derive or validate `source/` and working `published/` artifact locations beneath that root without repeating provider policy in consumers;
- place local published media beside `documents/` and `search/` beneath `published/`;
- keep sub-scopes one level deep and stop creating an unsupported nested `sub-scopes/` directory beneath a sub-scope;
- update config validation, provider-neutral resolution, manifests, create, rename, delete, sub-scope lifecycle, builders, management reads, media publication, watcher behavior, and serving to use the new model;
- migrate Studio, Library, Analysis, Moments, Processing, external-local Notes, and the Analysis `tags` sub-scope;
- introduce `docs-viewer/data/generated/` as the narrow home for reproducible non-scope application data and move the semantic-reference target lookup beneath it;
- preserve and verify canonical source and locally published media bytes, then rebuild consumer-ready document, search, reference, and registered media-producer output at the configured destinations;
- remove retired role-first roots only after the new locations and consumer contracts pass focused verification;
- update the durable source-organisation, scope-lifecycle, generated-data, and configuration owners with shipped behavior.

## Not In Scope

- moving `site/assets/data/docs/scopes/<scope>/`, `site/assets/data/search/<scope>/`, or R2 `docs/<scope>/...` destinations;
- changing public URLs, `/docs/` service routes, media tokens, `doc_id`, parent relationships, payload schemas, search behavior, or exposure policy;
- changing the semantic-reference target lookup schema, rows, matching behavior, or manage-only access boundary;
- creating speculative `data/canonical/`, runtime-state, cache, staging, or other application-data classes without a concrete owner;
- adding R2 document source or document/search publication;
- adding a persistent build cache, storage-version history, automatic remote cleanup, or general backup product;
- changing imported-versus-Mermaid-built SVG identity and collision behavior;
- recursive sub-scopes, sub-scope media authoring, or unrelated lifecycle features;
- retaining a dual-schema loader, old-path fallback, symlink, copy, route alias, or read-through compatibility layer after cutover.

## Settled Decisions

- Scope configuration owns exactly one repository or external-local root for the local scope workspace. Source and working-publication paths are confined descendants of that root; public projections and R2 media remain explicit external destinations.
- The stable local hierarchy is `<scope-root>/source/<artifact-class>` and `<scope-root>/published/<artifact-class>`. Build is a transformation, not a persistent sibling role.
- Local published media is `published/media/<type>`, not `published/documents/media/<type>`. Scope-owned document reference artifacts remain under `published/documents/references/`.
- Repository-wide generated artifacts are registered separately rather than placed under an arbitrary scope. The semantic-reference target lookup moves to `docs-viewer/data/generated/semantic-references/target-lookup.json` and its registry URL becomes `/docs-viewer/data/generated/semantic-references/target-lookup.json`.
- `docs-viewer/data/` is the tracked application-level data boundary for artifacts that no scope owns. This delivery creates only `generated/`; config, scope content, and machine-local operational state retain their existing owners.
- The config schema changes once. Production code does not accept old and new scope-layout schemas concurrently.
- Migration uses a write-free preflight, collision refusal, temporary staging where byte verification needs it, one explicit config/code cutover, and post-cutover removal of old roots. Temporary staging is not a second canonical location.
- Public projections and R2 objects are verified as stable dependencies, not copied into the new local scope root.
- Detailed path and byte inventories stay in session evidence or a temporary linked working note if they must survive across sessions; this delivery records only durable decisions and gates.

## Code Authority To Inspect

- scope model and policy: `docs-viewer/config/scopes/docs_scopes.json` and `docs-viewer/services/docs_scope_config.py`;
- provider boundary: `docs-viewer/services/docs_artifact_locations.py`;
- manifest and lifecycle: `docs_scope_manifest.py`, `docs_scope_create.py`, `docs_scope_rename.py`, `docs_scope_delete.py`, `docs_sub_scope_lifecycle.py`, and `docs_scope_external_validation.py`;
- document, search, reference, and browser-config builders under `docs-viewer/build/docs_builder/` plus `docs-viewer/build/build_search.py`;
- media and build-source consumers: `docs_media_storage.py`, `docs_import_media.py`, `docs_mermaid_media.py`, `docs_staged_media_service.py`, and `docs_live_rebuild_watcher.py`;
- management route/static-file resolution and generated runtime configuration under `docs-viewer/services/`, `docs-viewer/runtime/`, and `site/docs-viewer/`;
- focused config, builder, media, lifecycle, route, and fixture contracts under `docs-viewer/tests/`.

These are responsibility areas, not a permanent file inventory. Literal old paths found in active code must be removed or routed through the owning config helper; examples and assertions should describe only the new contract after cutover.

## Done When

- every configured repository or external-local scope has one explicit scope root containing its source and working publication;
- source documents, source build media, local documents, local search, reference artifacts, and local published media occupy the agreed artifact-class paths;
- all active consumers and lifecycle operations resolve those locations through config rather than role-first constants;
- Analysis `tags` remains a one-level leaf sub-scope and new lifecycle operations do not create recursive sub-scope directories;
- canonical source and direct-published media match the preflight inventory, and reproducible output rebuilds successfully at the new locations;
- public projections, R2 keys, service routes, logical tokens, and document identities are unchanged;
- the semantic target lookup retains its schema and behavior at `docs-viewer/data/generated/semantic-references/target-lookup.json`;
- old role-first roots and compatibility mechanisms are absent;
- focused repository and real external-local verification passes;
- durable documentation describes only the shipped scope-first layout and its independently registered destinations.
