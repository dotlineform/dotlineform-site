# External-Local Scope Media Layout

## Outcome

External-local scopes use the same scope-owned media structure as repo-backed local scopes:

```text
<scope source root>/
  *.md
  media/
    img/
    files/
```

For the external workspace this is:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/source/<scope>/media/
```

The browser contract remains `/docs/media/<scope>/<img-or-files>/<filename>`.

For every configured local scope, `media/`, `media/img/`, and `media/files/` are materialized directories even when they contain no media. They are part of the visible workspace contract, not virtual paths that appear only on first write.

## Ownership

External describes the storage location, not a different ownership model. Ordinary imported media and authored figures are managed by Docs Viewer in both local scope types:

- create establishes the scope source root and both media-class directories;
- local service startup restores any missing configured media directories;
- rename moves the source root, including media;
- delete removes the source root, including media.

Config loading remains read-only. Materialization belongs to local-service startup and confirmed scope creation, where a failure to establish the configured directories is visible rather than deferred until an import. Repo-backed empty directories use tracked marker files so the structure also survives a clean clone.

Generated docs and search data remain in their existing derived `generated/` roots because repo-backed scopes also separate canonical source from rebuildable output.

The former external sibling path `docs-viewer/media/<scope>/` is not a compatibility lookup. Existing data must be moved before adopting this boundary; the configured external scope had no files there when this change was made.

## Separate Asset-Collection Boundary

[External Asset Collections](/docs/?scope=studio&doc=d-20260714-194217-b2bdd7) inventory user-owned files that are not ordinary scope media. Their roots stay outside `source/<scope>/`, are not named or derived by scope lifecycle, and are never moved or deleted merely because the associated scope is renamed or deleted.

## Code Owners

- `docs_media_storage.py` derives and confines scope-owned media beneath the source root.
- `docs_import_media.py` and `docs_import_markdown_package.py` materialize imports through that root.
- `docs_scope_rename.py` moves external source and generated roots; media needs no independent move.
- scope deletion removes the manifest-owned source root.

[Source Organisation](/docs/?scope=studio&doc=d-20260423-000000-8fd731), [Scope Lifecycle](/docs/?scope=studio&doc=d-20260515-000000-f43740), and [Media And Asset Handling](/docs/?scope=studio&doc=d-20260514-184303-7914e2) own the durable current contracts.
