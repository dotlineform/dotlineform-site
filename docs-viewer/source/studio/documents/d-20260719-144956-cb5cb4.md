# Scope-First Storage Layout

## Purpose

Make the scope, rather than the artifact role, the outer ownership and lifecycle boundary for scope-owned Docs Viewer data, and give the artifact classes within each lifecycle role a consistent hierarchy.

Each scope owns its canonical document and build source, local consumer-ready output, direct-published media, reproducible published output, and configured provider destinations. `source` and `published` describe how the scope treats an artifact; they do not own separate global pieces of every scope.

The feature therefore contains two related structural changes: the scope moves above the lifecycle role, and published media moves out of the document-publication tree to become a sibling of `documents/` and `search/` beneath `published/`. Source media is likewise a sibling of source `documents/`.

## Status

Active. External-local media configuration, creation, rename, serving, and tests now place published media beneath the external published-documents root. The existing Notes Mermaid-built and directly published SVGs have been moved and verified there, including a real Mermaid rebuild and local media-route resolution. The broader scope-first migration remains open in this feature.

## Future State

The provider-neutral shape is scope-first and artifact-class-first within each lifecycle role. The maximal repository-backed scope below shows every currently supported stable directory. Individual scopes only materialize the branches they configure or generate.

```text
docs-viewer/scopes/<scope>/
├── source/
│   ├── documents/
│   ├── media/
│   │   └── mermaid/
│   └── sub-scopes/
│       └── <sub-scope>/
│           └── documents/
└── published/
    ├── documents/
    │   ├── by-id/
    │   ├── references/
    │   │   ├── by-doc/
    │   │   ├── by-target/
    │   │   │   └── <target-kind>/
    │   │   └── index.json
    │   ├── sub-scopes/
    │   │   └── <sub-scope>/
    │   │       ├── by-id/
    │   │       └── manifest.json
    │   ├── .publish/
    │   │   └── recent.json
    │   ├── index-tree.json
    │   └── recent.json
    ├── search/
    │   ├── sub-scopes/
    │   │   └── <sub-scope>/
    │   │       └── index.json
    │   └── index.json
    └── media/
        ├── files/
        ├── html/
        ├── img/
        └── svg/
```

`mermaid/` is the only currently registered source-media build type. `files/`, `html/`, `img/`, and `svg/` are the currently supported published-media types; their provider and served-path configuration still determines whether and how each branch is exposed. Generated semantic-reference artifacts remain within `published/documents/references/`, because they are part of the document publication rather than canonical source or general media. `.publish/` is a persistent builder-owned helper for the selected public `recent.json` snapshot, not a separate lifecycle role or general build cache.

Application-level data that no scope owns uses a separate data boundary:

```text
docs-viewer/data/
└── generated/
    └── semantic-references/
        └── target-lookup.json
```

The semantic-reference target lookup combines catalogue and configured-scope inputs for the manage-only source editor, so assigning it to one scope would be false ownership. `generated/` is accurate here as provenance for a reproducible application-level artifact; it is not an alternative name for scope publication. The cutover materializes only `data/generated/`. Authored configuration stays in `config/`, scope-owned artifacts stay in `scopes/`, and machine-local logs, caches, or staging remain outside this tracked application-data boundary. Other `data/` classes should be added only when a concrete non-scope dataset establishes their ownership and lifecycle.

Sub-scopes are one level deep. A configured `<sub-scope>/documents/` is a leaf source-document set; there is no recursive child collection or builder support for a sub-scope of a sub-scope. The current lifecycle code mechanically materializes an unused nested `sub-scopes/` directory beneath a new sub-scope source, but that directory is not a supported content location. The future-state contract omits it, and the migration should stop creating it.

For an external-local scope the same contract resolves beneath the external workspace, for example:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/scopes/notes/
├── source/
└── published/
```

The enclosing `<scope>` container is the create, rename, copy, backup, and delete boundary through the configured provider adapter. Its source and published children must not overlap.

A future provider-native source location must preserve the same ordering rather than recreate a provider-specific role-first tree:

```text
<configured provider root>/scopes/<scope>/source/...
```

For example, adding R2 source support would map canonical source identities beneath that scope-first object-key root. This feature establishes the location contract but does not add R2 source support.

Public exposure remains an independent provider concern. A public scope can keep its source and working publication beneath its canonical scope root while registered public projections and remote media remain at their deployment destinations:

```text
site/assets/data/docs/scopes/<scope>/...
site/assets/data/search/<scope>/...
R2: docs/<scope>/...
```

The scope owns those identities and location declarations through configuration; the repository deploy root or remote provider owns their physical storage. The feature does not introduce duplicate local copies of public assets.

Build remains an optional transformation between source and published rather than a persistent directory role. Public URLs, document identities, media tokens, provider policy, and management/public exposure remain unchanged.

## Feature Boundary

- Give every scope an explicit configured scope root independently of storage provider.
- Place Docs Viewer-owned source and working published data beneath that root using the same role names for every scope.
- Organise each role by artifact class, with media beside documents rather than physically nested inside the document-publication root.
- Keep configured sub-scopes one level deep and stop materializing an unsupported descendant `sub-scopes/` directory beneath them.
- Make scope create, rename, delete, configuration validation, builders, media publication, and serving resolve paths through that provider-neutral boundary.
- Keep public projections and R2 objects at their configured provider destinations and lifecycle-manage them as registered scope-owned outputs.
- Move the repository-wide semantic-reference target lookup to `docs-viewer/data/generated/semantic-references/target-lookup.json` and update its registered manage-only browser URL.
- Keep `docs-viewer/data/` for explicitly owned non-scope application data rather than using it as a general miscellaneous-data root.
- Move existing local bytes through one verified migration without compatibility aliases or duplicate canonical locations.
- Do not add R2 source support, a persistent build cache, or a different imported-versus-built SVG identity policy as part of the folder move.

## Success

- A scope's repository or external-local source and working publication resolve from one explicit scope container; configured public projections and R2 media remain independent destinations.
- Documents, search, references, source build inputs, and every supported published-media type have one explicit place in the scope hierarchy.
- Source contains only canonical source material; published contains consumer-ready artifacts.
- Direct-published media is not mistaken for reproducible build output and is preserved by lifecycle operations.
- Public projection paths and remote object keys remain stable.
- Configuration rejects source/published overlap and no longer needs to treat a source root as the scope container.
- The role-first top-level `source/` and `published/` trees are gone; the application-level target lookup is reproducible beneath `data/generated/`.

## Current State

The current repository is mostly organised by artifact lifecycle role first:

`<workspace> -> source | published -> <scope> -> content`

The paths encode several independent dimensions:

- **artifact lifecycle role**: `source` or `published`; build is an optional transformation between them, not currently a persistent storage role; the lifecycle itself is the set of operations that create, rename, copy, back up, or delete a scope.
- **scope**: ownership such as Studio, Analysis, or Notes;
- **artifact type**: documents, search, or a media type;
- **storage provider**: repository, external-local, or R2;
- **exposure**: management-only or public.

Within those dimensions, three locations need to be distinguished:

- **source**: canonical Markdown and registered build inputs such as Mermaid `.mmd` files;
- **working publication**: consumer-ready document payloads, search indexes, and locally published media used by the management viewer;
- **public projection**: the selected public document and search payloads tracked beneath `site/assets/`.

Public exposure and physical storage are configured independently. Published media can be stored in the repository, in the external-local workspace, or in R2.

### Public scopes

Analysis is a representative public scope.

#### Source

- documents: `docs-viewer/source/analysis/documents`
- registered Mermaid source: `docs-viewer/source/analysis/media/mermaid`
- scope root: `docs-viewer/source/analysis`

The source tree also contains materialised media directories. Their presence does not make them the configured published destinations.

#### Working publication

- documents: `docs-viewer/published/docs/analysis/by-id`
- search: `docs-viewer/published/search/analysis/index.json`

The extra `docs/` level distinguishes document payloads from the sibling `search/` publication tree.

#### Public projection

- documents: `site/assets/data/docs/scopes/analysis/by-id`
- search: `site/assets/data/search/analysis/index.json`

The public projection is distinct from working publication and can contain a selected subset of the working documents.

`site/assets/data/docs/scopes/` is the namespace for Docs Viewer scope-owned public data. The parent `site/assets/data/docs/` can also contain document-domain data that is not owned by one Docs Viewer scope; `public-reports.json` is a current example. The `scopes/` level is therefore an intentional ownership boundary, not merely an extra copy of the scope name.

#### Published media

Analysis media does not have one physical root:

- images: `site/assets/data/docs/scopes/analysis/media/img`
- SVG: `site/assets/data/docs/scopes/analysis/media/svg`
- files: R2 object prefix `docs/analysis/files`

Repository-hosted public media sits alongside the public scope projection, while configured R2 media remains outside the repository.

### Local scopes (in-repo)

Studio is a representative repository-backed management scope.

#### Source

- documents: `docs-viewer/source/studio/documents`
- registered Mermaid source: `docs-viewer/source/studio/media/mermaid`
- scope root: `docs-viewer/source/studio`

#### Working publication

- documents: `docs-viewer/published/docs/studio/by-id`
- search: `docs-viewer/published/search/studio/index.json`
- media: `docs-viewer/published/docs/studio/media`

Studio has no public projection. Its source and published media are physically separate.

### External-local scopes

Notes is the current external-local scope. The paths below are relative to `$DOTLINEFORM_PROJECTS_BASE_DIR`.

#### Source

- documents: `docs-viewer/source/notes/documents`
- Mermaid source: `docs-viewer/source/notes/media/mermaid`
- configured source root: `docs-viewer/source/notes`

#### Working publication

- documents: `docs-viewer/published/docs/notes/by-id`
- search: `docs-viewer/published/search/notes/index.json`
- images: `docs-viewer/published/docs/notes/media/img`
- SVG: `docs-viewer/published/docs/notes/media/svg`
- files: `docs-viewer/published/docs/notes/media/files`

External-local source and published roles are now separate. The validator confines external-local media beneath the scope's published media root. Rename moves that media with `published/docs/<scope>`, while deletion removes the same manifest-owned published root.

### Shared SVG namespace

Imported SVG and Mermaid-built SVG publish into the same configured `published.media.svg` namespace. For Notes that is `docs-viewer/published/docs/notes/media/svg`; for Studio it is `docs-viewer/published/docs/studio/media/svg`.

Mermaid output uses the source basename deterministically: `<name>.mmd` publishes as `<name>.svg`. A directly imported `<name>.svg` can therefore target the same identity. Interactive staging reports changed-byte collisions and requires replacement confirmation, while an ordinary Mermaid rebuild publishes its deterministic output by replacement. The collision concern applies to every Mermaid-enabled scope and is independent of the scope-first storage change.

### Build and temporary locations

There is no persistent or scope-owned `build/<scope>` artifact tree in the current implementation:

- document and search builders calculate their write plans and write consumer-ready payloads directly to the configured published locations;
- public scopes can retain a builder-owned `.publish/recent.json` beneath their working document publication so the selected public recent-documents snapshot remains distinct from the management snapshot;
- Mermaid renders intermediate SVG into an operating-system temporary directory, sanitizes it, and then writes the verified SVG to the configured published media location;
- provider adapters can stage remote artifacts in operating-system temporary directories;
- local publication and lifecycle operations can use short-lived sibling `.tmp` files for atomic replacement.

`.publish/` is persistent and structurally stable enough to appear in the full hierarchy, but it remains an internal publication helper rather than a lifecycle role or general build cache. The other temporary paths are implementation details and are removed or replaced during the operation, so they are not stable enough to count as filesystem roles. `docs-viewer/build/` is structurally stable, but it contains builder code and the Mermaid toolchain rather than per-scope build artifacts. The persistent Data Sharing import and review staging roots are workflow inputs and review state, not Docs Viewer build output.

### Repository-wide generated data

The semantic-reference target lookup is generated at `docs-viewer/published/semantic-references/target-lookup.json`. It combines catalogue records with configured Docs Viewer scope output and serves the manage-only semantic-token picker. It is not owned by one scope, so its move to the application-level `data/generated/` boundary is separate from the scope-root byte moves but part of the same cutover.

### Current-state conclusions

- The repository currently uses a role-first `source/<scope>` and `published/<artifact-type>/<scope>` layout.
- Build is currently a transformation between source and published, without a persistent artifact tree of its own.
- Public scopes add a separate public projection and may split media across repository and R2 locations.
- Local in-repo scopes keep source and published media separate.
- External-local scopes now keep source separate from documents, search, and media in their external published roots.
- The semantic-reference target lookup is an active repository-wide published artifact rather than a scope-owned output.
- Source versus published is the durable semantic distinction; public versus management exposure is a separate concern.
