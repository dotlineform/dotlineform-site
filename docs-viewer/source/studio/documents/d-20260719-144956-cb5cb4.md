# Scope-First Storage Layout

## Purpose

Make the scope, rather than the artifact role, the outer ownership and lifecycle boundary for Docs Viewer-owned data.

Each scope owns its canonical source, local consumer-ready output, direct-published media, reproducible build output, and configured provider destinations. `source` and `published` describe how the scope treats an artifact; they do not own separate global pieces of every scope.

## Future State

The provider-neutral shape is scope-first. For repository storage it appears as:

```text
docs-viewer/scopes/<scope>/
├── source/
│   ├── documents/
│   └── media/
│       └── mermaid/
└── published/
    ├── documents/
    ├── search/
    └── media/
```

For an external-local scope the same contract resolves beneath the external workspace, for example:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/scopes/notes/
├── source/
└── published/
```

The enclosing `<scope>` container is the create, rename, copy, backup, and delete boundary through the configured provider adapter. Its source and published children must not overlap.

A future provider-native source location must preserve the same ordering rather than recreate a provider-specific role-first tree:

```text
<configured provider root>/scopes/<scope>/source/...
```

For example, adding R2 source support would map canonical source identities beneath that scope-first object-key root. This feature establishes the location contract but does not add R2 source support.

Public exposure remains an independent provider concern. A public scope can keep its source and working publication beneath its canonical scope root while registered public projections and remote media remain at their deployment destinations:

```text
site/assets/data/docs/scopes/<scope>/...
site/assets/data/search/<scope>/...
R2: docs/<scope>/...
```

The scope owns those identities and location declarations through configuration; the repository deploy root or remote provider owns their physical storage. The feature does not introduce duplicate local copies of public assets.

Build remains an optional transformation between source and published rather than a persistent directory role. Public URLs, document identities, media tokens, provider policy, and management/public exposure remain unchanged.

## Feature Boundary

- Give every scope an explicit configured scope root independently of storage provider.
- Place Docs Viewer-owned source and working published data beneath that root using the same role names for every scope.
- Make scope create, rename, delete, configuration validation, builders, media publication, and serving resolve paths through that provider-neutral boundary.
- Keep public projections and R2 objects at their configured provider destinations and lifecycle-manage them as registered scope-owned outputs.
- Move existing local bytes through one verified migration without compatibility aliases or duplicate canonical locations.
- Do not add R2 source support, a persistent build cache, or a different imported-versus-built SVG identity policy as part of the folder move.

## Success

- A scope's Docs Viewer-owned data resolves from one explicit scope container regardless of provider.
- Source contains only canonical source material; published contains consumer-ready artifacts.
- Direct-published media is not mistaken for reproducible build output and is preserved by lifecycle operations.
- Public projection paths and remote object keys remain stable.
- Configuration rejects source/published overlap and no longer needs to treat a source root as the scope container.

## Current State

The current repository is mostly organised by artifact lifecycle role first:

`<workspace> -> source | published -> <scope> -> content`

The paths encode several independent dimensions:

- **artifact lifecycle role**: `source` or `published`; build is an optional transformation between them, not currently a persistent storage role; the lifecycle itself is the set of operations that create, rename, copy, back up, or delete a scope.
- **scope**: ownership such as Studio, Analysis, or Notes;
- **artifact type**: documents, search, or a media type;
- **storage provider**: repository, external-local, or R2;
- **exposure**: management-only or public.

Within those dimensions, three locations need to be distinguished:

- **source**: canonical Markdown and registered build inputs such as Mermaid `.mmd` files;
- **working publication**: consumer-ready document payloads, search indexes, and locally published media used by the management viewer;
- **public projection**: the selected public document and search payloads tracked beneath `site/assets/`.

Public exposure and physical storage are configured independently. Published media can be stored in the repository, in the external-local workspace, or in R2.

### Public scopes

Analysis is a representative public scope.

#### Source

- documents: `docs-viewer/source/analysis/documents`
- registered Mermaid source: `docs-viewer/source/analysis/media/mermaid`
- scope root: `docs-viewer/source/analysis`

The source tree also contains materialised media directories. Their presence does not make them the configured published destinations.

#### Working publication

- documents: `docs-viewer/published/docs/analysis/by-id`
- search: `docs-viewer/published/search/analysis/index.json`

The extra `docs/` level distinguishes document payloads from the sibling `search/` publication tree.

#### Public projection

- documents: `site/assets/data/docs/scopes/analysis/by-id`
- search: `site/assets/data/search/analysis/index.json`

The public projection is distinct from working publication and can contain a selected subset of the working documents.

`site/assets/data/docs/scopes/` is the namespace for Docs Viewer scope-owned public data. The parent `site/assets/data/docs/` can also contain document-domain data that is not owned by one Docs Viewer scope; `public-reports.json` is a current example. The `scopes/` level is therefore an intentional ownership boundary, not merely an extra copy of the scope name.

#### Published media

Analysis media does not have one physical root:

- images: `site/assets/data/docs/scopes/analysis/media/img`
- SVG: `site/assets/data/docs/scopes/analysis/media/svg`
- files: R2 object prefix `docs/analysis/files`

Repository-hosted public media sits alongside the public scope projection, while configured R2 media remains outside the repository.

### Local scopes (in-repo)

Studio is a representative repository-backed management scope.

#### Source

- documents: `docs-viewer/source/studio/documents`
- registered Mermaid source: `docs-viewer/source/studio/media/mermaid`
- scope root: `docs-viewer/source/studio`

#### Working publication

- documents: `docs-viewer/published/docs/studio/by-id`
- search: `docs-viewer/published/search/studio/index.json`
- media: `docs-viewer/published/docs/studio/media`

Studio has no public projection. Its source and published media are physically separate.

### External-local scopes

Notes is the current external-local scope. The paths below are relative to `$DOTLINEFORM_PROJECTS_BASE_DIR`.

#### Source

- documents: `docs-viewer/source/notes/documents`
- Mermaid source: `docs-viewer/source/notes/media/mermaid`
- configured source root: `docs-viewer/source/notes`

#### Working publication

- documents: `docs-viewer/published/docs/notes/by-id`
- search: `docs-viewer/published/search/notes/index.json`
- images: `docs-viewer/source/notes/media/img`
- SVG: `docs-viewer/source/notes/media/svg`
- files: `docs-viewer/source/notes/media/files`

Notes document and search payloads use the external `published/` tree, but all configured published media is stored beneath the Notes `source/` root. The current validator requires external-local published media to remain beneath that root so it follows scope rename and deletion.

This is the principal structural inconsistency: the configured source root is serving both as a semantic source location and as the lifecycle container for published media.

### Shared SVG namespace

Imported SVG and Mermaid-built SVG publish into the same configured `published.media.svg` namespace. For Notes that is `docs-viewer/source/notes/media/svg`; for Studio it is `docs-viewer/published/docs/studio/media/svg`.

Mermaid output uses the source basename deterministically: `<name>.mmd` publishes as `<name>.svg`. A directly imported `<name>.svg` can therefore target the same identity. Interactive staging reports changed-byte collisions and requires replacement confirmation, while an ordinary Mermaid rebuild publishes its deterministic output by replacement. The collision concern applies to every Mermaid-enabled scope, although Notes additionally places the shared published namespace beneath `source/`.

### Build and temporary locations

There is no persistent or scope-owned `build/<scope>` artifact tree in the current implementation:

- document and search builders calculate their write plans and write consumer-ready payloads directly to the configured published locations;
- Mermaid renders intermediate SVG into an operating-system temporary directory, sanitizes it, and then writes the verified SVG to the configured published media location;
- provider adapters can stage remote artifacts in operating-system temporary directories;
- local publication and lifecycle operations can use short-lived sibling `.tmp` files for atomic replacement.

These temporary paths are implementation details and are removed or replaced during the operation, so they are not stable enough to count as filesystem roles. `docs-viewer/build/` is structurally stable, but it contains builder code and the Mermaid toolchain rather than per-scope build artifacts. The persistent Data Sharing import and review staging roots are workflow inputs and review state, not Docs Viewer build output.

### Current-state conclusions

- The repository currently uses a role-first `source/<scope>` and `published/<artifact-type>/<scope>` layout.
- Build is currently a transformation between source and published, without a persistent artifact tree of its own.
- Public scopes add a separate public projection and may split media across repository and R2 locations.
- Local in-repo scopes keep source and published media separate.
- Notes keeps documents and search separate but publishes media beneath its configured source root.
- Source versus published is the durable semantic distinction; public versus management exposure is a separate concern.
