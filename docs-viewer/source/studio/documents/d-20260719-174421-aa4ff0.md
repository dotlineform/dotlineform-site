# Word Document Import Delivery

## Outcome

A Docs Viewer manager can select a staged `.docx`, inspect the same canonical-Markdown preview used by other ordinary source formats, and create or overwrite one document through the existing Docs Import service. Mammoth contributes semantic HTML and conversion messages only. Current HTML conversion, supported embedded-image publication, document planning, collision decisions, source writes, and rebuilds keep their existing owners.

## Delivery Boundary

In scope:

- pin Python Mammoth `1.12.0` in `requirements.txt`;
- register `.docx` as the stable `docx` ordinary source format without changing its separate eligibility as staged file media;
- add one local-service DOCX adapter that reads the confined direct-child source in binary mode and passes Mammoth's HTML fragment into the existing HTML-content preview;
- explicitly disable external file access and embedded document style maps while retaining Mammoth's default semantic style map;
- merge useful Mammoth messages into the existing import warning list;
- pass embedded PNG, JPEG, GIF, and WebP through the current inline-raster media planning and publication boundary;
- prove listing, preview, create, overwrite, media-before-source ordering, failure behavior, and the existing browser/service contract with focused evidence;
- close out the feature in its durable operator, architecture, dependency, parent, and roadmap owners.

Out of scope:

- Mammoth Markdown output, Pandoc, Word-exported HTML cleanup, or a second Markdown converter;
- custom style maps, document transforms, style inference, or document-specific conversion profiles;
- comments, tracked changes, equations, Word metadata, footnote-specific product behavior, or fidelity repair;
- unsupported embedded image conversion, image optimisation, or a second media pipeline;
- a DOCX-specific modal, client-side conversion, remote upload surface, conversion job, process isolation, or timeout machinery;
- round-trip editing or preservation of Word layout, fonts, spacing, pagination, headers, or footers.

## Settled Decisions

- Call `mammoth.convert_to_html()` on the validated binary source and feed `result.value` into `generate_html_content_import_preview()`; do not call Mammoth's deprecated Markdown converter.
- Pass `external_file_access=False` and `include_embedded_style_map=False` explicitly. Use the library's default semantic style map without custom transforms.
- Keep `source_format: docx` visible through listing, preview, activity, and result payloads. The browser already consumes ordinary source formats generically and should not gain a DOCX workflow branch.
- Use the first semantic `h1` as the title when available, then fall back to the staged filename through the existing HTML-preview rules.
- Preserve meaningful Mammoth warning and error messages as plain warning strings without exposing physical paths or internal conversion objects.
- Publish only the raster MIME types already supported by inline media planning. An unsupported or unreadable embedded image must produce a warning and must not leave a raw data URL in canonical Markdown.
- Keep Mammoth's generated HTML and base64 image bodies inside the local management service. Preview/result payloads return the canonical Markdown preview, warnings, safe summaries, and media-plan metadata, but not those intermediate bodies.
- A malformed DOCX, failed conversion, invalid Markdown preview, or failed media publication must prevent the source write. Existing media-before-source ordering remains authoritative.

## Done When

- `.docx` is a first-class ordinary Docs Import source while remaining independently eligible for staged file-media publication;
- Mammoth is a small, pinned, local-service semantic-HTML adapter with external files and embedded style maps disabled;
- existing HTML-to-Markdown, inline-raster media, `ImportContent`, collision, source-write, and rebuild contracts remain authoritative;
- supported representative content becomes valid canonical Markdown and useful conversion warnings remain visible;
- unsupported structures and images degrade without unsafe links, raw image bodies, hidden filesystem access, or partial source writes;
- no DOCX-specific browser workflow, fidelity engine, or parallel mutation path is introduced.
