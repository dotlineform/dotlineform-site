# Docs Viewer Core Refactor Delivery Overview

## Purpose

Define how a large, long-running refactor becomes a series of bounded outcomes. The Core Refactor feature will not be short or simple; each implementation delivery must be as small, complete, and independently verifiable as its ownership boundary allows.

This overview does not track implementation state and is not itself a delivery. The [Core Refactor parent](/docs/?scope=studio&doc=d-20260718-114645-7b9d24) owns navigation, and the [Working Method](/docs/?scope=studio&doc=d-20260718-114201-4e7a93) owns branch, worktree, concurrency, and integration controls.

## Documentation Shape

| document | responsibility |
| --- | --- |
| concept | explain the core problem, target compositions, and product invariants |
| core definition analysis delivery | derive an evidence-backed initial core boundary from the current system and known consumers |
| core definition baseline | state the current accepted boundary and adjustment history needed by architecture and deliveries |
| transition architecture | map current and target ownership, dependency direction, consumers, and viable extraction seams |
| delivery overview | define decomposition and delivery-quality rules across the feature |
| testing | define evidence treatment and cadence without preserving obsolete implementation structure |
| delivery | own one complete implementation outcome, current state, next action, decisions, and completion gates |
| temporary working notes | retain detailed inventories or evidence needed during one delivery, then be retired |
| durable current-system owner | receive stable conclusions after implementation |

Do not create one delivery document for the whole refactor. The feature parent may remain active for a long period; an individual delivery should not.

## Core Definition Before Transition Architecture

The [Core Definition Analysis Delivery](/docs/?scope=studio&doc=d-20260718-121833-9fa2d4) is the first bounded delivery. It runs on `main` and produces Core Definition Baseline v1 before Transition Architecture or implementation begins.

The accepted baseline must classify the current capabilities and responsibilities, name core inclusion and exclusion criteria, map composition consumers and environment assumptions, test open proposals, and identify unresolved questions. It is explicitly adjustable when later evidence exposes a better boundary; adjustments must be recorded rather than emerging silently in code.

After that baseline exists, **Core Refactor Transition Architecture** must establish enough transition evidence to choose a safe first implementation slice:

1. the current capability and dependency map;
2. current and target ownership for core, integrated-only, and standalone composition responsibilities;
3. import, service, configuration, generated-data, process, and filesystem boundaries;
4. existing consumers and open roadmap proposals affected by each boundary;
5. transition constraints and any responsibilities that must move together;
6. candidate independently integrable slices and the smallest safe first delivery.

The transition architecture may describe a long route, but it must not become a task tracker or imply that every candidate slice is already approved.

## Concrete Starting Point

The current starting point is main-owned planning, not worktree creation. No Core Refactor implementation delivery, long-lived feature branch, or refactor worktree should be activated yet.

```text
commit planning baseline on main
  → complete Core Definition Analysis on main
  → accept Core Definition Baseline v1
  → write Transition Architecture on main
  → define and ready the first implementation delivery
  → pass the Worktree Creation Gate
  → create the worktree
  → begin implementation
```

The [Working Method](/docs/?scope=studio&doc=d-20260718-114201-4e7a93) owns the concrete creation gate and activation checks. This sequence may pause while bounded work continues elsewhere on `main`; the pause does not create a dormant refactor branch that must be maintained.

## Delivery Contract

A Core Refactor delivery should meet all of these conditions:

- change one named ownership boundary or produce one complete enabling contract;
- leave the integrated application working at the end of the delivery;
- leave no responsibility half-owned by old and new paths;
- avoid a global standalone-mode branch inside shared behavior;
- include the smallest evidence that proves the new boundary;
- identify a recovery point and be independently mergeable or revertible where practical;
- complete the two-way assessment between the slice and open feature proposals;
- update the durable authority or navigation affected by the completed outcome.

A proposed delivery is too broad when it is described only as “extract the core,” spans several unrelated ownership changes, requires later work to restore the current application, or cannot be verified until the entire standalone product exists. Some seams may be larger than ordinary feature work because they are indivisible; the delivery must name that invariant and why a smaller cut would leave dishonest ownership.

## Discovery Handling

Core analysis and implementation will expose modules, configuration, compatibility paths, services, or product limitations that deserve refactoring, replacement, or retirement in their own right. Discovery does not automatically widen the active delivery.

Use one decision test first:

> Would deferring this finding make the current core-refactor outcome false, unsafe, or leave split ownership?

| classification | treatment |
| --- | --- |
| boundary-enabling | include explicitly in the current delivery because its outcome cannot be complete without it |
| independent prerequisite | create a separate bounded delivery and complete it before the affected refactor slice |
| independent improvement | route it to its normal roadmap or maintenance owner and continue without it |
| retirement caused by the new boundary | remove it in the delivery that eliminates its final consumer, after evidence confirms that ownership |
| unproven or speculative | retain only as an analytical finding until evidence supports action |

An actionable finding must record:

- the evidence and current owner;
- why it was discovered and whether it blocks the core boundary;
- its classification, proposed treatment, and future owner;
- dependency or ordering effects;
- impact on open proposals and the accepted Core Definition;
- evidence required before replacement or retirement.

Use a temporary discovery register beneath the active analysis or delivery when the volume justifies one. It is not a permanent backlog: promote real work to the appropriate owner, update the accepted Core Definition or Transition Architecture when the boundary changes, and retire incidental notes at closeout.

If a discovery materially changes an active delivery's outcome or completion gates, stop and re-scope that delivery explicitly before continuing. Do not let “found while refactoring” become hidden authority for a larger implementation.

## Integration Strategy

Do not equate a long-lived feature with one long-lived unmerged diff. Merge behavior-preserving foundations into `main` at explicit validated integration points when they are independently useful and safe. This reduces historical divergence and lets subsequent work build on the real current architecture.

If several changes genuinely must land together, the transition architecture must identify the intentionally coupled sequence, its temporary state, verification point, and recovery path. Do not create accidental coupling by scattering half-moves across nominally separate deliveries.

`main` remains open to bounded Docs Viewer, Studio, Analytics, and repository work. The Working Method's reciprocal assessment determines whether each incoming feature proceeds, coordinates with a checkpoint, or waits, and whether each refactor slice constrains or reshapes an open proposal.

## Candidate Transition Areas

Current discussion suggests the following areas, but they are not yet delivery titles or an approved sequence:

- capability, consumer, and dependency baseline;
- explicit application environment and logical root contracts;
- composition roots and optional integration registration;
- package-safe builders and in-process service boundaries;
- core scope, reader, management, import, and export ownership;
- recomposition and parity proof for the repository-integrated application;
- executable external-local-only composition handed to the Standalone Docs Viewer feature.

The transition architecture may combine, split, reorder, or reject these candidates after inspecting the real dependency graph and current roadmap proposals.

## Implementation Delivery Creation Gate

Create the first implementation delivery only when:

- Core Definition Baseline v1 has been accepted;
- the transition architecture names its exact current and target owners;
- the slice ends in a working integrated application;
- affected proposals and concurrent work have been assessed in both directions;
- its verification and recovery boundaries are concrete;
- later slices are optional from the perspective of the resulting `main` state.

## Status

The delivery model and Core Definition Analysis are proposed. Commit the planning baseline and activate the analysis next; no core definition baseline, transition architecture, implementation delivery, long-lived feature branch, or refactor worktree exists yet.
