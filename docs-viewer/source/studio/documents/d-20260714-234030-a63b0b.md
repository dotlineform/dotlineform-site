# External Asset Collection Association Delivery

## Outcome

An eligible local scope can explicitly reference one independently named user-owned collection while Docs Viewer owns only the association and report state.

This delivery is complete when the association and its lifecycle consequences are implemented and tested. It does not partially implement collection scanning or asset operations.

## Why Now

This is the proposed first delivery under [External Asset Collections](/docs/?scope=studio&doc=d-20260714-194217-b2bdd7).

Ordinary external-local media uses configured published locations confined beneath its Docs Viewer-managed scope container and follows scope lifecycle. A collection is the opposite boundary: it is outside that container, independently named, and only inventoried by Docs Viewer.

The [feature architecture](/docs/?scope=studio&doc=d-20260714-234030-19a064) describes the larger proposed structure. This delivery owns only the lifecycle boundary below.

## In Scope

- define a validated association between an eligible local scope, one collection identity, a marker-relative collection root, and a report document id
- confine the collection root beneath an agreed user-assets base without deriving its name from the scope id or storing an absolute path
- keep the collection root outside lifecycle-owned manifest file records
- make delete preview distinguish removal of Docs Viewer-owned association/report state from the untouched collection root
- let scope rename update the association's scope identity without relocating or renaming the collection root
- project only the server-owned lifecycle state needed to test and explain this contract
- add focused config, manifest, delete, rename, containment, and capability tests

Prefer the simplest current representation. Do not add migration aliases, unused versioning, or a general association framework without a second proven need.

## Not In Scope

- collection scanning or a shared filesystem inventory primitive
- the Processing inventory report
- local asset open/download routes
- New Scope or retrofit UI
- enable, disable, or relocation workflows
- wrapper documents, promotion, sub-scopes, editing, moving, or deleting assets
- public routes, R2 publication, caching, jobs, thumbnails, or content extraction

Those outcomes remain separate future deliveries. Finding that one is useful does not widen this delivery.

## Code Authority To Inspect

- `docs-viewer/services/docs_scope_config.py`
- `docs-viewer/services/docs_scope_manifest.py`
- `docs-viewer/services/docs_scope_delete.py`
- `docs-viewer/services/docs_scope_rename.py`
- `docs-viewer/services/docs_media_storage.py`
- `docs-viewer/services/docs_management_capabilities_service.py`
- focused lifecycle and capability tests under `docs-viewer/tests/`

These are starting points, not an exhaustive update list. Exact implementation follows the current code.

## Done When

- one explicit server-owned record associates the scope and collection without treating the collection as a deletable lifecycle file
- path derivation is confined beneath the configured projects root and exposes no absolute path to the browser
- delete preview and apply remove only Docs Viewer-owned association/report state and demonstrably leave the collection root untouched
- rename updates scope-owned association state without moving the collection root
- the focused verification set passes
- [Scope Lifecycle](/docs/?scope=studio&doc=d-20260515-000000-f43740) records the shipped ownership and rename/delete behavior
- this delivery is closed completely; later asset work remains with the feature parent

## Durable Documentation Boundary

[Scope Lifecycle](/docs/?scope=studio&doc=d-20260515-000000-f43740) is the durable owner for this delivery. A later user-visible inventory delivery may create the focused External Asset Collections durable document, but this delivery does not require speculative updates across media, reports, endpoints, setup, or Processing docs.
