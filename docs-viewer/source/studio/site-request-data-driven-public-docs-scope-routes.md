# Portable Static Docs Viewer Install Request

Status:

- planned

## Summary

Define a host-neutral static install contract for portable Docs Viewer public routes.

A public read-only Docs Viewer scope should be installable into any static host by copying or serving plain HTML, CSS, JavaScript, route config, and generated payloads.
It must not depend on Jekyll/Liquid, the dotlineform public-site builder, generated Python source, or hardcoded per-scope Python route entries.

## Context

The original portable Docs Viewer assumptions were shaped by installing into a Jekyll site.
That is no longer the right portability target.
After the static public-site migration, the better target is a plain static host contract.

The current dotlineform public site renders public Docs Viewer route shells through `public-site/build/public_site_builder/`.
That is acceptable as a host adapter for this repository, but it should not be the canonical Docs Viewer install model.

The Docs Viewer New Scope lifecycle path also still contains a retired public-route assumption: for public read-only scopes it plans and writes a Markdown route stub under the requested public route path.
That was appropriate for the previous Jekyll route model, but it is wrong for a host-neutral static install contract.
New Scope public-readonly creation is currently disabled until this contract exists.

## Problem

Docs Viewer portability is blocked if public scope installation depends on a host-specific site builder.

The current repo has three stale or host-specific extension points:

- the old Jekyll/Liquid route stub model is retired
- dotlineform public Docs Viewer routes are currently represented as explicit Python builder entries
- New Scope public-readonly apply still has a legacy Markdown route-stub implementation path, now blocked

If each new public scope requires a host-builder code edit, the public route model becomes source-code-driven rather than config-driven.
If New Scope generates Python or host-specific route files, local management UI becomes a generator for installed public pages.

## Goals

- Define a portable static Docs Viewer install contract.
- Make public Docs Viewer route installation host-neutral.
- Provide a plain static public route shell contract using HTML, `data-*` attributes, route config, runtime assets, and generated payloads.
- Treat the dotlineform public-site builder as one host adapter, not the core portability model.
- Keep public route rendering and public-route discovery data-driven for host adapters.
- Update New Scope public-readonly preview/apply so it creates or updates Docs Viewer data/config records only after the portable contract exists.
- Ensure public scope lifecycle write sets clearly separate:
  - source docs
  - scope config
  - route config
  - host adapter config or derived host inputs
  - working generated outputs
  - published public snapshots
- Preserve manage-mode scope creation behavior.
- Preserve public read-only behavior with no management assets, backend probes, or local write capability.
- Keep existing Library and Analysis public routes working during the transition.

## Non-Goals

- Do not generate Python scripts or Python route entries from the New Scope action.
- Do not reintroduce Jekyll, Liquid route stubs, or persistent Markdown public route files.
- Do not make the dotlineform public-site builder a required dependency for portable Docs Viewer installs.
- Do not change public Docs Viewer runtime behavior unless required by the route-config refactor.
- Do not redesign Docs Viewer scope storage or workspace mounting; that belongs to the workspace-mount architecture request.
- Do not make every Docs Viewer scope public by default.
- Do not introduce a general plugin framework for arbitrary host behavior.

## Decisions Needed

Before implementation, decide:

- which files are the minimal portable public install unit
- whether public route records are edited directly in `docs-viewer/config/routes/*.json` or generated from `docs-viewer/config/scopes/docs_scopes.json`
- whether route shell HTML is copied from a static Docs Viewer template, authored by the host, or generated by a host adapter
- what exact `data-*` attributes a public route shell must provide
- how hosts discover or copy Docs Viewer runtime assets, CSS, UI text, route config, generated docs payloads, and search payloads
- whether dotlineform `public-site/config/public-site.json` should list public Docs Viewer assets per scope or derive them from scope config at build time
- whether public route audit requirements are static configured lists, derived generated expectations, or both in the dotlineform adapter
- whether New Scope public-readonly creation stays blocked until the host-neutral contract and at least one host adapter exist
- whether scope deletion should remove public route records and host-adapter config entries for user-created public scopes

## Candidate Implementation Slices

1. Audit current public Docs Viewer install assumptions and host-specific dependencies.
2. Define the portable static install contract.
3. Define the public Docs Viewer route metadata source of truth.
4. Provide or document a plain static public route shell template.
5. Refactor the dotlineform public-site builder into a host adapter that consumes the portable contract.
6. Refactor dotlineform public-site config/audit handling for public Docs Viewer scope assets.
7. Update New Scope public-readonly preview/apply to write data/config records only.
8. Update scope delete behavior for user-created public scopes.
9. Update stable docs and route lifecycle tests.

## Verification Expectations

- Focused tests for the portable public route shell contract.
- Focused tests for host-neutral public route metadata.
- Focused Python tests for dotlineform host-adapter route expansion.
- Focused tests proving New Scope public-readonly does not write Markdown route stubs or Python source.
- Focused tests for route config records produced or edited by the lifecycle path.
- Dotlineform public-site build and artifact audit.
- Public Docs Viewer browser smoke for existing `/library/` and `/analysis/`.
- A temporary fixture or test scope proving a new public read-only route can be installed without editing Python route code or requiring Jekyll/Liquid.
- A host-neutral fixture proving the public route shell can run when served as plain static HTML.
