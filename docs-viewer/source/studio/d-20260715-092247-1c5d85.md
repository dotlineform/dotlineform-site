# LLM Integration Architecture

This is proposed structure, not an implemented shared LLM platform.

## Workflow Boundary

```text
owning domain selects explicit source context
  -> task builds structured input
  -> provider returns structured proposal data
  -> task validates the untrusted response
  -> person reviews and edits proposals
  -> accepted changes pass through the domain writer
  -> existing builders regenerate derived output
```

## Ownership

- Analytics owns tag, catalogue-analysis, and dimension meaning.
- Document source owners retain summaries, hierarchy, and document content.
- Each workflow owns its source selection, privacy decision, schemas, review surface, validation, apply rules, and audit decision.
- A provider adapter may own transport, structured-response parsing, timeouts, and common diagnostics after a real workflow proves that seam.

Shared transport does not imply shared persistence, user interface, or governance.

## Extension Method

Add a task-specific workflow with explicit input and proposal schemas. Reuse the owning domain's current read and write services. Extract provider-neutral invocation only when a second task demonstrates the same transport boundary.

## Weak Spots

- Provider and model selection are unresolved and temporally unstable.
- Privacy and data-retention constraints differ by workflow.
- Batch review needs a usable scale boundary.
- Provider output can validate structurally while remaining semantically wrong.
- Audit and reproducibility expectations are not yet established.
- Premature shared machinery could obscure domain-specific responsibilities.
