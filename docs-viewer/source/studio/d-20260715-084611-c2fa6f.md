# Search Mechanics Review Delivery

This is a paused mechanics review. The current two-domain, static-index structure is sound, but its inclusion, ranking, and efficiency policy grew through implementation slices and needs deliberate refinement.

It is not a proposal to merge Catalogue and Docs Viewer search, replace the static architecture, or redesign everything in one delivery.

## Review Outcome

Produce one short decision document that answers:

- which sources and fields should be included in each index
- what should be excluded even when it could improve recall
- which derived fields make retrieval clearer and which only duplicate payload
- which normalization and ranking rules should be shared or deliberately different
- how representative known-item and discovery queries should rank
- how index generation, shape, and client loading can be made more efficient
- what result context or filtering is needed to make matches understandable
- what privacy and publication boundaries constrain indexed data
- which mechanical improvements are worth promoting into separate deliveries

The review ends with a prioritized sequence of complete outcomes. It does not jump directly into implementation.

## Current Starting Point

- Catalogue Search indexes public work/series projections and ranks them on `/catalogue/search/`.
- Docs Viewer Search indexes document navigation metadata per scope and ranks inside the viewer.
- both use deterministic score bands and broad derived terms
- normalization and ranking are similar but duplicated
- Catalogue build dependencies are registry-backed; relevance policy is code-backed
- document bodies, catalogue prose, and Analytics tags are not current search sources
- both systems load their index into the browser

See [Search](/docs/?scope=studio&doc=d-20260331-000000-5dcf32) for the current authority map.

## Questions Worth Testing

- Are exact-id/title lookup and exploratory discovery being forced through one ranking model?
- Which apparent relevance problems are actually missing or poor source metadata?
- Should tags, prose, summaries, or aliases be searchable, and what becomes public if they are?
- Do users need field-aware filtering or only better ranking and result context?
- Which repeated values can be normalized once or omitted from the public payload?
- Can index generation and client normalization do less work without weakening retrieval?
- Should the domains share tested normalization/ranking primitives, or only a policy vocabulary?

## Guardrails

- Use representative real queries and observed failures, not invented completeness.
- Keep current behaviour documented separately from proposals.
- Do not introduce semantic/LLM search as an assumed destination.
- Do not add every possible field for recall; index size and result quality are part of the policy.
- Preserve the current domain ownership and static-index execution path unless evidence reveals a concrete limit.
- Promote each accepted improvement as a finishable delivery with its own evidence.

## Resume Condition

Start when search quality is an active priority and there is time to gather representative queries across both domains. Until then, fix clear regressions against current behaviour without expanding this review into an implementation tracker.
