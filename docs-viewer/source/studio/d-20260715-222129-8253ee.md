# Recent Added And Edited

## Outcome

The Docs Viewer **Recent** view offers two explicit options:

- **Added** — documents ordered by canonical `added_date`, newest first.
- **Edited** — documents ordered by canonical `last_updated`, newest first.

Both options use the current scope, visibility rules, result limit, and `recent-results` host. Switching option does not change the displayed document. Edited order comes from document metadata, not the opaque `doc_id` or live Git history.

## Current Boundary

The current route feature, provider, payload, and controller expose only recently added documents through `docs_recently_added_v1`. Search already consumes `last_updated`, but Recent has no edited ordering.

This feature is proposed and has no delivery document. Implementation should decide whether one small generated payload can carry both orderings or whether two focused payloads are clearer. Do not add date fields to `index-tree.json` solely to support Recent.

## Durable Home

[Generated Data Providers](/docs/?scope=studio&doc=d-20260605-125108-c68916) owns the generated payload and provider contract after implementation. Ordinary control placement and rendering remain self-documented in the current view/control code.
