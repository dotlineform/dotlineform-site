# Recent Documents

## Outcome

Docs Viewer has one **Recent** view whose meaning follows the route context rather than a user-selected mode.

- public `/moments/` is catalogue-like and shows works ordered by canonical `added_date`, newest first;
- public `/library/` and `/analysis/` are changing document collections and show documents ordered by canonical `last_updated`, newest first;
- `/docs/` is always editorial and orders every public, local, or external-local scope by canonical `last_updated`, newest first.

The same scope can therefore mean recently added at `/moments/` and recently edited at `/docs/?scope=moments`. The control remains simply **Recent**; there is no Added/Edited toggle or parallel user-facing view. New documents appear naturally in either ordering because creation is also their first update.

Any route that enables Recent declares an allowlisted `added` or `edited` basis in its route policy. Moments is the first Docs Viewer `added` case, not a hard-coded route-id or scope-id exception. A future catalogue-like Docs Viewer route can adopt the same behavior through configuration without adding another application branch.

The view keeps the current scope, visibility rules, result limit, `recent-results` host, and displayed-document behavior. Ordering comes from canonical document metadata, not the opaque `doc_id` or live Git history. Catalogue-owned `/series/` follows the same recently-added principle as public Moments but remains outside this Docs Viewer delivery. Docs Review has no Recent feature unless it later develops a distinct review-package need.

## Delivery

[Recent Documents Delivery](/docs/?scope=studio&doc=d-20260715-225512-80198b) owns the complete timestamp amendment and context-sensitive Recent-view change. It is one proposed delivery because trustworthy update timestamps and the views that consume them solve the same recent-document problem.

## Current Boundary

The current route feature, provider, payload, and controller expose recently added documents through `docs_recently_added_v1`, ordered by `added_date` in every context. `last_updated` is not yet a reliable activity timestamp: precision and automatic maintenance vary by write path.

The delivery first makes `last_updated` trustworthy, then replaces the current contract with one generically named Recent projection at each served context. Do not add date fields to `index-tree.json` solely to support this view or couple it to the search payload.

## Durable Home

[Generated Data Providers](/docs/?scope=studio&doc=d-20260605-125108-c68916) owns the generated payload and provider contract after implementation. The durable document-metadata owner records the timestamp contract. Ordinary control placement and rendering remain self-documented in the current view/control code.
