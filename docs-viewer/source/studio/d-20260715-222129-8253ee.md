# Recently Edited

## Outcome

The Docs Viewer **Recently Edited** view shows documents ordered by canonical `last_updated`, newest first.

There is one Recent view and no Added/Edited toggle. New documents appear naturally because creation is also their latest update. The question is simply: which documents were touched most recently?

The view keeps the current scope, visibility rules, result limit, `recent-results` host, and displayed-document behavior. Ordering comes from canonical document metadata, not the opaque `doc_id` or live Git history.

## Delivery

[Recently Edited Delivery](/docs/?scope=studio&doc=d-20260715-225512-80198b) owns the complete timestamp amendment and Recent-view change. It is one proposed delivery because trustworthy update timestamps and the view that consumes them solve the same problem.

## Current Boundary

The current route feature, provider, payload, and controller expose recently added documents through `docs_recently_added_v1`, ordered by `added_date`. `last_updated` is not yet a reliable activity timestamp: precision and automatic maintenance vary by write path.

The delivery first makes `last_updated` trustworthy, then replaces Recently Added with one small Recently Edited projection. Do not add date fields to `index-tree.json` solely to support this view or couple it to the search payload.

## Durable Home

[Generated Data Providers](/docs/?scope=studio&doc=d-20260605-125108-c68916) owns the generated payload and provider contract after implementation. The durable document-metadata owner records the timestamp contract. Ordinary control placement and rendering remain self-documented in the current view/control code.
