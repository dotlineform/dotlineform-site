# Pytest

## Role

Pytest is the Python collection/debugging layer beneath the optional profile runner. It is installed through the repo dependency baseline and is not a competing top-level test framework.

Use it for deterministic models, planners, parsers, services, API adapters, config validation, generated contracts, and filesystem/write boundaries.

```bash
$HOME/miniconda3/bin/python3 -m pytest <test-path> -q
$HOME/miniconda3/bin/python3 -m pytest <test-path> -k <behavior>
```

## Test Shape

- keep tests under the owning app's `tests/python/` tree;
- prefer plain `assert`, focused fixtures, parametrization, and temporary directories;
- avoid import-time side effects, implicit cwd dependence, fixed ports, real canonical writes, and shared mutable module state;
- test dry-run/no-write and error paths for mutation services;
- assert positive current contracts rather than enumerating every retired path or field;
- keep direct `__main__` entrypoints only where they remain useful; pytest ignores them.

## Boundary With Browser Tests

If UI behavior matters because it changes a request payload, response interpretation, or shared component contract, test that lower-level contract here or in a focused JavaScript module check. Do not recreate modal timing or screen click order in Python.

## Runner Integration

`admin-app/commands/run_checks.py` groups selected pytest paths with syntax, config, generator, and smoke commands. Its source and `--list` output are the current profile authority.

Run a profile for broad coordinated evidence; run a focused pytest path while developing or when one Python owner changed.

## Weak Spots

- Some tests are simultaneously pytest modules and directly executable scripts, which can encourage custom bootstrapping.
- Broad grouped collections can conceal which owner actually supplied evidence; report the focused failing/passing path when useful.
- Pytest fixtures should not become a compatibility reason to retain old production APIs.
