# Development Workflow

## Purpose

Use this page to move work from an idea to one complete, verifiable delivery. Use [Development Checklist](/docs/?scope=studio&doc=d-20260523-190651-7157ec) for implementation guardrails after the owning boundary is known.

## Fast Path

1. Name the user/maintenance outcome and the current code/config owner.
2. Decide whether this is a small direct change, concept exploration, or roadmap feature.
3. Resolve decisions that would fundamentally change the outcome.
4. Split the work until one delivery can finish completely.
5. Implement through current owners; do not create transitional aliases as an end state.
6. Run the smallest evidence that proves the changed contract.
7. Update one durable documentation owner when its behavior/navigation changed.
8. Close only when the complete outcome ships; put later outcomes back under the feature parent.

## Concept, Architecture, Roadmap, Delivery

```text
free-flowing discussion
  -> feature snapshot (one coherent boundary)
  -> app roadmap (suggested feature order and dependencies)
  -> optional concept and architecture
  -> zero, one, or several complete deliveries
  -> code + focused evidence
  -> durable owner updated
```

### Concept

Use a concept document when useful questions remain open. It may explain the long-term aim without pretending the entire aim is one delivery.

After implementation, retain it when the feature still benefits from a short durable explanation of purpose, mental model, and invariants. Rewrite it as current documentation, remove resolved speculation, and keep its immutable `doc_id`. Do not retain it as a decision diary.

### Feature Parent

A feature may be a half-page snapshot with no child documents. It can be completed immediately or parked without becoming a holding document for every later thought.

When a feature needs separate concept, proposed architecture, or delivery documents, keep one short parent under its [app roadmap](/docs/?scope=studio&doc=d-20260428-000000-f5ff18) to route between them. The parent assumes its required environment exists, does not track scheduling dependencies, and does not repeat its children.

“Feature” means any change complicated enough to need dedicated documentation. It does not imply a large or visible capability. Use the same feature boundary for refactors, maintenance improvements, workflows, and user-facing work; do not add epic, initiative, enabler, story, or improvement layers around it.

### Roadmap

An app roadmap is one short table of features in suggested order. It owns scheduling dependencies and implementation readiness without encoding internal delivery plans, abbreviations, or implementation detail.

### Delivery

One delivery owns one complete result and one documentation boundary. It may cross modules, but every change must serve the same verifiable outcome. Use a child task tracker only when coordination inside that bounded result genuinely needs one.

A delivery document tracks scope, ordered steps, current and next state, decisions, and completion gates. It does not accumulate command transcripts, verification logs, scan or audit results, technical tables, or detailed working inventories.

Codex may create temporary sibling working-notes or verification documents without separate approval when detailed material must survive between sessions. Create them only when useful, link them from the active delivery, and treat them as disposable. Ordinary command and result reporting can remain in the session handoff or final report. At closeout, transfer only stable conclusions and remaining weak spots to the durable owner, then retire the temporary siblings with the delivery.

A complex feature may have several delivery documents or explicitly named phases. Each must remain independently complete; the feature parent owns their internal order and navigation. Do not append additional outcomes to a parked delivery or reproduce the internal sequence in the app roadmap.

Before a feature is safe to implement, omit its delivery or mark a delivery **proposed** when writing it helps test feasibility, value, and boundaries. Promote it to planned only when the roadmap dependencies are satisfied; mark it active only when work starts.

Small fixes, documentation cleanup, and mechanical maintenance normally need none of this machinery.

## Shape The Slice

Before editing:

- identify authoritative source, generated outputs, config, browser/server boundary, and write owner;
- name the responsibility added/moved and the module that will own it afterward;
- note the main benefit and meaningful risks;
- separate shell/rendering, validation/planning, mutation, and generated follow-through;
- prefer existing primitives/providers/services over route-local duplication;
- investigate first when deleting config or compatibility code; history explains but does not prove current use.

If implementation reveals another useful outcome, add or split it under the feature parent. Do not silently widen the active delivery.

## Verify Proportionally

Choose the lowest layer that proves the durable contract:

| Change | Usual evidence |
| --- | --- |
| documentation | source review, links/build when publication follow-through is in scope, `git diff --check` |
| pure model/parser/planner | focused unit/pytest |
| local service/API | direct function or HTTP contract test |
| generator | dry run, output-shape check, then explicit write if required |
| route/module integration | focused browser smoke only when the browser boundary is the risk |
| visual/tactile UI | manual or temporary browser inspection |
| broad cross-app contract | narrowest relevant `run_checks.py` profile |

Loopback/browser checks may need elevated local permissions. Do not run broad profiles merely to produce more evidence.

[Testing](/docs/?scope=studio&doc=d-20260501-174746-efd581) owns test policy; [Run Checks](/docs/?scope=studio&doc=d-20260501-174746-f92667) owns current profile inventory.

## Documentation Boundary

Update a durable doc only when its current workflow, architecture, methodology, extension rule, weak spot, or navigation changed.

- Code/config/tests remain exact authority.
- Overview docs map capabilities and owners; they do not list every module/field/route.
- Focused contract docs may contain exact inventories when the inventory is the point.
- A delivery document records outcome decisions, not permanent implementation detail.
- Avoid generic Related lists; link only to an authority, prerequisite, or next action.
- Do not update several docs “in case” some wording becomes useful later.

Generated docs/search payloads follow source changes only when the task or watcher owns that follow-through.

## Closeout

Report:

- the outcome and important changed owners;
- focused verification and result;
- generated artifacts written or intentionally not rebuilt;
- known gaps/risks;
- any separate roadmap outcome created.

Mark a delivery done only when its whole outcome works, durable docs describe shipped behavior, and remaining work is genuinely separate. Never hide unfinished phases inside a completed delivery.

At feature closeout, promote a still-useful concept into the durable app tree. Retire the temporary feature router, proposed architecture, delivery, and tracker documents once they no longer route unresolved work.
