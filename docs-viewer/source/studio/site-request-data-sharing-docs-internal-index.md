# Data Sharing Docs Internal Index Request

Status:

- draft
- related request: [Docs Viewer Public Index Slimming Request](/docs/?scope=studio&doc=site-request-docs-viewer-public-index-slimming)

## Summary

Create an internal/local generated docs metadata index for Data Sharing and other first-class tooling consumers so they do not rely on public Docs Viewer artifacts.

Data Sharing needs access to richer document metadata than a public read-only tree index should carry.
That data must come from a local/tooling-owned projection under `docs-viewer/` or another first-class internal generated location, not from `assets/data/docs/scopes/<scope>/index.json`.
Public docs `index.json` is being retired rather than slimmed into a Data Sharing dependency.

## Goals

- define a local/internal generated docs metadata index for Data Sharing and tooling consumers
- move Data Sharing selectable-record and package-preparation reads away from public docs indexes
- preserve Data Sharing access to metadata such as summary, updated dates, source paths, content text length, visibility state, and payload paths where needed
- keep the Data Sharing adapter first-class instead of treating public generated artifacts as the convenient source of truth
- unblock retirement of public `assets/data/docs/scopes/<scope>/index.json` files
- keep the new internal index generated by owned Docs Viewer build/write flows, not by ad hoc Data Sharing scraping
- keep Data Sharing functionality working after public docs `index.json` no longer exists

## Non-Goals

- no public Docs Viewer UI changes in this request; public index and info-panel behavior belongs to [Docs Viewer Public Index Slimming Request](/docs/?scope=studio&doc=site-request-docs-viewer-public-index-slimming)
- no change to the public by-id rendered payload contract unless a separate public runtime request requires it
- no move of Data Sharing out of Analytics ownership
- no broad redesign of Data Sharing package formats beyond replacing the upstream docs metadata source
- no requirement that internal/manage indexes be as small as public indexes

## Current Problem

Data Sharing currently benefits from metadata that happens to be present in public docs scope indexes.
That is the wrong dependency direction.

Public `assets/data/docs/scopes/<scope>/index.json` files are being removed from the Docs Viewer runtime contract.
Public navigation will use `index-tree.json`, selected-document reader metadata will come from by-id payloads, search will use the existing search payload, and recently-added will use a small build-time payload.
None of those public runtime payloads should become Data Sharing's rich metadata source.

Data Sharing is a first-class local/tooling workflow and needs a richer, stable metadata source that is not constrained by public payload size or public-safe field exposure.
Current functionality that must be moved includes selectable document records, package/export preparation, and returned-package review context that checks current Library docs, parent relationships, viewability, payload existence, current summaries, rendered content, and content text length.

## Proposed Direction

Add a generated internal docs metadata projection for each relevant docs scope.
The exact path should be decided during implementation, but candidate shapes include:

- `docs-viewer/generated/docs/<scope>/metadata-index.json` as a sibling to rendered manage payloads
- `docs-viewer/generated/docs/<scope>/tooling-index.json` if the projection should serve broader tooling beyond Data Sharing
- a Data Sharing adapter-owned generated projection if ownership review shows the shape is specific to Data Sharing rather than broadly useful tooling metadata

The preferred starting point is a Docs Viewer-owned internal metadata index because the builder already owns source parsing, rendered payload paths, content text extraction, and scope configuration.
Do not name the replacement `index.json`; that name is being removed as a generic docs-index dependency to make accidental fallback to retired public indexes easier to catch.

## Candidate Fields

The internal index can carry fields that are inappropriate or too heavy for public indexes:

- `doc_id`
- `title`
- `summary`
- `added_date`
- `last_updated`
- `parent_id`
- `viewable`
- `ui_status`
- `source_path`
- rendered payload path or URL
- `content_text_length`
- scope id
- any builder diagnostics or provenance needed by tooling

The final field list should come from Data Sharing adapter needs, Docs management needs, and import/export review needs, not from what the public viewer happens to expose today.

## Current Dependencies To Replace

The implementation audit should start with these known Data Sharing document paths:

- `docs-viewer/services/docs_data_sharing/package.py` builds selectable records from `docs_generated_reads.read_generated_docs_index(...)` and exposes `summary`, `parent_id`, `viewable`, `published`, and `content_text_length`.
- `docs-viewer/services/docs_export.py` loads `assets/data/docs/scopes/<scope>/index.json` for package preparation and uses it for selected ids, hierarchy expansion, parent/child fields, summary/current-summary fields, visibility filtering, and rendered by-id payload lookup for source text extraction.
- `docs-viewer/services/docs_import.py` loads `assets/data/docs/scopes/<scope>/index.json` for returned-package review context, including current doc existence, current viewability, parent existence, payload existence, and current summary comparison.

These reads need to move to the internal metadata source, with rendered content or payload paths resolved from that source or another local/manage generated contract.
They should not switch to public `index-tree.json`, public by-id payloads as an implicit index, public search index, or the recently-added payload.

## Implementation Tasks

Allowed statuses are `planned`, `in progress`, `done`, and `deferred`.

| ID | status | action |
| --- | --- | --- |
| 1 | planned | Audit current Data Sharing document adapter reads and list every field currently taken from public docs indexes or public by-id payloads. |
| 2 | planned | Decide whether the internal index is Docs Viewer-owned, Data Sharing adapter-owned, or split into a broad Docs Viewer metadata index plus adapter-specific package projections. |
| 3 | planned | Define the internal index path, schema version, scope coverage, and field contract. Do not reuse the generic `index.json` filename for this replacement. |
| 4 | planned | Extend the Docs Viewer builder/write flow or Data Sharing adapter flow to generate the internal index without depending on public artifacts. |
| 5 | planned | Move Data Sharing selectable-record, package-preparation/export, and returned-package review-context reads to the internal index. |
| 6 | planned | Add tests that fail if Data Sharing relies on public `assets/data/docs/scopes/<scope>/index.json`, public `index-tree.json`, public search indexes, or public recently-added payloads for tooling metadata. |
| 7 | planned | Add tests for internal index generation, missing/stale index handling, and selected Data Sharing adapter behavior. |
| 8 | planned | Remove Data Sharing-driven blockers from public index retirement, including `content_text_length` if no public runtime call site needs it. |
| 9 | planned | Update Data Sharing docs, Docs Viewer data-model docs, and [Config Data Sharing Adapters](/docs/?scope=studio&doc=config-data-sharing-adapters) after the contract is stable. |

## Acceptance Criteria

- Data Sharing no longer depends on public docs scope index rows for non-public or tooling metadata
- Data Sharing prepare/export/selectable-record and returned-package review-context workflows keep current functionality after public docs `index.json` is removed
- internal docs metadata has an explicit schema, owner, and generated path
- the internal index includes the fields Data Sharing needs without bloating public runtime payloads
- public index retirement can proceed independently of Data Sharing metadata needs
- tests cover both the new internal metadata source and the absence of public-index coupling
