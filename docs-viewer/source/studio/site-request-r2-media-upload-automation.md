# R2 Media Upload Automation Request

Status:

- the publisher is available through the CLI and a local Work editor action
- `.env.local` and a bucket-scoped R2 account token are configured locally
- catalogue media staging is external under `$DOTLINEFORM_PROJECTS_BASE_DIR/catalogue/media/`
- dry-run, unchanged comparison, forced replacement, and direct R2 verification have been completed for a catalogue work
- cache-safe public primary URLs now use the confirmed canonical `media_version`
- the publisher now promotes `media_version` only after a complete successful R2 primary-set write and rebuilds the affected per-work JSON
- all existing media-bearing works and work details are backfilled to `media_version: 1`
- stale catalogue `--kind moments` support has been removed from the publisher
- Work `Save` continues into R2 preview/confirmation when media publishing is eligible and remains enabled for a pending retry after cancellation or failure
- new Work-detail sections publish all newly created detail primary sets automatically; replacement of an existing individual detail remains CLI-only
- confirmed section and Work deletes clean exact R2 primary variants after the canonical/local delete, with manual-attention warnings instead of rollback or retry state
- the Docs Viewer media follow-up now has a scope-aware storage contract; implementation remains future work

## Summary

This request tracks adding a script-managed publishing step for media assets that currently need to be copied to Cloudflare R2 manually.

The first supported use case is catalogue primary images generated by the local media pipeline.
The implementation also establishes a credential and upload architecture that can later support docs images without reworking the security model.

The upload, Work editor integration, and cache-safe URL-version boundary are implemented.
Successful complete-set uploads promote the canonical version and regenerate the affected per-work public payload; failed or partial attempts leave the confirmed version unchanged for a direct rerun.

Implemented entrypoint:

```bash
$HOME/miniconda3/bin/python3 studio/services/media/publish_media_to_r2.py --scope catalogue --kind works --id 01007
```

## Problem

The catalogue media pipeline stages source images, generates primary derivatives, and copies thumbnails into repo-owned public asset folders.
Primary derivatives remain under `$DOTLINEFORM_PROJECTS_BASE_DIR/catalogue/media/` as a manual handoff point for remote R2 publishing.

That leaves a recurring manual step:

- find the generated primary derivatives
- upload them to the correct R2 bucket prefix
- avoid missing one size variant
- avoid overwriting the wrong remote object
- keep local logs clear enough to know what was published

This is manageable while changes are small, but it is easy to forget and will become more awkward once docs imports can create image assets too.

Overwriting the remote object is not sufficient to make the replacement visible immediately.
Cloudflare may continue serving the previous custom-domain response from its edge cache until that cache entry expires, even though the new object is already present in R2.

## Goals

Add a project-local script that can publish approved media assets to R2.

The workflow should:

- read R2 credentials from the local environment or a local-only secret file, never from tracked repo files
- support a dry-run preview before upload
- upload catalogue primary derivatives from the existing staging tree
- report skipped, uploaded, unchanged, and failed objects clearly
- avoid logging secrets or full credential-bearing URLs
- produce deterministic object keys
- make successful primary-image replacements visible through a deterministic media-version query
- keep the confirmed public media version separate from staged or incompletely uploaded media
- provide an extension point for later docs media uploads

## Non-Goals

The first implementation should not:

- upload canonical source images
- delete remote R2 objects broadly or implicitly; exact-id primary-variant deletion is supported through `--delete`
- expose R2 credentials to browser-side Studio code
- require credentials in generated docs, public JSON, or tracked config
- replace the existing local thumbnail copy behavior
- reduce the current CDN TTLs or add routine cache-purge API calls
- publish `/moments/` media through the catalogue adapter; Moments is now owned by the Docs Viewer scope
- add docs image uploading before the docs asset import model is settled

## Credential Model

R2 credentials should be provided through environment variables that match the cloud/runtime convention already documented for the repo:

- `R2_ACCOUNT_ID`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET`
- `R2_ENDPOINT`

Local setup uses the gitignored `.env.local` file for those same R2 variables.
The file must be loaded only by CLI scripts or local server-side wrappers.
The script should fail closed when credentials are missing, with setup guidance that names the missing variables but never prints existing secret values.

The R2 API key should be scoped as narrowly as Cloudflare allows.
Preferred permissions:

- object read/write for the target bucket
- object list for preview and unchanged checks
- no account-wide permissions beyond what R2 S3-compatible access requires
- delete permission only when exact-id remote cleanup is intentionally used

## Proposed Architecture

Introduce a media publisher module with a small registry of source adapters and remote destinations.

Conceptual shape:

```text
media publisher
  credentials provider
  R2 client
  source adapter registry
    catalogue primary image adapter
    future docs asset adapter
  upload planner
  upload executor
  report writer
```

The script should separate:

- source discovery
- object-key mapping
- local/remote comparison
- dry-run reporting
- actual upload
- result logging

That separation matters because catalogue primary images and future docs assets have different source roots, but should share credentials, safety checks, and upload behavior.

## External Media Workspace

Catalogue media staging uses one fixed external root:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/catalogue/media/
```

The entire former `var/catalogue/media/` working tree is projected beneath that root, including staged source copies, generated primary and thumbnail variants, and staged work files.
There is no repo-local fallback or migration read path.
These artifacts are rebuildable, potentially large, and untracked, so keeping them inside the checkout provided no source-control benefit and allowed forgotten primary files to inflate the repository working directory.

`studio/shared/python/external_workspace_paths.py` owns the common `DOTLINEFORM_PROJECTS_BASE_DIR` resolution and confinement contract used by external workspaces.
Domain adapters declare their fixed child roots: Catalogue uses `catalogue/media`, Data Sharing uses `data-sharing`, and Docs Viewer uses `docs-viewer` plus `docs-export` for static exports.
Those adapters preserve domain-specific messages and configuration, but do not repeat base-directory or confinement logic.

Local Studio previews request `/studio/media/catalogue/...`.
The local Studio server maps that URL directly to the allowlisted external catalogue-media root; it does not copy the image back into the repository or expose the rest of the projects directory.

## First Use Case: Catalogue Primary Images

The catalogue adapter should discover generated primary derivatives under:

- `$DOTLINEFORM_PROJECTS_BASE_DIR/catalogue/media/works/srcset_images/primary/`
- `$DOTLINEFORM_PROJECTS_BASE_DIR/catalogue/media/work_details/srcset_images/primary/`

Initial scope flags should allow targeted uploads such as:

- all catalogue primaries
- one work id
- one work-detail id
- one media kind

The object-key mapping should match the public URL conventions already used by catalogue pages.
The implemented mappings are:

- `works/img/<work_id>-primary-<width>.webp`
- `work_details/img/<detail_uid>-primary-<width>.webp`

The upload planner should verify that expected size variants exist for each selected item before upload, or clearly report partial sets.
Partial uploads should either be blocked by default or require an explicit flag.

## Moments Scope Boundary

`/moments/` was part of the catalogue media model when this request was first written.
It has since become a Docs Viewer scope, so Moments media belongs with the future docs-owned asset workflow rather than the catalogue R2 adapter.

The stale `--kind moments` publisher path has been removed.
It is not part of the catalogue media-version contract.

## Delivery Cache And Media Version

The public R2 custom domain and GitHub Pages have independent cache behaviour.

For R2 primary images, the live custom-domain response was observed with `Cache-Control: max-age=14400`.
After a primary object was replaced successfully, the unversioned URL continued to return the old cached image while the same URL with a new query value returned the replacement.
The current Cloudflare cache key therefore distinguishes the version query, which gives the catalogue a simple cache-safe replacement contract without changing TTLs or adding purge credentials.

Public catalogue primary URLs should use:

```text
https://media.dotlineform.com/works/img/00008-primary-1200.webp?v=2
```

The version contract is:

- `media_version` is a positive monotonically increasing integer for the independently replaceable primary-image set
- existing media-bearing works and work details are backfilled to `media_version: 1`; an empty string is not a valid steady state
- the confirmed version is stored in the canonical record and emitted in `site/assets/works/index/<work_id>.json`
- work-detail media can carry its own version in the detail entry nested within that same per-work payload
- all widths in one primary set use the same version
- ordinary metadata or prose saves do not increment it
- selecting and generating replacement media makes Studio display the next candidate version as `confirmed + 1`
- Save persists the source selection and staged derivatives but leaves the confirmed version unchanged
- before writing to R2, the publisher verifies that the complete required local variant set exists
- after every required R2 variant uploads successfully, the publisher promotes the candidate version in canonical data and rebuilds the affected per-work public JSON
- a failed, partial, or not-yet-run upload leaves the confirmed version number unchanged, reports the failure clearly, and is recovered by rerunning the publisher
- public primary `src` and `srcset` composition always appends the confirmed version as `?v=<media_version>`
- aggregate lookup, search, recent, and series indexes do not need `media_version` because they render repo-owned thumbnails rather than R2 primary images

The query is a cache-key version, not an archive of the corresponding R2 object.
The workflow does not add rollback, temporary-object, or atomic multi-object publishing machinery; if a rare mid-upload failure occurs, the operator reruns the reported upload.
`studio/services/catalogue/catalogue_media_version.py` owns the narrow promotion seam.
It advances the canonical work or nested work-detail version only when remote bytes changed, then rebuilds the owning `site/assets/works/index/<work_id>.json` without regenerating local media or refreshing unrelated source dimensions.
An all-unchanged rerun rebuilds the current confirmed version without incrementing it, which also provides the recovery path when an earlier public-JSON rebuild failed after canonical promotion.

Catalogue thumbnails are committed under `site/assets/...` and served separately by GitHub Pages through its CDN.
The live thumbnail response was observed with `Cache-Control: max-age=600`, so an overwritten thumbnail may remain stale for about ten minutes after the regenerated file is committed and deployed.
That short delay is acceptable: thumbnails remain unversioned, and the version field does not need to be copied into aggregate indexes solely for thumbnail cache busting.

## Docs Viewer Media Follow-Up

Docs Viewer should use one scope-aware media policy rather than one physical storage location for every scope.
The storage labels describe application publication behavior, not a guarantee that bytes never exist in third-party infrastructure: the repository is public, and `$DOTLINEFORM_PROJECTS_BASE_DIR` may itself be synchronized by iCloud or another filesystem provider. There is currently no sensitive Docs Viewer material that requires a stronger security boundary.

The agreed defaults are:

| Docs Viewer scope | Ordinary images and files | Application publication behavior |
| --- | --- | --- |
| public scopes such as Library, Analysis, and Moments | Cloudflare R2 under `docs/<scope>/...` | published through the shared media origin |
| repo-backed local scopes such as Studio and temporary technical documentation | repo-owned assets when needed | retained with the repository and not intentionally copied to R2 by the Docs media workflow |
| external-local scopes | `$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/media/<scope>/...` | served only through the local Docs Viewer service and not copied into the repo, public site, or R2 |

Thumbs and interactive HTML remain separate asset classes because they have their own rendering and sandbox contracts.
They must still follow the active scope's publication boundary: an external-local scope must not place either class in a repo/public location if support is added later.

### Public Scope R2 Contract

Public Docs Viewer scopes should use the existing literal token form:

<pre><code>&#91;&#91;media:docs/&lt;scope&gt;/img/&lt;filename&gt;&#93;&#93;
&#91;&#91;media:docs/&lt;scope&gt;/files/&lt;filename&gt;&#93;&#93;</code></pre>

The builder already resolves those tokens through the configured shared media base, and Moments already uses this convention for manually copied R2 images.
Library and Analysis can use the same object-key family so public scopes do not need scope-specific media URL logic.

Automated Docs uploads should:

- activate the reserved `r2_upload` Docs Import storage mode
- reuse the existing server-owned R2 credentials and client rather than exposing credentials to browser code
- upload only the media plans owned by the current import/apply action
- use immutable fingerprinted filenames for automatically materialized media, so replacement produces a new URL without adding Catalogue-style `media_version` fields
- commit a source link to new remote media only after every required upload for that import record succeeds
- report safe scope/file identities without local absolute paths, object credentials, signed URLs, or checksums

Automatic remote deletion is not part of the first Docs slice.
Docs assets may be shared across documents, so document deletion does not prove that an R2 object is unreferenced. A later reference/orphan report can identify cleanup candidates without adding rollback or ownership machinery to the initial uploader.

### External-Local Media Contract

External-local media should be co-located with its external source and generated data under the configured projects base:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/media/<scope>/img/
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-viewer/media/<scope>/files/
```

This requires a new Docs Import storage mode, provisionally `external_assets`, that:

- derives the target root from the existing marker-rooted external scope configuration rather than accepting an arbitrary absolute path
- copies imported media into the allowlisted scope media root
- emits a scope-relative media reference rather than an R2 token or `/assets/...` URL
- serves media through a confined local Docs Viewer service route
- keeps absolute filesystem paths out of generated payloads and browser configuration
- rejects `r2_upload` and public repo-asset destinations for an external-local scope

The external-local rule is about keeping the application workflow local to that scope. It does not claim that the underlying filesystem is physically offline or unsynchronized.

### Existing Media And Migration

There is very little existing Docs Viewer media:

- Moments media is already manually stored in R2 and already uses the target token convention
- only a small amount of other public-scope media exists
- Studio and other local scopes currently have no media to migrate

No bulk migration phase is required.
Existing public media can remain in place or be normalized as it is next touched, while new public imports use R2 by default once the adapter is implemented.
Existing repo-held technical-documentation references may remain repo-owned.

The smallest implementation sequence is:

1. add and validate the `external_assets` mode so external-local imports cannot select a public destination
2. add the confined local media read route for external-local scopes
3. activate `r2_upload` for public-scope Docs Import plans using immutable filenames
4. add a small exact-scope CLI path for manually authored public Docs assets when the import workflow is not involved
5. leave deletion cleanup and migration automation out until actual usage demonstrates a need

## CLI Shape

The default should be dry-run.

Useful options:

- `--scope catalogue`
- `--scope docs`
- `--kind works|work_details`
- `--id <id>`
- `--all`
- `--changed-only`
- `--force`
- `--write`
- `--report-json <path>`

## Upload Safety

The script should:

- read only from allowlisted local source roots
- refuse path traversal and symlink escapes
- compare local file size and checksum or ETag-compatible digest where practical
- skip unchanged remote objects by default
- require `--force` to overwrite changed remote objects when remote comparison finds a conflict
- use concise logs that show ids, local relative paths, object keys, and statuses
- never print secret values
- never include signed URLs in logs

## Studio Integration

The Work editor uses `Save` as the sole media-publish entrypoint.
Save runs its existing workflow first, then evaluates the unchanged media-publish eligibility contract: single saved published Work, ready local primary set, no unsaved source changes, available service, and idle route.
An ineligible result ends the composed workflow quietly.
An eligible result continues into R2 preview and confirmation and marks media publishing pending.
Cancellation, a blocked preview, or failure leaves that state in place, so Save remains enabled and its no-source-change path can retry the eligibility handoff.
Successful publishing, loading another record, or leaving single-record mode clears the pending state.
It uses a server-owned preview/apply flow:

1. `POST /studio/api/catalogue/media-publish-preview` checks the expected confirmed media version, validates the complete local width set, and compares the variants with R2.
2. Studio shows one terse mandatory `Publish R2 media?` or `Replace R2 media?` confirmation. Changed remote objects use the explicit `Replace media` action; detailed width and version counts stay out of this routine modal.
3. Preview returns an opaque fingerprint of the exact local/remote comparison. `POST /studio/api/catalogue/media-publish-apply` repeats that comparison and rejects local-byte or remote-state drift before any upload.
4. Apply performs the upload through the same media-owned publisher used by the CLI. Forced replacement also requires `confirm_overwrite: true` at the service boundary.
5. A complete result promotes or retains the confirmed canonical version, rebuilds focused public work JSON, records the action in Studio Activity, and returns the refreshed Work record.
6. The editor reloads the focused record and preview so the confirmed `?v=<media_version>` URL is immediately visible.

The browser sends only the work id, expected media version, opaque preview fingerprint, explicit overwrite decision, and Studio Activity context.
It never receives raw R2 credentials, credential-bearing URLs, local paths, object keys, or checksums; `.env.local` is read only by the Local Studio service and media publisher.

Existing Work-detail replacement continues to use the CLI because there is intentionally no individual detail mutation action.
The integrated Work editor does own aggregate detail-section creation and deletion:

- section creation writes and builds every selected detail first, then publishes those exact detail ids in one server-owned R2 run
- section deletion completes its canonical/public/local aggregate delete first, then removes every affected detail primary set from R2
- Work deletion cascades through the Work primary set and every remaining dependent detail primary set
- missing remote objects count as successful cleanup
- a rare remote failure leaves canonical creation/deletion complete and returns only safe affected ids for a manual Cyberduck or CLI follow-up; there is no rollback, pending job, or automated retry state

## Acceptance Criteria

The first catalogue upload milestone is implemented:

- dry-run lists catalogue primary derivatives that would upload to R2
- write mode uploads selected catalogue primary derivatives only when `--write` is passed
- remote delete mode removes selected catalogue primary derivatives only when `--delete --write` is passed with an exact kind and id
- missing credentials fail with a clear non-secret error
- unchanged remote files are skipped by default
- changed remote files require `--force`
- object keys are deterministic and documented in [Publish Media To R2](/docs/?scope=studio&doc=scripts-publish-media-to-r2)
- logs and optional JSON reports avoid secrets and signed URLs
- tests cover credential loading, source-root allowlisting, object-key mapping, dry-run planning, and overwrite behavior

The cache-safe delivery follow-up is complete:

- existing media-bearing works and work details are backfilled to `media_version: 1`
- the confirmed `media_version` is emitted in the appropriate per-work JSON only
- public R2 primary `src` and `srcset` URLs append that confirmed version
- Studio displays the current confirmed version and a candidate next version when replacement media is staged
- Save does not advance the confirmed version
- the publisher validates a complete local variant set before upload
- a complete successful upload advances the canonical version and rebuilds the affected per-work JSON
- a failed or partial upload reports failure without advancing the confirmed version and can be rerun directly
- stale catalogue support for `--kind moments` has been removed

The Local Studio integration is complete for Works:

- a published Work with a ready complete local primary set is eligible for media publishing after Save
- Save automatically continues into the media-publish workflow when eligible and exits quietly when it is not
- there is no separate media-publish button; pending cancellation and failure retries keep Save enabled until publishing succeeds or the record context changes
- unsaved, unpublished, unavailable, or incomplete Work media cannot be applied from the editor
- preview is write-free and confirmation is mandatory
- apply revalidates the opaque preview fingerprint and stops before upload when local bytes or remote state drift
- changed remote objects require explicit replacement confirmation
- the local service owns credential reads, R2 comparison, upload, version promotion, and public JSON regeneration
- the browser response omits credentials, signed URLs, local paths, object keys, and checksums
- successful apply refreshes the editor to the confirmed media version and writes a unified Studio Activity row
- new detail-section creation publishes every created detail primary set after the complete local build succeeds
- detail-section deletion removes every affected detail primary set after canonical/public/local deletion
- Work deletion also cleans all remaining dependent detail primary sets without requiring the user to delete sections first
- existing individual detail replacement remains CLI-only

## Benefits

- removes the manual R2 copy step for catalogue primary images
- reduces the chance of missing or mismatched primary image variants
- creates one secure credential boundary for future media publishing
- gives docs image publishing a planned extension point instead of a separate ad hoc script

## Risks

- incorrect object-key mapping could overwrite or publish media in the wrong R2 location
- overly broad R2 credentials would increase blast radius if leaked
- remote comparison behavior may differ from local checksum expectations
- promoting a version before all variants are uploaded could cache an incomplete replacement under the new public URL
- GitHub-hosted thumbnails may briefly lag behind the versioned R2 primary after a deployment
- adding docs assets later could require link-rewrite decisions that are not part of the first catalogue milestone
