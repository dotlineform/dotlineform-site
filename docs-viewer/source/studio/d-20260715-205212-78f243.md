# Control Surfaces

## Purpose

Give every Docs Viewer shell, panel, and hosted-view control an intentional owner and rendering surface so adding a control does not require another one-off toolbar model.

This is the feature parent. It keeps the architectural cleanup separate from the first product feature that needs it: index multiple selection.

## Feature Documents

- [Concept](/docs/?scope=studio&doc=d-20260715-205212-4782bc) — the recurring problem, intended control taxonomy, boundaries, and success condition.
- [Architecture](/docs/?scope=studio&doc=d-20260715-205212-66b4a5) — proposed registration, host, projection, dispatch, and extension boundaries.
- [Foundation Delivery](/docs/?scope=studio&doc=d-20260715-205212-3268e2) — migrate the existing cross-shell control surfaces and remove obsolete plumbing without adding new product behavior.
- [Docs Viewer Roadmap](/docs/?scope=studio&doc=d-20260714-234030-1e0af6) — scheduling, dependencies, and suggested feature order.

## Status

- Feature: ready and not active.
- The foundation delivery is bounded and safe to implement independently; it is not a permanent audit.
- This feature does not add selection, checkboxes, or group movement.

## Durable Home

The current [Panel Hosts](/docs/?scope=studio&doc=d-20260528-224754-964108) document is the eventual durable owner. At closeout it should become **Panel Hosts and Control Surfaces** and absorb the shipped taxonomy, ownership, extension method, and known weak spots.

The feature parent, proposed architecture, and delivery should retire after that transfer. Retain the concept only if it still explains something useful that the durable owner does not; do not preserve a second copy of the architecture.
