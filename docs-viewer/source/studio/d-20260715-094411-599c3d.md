# Document Identity Architecture

This is proposed structure for DI-01. The code remains current authority until the delivery ships.

## Current Starting Point

Docs Viewer already treats `doc_id` as a stable lookup key across hierarchy, URLs, generated payloads, semantic references, exports, reports, and reviewed updates. Metadata changes preserve it, and the live watcher does not assume that source filename equals `doc_id`.

The remaining semantic coupling is concentrated at creation boundaries:

- `docs_source_model.ensure_unique_stem()` derives a new identifier from title;
- `docs_management_mutations.plan_create()` uses that identifier as both `doc_id` and filename;
- ordinary import previews derive proposed identifiers from staged filenames or titles;
- import collision handling exposes replacement title-like IDs;
- Docs search currently includes `doc_id` in general search terms.

## Proposed Ownership

### Identity Allocation

`docs_source_model.py` owns one generator used by every local new-document boundary.

The generator:

1. receives or creates the same second-resolution timestamp written to `added_date`;
2. formats `d-YYYYMMDD-HHMMSS-`;
3. appends six hexadecimal characters from a standard cryptographic random source;
4. checks the result against current identities and source filenames;
5. retries on collision;
6. returns the ID without writing.

Planning remains write-free. Apply writes the already planned ID; it must not silently generate a different one after confirmation.

### Creation Boundaries

- New Document allocates one ID and uses it as the Markdown filename.
- Ordinary single-source and collection imports allocate local IDs for genuinely new documents.
- Overwrite and returned-package update workflows preserve the selected canonical target ID.
- Import media may use the new ID when naming newly materialized assets, but existing media filenames are not identity and are not renamed by the migration.
- Titles remain independently editable metadata.

### Added-Date Normalization

The migration planner normalizes every repository-owned document to `YYYY-MM-DD HH:MM:SS` and records an evidence method:

- `front-matter-time`;
- `git-time-on-matching-date`;
- `midnight-default`.

Current minute-only timestamps gain `:00`. Date-only metadata uses a matching Git addition time when available and otherwise keeps its date at midnight.

### Migration Planner

A focused one-off planner scans configured repository-owned scope and sub-scope sources and produces an ignored review artifact under `var/docs/document-identity/`.

Each mapping row contains:

- scope and sub-scope where relevant;
- source path and title;
- old and new `doc_id`;
- original and normalized `added_date`;
- timestamp evidence method;
- old and new source filename;
- parent and reference effects.

The map is keyed by scope plus old identity. It must be stable between dry-run review and apply.

Before apply, validate:

- every code and destination filename is unique and safe;
- every current parent resolves through the map;
- every configured default, non-loadable, and manage-only identity resolves;
- every recognized Docs Viewer `doc` or `subdoc` link can be rewritten;
- no source write or destination path escapes its configured root;
- external-local sources are excluded from DI-01.

### Apply And Rebuild

Apply rewrites canonical source and config from the reviewed map:

```text
reviewed identity map
  -> front-matter doc_id and parent_id
  -> source filename moves
  -> Docs Viewer doc/subdoc links
  -> configured identity references and fixtures
  -> full affected-scope docs builds
  -> full affected-scope search builds
  -> broken-link and old-identity scans
```

Generated payloads are rebuilt from source and stale old-ID files are removed. They are not migrated as authority.

### Search

General Docs search should use title, parent title, and useful document metadata without tokenizing opaque IDs. Exact full-ID matching may be retained as a separate diagnostic path or exact token.

## External And Historical Boundaries

- Configured external-local scopes require a separate delivery because their source, workspace, backup, and lifecycle ownership differs.
- Pre-migration staged packages, review sessions, exports, bookmarks, and copied URLs may contain old IDs. Historical artifacts are not rewritten; active work is recreated or repaired deliberately.
- Public inbound links change once. No permanent alias registry is introduced.
- Git history and the migration map provide recovery if a source or reference mapping is wrong.

## Pressure Points

- Current validators accept slightly different character sets; the selected format fits their common safe subset, but allocation should have one canonical validator.
- Import workflows distinguish create, overwrite, and returned canonical identity in several adapters; the boundary must be explicit rather than inferred from one generic record shape.
- A mass rename may exceed targeted watcher assumptions, so implementation should suppress incidental rebuilds and perform deliberate full follow-through after canonical source is coherent.
- Search currently derives terms directly from `doc_id`; changing relevance must retain any genuinely useful exact diagnostic lookup.
- Front-matter and Git dates are provenance signals, not guaranteed historical truth.

## Retirement

After DI-01, stable ownership and weak spots move into the retained concept/current overview. Delete this proposal once it no longer helps choose or implement unresolved structure.
