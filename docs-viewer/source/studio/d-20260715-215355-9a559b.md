# Local Tree Move Projection Architecture

## Architectural Outcome

A committed move should flow through one model-first browser projection. The DOM reflects the updated model; it is not a second authority.

## Execution Path

```text
drag intent
  -> server validates and commits the source change
  -> targeted docs and search rebuild
  -> canonical move record returned
  -> browser updates document and child-collection models
  -> existing subtree moves to its sorted destination
  -> affected parent chrome and displayed metadata re-project
```

The normal successful path does not read `index-tree` or run the complete sidebar renderer.

## Ownership

| responsibility | owner |
| --- | --- |
| validation, source mutation, rebuild, and canonical response | current move service |
| document, parent, and ordered-child state | shared index model owner |
| subtree placement and affected parent chrome | focused tree projection owner |
| workflow, busy state, cache invalidation, and recovery | management action controller |
| displayed-document and ancestry projection | current route, breadcrumb, and info owners |

The projection updates the moved document's parent, removes and inserts its id in the affected child collections, and uses the current deterministic title/doc-id ordering. It detaches and reinserts the existing `<li>` subtree when mounted, preserving node identity and unrelated expansion state.

## Recovery Boundary

Projection begins only after the server reports a successful commit, so there is no optimistic rollback. If the committed record cannot be reconciled with the current model, report the failure and perform one authoritative index reload. Recovery must remain visible and separately tested so it does not conceal a reproducible projection defect.

## Extension And Weak Spots

- A later group move may supply several canonical move records to the same projection boundary.
- Create, delete, rename, import, and external file changes do not join this feature automatically.
- DOM-only movement would split authority from the index model.
- Recreating the moved subtree would discard useful local node state.
- Parent chrome and displayed ancestry can become stale unless both old and new relationships are projected.
- Broad cache reads after the move would reintroduce the refresh cost under another name.
