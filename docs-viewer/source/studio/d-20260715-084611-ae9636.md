# Studio Overview

## What Studio Is

Studio is the local catalogue authoring and maintenance app. It provides a browser shell for catalogue workflows and a loopback Python service for validated reads, writes, builds, publication plans, media operations, and project reports.

Studio is not part of the deployed public site. Analytics, Admin, Docs Viewer, and their write authority are sibling local apps rather than Studio route modules.

## Capabilities

Studio currently supports:

- browsing catalogue works and drafts
- creating and editing works, work-detail sections, and series
- bulk work import through preview and confirmed apply
- reviewing the catalogue field registry
- generating and opening the Project State report
- previewing and applying catalogue deletes, builds, publication, and media publication
- linking to public catalogue results through the configured public-preview base

The checked-in route registry is the exact authority for mounted browser pages. The catalogue API and service tests are the authority for available mutations.

## Execution Path

```text
browser requests /studio/... route
  -> Studio app server serves studio-shell.html
  -> studio-app.js loads /studio/runtime-config.json
  -> app.routes resolves the route
  -> configured HTML template is mounted
  -> configured route script boots
  -> static fallback read or /studio/api/catalogue/... request
  -> catalogue API validates and delegates
  -> focused catalogue service reads or changes canonical data
  -> required generated catalogue, search, media, or public output follows
```

The browser may use checked-in or generated JSON for safe fallback reads. Writes never use those browser paths as authority; they go through named local endpoints and server-side validation.

## Authority Boundaries

### Routes And Browser Boot

- `studio/app/frontend/config/studio-config.json` owns source route records and browser-safe data paths.
- `studio/app/server/studio/studio_app_config.py` validates the route registry and injects runtime endpoints, site bases, media settings, pipeline values, and the resolved view list.
- `studio/app/frontend/studio-shell.html` is the stable outer shell.
- `studio-app.js` composes the shell; the configured route template owns stable route markup; the route script owns behavior and dynamic state.

### Reads And Writes

- `studio-data.js` and route-specific clients own browser reads and response shaping.
- `studio-transport.js` and focused editor service clients own browser transport.
- `studio_catalogue_api.py` is the narrow HTTP adapter for catalogue reads and writes.
- `studio/services/catalogue/` owns validation, plans, source mutation, build follow-through, publication, media, activity, and generated catalogue behavior.

### Data

- `studio/data/config/` owns checked-in Studio domain config.
- `studio/data/canonical/` owns canonical catalogue records.
- `studio/data/generated/` owns rebuildable local lookup and activity projections.
- `site/` owns tracked public output; it is not a Studio source tree.
- external project and media roots come from environment-backed workspace helpers, never browser config absolute paths.

## Configuration And Extension

To add or change a Studio route:

1. add or update its `app.routes` record
2. provide the configured template and route script
3. let server validation reject missing or unsupported fields
4. add the focused browser/service behavior the route actually needs
5. verify route boot and ready-state integration

To add a catalogue operation:

1. define the focused catalogue service or extend its current owner
2. expose a narrow API adapter path only when browser access is needed
3. project the endpoint through runtime config when the browser must discover it
4. keep mutation validation and file authority on the server
5. update the workflow’s durable owner and focused tests

Do not use route config, UI visibility, or a browser data path as proof of write capability.

## Sibling-App Boundaries

- [Docs Viewer](/docs/?scope=studio&doc=d-20260424-000000-50b63f) owns `/docs/`, documentation management, Docs Import, review, and scope lifecycle.
- [Analytics](/docs/?scope=studio&doc=d-20260506-181900-d08dbe) owns analytics and Data Sharing browser routes and APIs.
- [Admin](/docs/?scope=studio&doc=d-20260606-212412-01ed44) owns operational checks, audits, and admin reports.
- The public site owns deployed reading routes and tracked public assets.

Studio may link to these surfaces. It does not inherit their route registries, frontend modules, service endpoints, or authority.

## Known Weak Spots

- Runtime config is assembled from checked-in Studio config, Python-owned endpoint/media constants, environment, and `_data/pipeline.json`; there is no single source file showing the final payload.
- The route registry is copied into `app.runtime.views` for current helpers and tests, so two runtime projections describe the same routes.
- `studio_app_config.py` retains one route compatibility key for catalogue field registry lookup.
- Static fallback reads and server-backed reads coexist. A route change must confirm which path is authoritative for that workflow.
- The catalogue API adapter still imports a broad catalogue service surface; exact domain ownership lives below it rather than in the adapter.
- Some older workflow docs describe retired route modules or repeat module inventories and need consolidation against the current registry.
