# Control Surfaces Concept

## Problem

Docs Viewer has several useful control patterns, but not one complete control-surface model.

- The index header hard-codes panel collapse and expand buttons.
- The viewer toolbar hard-codes search, recent, and index-view controls.
- The main document toolbar reads some controls from the view registry and maps known renderer keys to buttons.
- Management actions, menus, context actions, scope selection, and theme switching use additional render and event paths.
- Action definitions already separate operation targeting from visual placement, but not every control surface reaches them consistently.

Each current path works. The gap appears when a new control needs a home: adding it directly is easy, but repeats ownership, eligibility, rendering, state, and event decisions that the next control will repeat again.

Index selection exposes the problem concretely. A manage-only **Select** control belongs to the active index-tree view, not to global viewer controls, panel layout chrome, or the management workflow toolbar.

## Desired Outcome

Every control visible outside a self-contained component has an obvious category and owner:

- **app controls** provide app-wide navigation, discovery, scope, or presentation behavior;
- **panel chrome** changes panel layout or visibility;
- **view controls** act on the active view or one of its modes;
- **workflow actions** invoke an operation with an explicit target and cardinality rule;
- **component controls** remain local to a report, modal, picker, tree row, or other self-contained widget.

Stable toolbar hosts render eligible app, panel, or view controls. Code-owned registration determines what may appear. Focused controllers own live state. Workflow action definitions continue to own targets separately from placement.

The result should make a future control routine to add without making every button globally configurable.

## Invariants

- Route config may enable features or hide a known control id; it cannot define toolbar DOM, module URLs, icons, or executable behavior.
- Public, manage, and review entrypoints contribute only controls they are allowed to load.
- Availability and live state are distinct: registration decides eligibility; a controller decides hidden, disabled, pressed, busy, count, and label state.
- Panel layout controls do not acquire document-action targeting merely because they are buttons.
- One workflow action resolves the same targets wherever it appears in a toolbar, menu, or context menu.
- Optional toolbar mounts remain real optionality; a route must not render replacement chrome merely to satisfy code assumptions.
- Adding the foundation does not change current visible behavior.
- No compatibility path remains after existing control surfaces migrate.

## Product Boundary

This is a shell and host architecture feature, not a universal button framework.

Report filters, modal confirmation buttons, semantic-picker rows, search result controls, bookmark rows, and tree-node expand/collapse buttons remain owned by their components. They may reuse presentation primitives where useful, but they do not need global registration.

The feature does not add index selection, a graph view, new workflows, configurable toolbar ordering, plugins, arbitrary JSON-defined controls, or a visual redesign.

## Success Condition

After migration, adding the index-tree **Select** control should require:

- one manage-owned control contribution;
- one focused renderer/controller implementation;
- projection into the existing index-view control host.

It should not require a new mount, a new toolbar shape, direct global-runtime DOM wiring, or a duplicate action-target rule.
