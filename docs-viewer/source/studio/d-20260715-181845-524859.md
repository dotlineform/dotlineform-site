# Document Visualisations Concept

## Why Consider It

High-level diagrams could make execution paths, ownership, and public/manage/server boundaries easier to understand than prose or text art.

There is also a concrete data-driven destination: a Docs Viewer panel that visualises the network formed by document semantic references, catalogue Works, and Series. `/analysis/` may eventually publish or extend that view.

These needs should not be collapsed into one renderer merely because both produce graphics.

## Authored-Diagram Decision

Use D2 as the default source language for authored Studio architecture and workflow diagrams.

D2 source is the durable authority. The D2 CLI renders it to SVG before Docs Viewer reads the document; Docs Viewer displays the SVG as an ordinary image and does not parse D2 or load a diagram runtime in the browser.

This is a default for maintainable technical diagrams, not a ban on ordinary images, inline SVG, or bespoke interactive assets where those are the better fit. It also has no bearing on the future D3 semantic graph: D2 produces authored figures, while D3 would render live repository data in a dedicated panel.

## Three Different Capabilities

### Authored Diagrams

Text in Markdown describes a flowchart, sequence, state, relationship, or architecture map. The diagram changes when the author edits that text.

This is the likely fit for Docs Viewer architecture and workflow documentation.

### Build-Rendered Figures

A command turns diagram source into SVG or another static asset during a docs or site build. Readers receive the result without downloading the rendering library.

This improves runtime independence and export behavior, but adds a local and CI build dependency plus generated-asset ownership.

### Data-Driven Visualisations

A specification or browser module reads generated data and produces charts, plots, networks, or interactive exploration. The data owner drives the result; Markdown may explain or host it but is not the data source.

This is the long-term fit for the semantic relationship graph and possible public `/analysis/` use. It should use a focused panel or visualisation owner rather than arbitrary executable code embedded in Markdown.

These capabilities may share presentation styles, but they do not need one renderer, source format, or delivery.

## Semantic Relationship Graph

The intended graph is a projection over relationships the repository already owns:

- semantic-reference artifacts provide document-to-Work, document-to-Series, document-to-Moment, and future registry-backed edges;
- catalogue data owns Work-to-Series membership and canonical Work/Series identity;
- generated browser-safe records provide titles, routes, and compact display metadata;
- a Docs Viewer panel owns graph layout, filtering, selection, and navigation.

Likely initial node types are documents, Works, and Series. Moments, tags, and other semantic kinds can join only through their existing registry and data owners.

D3 is a strong candidate because this is an interactive network/index view rather than a conventional chart. The visualisation would consume a bounded graph projection shaped for the panel; it should not make the browser reconstruct canonical relationships from unrelated payloads or create a second semantic-reference model.

The graph is initially a read and navigation surface. Editing tokens, catalogue membership, or canonical records remains in the existing authoring applications.

## Consequences Of Choosing D2

- **New local tool:** machines that author or render diagrams need the standalone D2 CLI. Ordinary Docs Viewer readers do not.
- **Pinned, offline build:** the repository must name a supported D2 version and provide one project command that checks it. A docs build must never download or update D2 implicitly.
- **No Node consequence:** D2 does not require `package.json`, npm, or browser JavaScript. A future D3 delivery may create a JavaScript dependency for its own reasons.
- **Build-rendered SVG:** SVG is the default output. It stays sharp at different sizes and avoids the headless-browser dependency used by D2's PNG export.
- **Deliberately narrow format:** the first Docs Viewer contract is one D2 board rendered to SVG. D2 can also produce compositions, presentations, animation, PNG, PDF, and other outputs; those capabilities do not become Docs Viewer requirements merely because the CLI provides them.
- **Two owned artifacts:** reviewable `.d2` source is canonical; SVG is derived. Both need explicit Docs Viewer-owned locations, and Studio-only diagrams must not drift into the public `site/` tree.
- **Normal Markdown image:** documents refer to the rendered SVG through the existing media path rather than injecting generated SVG markup. The surrounding heading, caption, and prose carry the accessible explanation.
- **Reproducible rendering:** layout engine, theme, padding, fonts, and other material options belong in the project command rather than ad hoc author commands.
- **Failure by exit status:** validation must use the CLI result, not merely the presence of an SVG, because D2 can leave partial output after a render error.
- **Watcher/build work:** the implementation must decide when changed `.d2` sources are rendered, how stale SVG is detected, and how a diagram error is reported without corrupting unrelated document output.
- **Publication remains explicit:** Studio diagrams remain local. If a public scope later adopts D2, its rendered SVG must enter that scope's normal asset and manual publication gate; the D2 executable still does not ship to the public site.
- **Static export:** exports should carry the rendered SVG and retain the `.d2` source only in complete source-and-asset packages where canonical dependencies are expected.

The [D2 CLI](https://d2lang.com/tour/man/) and [export documentation](https://d2lang.com/tour/exports/) are the upstream authority for command and format behavior. The repository's eventual wrapper and pinned version will be the authority for this project.

## Invariants

- Markdown remains readable and editable without a graphical renderer.
- A broken diagram does not prevent the rest of its document from rendering.
- Ordinary documents do not load a visualisation dependency they do not use.
- Public routes receive no local-only renderer, service, or data source by accident.
- Data-driven visualisations preserve the canonical data owner's authority.
- The semantic graph consumes existing reference and catalogue identities rather than defining parallel nodes or edges.
- Markdown does not become a route for arbitrary JavaScript execution.
- Versions are pinned and generated results are reproducible enough to review through source control and the publication gate.

## Questions Before A Delivery

1. Which two or three current documents should prove the D2 convention?
2. Where should canonical `.d2` sources and derived Studio SVGs live?
3. Should D2 rendering be part of the normal docs build, a preceding focused command, or both through one shared service?
4. How should the pinned version be installed and checked across local and GitHub workflow environments?
5. Which layout, theme, sizing, and accessibility defaults are enough for the first delivery?
6. For the semantic graph, what question should the first view answer that the current reference report cannot?
7. Which node and edge types make a bounded useful first graph, and how large is the real dataset?
8. Should the graph begin as a local Docs Viewer panel before any public `/analysis/` promotion?

## Separate Delivery Candidates

- Authored documentation diagrams: install and pin D2, define the `.d2`-to-SVG ownership and build contract, and prove it on a small set of architecture/workflow documents.
- Semantic relationship graph: prove one read-only Docs Viewer panel over an explicit generated graph projection, initially limited to the node and edge types needed for a useful index view.

These should remain separate roadmap deliveries even if both eventually use repository-level JavaScript tooling.

When those answers point to one complete result, add an architecture document if needed and promote one bounded delivery into the [Docs Viewer Roadmap](/docs/?scope=studio&doc=d-20260714-234030-1e0af6).
