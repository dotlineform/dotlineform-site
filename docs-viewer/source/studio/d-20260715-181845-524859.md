# Document Visualisations Concept

## Why Consider It

High-level diagrams could make execution paths, ownership, and public/manage/server boundaries easier to understand than prose or text art.

There is also a concrete data-driven destination: a Docs Viewer panel that visualises the network formed by document semantic references, catalogue Works, and Series. `/analysis/` may eventually publish or extend that view.

These needs should not be collapsed into one renderer merely because both produce graphics.

## Authored-Diagram Decision

Use Mermaid as the default source language for authored architecture and workflow diagrams in any Docs Viewer scope.

Mermaid source is the durable authority. An author-local command renders it to static SVG before Docs Viewer reads the document; Docs Viewer displays the SVG as an ordinary image and does not parse Mermaid or load a diagram runtime in the browser.

This is a default for maintainable technical diagrams, not a ban on ordinary images, inline SVG, D2 where Mermaid proves insufficient, or bespoke interactive assets where those are the better fit. It also has no bearing on the future D3 semantic graph: Mermaid produces authored static figures, while D3 would render live repository data in a dedicated panel.

## Three Different Capabilities

### Authored Diagrams

Text in Markdown describes a flowchart, sequence, state, relationship, or architecture map. The diagram changes when the author edits that text.

This is the likely fit for Docs Viewer architecture and workflow documentation.

### Build-Rendered Figures

A command turns diagram source into SVG or another static asset before publication. Readers receive the result without downloading the rendering library.

This improves runtime independence and export behavior. For the current need it adds an author-local tool and generated-asset ownership, but not a CI or repository package dependency.

### Data-Driven Visualisations

A specification or browser module reads generated data and produces charts, plots, networks, or interactive exploration. The data owner drives the result; Markdown may explain or host it but is not the data source.

This is the long-term fit for the semantic relationship graph and possible public `/analysis/` use. It should use a focused panel or visualisation owner rather than arbitrary executable code embedded in Markdown.

These capabilities may share presentation styles, but they do not need one renderer, source format, or delivery.

## Semantic Relationship Graph

The intended graph is a projection over relationships the repository already owns:

- semantic-reference artifacts provide document-to-Work, document-to-Series, document-to-Moment, and future registry-backed edges;
- catalogue data owns Work-to-Series membership and canonical Work/Series identity;
- generated browser-safe records provide titles, routes, and compact display metadata;
- a Docs Viewer panel owns graph layout, filtering, selection, and navigation.

Likely initial node types are documents, Works, and Series. Moments, tags, and other semantic kinds can join only through their existing registry and data owners.

D3 is a strong candidate because this is an interactive network/index view rather than a conventional chart. The visualisation would consume a bounded graph projection shaped for the panel; it should not make the browser reconstruct canonical relationships from unrelated payloads or create a second semantic-reference model.

The graph is initially a read and navigation surface. Editing tokens, catalogue membership, or canonical records remains in the existing authoring applications.

## Mermaid-First Static SVG Boundary

- **Local authoring tool:** machines that author or refresh diagrams need Mermaid CLI and its own local prerequisites. Ordinary Docs Viewer readers do not.
- **No repository runtime:** the current boundary does not add `package.json`, npm installation, Mermaid JavaScript, or a browser rendering promise to Docs Viewer or the public site.
- **Static SVG:** SVG is the default output because it stays sharp at different sizes and behaves as an ordinary document image.
- **Two owned artifacts:** reviewable `.mmd` source is canonical; SVG is derived. Commit the SVG when it has a repo-backed media root; public-scope SVGs instead pass through the existing publication gate.
- **Normal Markdown image:** documents refer to the rendered SVG through the existing media path rather than injecting generated SVG markup. The surrounding heading, caption, and prose carry the accessible explanation.
- **Explicit publication:** a public-scope SVG passes through that scope's existing media staging and R2 publication gate. Mermaid, Node, npm, Puppeteer, and D3 do not ship with it.
- **No automatic rendering yet:** the Docs watcher and normal docs build do not discover or render Mermaid source. A first delivery may add a small command, but should not introduce a package manifest merely to install the tool.
- **Readable source matters:** prefer diagram forms whose nodes and relationships remain understandable in the `.mmd` file. Use D2 only after a real Mermaid limitation is demonstrated.
- **Separate D3 delivery:** interactive graph layout, selection, navigation, movement, or live repository data belongs to a later runtime feature with its own dependency and accessibility decisions.

The [Mermaid CLI](https://github.com/mermaid-js/mermaid-cli) is the upstream authority for command behavior. If the repository later gains a wrapper, its supported version and render options become the project authority.

## Suggested Repository Convention

Use the existing configured source and media roots rather than creating a Studio-only asset system:

```text
<scope source root>/mermaid/<diagram-name>.mmd
<scope media root>/img/<diagram-name>.svg
```

For a repo-backed scope this maps to `docs-viewer/source/<scope>/mermaid/` and `docs-viewer/source/<scope>/media/img/`. For an external-local scope it maps to `<source root>/mermaid/` and `<source root>/media/img/`. The configured local `media/img/` directory already exists; create `mermaid/` when the scope gains its first diagram source.

Public scopes keep the same scope ownership and basename, but the SVG goes through the existing exact-file staging and R2 publication flow instead of being served from repo media. Public Markdown uses `[[media:docs/<scope>/img/<diagram-name>.svg]]`; repo-backed and external-local Markdown uses `/docs/media/<scope>/img/<diagram-name>.svg`.

Additional conventions:

- use lowercase kebab-case names describing the diagram, not the rendering tool;
- keep one `.mmd` source and one `.svg` output with the same basename;
- keep source under the scope that owns the explanation rather than in `site/` or a global miscellaneous-assets folder;
- do not edit the generated SVG by hand;
- render with explicit background, theme, dimensions, and other material options once a project command exists;
- include useful surrounding prose and alt text; the SVG is not the sole statement of the document's meaning;
- if two scopes need the same diagram, name one owning source rather than silently forking two `.mmd` files.

## Invariants

- Markdown remains readable and editable without a graphical renderer.
- A broken diagram does not prevent the rest of its document from rendering.
- Ordinary documents do not load a visualisation dependency they do not use.
- Public routes receive no local-only renderer, service, or data source by accident.
- Data-driven visualisations preserve the canonical data owner's authority.
- The semantic graph consumes existing reference and catalogue identities rather than defining parallel nodes or edges.
- Markdown does not become a route for arbitrary JavaScript execution.
- The supported local tool version and material render options are recorded once the first project command exists, and generated results are reproducible enough to review through source control and the publication gate.

## Questions Before A Delivery

1. Which one or two current documents should prove the Mermaid convention?
2. Is a direct local `mmdc` command sufficient initially, or would a small repository wrapper materially improve consistency?
3. Which Mermaid CLI version, theme, sizing, and accessibility defaults are enough for the first diagram?
4. Should a later check detect an SVG older than its `.mmd` source without making rendering part of every docs build?
5. For the semantic graph, what question should the first view answer that the current reference report cannot?
6. Which node and edge types make a bounded useful first graph, and how large is the real dataset?
7. Should the graph begin as a local Docs Viewer panel before any public `/analysis/` promotion?

## Separate Delivery Candidates

- Authored documentation diagrams: prove the `.mmd`-to-SVG convention on one useful diagram, record the local tool/version command, and add only the consistency checks the use case demonstrates it needs.
- Semantic relationship graph: prove one read-only Docs Viewer panel over an explicit generated graph projection, initially limited to the node and edge types needed for a useful index view.

These should remain separate roadmap deliveries. Static Mermaid publishing does not itself justify repository-level JavaScript tooling; the interactive D3 graph may do so later.

When those answers point to one complete result, add an architecture document if needed and promote one bounded delivery into the [Docs Viewer Roadmap](/docs/?scope=studio&doc=d-20260714-234030-1e0af6).
