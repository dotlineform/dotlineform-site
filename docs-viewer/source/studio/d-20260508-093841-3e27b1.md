# Publish Media To R2

## What It Owns

`studio/services/media/publish_media_to_r2.py` previews, uploads, replaces, or exactly deletes approved Cloudflare R2 objects.

It supports two deliberately different modes:

- **catalogue media** publishes complete Work or Work-detail primary variant sets and coordinates confirmed media versions;
- **Docs media** publishes one exact staged file to one configured public Docs Viewer scope/class.

The shared implementation owns credentials, object mapping, remote comparison, upload/delete requests, safe reporting, and write confirmation. The source/workflow rules remain with the catalogue or Docs feature that invokes it.

## Catalogue Flow

Local source is the externally staged primary tree below `$DOTLINEFORM_PROJECTS_BASE_DIR/catalogue/media/`.

Object families are derived from checked pipeline and site configuration, for example:

```text
works/img/<work_id>-primary-<width>.webp
work_details/img/<detail_uid>-primary-<width>.webp
```

The normal required catalogue widths are resolved from pipeline configuration. A group missing a required width is blocked unless an operator explicitly allows a partial CLI upload; partial groups are never promoted as the confirmed public version.

Preview one Work, then apply after reviewing it:

```bash
$HOME/miniconda3/bin/python3 studio/services/media/publish_media_to_r2.py --scope catalogue --kind works --id 01007
$HOME/miniconda3/bin/python3 studio/services/media/publish_media_to_r2.py --scope catalogue --kind works --id 01007 --write
```

Changed remote objects require `--force`. Exact deletion requires `--delete --write` plus one kind and ID; broad `--all` deletion is intentionally unavailable.

## Confirmed Versions

Works and details carry positive canonical `media_version` values. Public primary URLs append the confirmed version as a cache token.

- A complete upload that changes at least one required remote object increments the item's version once.
- An all-current result rebuilds the focused public payload at the existing version.
- Failed, blocked, or partial groups do not advance the version.
- Ordinary source save and local derivative generation do not advance it.
- Work-detail versions are projected inside the owning Work JSON.

The publisher updates the canonical version only after the complete remote set succeeds, then rebuilds the focused Work payload. This keeps “new local bytes exist” separate from “public clients may safely request a new confirmed version.”

## Work Editor Composition

The Work editor calls importable preview/apply services rather than shelling out or implementing R2 logic in the browser.

For an eligible saved published Work, `Save` can continue into media publication. The browser sees compact per-width state and confirmed version only; credentials, local paths, object keys, checksums, and signed requests remain server-side.

Preview is write-free and returns an opaque fingerprint of the compared local/remote state. Apply repeats version and comparison guards and rejects drift before upload. New publication and replacement require distinct confirmations.

If the composed media step is cancelled or fails, the canonical source save remains complete and the media action stays retryable. It is not represented as a rollback of the source edit.

Detail-section creation publishes the newly created complete detail sets after the local build. Section or Work deletion performs the authoritative local transaction first, then attempts exact remote cleanup. Missing objects count as clean; remote failure becomes a visible manual-cleanup warning rather than restoring deleted canonical source.

## Docs Exact-File Flow

Docs publishing reads one direct-child file from the shared import-staging drop zone and maps it to:

```text
docs/<scope>/<img-or-files>/<filename>
```

Preview and apply one exact file:

```bash
$HOME/miniconda3/bin/python3 studio/services/media/publish_media_to_r2.py --scope docs --docs-scope library --kind img --staged-filename diagram.png
$HOME/miniconda3/bin/python3 studio/services/media/publish_media_to_r2.py --scope docs --docs-scope library --kind img --staged-filename diagram.png --write
```

The Docs scope must explicitly use the matching R2 upload/media-prefix policy. There is no Docs `--all` or remote-delete mode. Changed objects require `--force`, though a new filename is preferable when cache-safe replacement matters.

This command provides a transport capability; it does not decide which Docs assets are supported, how they are linked, or which document owns them. Those rules belong to the Docs media/asset-collection workflow.

## Credentials And Safety

Local runs load R2 settings from `.env.local`, with optional additional `--env-file` inputs and environment fallback for hosted runs. Missing variables are named; configured values are never printed.

The publisher:

- confines discovery to allowlisted staging roots and rejects traversal/symlink escapes;
- defaults to dry-run and requires `--write` for remote mutation;
- compares size and ETag/MD5 before deciding current, new, or changed state;
- skips current objects and blocks changed ones without explicit force;
- keeps reports/logs to non-secret identifiers and statuses;
- promotes catalogue versions only after complete success.

Run `--help` for the exact current flags. [Srcset Builder](/docs/?scope=studio&doc=d-20260331-000000-0a1f71) owns derivative creation; [Scoped JSON Catalogue Build](/docs/?scope=studio&doc=d-20260418-000000-4ec702) owns local/public build planning.
