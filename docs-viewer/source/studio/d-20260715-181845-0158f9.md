# Document Visualisations

## Purpose

Add useful authored diagrams and data-driven visualisations through boundaries suited to each purpose.

The concrete long-term data use is a Docs Viewer graph panel connecting documents, catalogue Works, and Series through semantic references and catalogue-owned relationships. Authored architecture and workflow diagrams remain a separate branch because they have different source, build, and runtime consequences.

## Feature Documents

- [Concept](/docs/?scope=studio&doc=d-20260715-181845-524859) — the semantic relationship graph, authored-diagram branch, candidate tools, build consequences, invariants, and questions to answer before delivery.
- [Docs Viewer Roadmap](/docs/?scope=studio&doc=d-20260714-234030-1e0af6) — relative importance and promotion into a finishable delivery.

## Status

- Feature: proposed and unsequenced.
- Authored-diagram decision: Mermaid source rendered locally to static SVG before Docs Viewer reads it.
- Mermaid is an author-local tool, not a Docs Viewer or public-site runtime. Static SVG publishing does not require `package.json`, a browser diagram renderer, or a Docs Viewer build dependency.
- The proposed convention is scope-relative: canonical `.mmd` source under the scope source root, and derived SVG through that scope's existing media and publication path.
- No Mermaid installation, supported-version check, rendering command, source folder, first diagram, or automated build integration has been implemented yet.
- D2 remains a possible fallback only if a concrete diagram exposes a Mermaid layout or expression limit.
- D3 remains the meaningful candidate for the separate semantic relationship graph.
- A first delivery should follow one proven use case rather than install a general visualisation stack.

Add architecture only when a rendering and ownership boundary is worth proposing. Add a delivery only when one complete outcome can be named and verified.
