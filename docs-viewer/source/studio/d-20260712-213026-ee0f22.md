# Workspaces

Workspaces are coordinated working environments inside an application.
They combine an activity-specific main view with the panel layout, controls, settings, commands, keyboard behavior, and preview or output surfaces needed to complete that activity.

This use of `workspace` describes an application UI concept.
It is separate from filesystem workspaces, external staging roots, and project directories.

## Purpose

The workspace model exists for activities that no longer fit comfortably into one document view, modal, or isolated toolbar action.

A workspace may need to:

- replace the current document mode temporarily
- hide, collapse, restore, or repurpose surrounding panels
- divide its main region into editor, preview, visualisation, or inspection surfaces
- place activity-specific settings in an existing hosted panel
- register controls and keyboard commands through shared application contracts
- load specialised frontend modules and CSS only while they are needed
- produce more than one kind of derived output

The first planned consumer is the [Moments Composition Workspace](/docs/?scope=studio&doc=d-20260712-000000-45286c).

## Core Boundaries

- The application shell owns panel regions, layout projection, toolbar hosts, command dispatch, lifecycle transitions, and restoration of prior state.
- A workspace owns its internal UI, working state, preview mounts, and coordination of its specialised tools.
- A workspace must request layout through a named contract; feature code must not hide or resize unrelated panels directly.
- Workspace layout is transient. Entering a workspace must not overwrite the user's persisted reading or panel preferences.
- Controls and keyboard shortcuts invoke the same registered commands.
- Route and scope config may enable, disable, or narrow known workspace capabilities, but executable modules and handlers remain code-owned.
- First-party tool registries are explicit repo module registries, not generic plugin systems or arbitrary config-driven module loaders.
- Public read-only presentation remains separate from management-only editing and workspace services.

## Content And Output

Some workspaces operate only on an existing canonical source format.
Others may need to distinguish:

1. canonical authored content
2. a tool- or algorithm-specific recipe
3. materialised HTML, DOM, SVG, canvas, image, data, or other derived output

Derived output must not become the only recoverable copy of canonical content.
Output-specific limitations, accessibility fallbacks, publication behavior, and rebuild ownership belong to the focused workspace or engine contract.

## Working Method

Workspace architecture should grow through vertical implementation:

1. name the concrete user activity
2. make the smallest shared refactor needed to host it cleanly
3. implement one usable vertical feature
4. review the hardcoded assumptions the feature exposed
5. refactor the newly proven boundary
6. add a deliberately different feature to test that boundary

Do not attempt to predict every possible workspace or algorithm before implementing the first one.
Do not allow a first implementation to become an implicit permanent contract merely because only one consumer exists.
