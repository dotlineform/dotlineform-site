# Control Surfaces Architecture

## Architectural Outcome

Docs Viewer should have stable control hosts and one code-owned definition/projection model for app and hosted-view controls. Panel chrome and component-local controls remain explicit adjacent categories rather than being forced into a command registry.

The existing view registry and action definitions are the starting structure. This feature should complete and simplify that structure, not add a second competing registry.

## Current Shape

The current code already provides useful parts:

- the view registry normalizes views, modes, controls, app kinds, features, backend requirements, and route-policy visibility;
- the main-view renderer and management contribution render registered document controls;
- action definitions own active-document, selection, and scope targets plus selection cardinality;
- panel layout projects index and info state;
- app-shell renderers provide stable index, main, info, and top-bar regions.

The incomplete seams are also concrete:

- the index header is a styled control container rather than a registered toolbar host;
- viewer and management controls use separate hard-coded rendering paths;
- several handlers are wired through individual DOM references in the global runtime;
- control renderer mappings and live-state projection are surface-specific;
- `toolbar` and `toolbarView` are normalized for hosted views but do not drive a mounted toolbar.

## Ownership Model

| category | owner | examples | registration rule |
| --- | --- | --- | --- |
| app control | app shell or app-kind contribution | search, recently added, scope, theme | code-owned definition projected into a stable app surface |
| panel chrome | panel host/layout owner | collapse index, expand index, close info | shell-owned control projected from panel state |
| view control | active view or mode contribution | bookmark, info, edit, source, save, Select | code-owned definition eligible only for its owner view/mode |
| workflow action | action definition and focused workflow controller | import, publish, move, delete | one target/cardinality rule, reusable across placements |
| component control | focused component | report filter, modal action, tree-node toggle | local rendering and state; no global registration |

Visual placement never becomes operation authority. A registered control may reference an action id, but the action definition continues to resolve targets independently.

## Definition And Projection

Build on the existing code-owned definition contributions from shared, manage, and review entrypoints.

A host-level control definition needs only the fields that affect eligibility and rendering:

- stable control id and accessible label;
- owning app surface, panel/view, and optional mode;
- app kinds, route features, route-policy visibility, and genuine backend capability requirements;
- renderer key and optional action id.

Definition order is the default presentation order. Do not add user-configurable placement, arbitrary numeric ordering, or route-authored icons without a real requirement.

The registry projects eligible controls for the current app, route, active view, and mode. A focused controller then projects live control state by id:

```text
code-owned definitions + route/app/view context
                    |
                    v
          eligible control projection
                    |
                    v
             stable surface host
                    |
                    v
        focused controller live state
```

Live state includes only what the renderer needs, such as hidden, disabled, pressed, busy, label, count, and expanded state. Domain state remains in its existing controller rather than moving into the registry.

## Hosts And Rendering

The app shell should expose stable mounts for:

- app-wide viewer controls;
- management app/scope controls;
- active index-view controls;
- active main-view or mode controls.

Panel chrome may share the same visual toolbar container while retaining a separate host-owned group. For example, index collapse/expand remains layout chrome while **Select** is projected into the adjacent active-view control mount.

Renderers consume projected definitions and produce consistent identity, accessibility, and state hooks. Entrypoint-owned renderer contributions prevent management-only modules from entering public routes.

The presence of eligible controls should determine whether a view-control group exists. Retire inert `toolbar` and `toolbarView` capability fields unless implementation finds a distinct behavior they genuinely own.

## Events And Actions

Rendered controls expose stable control ids and, when applicable, action ids.

- A workflow control dispatches through the existing action-target contract before its focused controller performs work.
- A view-state control dispatches to its owning controller and requests a new projection.
- Panel chrome dispatches to the panel-layout owner.
- Menus and context menus may render differently, but a shared action id must resolve the same target set and availability.

Avoid adding another global switch statement for every new control. Hosts may delegate events, while owning entrypoints/controllers provide the behavior associated with registered renderer or control ids.

## Config And Capability Boundary

- Code registration allowlists executable controls and renderers.
- Route feature policy decides whether a feature is eligible on that route.
- Route view policy may hide a registered control id.
- Backend capabilities describe actual service support, not cosmetic toolbar preferences.
- Scope config may affect workflow availability through its existing projected capabilities; it does not define UI controls.

## Weak Spots To Resolve In Delivery

- Existing control normalization assumes a view owner, while app controls and panel chrome have different owners.
- Some current renderers mix mount creation, control creation, and feature decisions.
- Global runtime references couple event binding to fixed element ids.
- Source mode and other focused controllers hold direct button references for live state.
- Public routes intentionally omit some toolbar/path UI, so migration must preserve null-safe optional mounts.
- Action availability can drift between toolbar, menu, and context-menu placements if projection remains duplicated.

These are migration pressure points, not a permanent file inventory.

## Durable Documentation Boundary

After delivery, fold the implemented structure, extension method, and remaining weak spots into [Panel Hosts](/docs/?scope=studio&doc=d-20260528-224754-964108), renamed **Panel Hosts and Control Surfaces**. Exact definitions, renderer mappings, and state shapes remain code authority.
