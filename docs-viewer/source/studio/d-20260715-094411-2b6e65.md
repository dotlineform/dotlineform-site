# Document Identity

## Why It Exists

A document's identity should survive changes to its title, meaning, hierarchy, feature ownership, and source organisation.

Titles provide human navigation through the index, search, source view, and editor integrations. `doc_id` exists for durable references and machine contracts; putting mutable meaning into it causes link churn without improving discovery.

## Identity Contract

New local documents use:

```text
d-YYYYMMDD-HHMMSS-xxxxxx
```

- `d-` identifies a document code.
- The date and time match the document's second-resolution `added_date`.
- Six lowercase hexadecimal characters separate same-second and parallel creations.
- Allocation checks current identities before use; it does not require a central sequence or registry.
- The source Markdown filename matches the immutable code.
- Runtime lookup remains scope-local, although the shape makes accidental cross-scope collisions unlikely.
- Generated-data builds and generated document reads reject IDs outside this format.

The identifier is not a title, slug, access token, security boundary, document type, app owner, hierarchy path, or promise about current content.

`docs-viewer/services/docs_document_identity.py` owns the format, timestamp, validation, and allocation primitives. Creation workflows own when an identity is allocated.

## Creation And Update Rules

- New Document, ordinary new imports, collection records that become new local documents, and the first document in a new scope allocate a local identity.
- Scope identity stays human-readable; its first document identity does not derive from the scope name.
- Import filenames, external record IDs, and titles are source context. A new local document records a different source ID as provenance rather than adopting it.
- Overwrite and returned-package update workflows preserve the selected canonical document identity, filename, `added_date`, and placement rules.
- Rename changes `title`; move changes `parent_id`; neither changes identity or filename.
- Duplicate allocates a new identity. Delete removes the identity.
- Planning stays write-free. Multi-step preview/apply workflows carry the planned identity into apply rather than silently allocating another after confirmation.

Repository-owned and external-local scopes use the same identity contract and allocator. Migrating an external-local scope remains an explicit, selected operation because its source and generated files sit outside the repository boundary.

The external migration plan:

- records configured workspace markers rather than machine-specific absolute paths;
- rewrites hierarchy, scope defaults, filenames, and known viewer links together;
- can consume an earlier migration map when external documents link into an already-migrated scope;
- retains no runtime aliases after verification and rebuild.

The Notes scope was migrated after the repository corpus. Its date-only front matter used midnight because repository Git history is not evidence for external source files.

## Date Provenance

New documents write `added_date` and `last_updated` as:

```yaml
added_date: YYYY-MM-DD HH:MM:SS
last_updated: YYYY-MM-DD HH:MM:SS
```

Creation gives both fields the same captured local timestamp. After creation, only a change to the canonical body, `title`, or `summary` advances `last_updated` through the shared source-model comparison and timestamp owner. The edit basis is intentionally a positive allowlist.

Direct source edits outside a managed write must advance `last_updated` themselves when that positive edit content changes. The live watcher compares the same content and reports missing timestamp evidence; it does not silently rewrite canonical source. Legacy date-only values remain valid source metadata but do not qualify for edited-Recent ordering until a qualifying edit supplies a full timestamp.

The repository migration normalized existing `added_date` values by preserving explicit times, using Git addition time when a date-only value matched the Git date, and otherwise using midnight. Deliberately invented historical dates were retained.

That migration was a provenance normalization, not a claim of perfect historical certainty. If a future legacy source has no usable date, assign an explicit conventional value rather than deriving identity from unstable filesystem metadata.

The later bounded Recent seed was deliberately narrower: it followed renames to the latest Git author time at which body, `title`, or `summary` changed, and wrote only the union needed to populate current edited projections. Dirty qualifying edits used actual write time and the rest of the corpus stayed unchanged. Git history is migration evidence, not a live ordering source.

## Search And Display

Titles and parent titles remain the visible navigation model. Search indexes the complete opaque ID as one exact lookup token for URLs, logs, reports, and diagnostics; date and random-suffix fragments do not enter ordinary relevance terms.

No compatibility aliases, legacy-format lookup, or redirect registry were retained after migration. The migration command can read old IDs only while applying a reviewed rewrite plan; it is not part of runtime document resolution. Git history and the ignored reviewed migration maps are the repair surface for one-off mistakes.

Docs Review package-record identities, package identities, semantic-reference target IDs, and reference slugs have separate owners and formats. They are not document identity exceptions. Docs Import maps a package create to a new local identity and an overwrite to the existing canonical identity.

## Extension Rule

When adding a workflow that can create documents:

1. decide whether the operation creates a genuinely new local document or updates an existing canonical one;
2. allocate only for create, using the shared identity owner and the same timestamp written to `added_date`;
3. preserve canonical identity for update;
4. keep source-specific IDs in provenance;
5. prove preview/apply identity stability and filename agreement in focused tests.

Generated payloads, search entries, links, exports, logs, and reports consume source identity. They do not allocate or redefine it. Exact adapters and validators remain code/test authority.
