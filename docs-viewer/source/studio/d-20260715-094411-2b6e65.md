# Document Identity Concept

## Purpose

A document's identity should survive changes to its title, meaning, hierarchy, feature ownership, and source organisation.

Docs Viewer now provides the useful navigation layer: index titles, search, source view, management tools, and VS Code integration. A title-like `doc_id` and filename no longer improve discovery enough to justify the link churn and semantic drift they create.

## Identity Contract

Every newly allocated local identity uses:

```text
d-YYYYMMDD-HHMMSS-xxxxxx
```

Example:

```yaml
doc_id: d-20260715-094411-2b6e65
title: Document Identity Concept
```

- `d-` identifies a document code.
- The date and time record identity issuance to second precision.
- The six lowercase hexadecimal characters prevent same-second and parallel-creation collisions.
- The complete code is validated against current scope identities before use.
- The source Markdown filename matches the immutable code.
- Identity is scope-local in current runtime contracts, even though the generated shape makes accidental cross-scope collisions unlikely.

The identifier is not a title, slug, access token, security boundary, document type, app owner, hierarchy path, or promise about current content.

## Added Date

`added_date` is useful provenance and should have one complete shape:

```yaml
added_date: YYYY-MM-DD HH:MM:SS
```

For the current repository corpus:

1. preserve an explicit front-matter time and add `:00` when only seconds are missing;
2. when front matter has only a date and Git first added the current document on that date, use the Git time;
3. otherwise preserve the chosen date and use `00:00:00`;
4. retain deliberately invented dates, including the Moments dates, rather than replacing them with later repository-import times.

The migration plan records whether each normalized timestamp came from front matter, matching-date Git history, or the midnight default. Exact historical certainty is not required; consistent useful provenance is.

Every current repository document already has a date. If a future legacy source genuinely has none, assign an explicit conventional date such as `1900-01-01 00:00:00` rather than hiding uncertainty behind filesystem metadata.

The document code derives its timestamp component from the normalized `added_date`. The random suffix remains necessary because several documents may share the same second.

## Editing Consequences

- Rename changes `title`, never `doc_id` or source filename.
- Move changes `parent_id`, never identity.
- Duplicate allocates a new identity.
- Delete removes the identity and allows genuinely meaningful inbound links to fail visibly.
- Ordinary HTML, Markdown, text, document, or media-wrapper imports allocate a new local identity independent of the staged filename or title.
- A returned package updating an existing canonical document preserves that document's supplied identity.
- A package record that becomes a new local document receives a local identity; any external identifier is provenance, not the local `doc_id`.

No compatibility aliases or redirects are retained after the migration. The reviewed migration map and Git history are the repair surface for one-off mistakes.

## Search And Display

Titles and parent titles remain the human navigation model. Opaque IDs should not contribute fragmented date and random-suffix tokens to ordinary relevance scoring. Exact-ID lookup may remain available for logs, URLs, reports, and diagnostics.

## Lasting Invariants

- Allocate once; never rename an identity.
- Keep identity free of mutable meaning.
- Display titles to people and codes to contracts.
- Preserve canonical identity through review and update workflows.
- Allocate local identity at the boundary where a genuinely new local document is created.
- Keep the generator centralized and the format consistent across creation paths.
- Treat generated payloads, search entries, references, and exports as consumers of source identity, not alternate owners.

## Non-Goals

- a global document registry or central sequence;
- meaningful or memorable URLs;
- aliases for previous title-derived IDs;
- rewriting Git history, completed logs, or historical packages;
- renaming unrelated media merely because its filename contains an old document slug;
- using predictability or obscurity as authorization.
