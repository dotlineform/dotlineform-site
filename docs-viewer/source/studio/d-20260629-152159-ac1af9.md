# Export

Docs Viewer export converts existing generated Docs Viewer payloads into a standalone static HTML folder for a repo-backed local scope.

The static HTML export is intentionally separate from public-scope Publish and from Analytics Data Sharing exports.
It does not deploy content, copy public route assets, package JSONL data, parse Markdown, or rebuild generated docs.
It reads generated Docs Viewer payloads and writes local HTML files that can be opened directly from `file://`.

## Scope

The export is available only for repo-backed local Docs Viewer scopes, such as `studio`.

Supported scope requirements:

- `scope_type: local`
- `include_scope_param: true`
- generated docs output is inside the repository
- generated payloads include `index-tree.json` and `by-id/<doc_id>.json`
- `$DOTLINEFORM_PROJECTS_BASE_DIR` is set to an absolute path

Rejected scope families:

- public read-only scopes such as `library`, `analysis`, and `moments`
- `local_external` scopes
- scopes whose source or generated output is outside the repository

External local scopes need a separate portability decision because their data and filesystem ownership live outside this repository.

## Output

Apply writes to:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-export/<scope>/
```

For `studio`, the output shape is:

```text
$DOTLINEFORM_PROJECTS_BASE_DIR/docs-export/studio/
  index.html
  styles.css
  docs/
    <doc_id>.html
```

Every apply run wipes the destination scope folder first, then writes the current export.
There is no stale-file manifest, partial sync, rollback record, or delete bookkeeping.
Delete Scope does not remove static export output; the user owns cleanup of stale export folders.

## Input Payloads

The exporter reads existing generated Docs Viewer payloads:

```text
docs-viewer/generated/docs/<scope>/index-tree.json
docs-viewer/generated/docs/<scope>/by-id/<doc_id>.json
```

`index-tree.json` supplies document order and hierarchy.
Each referenced by-id payload supplies the rendered `content_html` and metadata used by the document page.

The exporter does not read `docs-viewer/source/<scope>/*.md`, parse front matter, render Markdown, or apply viewability filtering.
All generated docs in the selected scope tree are exported, including docs that are not public-viewable.

## Service Owner

The backend owner is:

```text
docs-viewer/services/docs_static_html_export.py
```

Core responsibilities:

- validate that the requested scope is repo-backed local
- resolve generated input paths from `docs-viewer/config/scopes/docs_scopes.json`
- resolve the destination under `$DOTLINEFORM_PROJECTS_BASE_DIR/docs-export/<scope>/`
- load `index-tree.json`
- collect ordered doc ids from the nested tree
- validate doc ids as safe HTML filenames
- load each `by-id/<doc_id>.json` payload
- render root `index.html`
- render one document HTML page per doc
- render shared `styles.css`
- rewrite internal Docs Viewer links
- replace the destination folder on apply
- delete the destination folder on delete

The service is independent from Docs Viewer frontend modules.

## Management API

The management-only endpoints are:

```text
POST /docs/export/static-html/apply
POST /docs/export/static-html/delete
```

Apply request shape:

```json
{
  "scope": "studio",
  "action": "export"
}
```

Apply response includes:

- `schema_version: docs_static_html_export_v1`
- `operation: apply`
- `scope`
- `document_count`
- `file_count`
- `default_doc_id`
- absolute `destination`
- display-safe `destination_label`
- `summary_text`

Delete request shape uses the same `scope`.
Delete only needs a valid repo-backed local scope and destination path; it does not require generated payload files to still exist.

Route constants live in:

```text
docs-viewer/services/docs_management_routes.py
```

Dispatch is owned by:

```text
docs-viewer/services/docs_management_service.py
```

## Capability Projection

Management capabilities expose static HTML export availability per scope.

Global capability:

```json
{
  "static_html_export": {
    "apply": true,
    "delete": true
  }
}
```

Per-scope capability:

```json
{
  "static_html_export": {
    "apply": true,
    "delete": true,
    "destination": "/docs-export/studio/",
    "document_count": 232,
    "default_doc_id": "docs-viewer",
    "error": ""
  }
}
```

Capability projection is owned by:

```text
docs-viewer/services/docs_management_capabilities_service.py
```

The frontend uses this capability to show the `Export` action only when the active scope supports static HTML export.

## Frontend Ownership

The management Actions menu includes an `Export` item in:

```text
docs-viewer/runtime/js/management/docs-viewer-management-actions-renderer.js
```

The action is hidden unless `scopeStaticHtmlExportSupported()` returns true for the current scope.
Capability helpers live in:

```text
docs-viewer/runtime/js/management/docs-viewer-management-capabilities.js
```

The client call is:

```text
applyManagedDocsStaticHtmlExport()
```

in:

```text
docs-viewer/runtime/js/management/docs-viewer-management-client.js
```

The action controller opens a confirmation modal showing:

- source scope
- document count
- default document
- destination folder

If the user confirms, the modal closes and the action controller sends the apply request.
While the export is running, manage mode uses the existing `managementBusy` state, which sets the waiting cursor through `docs-viewer/static/css/docs-viewer-manage.css`.

The export action stays separate from public-scope `Publish`.

## HTML Rendering

The root `index.html` renders a nested tree from `index-tree.json` and links to:

```text
docs/<doc_id>.html
```

Each document page renders:

```html
<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>{{ escaped title }}</title>
  <link rel="stylesheet" href="../styles.css">
</head>
<body>
  <nav><a href="../index.html">Index</a></nav>
  <main>
    {{ content_html }}
  </main>
</body>
</html>
```

The page template does not inject an additional heading.
Generated `content_html` already includes the source document heading.

Metadata such as `<title>` is HTML-escaped.
`content_html` is inserted as generated by the existing Docs Viewer builder.

## Link Rewriting

The exporter rewrites internal Docs Viewer links for the exported scope:

```text
/docs/?scope=<scope>&doc=<doc_id>
```

Document-page links become:

```text
<doc_id>.html
```

Root index links are rendered as:

```text
docs/<doc_id>.html
```

Fragments are preserved.
External links and links to other scopes are left unchanged.
The exporter does not validate whether rewritten doc links point to an exported document.
