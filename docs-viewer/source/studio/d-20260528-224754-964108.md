# Docs Viewer Panel Hosts and Control Surfaces

Docs Viewer has three panel regions—index, main, and info—and four stable control surfaces: app viewer, app management, active index view, and active main view.

The app shell owns regions and mounts. The code-owned registry owns definitions and eligibility. Surface hosts own rendering and delegated identity. Focused controllers own live state and behaviour. Workflow action definitions separately own operation targets and cardinality.

## Ownership Model

| category | owner | examples | rule |
| --- | --- | --- | --- |
| app control | shared app or app-kind contribution | search, recently added, scope, theme, review package controls | register on `app-viewer` or `app-management` |
| panel chrome | panel host or layout owner | collapse/expand index, close info | keep adjacent to the surface registry |
| view control | active view or mode contribution | bookmark, info, edit, source, save | register on `index-view` or `main-view` with its view/mode owner |
| workflow action | action definition and focused workflow controller | import, publish, move, delete | resolve targets independently of placement |
| component control | focused component | report filter, modal action, tree-node toggle | render locally; do not register globally |

Visual placement never becomes operation authority. A registered control may reference an action id, but it cannot redefine that action's target or cardinality.

## Definition, Projection, And Rendering

```text
shared definitions + entrypoint contribution
                    |
                    v
      registry eligibility projection
  app + feature + capability + route policy
             + active view/mode
                    |
                    v
          stable control-surface host
                    |
                    v
      entrypoint-safe renderer contribution
                    |
                    v
       focused live state and dispatch owner
```

A control definition declares:

- a stable id, label, renderer key, and optional action id;
- `ownerType` as `app` or `view`;
- one stable `surfaceId`;
- an owner view and optional modes for view controls;
- app kinds, route features, route-policy visibility, and genuine backend capability requirements.

Definitions are code-owned. Route config may enable features or hide known ids; it cannot supply renderer code, module URLs, icons, or new controls.

Definition order is presentation order. Boot composition keeps shared app controls first, then entrypoint-owned controls, then shared view controls. Do not add user-configurable or numeric ordering without a concrete requirement.

Surface hosts render eligible definitions, preserve stable control/action data identity, apply accessible live state, hide empty mounts, and delegate `click`, `change`, and `input` events. A renderer contribution may be shared or supplied only by the manage/review entrypoint allowed to load it.

Live state is keyed by control id and limited to renderer-facing values such as `hidden`, `disabled`, `pressed`, `busy`, `expanded`, `label`, and `count`. Domain state stays in the focused controller. Where two owners contribute a real shared concern—for example management availability plus source-save readiness—the runtime merges their projected state rather than moving domain state into the registry.

## Stable Surfaces

- `app-viewer`: shared discovery/index-view controls and review package controls.
- `app-management`: manage toolbar, scope, theme, and workflow placements; absent from public/review shells.
- `index-view`: controls owned by the active index view. Panel collapse/expand remains adjacent panel chrome.
- `main-view`: bookmark/info plus manage edit/source/save controls filtered by active view and mode.

An optional mount remains genuine optionality. A route with no eligible main-view controls does not render replacement chrome merely to satisfy a reference.

Composite controls are acceptable when their internals form one focused component. The Docs Review package group is one review app control; its selector, repair/assets buttons, and links remain owned by the review controller.

## Actions And Invocation Context

Action definitions are the authority used by toolbar, menu, main-view, and context placements before dispatch.

- displayed-document controls use `activeDocId`;
- batch actions use the independently supplied `selectedDocIds` checkbox set;
- scope actions have no document target;
- context-menu operations pass `invocationDocId`, which supplies the primary invocation document without becoming checkbox selection.

The checkbox set starts empty until the selection feature supplies it. Do not infer selection from the displayed document, active-row styling, context invocation, or checkbox DOM. Exact-one and multi-selection cardinality failures must remain explicit disabled resolutions.

## Panels And Lifecycle

### Index

The index panel renders the tree or manage-only graph placeholder. `layoutStates` is its only hosted-view layout capability. Collapse/expand belongs to panel layout; index-view switching is an app-viewer control; future index-tree actions belong in the adjacent `index-view` surface.

### Main

The main panel represents rendered documents, search results, and recent results. Markdown source is a manage-only mode inside `rendered-document`. The document-view coordinator keeps active view/mode, main-view controls, and the default info view synchronized.

### Info

The info host loads, mounts, updates, unmounts, and disposes its active view inside the panel body. The panel renderer owns close/status chrome. Current views include public-safe metadata and the manage-only semantic token picker.

An independently hosted module may implement:

```text
load() -> mount(context) -> update(context) -> unmount(context) -> dispose(context)
```

Main views and modes may also implement `beforeLeave(context)`. Use lifecycle only for modules that own mounted resources or state; keep stateless rendering and projection helpers stateless.

Hosted modules receive projected context rather than broad runtime state. Registration or receiving a client handle never authorizes a write; server endpoints remain authoritative.

## Adding A Control

1. Classify it as app control, panel chrome, view control, workflow action, or component control.
2. Add a shared definition only when it is public-safe; otherwise contribute it from the manage or review entrypoint.
3. Reuse the existing surface mount and declare app/feature/capability/view/mode eligibility.
4. Add a renderer contribution keyed by the registered renderer id.
5. Have the focused owner publish live state by control id and handle delegated dispatch.
6. If it invokes a workflow, reference the existing action id and resolve its target context before work begins.
7. Preserve the public import boundary and add the narrowest registry/host/route evidence for the new seam.

A future manage-only index-tree **Select** control should therefore require one definition contribution, one focused renderer/controller, and projection into the existing `index-view` host—no new toolbar mount or global DOM reference.

## Weak Spots

- Main views still combine hosted lifecycle modules with established document/search/report controllers.
- `index-graph` remains a layout placeholder, not evidence of a generic visualization extension point.
- The shared hosted-view context is broader than every consumer needs and should narrow when stable per-view contracts emerge.
- Checkbox selection has no UI yet, so selection-targeted actions remain unavailable until that separate feature supplies state.
- Review package controls are intentionally one composite app control; split them only if independent placement or state ownership becomes useful.
