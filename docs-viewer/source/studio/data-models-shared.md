# Shared Artifact Patterns

## Why Artifacts Are Split

The public site and Docs Viewer repeatedly separate stable routing, lightweight discovery, record-local content, and search:

```text
canonical source
  -> stable route shell
  -> lightweight index/tree/recent payload
  -> per-record by-id payload
  -> separate search projection
```

This avoids a single large runtime dump, keeps URLs stable, and lets list/detail/search consumers load only their contract.

## Pattern

### Route Shell

Tracked public HTML provides a stable URL and boot boundary. It owns presentation/runtime installation, not canonical record content.

### Index Or Tree

Generated list/navigation payloads carry only the identity, ordering, hierarchy, labels, and small metadata needed for discovery. Exact shapes are domain-owned; a Docs Viewer tree is not a catalogue index schema.

### Per-Record Payload

Generated by-id/per-record JSON carries heavier record-local structure and rendered HTML. Consumers fetch it after selecting an identity.

### Search Projection

Search has its own normalized/ranked artifact because discovery text and fields are not the same as navigation or detail contracts.

## Common Choices

- a small header/schema/version/freshness/count block where the consumer needs it;
- object maps for canonical ID lookup;
- arrays when order/ranking is part of the contract;
- build-time rendered HTML when the browser should not parse Markdown;
- source-specific generated roots for public versus local/management data.

These are tendencies, not one universal JSON schema. Read the owning builder/config/tests for exact fields.

## Ownership Rules

- canonical source and checked config are edited;
- builders own projections and skip/write behavior;
- public artifacts live under `site/`; local Docs Viewer generated reads live under their configured generated root;
- Data Sharing reads canonical Docs Viewer source context, not generated docs payloads;
- search reads its declared source artifacts and remains a separate projection;
- route consumers should not infer authority from whichever payload is easiest to fetch.

## Assurance

Each domain should provide:

- one focused source/model doc;
- one builder/projection contract;
- schema/fixture checks for public/browser-visible fields;
- cross-reference/audit checks where artifacts join;
- a clear rebuild path from canonical source.

[Projection Contract](/docs/?scope=studio&doc=data-models-projection-contract) explains the audit boundary. App/domain overviews point to their exact builders.

## Weak Spots

- One conceptual entity spans several artifacts, so partial rebuilds can leave temporary disagreement.
- Similar headers/IDs across domains can look like a shared schema when they are not.
- Legacy/compatibility fields in generated payloads can survive after consumers disappear.
- Central inventory pages go stale quickly; use builders/config/search to discover exact current files.
