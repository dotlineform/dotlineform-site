PImage blendTwo(PImage base, PImage top, float opacity) {
  PImage out = createImage(base.width, base.height, ARGB);
  base.loadPixels();
  top.loadPixels();
  out.loadPixels();
  for (int i = 0; i < base.pixels.length; i++) {
    color cb = base.pixels[i];
    color ct = top.pixels[i];
    float r = red(cb)   * (1 - opacity) + red(ct)   * opacity;
    float g = green(cb) * (1 - opacity) + green(ct) * opacity;
    float b = blue(cb)  * (1 - opacity) + blue(ct)  * opacity;
    out.pixels[i] = color(r, g, b);
  }
  out.updatePixels();
  return out;
}

Great point. Here’s the situation:
	•	**PNG inputs** → may include transparency (alpha channel). Processing loads them with ARGB, so transparent pixels will contribute less (e.g., fully transparent = ignored).
	•	**JPEG inputs** → no alpha channel, always fully opaque. Processing loads them as RGB but internally fills alpha = 255, so they’re treated as solid.

 <span style="font-family: .AppleColorEmojiUI;">
     👉
 </span> In your minimal code, you’re blending with a fixed opacity (0.5) and **ignoring alpha**. That means both PNGs and JPEGs are treated the same: fully opaque, except transparent PNG regions still let the lower layer show through. No changes are needed for JPEG support.
