# Layer blending



```
Layering photos in processing: is it possible to emulate the way Photoshop can blend layers together? 

```


Yes — Processing can absolutely emulate Photoshop-style **layer blending**, though with some limitations compared to Photoshop’s long list of modes.

At its core, Photoshop blending works by taking pixel values from a **base layer** and a **blend layer**, then combining them according to a formula (multiply, screen, overlay, etc.). Processing gives you a few tools to do the same:

⸻

**1. Using blend() function**

Processing has a built-in [blend()](https://processing.org/reference/blend_.html) function that lets you combine two images with specific blending modes:


```
blend(srcImg, 0, 0, srcImg.width, srcImg.height,
      0, 0, destImg.width, destImg.height,
      MULTIPLY);  
```


```
*// options: BLEND, ADD, SUBTRACT, LIGHTEST, DARKEST, DIFFERENCE, MULTIPLY, SCREEN, OVERLAY, etc.*
```


```


```


Supported modes are more limited than Photoshop, but you do get **ADD, SUBTRACT, MULTIPLY, SCREEN, OVERLAY, DARKEST, LIGHTEST, DIFFERENCE**.

⸻

**2. Using tint() + image()**

You can draw multiple images on top of each other with transparency:


```
tint(255, 127); 
```


```
*// half transparent*
```


```

image(img1, 0, 0);
image(img2, 0, 0);

```


This gives simple **opacity control**, like lowering a layer’s opacity in Photoshop.

⸻

**3. Manual per-pixel blending**

For full Photoshop-like control, you can access pixel arrays and apply custom formulas. For example, a Multiply blend:


```
img1.loadPixels();
img2.loadPixels();
PImage output = createImage(img1.width, img1.height, RGB);
output.loadPixels();

for (int i = 0; i < img1.pixels.length; i++) {
  color c1 = img1.pixels[i];
  color c2 = img2.pixels[i];
  
  float r = red(c1) * red(c2) / 255;
  float g = green(c1) * green(c2) / 255;
  float b = blue(c1) * blue(c2) / 255;
  
  output.pixels[i] = color(r, g, b);
}

output.updatePixels();
image(output, 0, 0);

```


With this approach, you can implement **all Photoshop blend modes** (Overlay, Soft Light, Hard Light, Color Burn, etc.) by using their published formulas.

⸻

**4. PShader for GPU blending**

If you want speed, you can write a GLSL fragment shader (Processing PShader) to do blending in real time. That way you can overlay multiple layers at 60fps, which is useful for video or interactive pieces.

```
void permute(int[] arr, int l) {
  if (l == arr.length - 1) {
    perms.add(arr.clone());
    return;
  }
  for (int i = l; i < arr.length; i++) {
    swap(arr, l, i);
    permute(arr, l + 1);
    swap(arr, l, i);
  }
}

void swap(int[] a, int i, int j) {
  int tmp = a[i]; a[i] = a[j]; a[j] = tmp;
}

boolean endsWithAny(String s, String[] suffixes) {
  for (String suf : suffixes) if (s.endsWith(suf)) return true;
  return false;
}

String joinInt(int[] arr, String sep) {
  String[] ss = new String[arr.length];
  for (int i = 0; i < arr.length; i++) ss[i] = str(arr[i]);
  return join(ss, sep);
}

```
```
Explain difference between combinations and permutations 

```

Great question — this distinction is exactly what matters when deciding how many composites you’ll end up generating.

⸻

**Permutations**
	•	**Definition:** All possible **orderings** of a set of items.
	•	**Key:** Order **matters**.
	•	**Formula:**
P(n, r) = \frac{n!}{(n-r)!}
(for choosing r items from n in sequence)
	•	**Example with images {A, B, C}:**
	•	For all 3 images (r = 3):
	•	ABC, ACB, BAC, BCA, CAB, CBA
	•	That’s 6 permutations (= 3!).

 <span style="font-family: .AppleColorEmojiUI;">
     👉
 </span> In your Processing sketch, **permutations** mean you generate a composite for every possible *stacking order* of the source images. With n images, that’s n! composites.

⸻

**Combinations**
	•	**Definition:** All possible **selections** of items from a set.
	•	**Key:** Order **does not matter**.
	•	**Formula:**
C(n, r) = \binom{n}{r} = \frac{n!}{r!(n-r)!}
	•	**Example with images {A, B, C}:**
	•	Choosing 2 at a time:
	•	{A, B}, {A, C}, {B, C}
	•	Only 3 combinations (instead of 6 permutations).

 <span style="font-family: .AppleColorEmojiUI;">
     👉
 </span> For your composites, **combinations** mean: “Which set of images do I blend together?” without caring about their order. So {A, B, C} produces just **one composite**, not six different orderings.

⸻

**Quick Comparison**

| Input size (n)              | Permutations (n!)           | Combinations (just one set) |
|-----------------------------|-----------------------------|-----------------------------|
| 3 images                    | 6 composites                | 1 composite                 |
| 4 images                    | 24 composites               | 1 composite                 |
| 6 images                    | 720 composites              | 1 composite                 |


*(If you allow subsets — e.g. any 2, 3, or 4 images chosen out of 6 — the number of* ***combinations*** *grows, but still far more slowly than permutations.)*

⸻

 <span style="font-family: .AppleColorEmojiUI;">
     ✅
 </span> **Summary:**
	•	Use **permutations** if you want *every ordering* (layer A over B is different from B over A).
	•	Use **combinations** if you only care about *which images are present*, not their order.

```
Blending images needs low opacity so all the images contribute. Can processing emulate a Photoshop auto-levels or auto-contrast function?
```

Yes — you can get very close to Photoshop’s **Auto Levels** / **Auto Contrast** in Processing.

Photoshop’s *Auto Levels* basically does this:
	1.	Look at the histogram (distribution of brightness values) of each channel (R, G, B).
	2.	Find the **darkest pixel** (min value > 0) and the **brightest pixel** (max value < 255).
	3.	Linearly rescale everything so that the min maps to 0 and the max maps to 255.
	•	If you use one scale across all channels, that’s like **Auto Contrast** (preserving color balance).
	•	If you rescale each channel independently, that’s like **Auto Levels** (can shift color balance).

⸻

**1. Low Opacity Blending**

If you want all n layers to contribute without any one overpowering:
	•	Just lower the opacity (e.g. 1/n for each).
	•	In Processing:

```

PImage composite = createImage(width, height, RGB);
composite.loadPixels();
for (int i = 0; i < composite.pixels.length; i++) {
  float r = 0, g = 0, b = 0;
  for (int j = 0; j < images.size(); j++) {
    color c = images.get(j).pixels[i];
    r += red(c);
    g += green(c);
    b += blue(c);
  }
  r /= images.size();
  g /= images.size();
  b /= images.size();
  composite.pixels[i] = color(r, g, b);
}
composite.updatePixels();

```


That’s like “average blend” — equivalent to stacking all images with equal low opacity.

```
  float minR=255, minG=255, minB=255;

  for (int i = 0; i < img.pixels.length; i++) {

  for (int i = 0; i < img.pixels.length; i++) {
    color c = img.pixels[i];
    float r = red(c), g = green(c), b = blue(c);

    if (perChannel) {
      r = map(r, minR, maxR, 0, 255);
      g = map(g, minG, maxG, 0, 255);
      b = map(b, minB, maxB, 0, 255);
    } else {
      float minAll = min(minR, min(minG, minB));
      float maxAll = max(maxR, max(maxG, maxB));
      r = map(r, minAll, maxAll, 0, 255);
      g = map(g, minAll, maxAll, 0, 255);
      b = map(b, minAll, maxAll, 0, 255);
    }

    out.pixels[i] = color(constrain(r, 0, 255), constrain(g, 0, 255), constrain(b, 0, 255));
  }

  out.updatePixels();
  return out;
}
```

```
Can processing cope with UHD source images?

```

Yes — **Processing can handle UHD (3840 × 2160) source images**, but there are a few caveats depending on your machine, memory settings, and the exact operations you do.

⸻

 <span style="font-family: .AppleColorEmojiUI;">
     ⚙️
 </span> **1. Image Size in Memory**
	•	A single **3840×2160** image has **8.3 million pixels**.
	•	At 4 bytes per pixel (ARGB), that’s **~33 MB per image in RAM**.
	•	With 10 source images, you’re already at ~330 MB just for the pixel arrays (not counting overhead, composites, and temp buffers).

 <span style="font-family: .AppleColorEmojiUI;">
     👉
 </span> With multiple permutations and copies, memory usage can grow into the **GBs quickly**.

⸻

 <span style="font-family: .AppleColorEmojiUI;">
     🧵
 </span> **2. Processing’s Pixel Operations**
	•	PImage.loadPixels() → creates an int[] of length width*height.
	•	Operations like blending or auto-levels iterate over the full array each time. For UHD, that means looping through ~8.3 million values per layer per pass.
	•	On a modern CPU, it’s feasible, but each composite may take **seconds rather than milliseconds**.

 <span style="font-family: .AppleColorEmojiUI;">
     👉
 </span> If you’re generating *all* permutations of even 6 UHD images (720 composites), it could take **hours**.

⸻

 <span style="font-family: .AppleColorEmojiUI;">
     💻
 </span> **3. Practical Strategies**
	•	**Memory allocation:** Increase Processing’s Java heap size (Preferences → Run → VM Options or processing-java --jvm-args). e.g. -Xmx8G for 8 GB.
	•	**Batching:** Add MAX_PERMS to limit how many permutations you render in one run.
	•	**Average strategy:** Much lighter than sequential blending (since it just sums pixel channels once).
	•	**Save intermediates:** Write images to disk as soon as possible instead of keeping them all in RAM.
	•	**Downsample preview mode:** Work at 25–50% scale while tuning, then switch back to UHD.
	•	**Consider GPU/FFmpeg:** For very large batch composites, shaders or external tools (e.g. FFmpeg + blend filters) can be orders of magnitude faster.

⸻

 <span style="font-family: .AppleColorEmojiUI;">
     ✅
 </span> **Summary**
	•	**Yes, Processing can cope with UHD images.**
	•	You’ll need: plenty of RAM, possibly increased Java heap, and realistic limits on permutations.
	•	For “few dozen composites” it’s fine. For “hundreds of UHD permutations,” expect long runtimes and high memory load.

```
What is the meaning of stride?

```


Great question. In this context:

**Stride**
	•	A **stride** means you **step through a sequence with a fixed interval** instead of taking every element.
	•	Mathematically, if you have a list of items indexed 0, 1, 2, 3, …, and your stride is n, you only keep those where **index % n == 0**.

⸻

**Examples**
	•	Suppose you have **10 permutations**: P0, P1, P2, …, P9.
	•	**Stride = 1** → keep all (P0…P9).
	•	**Stride = 2** → keep every 2nd: (P0, P2, P4, P6, P8).
	•	**Stride = 3** → keep every 3rd: (P0, P3, P6, P9).

⸻

**Why use stride here?**
	•	If you have a huge set (e.g. **720 permutations**), rendering all can be slow.
	•	With **stride = 10**, you generate ~10% of them (72 images), evenly spaced across the whole set.
	•	It’s like skimming through the permutation space in a **regular, systematic way**, rather than taking everything.

⸻

     👉
 </span> In your Processing sketch, the “1-in-n” option is just another way of saying **stride = n**.

```
What is seeded random subset?

```


Good question 👍

A **seeded random subset** means:
	•	Instead of taking permutations evenly spaced by a stride (every 10th, every 20th), you pick a **random sample** of them.
	•	**Seeded** means you use a fixed random seed (a starting value for the random number generator). That way:
	•	With the same seed, you’ll always get the **same subset**.
	•	With a different seed, you’ll get a different (but still reproducible) subset.

⸻

 <span style="font-family: .AppleColorEmojiUI;">
     📊
 </span> **Example**

You have **720 permutations** of 6 images.
	•	**Stride = 10** → you get permutations #0, #10, #20, … evenly spaced (deterministic, but patterned).
	•	**Random subset, seed = 1234, 10%** → you get 72 permutations chosen randomly across the whole set.
	•	Run again with the same seed (1234) → identical 72.
	•	Change the seed to 5678 → a different 72, still reproducible.

⸻

 <span style="font-family: .AppleColorEmojiUI;">
     ✅
 </span> **Why useful**
	•	If you want **variety**, stride may look too regular.
	•	A seeded random subset lets you **sample the space more naturally**, but still keep reproducibility (so you can regenerate or extend results later).
