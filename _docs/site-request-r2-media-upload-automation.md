# R2 Media Upload Automation Request

Status:

- the first implementation is CLI-only.
- var/local/site.env has been implemenmted
- still need to create the R2 keys
- then need to switch UI over to using R2

## Summary

This request tracks adding a script-managed publishing step for media assets that currently need to be copied to Cloudflare R2 manually.

The first supported use case is catalogue primary images generated by the local media pipeline.
The implementation also establishes a credential and upload architecture that can later support docs images without reworking the security model.

Implemented entrypoint:

```bash
./scripts/media/publish_media_to_r2.py --scope catalogue --kind works --id 01007
```

## Problem

The catalogue media pipeline already stages source images, generates primary derivatives, and copies thumbnails into repo-owned public asset folders.
Primary derivatives remain under `var/catalogue/media/` as a manual handoff point for remote R2 publishing.

That leaves a recurring manual step:

- find the generated primary derivatives
- upload them to the correct R2 bucket prefix
- avoid missing one size variant
- avoid overwriting the wrong remote object
- keep local logs clear enough to know what was published

This is manageable while changes are small, but it is easy to forget and will become more awkward once docs imports can create image assets too.

## Goals

Add a project-local script that can publish approved media assets to R2.

The workflow should:

- read R2 credentials from the local environment or a local-only secret file, never from tracked repo files
- support a dry-run preview before upload
- upload catalogue primary derivatives from the existing staging tree
- report skipped, uploaded, unchanged, and failed objects clearly
- avoid logging secrets or full credential-bearing URLs
- produce deterministic object keys
- provide an extension point for later docs media uploads

## Non-Goals

The first implementation should not:

- upload canonical source images
- delete remote R2 objects broadly or implicitly; exact-id primary-variant deletion is supported through `--delete`
- expose R2 credentials to browser-side Studio code
- require credentials in generated docs, public JSON, or tracked config
- replace the existing local thumbnail copy behavior
- add docs image uploading before the docs asset import model is settled

## Credential Model

R2 credentials should be provided through environment variables that match the cloud/runtime convention already documented for the repo:

- `R2_ACCOUNT_ID`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET`
- `R2_ENDPOINT`

Local setup uses the gitignored `var/local/site.env` file for those same R2 variables.
The file must be loaded only by CLI scripts or local server-side wrappers.
The script should fail closed when credentials are missing, with setup guidance that names the missing variables but never prints existing secret values.

The R2 API key should be scoped as narrowly as Cloudflare allows.
Preferred permissions:

- object read/write for the target bucket
- object list for preview and unchanged checks
- no account-wide permissions beyond what R2 S3-compatible access requires
- delete permission only when exact-id remote cleanup is intentionally used

## Proposed Architecture

Introduce a media publisher module with a small registry of source adapters and remote destinations.

Conceptual shape:

```text
media publisher
  credentials provider
  R2 client
  source adapter registry
    catalogue primary image adapter
    future docs asset adapter
  upload planner
  upload executor
  report writer
```

The script should separate:

- source discovery
- object-key mapping
- local/remote comparison
- dry-run reporting
- actual upload
- result logging

That separation matters because catalogue primary images and future docs assets have different source roots, but should share credentials, safety checks, and upload behavior.

## First Use Case: Catalogue Primary Images

The catalogue adapter should discover generated primary derivatives under:

- `var/catalogue/media/works/srcset_images/primary/`
- `var/catalogue/media/work_details/srcset_images/primary/`
- `var/catalogue/media/moments/srcset_images/primary/`

Initial scope flags should allow targeted uploads such as:

- all catalogue primaries
- one work id
- one work-detail id
- one moment id
- one media kind

The object-key mapping should match the public URL conventions already used by catalogue pages.
The request intentionally leaves the exact prefix as an implementation decision to verify against the current R2 bucket layout before coding.

The upload planner should verify that expected size variants exist for each selected item before upload, or clearly report partial sets.
Partial uploads should either be blocked by default or require an explicit flag.

## Later Use Case: Docs Assets

The later docs adapter should upload docs-owned media only after the docs import media model is implemented.

Likely source roots:

- `assets/docs/imports/...`
- other stable docs asset folders under `assets/docs/...`

Docs asset upload should not change Markdown source links until the docs remote-media convention is confirmed.
The first docs milestone may only validate and report upload plans without rewriting doc bodies.

## CLI Shape

Candidate script:

```text
./scripts/media/publish_media_to_r2.py --scope catalogue --kind works --id 01007 --dry-run
./scripts/media/publish_media_to_r2.py --scope catalogue --all --write
```

The default should be dry-run.
Write mode should require an explicit flag such as `--write`.

Useful options:

- `--scope catalogue`
- `--scope docs`
- `--kind works|work_details|moments`
- `--id <id>`
- `--all`
- `--changed-only`
- `--force`
- `--write`
- `--report-json <path>`

## Upload Safety

The script should:

- read only from allowlisted local source roots
- refuse path traversal and symlink escapes
- compare local file size and checksum or ETag-compatible digest where practical
- skip unchanged remote objects by default
- require `--force` to overwrite changed remote objects when remote comparison finds a conflict
- use concise logs that show ids, local relative paths, object keys, and statuses
- never print secret values
- never include signed URLs in logs

## Studio Integration

The first implementation can be CLI-only.

A later Studio integration could expose a local-only command from catalogue editors or unified Studio Activity, but browser-side code must never receive raw R2 credentials.
If Studio triggers uploads, the local write service should own credential reads and upload execution.

## Documentation Requirements

Implementation should update:

- script reference docs for the new publisher
- local setup docs for R2 credential setup
- cloud environment docs for secret-store expectations
- catalogue media docs to replace the current manual R2 handoff note
- docs image docs later, when docs asset upload is enabled

Documentation must keep examples project-local and avoid user-specific paths.

## Acceptance Criteria

The first catalogue milestone is implemented:

- dry-run lists catalogue primary derivatives that would upload to R2
- write mode uploads selected catalogue primary derivatives only when `--write` is passed
- remote delete mode removes selected catalogue primary derivatives only when `--delete --write` is passed with an exact kind and id
- missing credentials fail with a clear non-secret error
- unchanged remote files are skipped by default
- changed remote files require `--force`
- object keys are deterministic and documented in [Publish Media To R2](/docs/?scope=studio&doc=scripts-publish-media-to-r2)
- logs and optional JSON reports avoid secrets and signed URLs
- tests cover credential loading, source-root allowlisting, object-key mapping, dry-run planning, and overwrite behavior

## Benefits

- removes the manual R2 copy step for catalogue primary images
- reduces the chance of missing or mismatched primary image variants
- creates one secure credential boundary for future media publishing
- gives docs image publishing a planned extension point instead of a separate ad hoc script

## Risks

- incorrect object-key mapping could overwrite or publish media in the wrong R2 location
- overly broad R2 credentials would increase blast radius if leaked
- remote comparison behavior may differ from local checksum expectations
- adding docs assets later could require link-rewrite decisions that are not part of the first catalogue milestone
